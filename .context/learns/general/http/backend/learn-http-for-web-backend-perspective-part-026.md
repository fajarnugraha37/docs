# learn-http-for-web-backend-perspective-part-026.md

# Part 026 — Observability: Logs, Metrics, Traces, and HTTP Diagnostics

> Series: `learn-http-for-web-backend-perspective`  
> Context: Java Software Engineer / Backend Engineer  
> Focus: HTTP observability as production diagnostic architecture, not merely "logging requests".

---

## 0. Why This Part Matters

A backend HTTP API that is semantically correct, secure, and well-designed can still become operationally useless if nobody can answer basic production questions:

- Which endpoint is failing?
- Is the failure caused by client misuse, gateway policy, application bug, downstream timeout, database saturation, or dependency outage?
- Did the request reach the application?
- Was the response generated by the app, proxy, gateway, load balancer, CDN, WAF, or service mesh?
- Is latency caused by queueing, network, TLS, body upload, controller execution, database, downstream HTTP call, serialization, or response write?
- Are 5xx errors real server bugs or gateway-generated failures?
- Are 4xx errors harmless validation failures or signs of attack/abuse?
- Are retries amplifying the outage?
- Which tenant, client, operation, region, or release is affected?
- Can we debug the incident without leaking sensitive data?

This part is about building a mental model and implementation discipline for observing HTTP systems in production.

Observability is not only "logs + metrics + traces". It is the ability to infer internal system behavior from external signals. For HTTP backends, the most important external signal is the request/response boundary.

---

## 1. Core Mental Model

An HTTP request is not just a function call.

It is a distributed transaction of resource consumption across many layers:

```text
client
  -> DNS
  -> TCP / TLS / QUIC
  -> CDN / WAF
  -> load balancer
  -> reverse proxy / ingress
  -> API gateway
  -> service mesh sidecar
  -> application server
  -> framework routing
  -> controller / handler
  -> application service
  -> database / cache / queue / downstream service
  -> serializer
  -> response write
  -> proxies back to client
```

Each layer may:

- accept or reject the request,
- mutate headers,
- rewrite path,
- terminate TLS,
- enforce auth/rate limits,
- buffer body,
- retry upstream,
- timeout,
- generate its own error response,
- log partial information,
- attach trace context,
- drop the connection.

A strong backend engineer does not ask only:

> "What did my controller log?"

They ask:

> "At which layer was the HTTP exchange accepted, transformed, delayed, rejected, retried, or terminated?"

---

## 2. Observability Is a Contract Too

Just as an API has method/status/header/body contracts, it should have an observability contract.

For every production HTTP endpoint, the system should define:

1. What is logged?
2. What is measured?
3. What is traced?
4. What is redacted?
5. What labels/tags are allowed?
6. Which IDs allow correlation?
7. Which failures are counted as client error vs server error vs dependency error?
8. Which status codes trigger alerts?
9. Which metrics define SLOs?
10. Which events must be auditable?

Without this discipline, observability becomes noisy, expensive, unsafe, and unreliable.

---

## 3. Three Pillars Are Not Enough

The usual phrase is:

```text
logs + metrics + traces
```

For HTTP backend production, expand it into:

```text
logs
metrics
traces
profiles
health checks
synthetic probes
access logs
application logs
audit logs
security events
release/deployment markers
configuration state
runtime saturation signals
```

The important distinction:

| Signal | Best For | Weakness |
|---|---|---|
| Logs | Narrative detail for specific events | Expensive, noisy, hard to aggregate |
| Metrics | Trends, alerting, SLOs | Low detail, can hide individual cases |
| Traces | Cross-service causality | Sampling gaps, instrumentation complexity |
| Access logs | HTTP boundary truth | Often lacks domain context |
| Audit logs | Legal/business accountability | Not designed for debugging performance |
| Profiles | CPU/memory hotspots | Usually not request-semantic |
| Health checks | Availability routing | Can lie if too shallow |
| Synthetic probes | User-visible path checks | Limited scenario coverage |

Top-tier backend observability is not about collecting everything. It is about collecting the right signals with clear meaning.

## 4. Access Log vs Application Log vs Audit Log

Many teams mix these together. That is a serious design smell.

### 4.1 Access Log

Access log records the HTTP exchange at an edge or server boundary.

Typical fields:

- timestamp,
- method,
- path template or raw path,
- status,
- response size,
- duration,
- remote IP,
- user agent,
- request ID,
- trace ID,
- host,
- protocol version,
- TLS information,
- upstream target,
- upstream duration.

Access logs answer:

- Did this HTTP request happen?
- What status was returned?
- How long did the boundary observe it?
- Which upstream handled it?
- Did the proxy or app generate the response?

### 4.2 Application Log

Application log records internal application events.

Examples:

- validation failed,
- authorization denied,
- case transition rejected,
- downstream call timed out,
- idempotency key replayed,
- database optimistic lock conflict,
- file scan failed,
- background job created.

Application logs answer:

- What decision did the application make?
- Which branch of business logic executed?
- Which exception occurred?
- Which dependency call failed?

### 4.3 Audit Log

Audit log records accountability-relevant business/security events.

Examples:

- user submitted evidence,
- officer approved enforcement action,
- supervisor reassigned case,
- sensitive document downloaded,
- user attempted unauthorized access,
- policy override was applied.

Audit logs answer:

- Who did what?
- To which resource?
- Under which authority?
- At what time?
- From where?
- Was it allowed or denied?
- What was the before/after business state?

Audit logs should be durable, tamper-resistant, and semantically stable.

They should not depend on debug log level.

## 5. The Golden Rule: Log Events, Not Text

Bad log:

```text
User failed to update case
```

Better structured log:

```json
{
  "event": "case.update.rejected",
  "reason": "version_conflict",
  "http.method": "PATCH",
  "http.route": "/cases/{caseId}",
  "http.status_code": 412,
  "case.id": "case_123",
  "actor.id": "user_456",
  "tenant.id": "tenant_789",
  "trace_id": "...",
  "request_id": "..."
}
```

The first log can be read by a human.

The second can be searched, aggregated, correlated, alerted on, and audited.

For production backend systems, prefer structured logs.

## 6. Request Identity: Request ID, Correlation ID, Trace ID

Teams often confuse these IDs.

### 6.1 Request ID

A request ID identifies one inbound HTTP request as observed by a boundary.

It is useful for:

- finding one request across logs,
- returning an error reference to client,
- correlating access log and application log.

Common headers:

```http
X-Request-ID: 01HV...
```

or internal names such as:

```http
X-Correlation-ID: ...
```

### 6.2 Correlation ID

A correlation ID links a group of related operations.

Example:

- one user action creates several HTTP calls,
- one batch job touches many resources,
- one business workflow spans multiple services.

Correlation ID can outlive one HTTP request.

### 6.3 Trace ID

A trace ID identifies one distributed trace.

In modern systems, this usually follows W3C Trace Context:

```http
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
tracestate: vendor-specific-data
```

Trace ID is used by tracing systems to connect spans across services.

### 6.4 Do Not Treat Them as the Same Thing

A simple mapping can work:

```text
request_id: one inbound request
trace_id: one distributed trace
correlation_id: one business/user workflow
```

A single user action may have:

```text
correlation_id = submit-evidence-flow-abc
trace_id       = trace for one request chain
request_id     = request at each edge/app boundary
```

## 7. Trace Context Propagation

For service-to-service HTTP calls, the backend must propagate trace context.

Example inbound request:

```http
GET /cases/123 HTTP/1.1
Host: api.example.gov
traceparent: 00-aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa-bbbbbbbbbbbbbbbb-01
```

When service A calls service B, it should continue the trace:

```http
GET /internal/cases/123/audit-summary HTTP/1.1
Host: audit-service
traceparent: 00-aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa-cccccccccccccccc-01
```

Same trace ID, new span ID.

This lets you see:

```text
API Gateway
  -> case-service controller
    -> authorization service
    -> database query
    -> document service
    -> audit service
```

Without propagation, each service looks isolated.

---

## 8. Span Modeling for HTTP Backends

A trace is only useful if spans represent meaningful boundaries.

Good span boundaries:

- inbound HTTP request,
- controller/handler execution,
- authorization decision,
- validation if expensive or external,
- database query/transaction,
- outbound HTTP call,
- cache call,
- message publish,
- file storage call,
- malware scan request,
- template rendering/export generation.

Bad span boundaries:

- every tiny private method,
- every getter/setter,
- every loop iteration,
- meaningless technical wrappers.

The goal is causality, not visual clutter.

## 9. Metrics: RED Method for HTTP

For request-driven services, RED is a useful starting point:

```text
Rate
Errors
Duration
```

### 9.1 Rate

How many requests are received?

Useful dimensions:

- route template,
- method,
- status class,
- client type,
- tenant tier,
- region,
- service version.

Avoid raw URL as a metric label.

Bad:

```text
http_requests_total{path="/cases/123"}
http_requests_total{path="/cases/456"}
```

Good:

```text
http_requests_total{route="/cases/{caseId}"}
```

### 9.2 Errors

Errors should be split by class:

```text
2xx success
3xx redirect
4xx client/request/policy error
5xx server/upstream error
```

But for backend diagnosis, also track:

- validation errors,
- auth failures,
- authorization denials,
- conflict errors,
- rate limit rejections,
- timeout errors,
- dependency failures,
- gateway-generated errors.

A spike in `400` may be harmless client bug or active probing attack.

A spike in `409` may indicate real business concurrency.

A spike in `401` may mean token expiry issue.

A spike in `429` may mean abuse or legitimate traffic growth.

### 9.3 Duration

Measure latency as distribution, not average.

Important percentiles:

```text
p50   median user experience
p90   common tail
p95   important service health signal
p99   tail latency / saturation / queueing
p999  extreme tail, often costly and noisy
```

Average latency can lie.

Example:

```text
100 requests:
- 99 requests take 20 ms
- 1 request takes 10 seconds

Average ≈ 119.8 ms
p99 ≈ 10 seconds
```

The average looks acceptable, but one user had a terrible experience.

## 10. USE Method for Saturation

For infrastructure/resource diagnosis, use:

```text
Utilization
Saturation
Errors
```

HTTP backend resources to monitor:

- CPU utilization,
- memory usage,
- GC pauses,
- thread pool active count,
- executor queue depth,
- connection pool active/idle/pending,
- event loop delay,
- request body buffer usage,
- file descriptor count,
- database pool usage,
- outbound HTTP connection pool,
- rate limiter rejections,
- load shedding rejections.

A service may have low CPU but still be down because:

- all threads are blocked,
- database connection pool is exhausted,
- outbound HTTP pool is saturated,
- event loop is blocked,
- request queue is full,
- disk temp space is exhausted by uploads.

At minimum:

```text
http.server.errors.by_status

```

Useful dimensions:

```text

deployment.environment
http.request.method
http.route
http.response.status_code
network.protocol.name

```

Be careful with:

```text

raw query
email
IP address

```

## 12. High Cardinality Trap

High cardinality means a metric label has many possible values.

Bad metric:

```text
http_requests_total{case_id="CASE-2026-000001"}
http_requests_total{case_id="CASE-2026-000002"}
http_requests_total{case_id="CASE-2026-000003"}
...
```

This can create millions of time series.

Consequences:

- monitoring backend becomes expensive,
- dashboards slow down,
- alerts become unreliable,
- storage grows rapidly,
- cardinality limits drop data.

Use high-cardinality fields in logs/traces, not metrics.

Good metric labels:

```text
route="/cases/{caseId}"
status_code="412"
method="PATCH"
tenant_tier="enterprise"
```

Good log fields:

```json
{
  "case.id": "CASE-2026-000001",
  "actor.id": "u-123",
  "tenant.id": "t-456"
}
```

## 13. Status Code Observability

Status codes are not only client contract. They are operational signal.

### 13.1 Useful Buckets

```text
2xx: success
3xx: redirect/cache/proxy behavior
400: malformed or invalid request
401: unauthenticated
403: authenticated but forbidden
404: missing or hidden resource
409: conflict / invalid state transition / duplicate
412: precondition failed / optimistic concurrency
415: unsupported media type
422: semantic validation failure if used by your API style
429: rate limited
499: client closed request (common in Nginx, non-standard)
500: application bug/unhandled failure
502: bad gateway/upstream protocol failure
503: unavailable/load shedding/maintenance
504: gateway timeout
```

### 13.2 Alerting Mistake

Bad alert:

```text
Alert when 4xx > 5%
```

Why bad?

- validation-heavy APIs may naturally produce 4xx,
- attackers can trigger 4xx and page your team,
- 404 from scanning bots may be irrelevant,
- 401 may spike due to expired token rollout.

Better:

```text
Alert on 5xx ratio by route and service.
Alert on 429 ratio for key clients or tenants.
Alert on 401 spike after auth release.
Alert on 403 spike for privileged operations.
Alert on 412 spike if concurrency conflicts exceed expected domain baseline.
Alert on 504 at gateway and compare app-side traces.
```

## 14. Latency Decomposition

End-to-end latency is not one number.

Break it down:

```text
client observed latency
  = DNS
  + TCP connect
  + TLS handshake
  + request upload
  + proxy queueing
  + gateway processing
  + upstream connect
  + app queueing
  + app handler
  + database/cache/downstream calls
  + serialization
  + response transfer
```

The application usually sees only part of this.

Example:

```text
Client sees: 4.5 seconds
Gateway access log upstream time: 4.1 seconds
Application handler metric: 120 ms
```

Possible explanation:

- request body upload was slow,
- proxy buffered large body,
- gateway queued before upstream,
- connection pool to upstream was exhausted,
- response write to slow client blocked.

Do not assume app handler latency equals user-visible latency.

## 15. Timeout Observability

A timeout should record:

```text
which timeout fired
configured timeout value
elapsed duration
upstream/downstream target
operation name
whether request was canceled
whether work may have committed
retry attempt number
idempotency key if present
```

Bad timeout log:

```text
Request timed out
```

Better:

```json
{
  "event": "downstream.timeout",
  "downstream.service": "document-service",
  "operation": "POST /documents/{id}/scan",
  "timeout.ms": 2000,
  "elapsed.ms": 2003,
  "attempt": 1,
  "idempotency_key.present": true,
  "http.route": "/evidence/{evidenceId}/submit",
  "trace_id": "..."
}
```

---

## 16. Retry Observability

Retries are invisible unless instrumented.

Record:

- attempt number,
- max attempts,
- retry reason,
- backoff delay,
- whether method/business operation is idempotent,
- idempotency key,
- downstream status/exception,
- final outcome.

Without retry observability, an outage often looks like random latency and traffic spike.

Example failure amplification:

```text
Normal traffic: 1,000 requests/sec
Downstream starts timing out
Each request retries 3 times
Downstream receives 4,000 attempts/sec
System collapses harder
```

Metrics should distinguish:

```text
logical_requests_total
physical_attempts_total
retries_total
retry_exhausted_total
```

## 17. Gateway vs Application Attribution

When debugging HTTP failures, determine who generated the response.

| Status | Possible Generator | Diagnostic Question |
|---|---|---|
| 400 | gateway/app | Did request fail parsing before app? |
| 401 | gateway/app | Was auth enforced at edge or app? |
| 403 | gateway/app/WAF | Was this policy, authz, or WAF? |
| 404 | gateway/app | Did route not match at edge or resource missing? |
| 413 | gateway/app | Which layer body limit rejected it? |
| 429 | gateway/app | Which rate limiter fired? |
| 502 | gateway/proxy | Did upstream close/reset/return invalid response? |
| 503 | gateway/app | Was service unavailable or app load-shedding? |
| 504 | gateway/proxy | Did upstream exceed gateway timeout? |

Add response headers or structured access log fields when possible:

```http
X-Request-ID: ...
X-Error-Origin: gateway
```

Be careful exposing internal architecture to public clients. This can be internal-only, logs-only, or sanitized.

## 18. Route Template Is Critical

For metrics, traces, and logs, prefer route template:

```text
/cases/{caseId}/evidence/{evidenceId}
```

not raw path:

```text
/cases/CASE-2026-000001/evidence/EVD-999
```

Raw path is useful in logs but dangerous as a metric dimension.

Spring MVC can usually expose the best matching pattern. Ensure your observability instrumentation captures it.

## 20. What Not to Log

Do not log:

- raw access tokens,
- refresh tokens,
- session IDs,
- API keys,
- passwords,
- one-time codes,
- full cookies,
- personal identity numbers,
- full credit card numbers,
- unredacted authorization headers,
- sensitive document content,
- full request/response body by default,
- secrets in query parameters,
- medical/legal/regulatory sensitive payload unless explicitly governed.

Common dangerous log:

```text
Authorization: Bearer eyJhbGciOi...
```

Instead:

```json
{
  "auth.scheme": "Bearer",
  "auth.token.present": true,
  "auth.subject": "u-123",
  "auth.issuer": "https://issuer.example",
  "auth.audience": "case-api"
}
```

## 21. Request/Response Body Logging

Default rule:

> Do not log HTTP bodies in production by default.

Body logging is tempting but dangerous.

Risks:

- sensitive data leakage,
- huge storage cost,
- performance degradation,
- compliance violation,
- accidental credential capture,
- duplicate body consumption bugs,
- memory pressure.

Safer alternatives:

- log schema validation errors by field name, not full payload,
- log hashes/fingerprints,
- log body size,
- sample only non-sensitive endpoints,
- use explicit debug window with approval,
- redact aggressively,
- store audit event separately from raw payload,
- use secure evidence/document storage for regulated content.

Example safe validation log:

```json
{
  "event": "request.validation.failed",
  "http.route": "/cases/{caseId}/evidence",
  "field_errors": [
    {"field": "documentType", "code": "required"},
    {"field": "submittedAt", "code": "invalid_format"}
  ]
}
```

## 22. Redaction Strategy

Redaction must be systematic, not ad-hoc.

Apply redaction at:

1. HTTP logging filter,
2. exception handler,
3. structured logging encoder,
4. tracing attributes,
5. audit event publisher,
6. gateway logs,
7. data pipeline/export.

Redact by:

- header name,
- field name,
- JSON path,
- data classification,
- annotation/metadata,
- allowlist rather than blocklist.

Allowlist is safer:

```text
Log only approved fields.
```

Blocklist is fragile:

```text
Log everything except password/token/secret.
```

Because new sensitive fields are added over time.

## 23. Exception Observability

Do not merely log stack traces.

For each exception, capture:

```text
exception class
stable error code
http status mapping
route template
operation name
business resource if safe
whether retryable
whether caused by downstream
trace id
request id
```

Example:

```json
{
  "event": "exception.mapped",
  "exception.type": "OptimisticLockingFailureException",
  "error.code": "CASE_VERSION_CONFLICT",
  "http.status_code": 412,
  "http.route": "/cases/{caseId}",
  "retryable": false,
  "trace_id": "..."
}
```

Avoid returning stack traces to clients.

Avoid logging the same exception repeatedly at every layer.

---

## 24. Log Level Discipline

Suggested default:

| Situation | Level |
|---|---|
| Normal request completion | INFO or access log only |
| Expected validation failure | INFO or DEBUG depending volume |
| Authorization denial | INFO/WARN depending sensitivity |
| Rate limit rejection | INFO/WARN depending volume |
| Business conflict | INFO |
| Downstream timeout | WARN |
| Retry exhausted | ERROR/WARN depending final outcome |
| Unhandled exception | ERROR |
| Security attack signal | WARN/ERROR/security event |

Common mistake:

- logging every 4xx as ERROR,
- logging expected conflicts as ERROR,
- logging full stack trace for validation failure,
- logging unhandled 500 only as INFO.

---

## 25. Sampling

Sampling is necessary for high-volume systems.

Types:

- trace sampling,
- log sampling,
- error-biased sampling,
- tail-based sampling,
- tenant/client-specific sampling,
- route-specific sampling.

Naive random sampling can miss rare errors.

Better:

```text
Sample successful high-volume GETs lightly.
Keep all 5xx traces.
Keep all security-relevant events.
Keep all audit events.
Keep all rare status code transitions.
Use tail sampling for slow traces.
```

Never sample away required audit events.

## 26. Health Checks Are Observability, Not Just Kubernetes Config

Health endpoints are part of HTTP backend operations.

Common endpoints:

```text
/livez
/readyz
/health
/actuator/health
```

### 26.1 Liveness

Answers:

> Should this process be restarted?

Should be shallow.

Examples:

- process event loop responsive,
- JVM alive,
- no fatal internal state.

Do not make liveness depend on database availability; otherwise dependency outage can trigger restart storms.

### 26.2 Readiness

Answers:

> Should this instance receive traffic?

May depend on:

- database connectivity,
- required config loaded,
- migrations complete,
- critical dependency reachable,
- local warmup complete,
- connection pools initialized.

### 26.3 Startup

Answers:

> Has the application finished startup initialization?

Useful for slow Java applications.

### 26.4 Deep Health

For humans/synthetic checks, not necessarily load balancer routing.

May check:

- auth service,
- database read/write,
- object storage,
- queue publish/consume,
- downstream dependencies.

## 27. Synthetic Probes

Synthetic probes simulate user/client behavior.

Examples:

```text
GET /health/simple
GET /cases?limit=1 as synthetic tenant
POST /auth/token in staging
POST /cases validation-only endpoint
download small known file
```

Good synthetic probes:

- represent important user journeys,
- run from multiple locations,
- use realistic auth,
- avoid mutating production state unless carefully isolated,
- measure user-visible latency,
- alert separately from internal metrics.

Synthetic checks catch problems that internal metrics may miss:

- DNS issue,
- TLS certificate expiry,
- CDN routing problem,
- gateway misconfiguration,
- auth provider problem,
- bad deployment path rewrite,
- public internet connectivity issue.

## 28. Security Observability

HTTP APIs must produce security-relevant signals.

Track:

- repeated authentication failures,
- authorization denials,
- BOLA/IDOR-like access attempts,
- invalid/malformed tokens,
- suspicious user agents,
- unusual path scanning,
- unsupported method attempts,
- high 404 from one IP/client,
- high 401/403 from one actor,
- rate limiter rejections,
- WAF blocks,
- CSRF failures,
- CORS origin rejections,
- request smuggling indicators,
- oversized body rejections,
- invalid content type spikes,
- file upload malware scan failures.

But do not turn every security signal into a page.

Security observability often feeds:

- SIEM,
- abuse detection,
- fraud/risk system,
- audit system,
- security dashboard,
- incident response workflow.

## 29. Auditability for Regulatory Systems

In regulatory/case-management systems, observability and audit overlap but are not identical.

For audit, capture:

```text
actor
authority/role at decision time
resource
operation
business state before/after
policy decision
reason/comment
timestamp
source channel
request id
trace id
client/application identity
```

Example audit event:

```json
{
  "event": "case.escalated",
  "case.id": "CASE-2026-000042",
  "from_state": "UNDER_REVIEW",
  "to_state": "ESCALATED_TO_LEGAL",
  "actor.id": "officer-123",
  "actor.role": "SUPERVISOR",
  "tenant.id": "agency-9",
  "reason.code": "HIGH_RISK_FINDING",
  "http.request_id": "req-abc",
  "trace_id": "trace-def",
  "occurred_at": "2026-06-19T10:00:00Z"
}
```

Do not rely on access logs to prove business decisions.

Access logs say request happened.

Audit logs say accountable business action happened.

## 30. Java / Spring MVC Implementation Pattern

A typical Spring MVC observability pipeline:

```text
Servlet container access log
  -> request ID filter
  -> trace context filter
  -> security filter chain
  -> logging/redaction filter
  -> DispatcherServlet
  -> controller
  -> service
  -> repositories/downstream clients
  -> exception handler
  -> response logging
```

### 30.1 Request ID Filter Sketch

```java
@Component
public class RequestIdFilter extends OncePerRequestFilter {
    public static final String HEADER = "X-Request-ID";

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain
    ) throws ServletException, IOException {
        String incoming = request.getHeader(HEADER);
        String requestId = isUsable(incoming) ? incoming : UUID.randomUUID().toString();

        MDC.put("request_id", requestId);
        response.setHeader(HEADER, requestId);

        try {
            filterChain.doFilter(request, response);
        } finally {
            MDC.remove("request_id");
        }
    }

    private boolean isUsable(String value) {
        return value != null && value.length() <= 128 && value.matches("[A-Za-z0-9._:-]+$");
    }
}
```

Important points:

- validate incoming request ID,
- avoid unbounded header value,
- put it in MDC,
- return it to the client,
- clean MDC in `finally`.

### 30.2 Controller Advice Error Mapping

```java
@RestControllerAdvice
public class ApiExceptionHandler {

    @ExceptionHandler(CaseVersionConflictException.class)
    ResponseEntity<ProblemDetail> handleVersionConflict(CaseVersionConflictException ex) {
        ProblemDetail problem = ProblemDetail.forStatus(HttpStatus.PRECONDITION_FAILED);
        problem.setTitle("Case version conflict");
        problem.setDetail("The case has changed since it was last read.");
        problem.setProperty("code", "CASE_VERSION_CONFLICT");
        problem.setProperty("requestId", MDC.get("request_id"));
        return ResponseEntity.status(HttpStatus.PRECONDITION_FAILED).body(problem);
    }
}
```

Ensure the same error code appears in:

- response body,
- logs,
- metrics,
- traces.

## 32. WebFlux Observability Considerations

Reactive systems add specific concerns:

- event loop blocking,
- context propagation,
- backpressure,
- cancellation,
- long-lived streams,
- lazy execution,
- operator chain errors,
- DataBuffer leaks,
- scheduler misuse.

### 32.1 MDC Problem

Traditional MDC is thread-local.

In reactive code, execution may move across threads.

So MDC does not automatically work unless integrated with Reactor context/logging instrumentation.

Bad assumption:

```text
MDC set once at request start will always be available downstream.
```

Better:

- use framework-supported context propagation,
- use Micrometer tracing/Reactor integration,
- avoid manual thread-local assumptions,
- test logs under async boundaries.

### 32.2 Event Loop Blocking

Track event loop blocking symptoms:

- high latency with low CPU,
- blocked Netty event loop warnings,
- p99 spike under load,
- slow streaming response,
- backpressure not honored.

Never run blocking JDBC/file calls directly on event loop.

## 34. Diagnosing Common Incidents

### 34.1 Spike in 504 Gateway Timeout

Questions:

1. Did the app receive the requests?
2. Are there matching traces?
3. Is app handler latency high?
4. Is gateway upstream timeout lower than app timeout?
5. Are DB/downstream calls slow?
6. Are app threads saturated?
7. Is connection pool exhausted?
8. Did a deployment change routing?

Signals:

- gateway access logs,
- app request metrics,
- traces,
- thread pool metrics,
- DB pool metrics,
- downstream client metrics.

### 34.2 Spike in 401

Questions:

1. Did token issuer rotate keys?
2. Did clock skew break expiry validation?
3. Did audience/issuer config change?
4. Are clients using expired tokens?
5. Did gateway/app auth policy diverge?

Signals:

- auth failure reason metric,
- issuer/audience mismatch logs,
- JWKS fetch errors,
- deployment marker,
- client distribution.

### 34.3 Spike in 412

Questions:

1. Is there real concurrent editing?
2. Did frontend/client stop sending `If-Match` correctly?
3. Did ETag generation change?
4. Did resource version update too frequently?
5. Is a background process touching the same rows?

Signals:

- route/status metrics,
- conflict reason logs,
- actor/client distribution,
- resource type,
- release marker.

### 34.4 High Latency with Low CPU

Possible causes:

- DB pool exhausted,
- downstream HTTP pool exhausted,
- thread pool blocked,
- queueing in gateway,
- slow request body upload,
- slow response consumer,
- lock contention,
- DNS/TLS/connect delay,
- event loop blocked,
- GC pauses not reflected in CPU.

## 35. Dashboards That Actually Help

A useful HTTP service dashboard includes:

1. Request rate by route/status class.
2. Error ratio by route/status.
3. Latency percentiles by route.
4. Top 5 slow endpoints.
5. Top 5 erroring endpoints.
6. Active requests.
7. Thread/event loop saturation.
8. DB pool usage.
9. Outbound dependency latency/error.
10. Rate limit/load shedding count.

13. 502/503/504 split by gateway vs app.

## 36. Alert Design

Bad alerts:

```text
CPU > 80%
Any 500 occurred
4xx > 100/min
Latency average > 200ms
Disk > 70%
```

Better alerts:

```text
p95 latency for critical route exceeds SLO for 10 minutes
5xx ratio for case-service exceeds 2% for 5 minutes
504 at gateway exceeds baseline and app traces are missing
DB pool pending connections > 0 for 5 minutes
error budget burn rate too high
rate limit rejects for premium tenants exceed threshold
synthetic case-read flow failing from two regions
```

Alert on user impact and saturation, not every noisy symptom.

## 37. SLO Thinking for HTTP APIs

SLO examples:

```text
99.9% of GET /cases/{caseId} requests return non-5xx within 300 ms over 30 days.
99.5% of POST /cases submissions return 201 or 202 within 2 seconds over 30 days.
99.9% of evidence download requests either succeed or return controlled 4xx within 1 second excluding client-aborted requests.
```

Define:

- eligible traffic,
- success criteria,
- latency threshold,
- measurement window,
- excluded cases,
- dependency on client behavior,
- business criticality.

Do not put every endpoint under the same SLO.

## 38. Observability for Async HTTP Patterns

For `202 Accepted` workflows:

```text
POST /exports -> 202 + Location: /exports/{id}
GET /exports/{id} -> status
GET /exports/{id}/file -> download
```

Observe separately:

- initial request accepted,
- job queued,
- job started,
- job progress,
- job completed/failed,
- result downloaded.

Metrics:

```text
export.requests.accepted
export.jobs.queued
export.jobs.duration
export.jobs.failed
export.downloads.count
```

Do not consider the API successful only because `POST` returned `202`. The user journey completes when the job result is available.

## 39. Observability for Streaming/SSE

For long-lived streams, normal request duration metrics can distort dashboards.

- stream opened,
- stream closed,
- close reason,
- active streams,
- events sent,
- bytes sent,
- heartbeat sent,
- write failures,
- client disconnects,
- authorization revocation during stream,
- proxy timeout closures.

Separate streaming route latency from normal request latency.

## 41. Privacy and Compliance Constraints

For regulated systems, observability itself can become regulated data.

Questions:

- Do logs contain personal data?
- Do traces contain resource IDs?
- Are tenant IDs sensitive?
- Are IP addresses considered personal data in your jurisdiction?
- Who can access production logs?
- How long are logs retained?
- Are logs encrypted?
- Are audit logs tamper-resistant?
- Can data subject deletion requests affect logs?
- Are debug logs disabled in production?

Engineering observability without governance can create legal and security risk.

## 42. Production Checklist

For each HTTP service, verify:

### Identity and Correlation

- [ ] Every request has request ID.
- [ ] Trace context is accepted and propagated safely.
- [ ] Correlation ID exists for multi-step workflows when needed.
- [ ] Error responses include safe request reference.

### Logs

- [ ] Logs are structured.
- [ ] Access logs and app logs can be correlated.
- [ ] Sensitive headers/body fields are redacted.
- [ ] Expected 4xx are not logged as noisy errors.
- [ ] Security events are distinguishable.
- [ ] Audit logs are separate from debug/app logs.

### Metrics

- [ ] Request rate by route/method/status.
- [ ] Latency percentiles by route.
- [ ] Error ratio by route/status class.
- [ ] Active requests.
- [ ] Thread/event loop saturation.
- [ ] DB/outbound pool saturation.
- [ ] Rate limit/load shedding metrics.
- [ ] No high-cardinality labels such as raw path/user ID/case ID.

### Traces

- [ ] Inbound HTTP requests create server spans.
- [ ] Outbound HTTP calls create client spans.
- [ ] DB/cache/message calls are instrumented.
- [ ] Trace context propagates across services.
- [ ] Error status and exception metadata are attached.
- [ ] Sampling keeps important failures.

### Alerts

- [ ] Alerts are tied to user impact/SLO/saturation.
- [ ] Gateway-generated and app-generated errors are distinguishable.
- [ ] Synthetic probes cover critical flows.
- [ ] Deployment markers appear on dashboards.

### Security and Compliance

- [ ] Authorization denials are observable.
- [ ] Auth failures have reason categories.
- [ ] CSRF/CORS/rate-limit events are observable.
- [ ] Log retention is defined.
- [ ] Access to observability data is controlled.

### Exercise 1: Route Metrics

Given these requests:

```text
GET /cases/1
GET /cases/2
GET /cases/3
PATCH /cases/1
PATCH /cases/2
```

Design metric labels that avoid high cardinality but still allow useful dashboarding.

Expected direction:


### Exercise 2: 504 Diagnosis

A client reports timeout. Gateway shows 504. Application logs show no error.

List five possible explanations.

Possible answers:

1. Request never reached app.
2. Gateway could not connect to upstream.
3. App processed request but trace/log sampling missed it.
4. Gateway timeout fired while app still running.
5. App accepted connection but request queued before handler.
6. Response write failed after app completed.
7. Service mesh sidecar timed out.
8. Load balancer routed to unhealthy target.

### Exercise 3: Safe Error Log

Rewrite this unsafe log:

```text
Failed request body: {"password":"secret","token":"abc","caseId":"CASE-1"}
```

Better:

```json
{
  "event": "request.validation.failed",
  "http.route": "/cases/{caseId}",
  "error.code": "INVALID_REQUEST",
  "field_errors": [
    {"field": "caseId", "code": "invalid_format"}
  ],
  "request_id": "..."
}
```

## 44. Key Takeaways

1. HTTP observability begins at the request/response boundary, but must connect through proxy, gateway, app, database, and downstream calls.
2. Access logs, application logs, and audit logs solve different problems.
3. Request ID, correlation ID, and trace ID are related but not identical.
4. Metrics should use low-cardinality labels such as route template, method, and status—not raw path or user/resource IDs.
5. Logs and traces can carry high-cardinality details, but must be redacted and access-controlled.
6. Status codes are operational signals; do not treat all 4xx or 5xx equally.
7. Latency must be decomposed across gateway, app, dependency, queueing, and network transfer.
8. Retry, timeout, load shedding, and rate limiting must be observable or they will hide failure amplification.
9. Observability design must include privacy, security, and compliance constraints.
10. A top-tier backend engineer designs APIs and their diagnostic surfaces together.
