Ini salah satu area paling sering disalahpahami oleh engineer yang datang dari SQL atau document database.

Di SQL, engineer sering berpikir:

```text

```

Di ScyllaDB/Cassandra-style LSM distributed database, lebih tepat:

```text

```

Delete adalah write.

TTL expiry adalah delete yang dijadwalkan.

Tombstone adalah data penting untuk mencegah deleted data hidup kembali.

---

## 1. Why Deletes Are Hard in Distributed LSM Systems

Dalam cluster RF=3:

```text
Replica A
Replica B
Replica C
```

Misal row:

```text
case_id = C123
status = OPEN
```

Ada di A/B/C.

Saat client delete:

```text
DELETE FROM case_current_by_id WHERE case_id = C123;
```

Bagaimana jika:

```text
Replica C sedang down saat delete.
```

A dan B menerima delete. C masih punya data lama.

Jika delete langsung menghapus fisik di A/B tanpa marker, nanti saat C kembali dan repair berjalan:

```text
C says: I have status=OPEN
A/B say: I don't have row
```

Tanpa tombstone, sistem bisa mengira C punya data yang “hilang” dan menyalinnya kembali ke A/B.

Ini disebut zombie data/data resurrection.

Karena itu delete harus direpresentasikan sebagai tombstone selama cukup lama.

---

## 2. Tombstone Mental Model

Tombstone adalah marker:

```text
data ini sudah dihapus pada timestamp T
```

Ia bukan “nothing”.

Ia adalah data metadata yang aktif memengaruhi read/compaction/repair.

ScyllaDB glossary describes a tombstone as a marker indicating data has been deleted, and warns that many tombstones may impact read performance and disk usage.

Simplified:

```text
old value:
status = OPEN, timestamp = 100

delete tombstone:
deleted status/row, timestamp = 200
```

During read:

```text
if tombstone timestamp > old value timestamp:
  old value suppressed
```

During repair:

```text
tombstone can be sent to stale replica
```

During compaction:

```text
old value and tombstone can eventually be purged if safe
```

---

## 3. Delete Is a Write

A delete mutation goes through write path:

```text
coordinator
replicas
commitlog
memtable
SSTable
compaction later
```

It consumes:

- write throughput,
- commitlog,
- memtable,
- disk,
- replication,
- compaction,
- repair.

Do not design high-volume systems assuming delete is free.

Common bad pattern:

```text
use ScyllaDB as queue
insert task
delete task when processed
```

This creates tombstone-heavy hot partitions.

---

## 4. Types of Tombstones

Tombstones can appear at different granularity.

### 4.1 Row Tombstone

Delete a row:

```sql
DELETE FROM case_current_by_id
WHERE tenant_id = ?
  AND case_id = ?;
```

### 4.2 Cell/Column Tombstone

Delete a column:

```sql
DELETE assignee_id
FROM case_current_by_id
WHERE tenant_id = ?
  AND case_id = ?;
```

### 4.3 Range Tombstone

Delete clustering range:

```sql
DELETE FROM case_events_by_case_month
WHERE tenant_id = ?
  AND case_id = ?
  AND bucket_month = ?
  AND event_time < ?;
```

### 4.4 Partition Tombstone

Delete entire partition:

```sql
DELETE FROM events_by_case_month
WHERE tenant_id = ?
  AND case_id = ?
  AND bucket_month = ?;
```

### 4.5 Collection Tombstone

Updating/replacing collections can create tombstones.

ScyllaDB KB notes that tombstones are generated by DELETE operations, TTLs, and INSERT/UPDATE operations that overwrite composite type values such as map, list, or UDT.

---

## 5. TTL Creates Tombstones

TTL means time-to-live.

Example:

```sql
INSERT INTO command_idempotency_by_id (...)
VALUES (...)
USING TTL 86400;
```

After TTL expires:

```text
row/cell is logically deleted
```

This creates tombstone behavior.

ScyllaDB docs note that TTL is determined by the coordinator node and recommends synchronized clocks when using TTL; docs also advise considering Time Window Compaction Strategy for TTL tables.

Important:

```text
TTL is not free automatic cleanup.
TTL is scheduled logical delete.
```

---

## 6. Table-Level TTL vs Per-Write TTL

Table-level:

```sql
CREATE TABLE sessions_by_token (
    token text PRIMARY KEY,
    user_id uuid,
    created_at timestamp
) WITH default_time_to_live = 3600;
```

Per-write:

```sql
INSERT INTO sessions_by_token (...)
VALUES (...)
USING TTL 3600;
```

Table-level TTL is good if all rows have same retention.

Per-write TTL is useful when retention differs, but random TTL values can make purge inefficient.

ScyllaDB compaction docs strongly recommend using a single TTL value for a given table, noting that multiple TTL values may make purging expired data inefficient because an SSTable remains until all its data is expired.

---

## 7. Tombstone Read Cost

During read, tombstones must be considered.

Example:

```text
partition has:
100 live rows
1,000,000 tombstones
```

Query:

```sql
SELECT *
FROM open_cases_by_assignee_day_bucket
WHERE tenant_id = ?
  AND assignee_id = ?
  AND bucket_day = ?
  AND bucket_id = ?
LIMIT 50;
```

Even with `LIMIT 50`, storage may scan many tombstones before finding live rows.

This can cause:

- high read latency,
- read timeouts,
- memory pressure,
- CPU usage,
- tombstone warnings.

Important:

```text
LIMIT limits returned live rows, not necessarily tombstones scanned.
```

---

## 8. Tombstone Scan Anti-Pattern

Bad design:

```text
work queue table
partition = queue_name
clustering = created_at
delete processed tasks
read next tasks from head
```

As tasks are processed, old head is full of tombstones.

Query “next 10 tasks” scans tombstones.

This is why Cassandra/ScyllaDB-style stores are not naive queue engines.

If you need queue semantics, use:

- Kafka,
- RabbitMQ,
- SQS/PubSub,
- dedicated task queue,
- or carefully bucketed worklist with tombstone strategy.

---

## 9. gc_grace_seconds

`gc_grace_seconds` is table option controlling how long tombstones are retained before they are eligible for garbage collection.

Default in Cassandra-compatible systems is often around 10 days, and ScyllaDB repair docs mention repair should run more often than gc_grace_seconds if you delete data frequently, with default 10 days as example.

Purpose:

```text
give time for tombstones to propagate to replicas that missed delete
```

If tombstone is purged too early and stale replica still has old data, repair can resurrect data.

### 9.1 Example

```text
Day 0: delete row, tombstone written to A/B, C down
Day 5: tombstone still retained
Day 6: C returns, repair runs, tombstone sent to C
Day 10+: tombstone may be eligible for purge
```

Good.

Bad:

```text
Day 0: delete row, C down
Day 1: tombstone purged due gc_grace too low/no repair
Day 2: C returns with old data
Repair can resurrect old data
```

---

## 10. Zombie Data

Zombie data is deleted data that reappears.

Classic cause:

```text
1. replica misses delete
2. tombstone is purged before replica repaired
3. stale replica later participates in repair/read
4. old data copied back
```

This is why repair and gc_grace_seconds are connected.

Zombie data is not a myth. It is a consequence of distributed delete semantics.

### 10.1 Zombie Risk Conditions

Risk increases when:

- deletes/TTL frequent,
- repair not run within gc_grace_seconds,
- nodes down longer than grace,
- gc_grace_seconds set too low,
- restore from old backup,
- manual SSTable manipulation,
- inconsistent schema/repair process,
- CL too low and replicas miss deletes.

---

## 11. Repair and Tombstones

Repair synchronizes replicas.

ScyllaDB repair docs state repair synchronizes data between nodes so replicas hold the same data and is necessary to ensure consistency across cluster nodes.

For delete safety:

```text
repair must propagate tombstones before they are purged
```

ScyllaDB nodetool repair docs advise running repair regularly; if deleting frequently, repair should be more often than gc_grace_seconds.

This is operationally critical.

Application engineer does not run repair daily, but must design tables knowing repair exists.

---

## 12. Hinted Handoff Is Not Enough

Hinted handoff can replay missed writes/deletes to a temporarily down replica.

But it is not a full replacement for repair.

Why?

- hints can expire,
- node down too long,
- coordinator storing hint may fail,
- topology changes,
- hints not comprehensive for all failure modes.

Therefore:

```text
tombstone safety still depends on repair discipline and gc_grace_seconds.
```

---

## 13. Compaction and Tombstone Purge

Tombstones are purged during compaction when safe.

ScyllaDB compaction KB describes that after compaction deletes all shadowed data from SSTables, purgeable tombstones are purged and SSTables holding them can be removed.

But tombstone cannot always be purged immediately after grace.

It must be compacted with the data it shadows.

If old data and tombstone are in different SSTables, compaction must bring them together.

ScyllaDB docs on tombstone GC with Incremental Compaction Strategy note that a droppable tombstone cannot be purged unless compacted with the data it shadows.

---

## 14. Droppable Tombstones

A tombstone can become droppable after conditions are met:

```text
grace period passed
shadowed data can be safely removed
compaction includes relevant SSTables
```

But if no compaction touches relevant files, tombstones may remain.

This is why tombstone-heavy tables can need compaction strategy attention.

Do not assume:

```text
gc_grace_seconds passed -> disk immediately freed
```

Wrong.

---

## 15. TTL and Compaction Strategy

Time-series TTL-heavy tables often need compaction aligned with time windows.

ScyllaDB docs recommend considering TWCS when using TTL.

TWCS can help when:

```text
data written by time
data expires by time
queries target recent windows
late writes limited
single TTL per table
```

But TWCS can be less effective if:

- data arrives very late,
- multiple TTL values,
- updates old windows,
- queries scan many windows,
- table mixes permanent and expiring data.

Compaction strategy deep dive is part 016.

For now:

```text
TTL design and compaction design are linked.
```

---

## 16. Multiple TTL Values Problem

If one SSTable contains rows expiring at very different times, the file cannot be fully dropped until all data in it is expired/purgeable.

Docs warn that multiple TTL values in a table may make purging expired data inefficient.

Bad:

```text
same table:
some rows TTL 1 hour
some rows TTL 90 days
some rows no TTL
```

Better:

- separate tables by retention class,
- use table-level default TTL,
- align buckets with retention,
- avoid random TTLs when high volume.

---

## 17. TTL Coordinator Clock

TTL is determined by coordinator node according to ScyllaDB docs.

Therefore synchronized clocks matter.

If node clocks differ:

- expiry time anomalies,
- data expires earlier/later,
- conflict/timestamp weirdness.

Use NTP/chrony and monitor clock drift.

For Java app:

```text
do not assume app clock solely determines TTL expiry
```

If retention needs exact business semantics, store explicit `expires_at` column too and enforce in app/workflow.

---

## 18. Nulls and Tombstones

In Cassandra-compatible systems, setting a column to null can create tombstone-like behavior.

Bad Java mapping:

```java
UPDATE user_profile
SET name = ?, email = ?, phone = ?
WHERE user_id = ?;
```

If mapper binds null for absent optional fields, it may delete columns.

This can create many tombstones unintentionally.

Better:

- update only changed columns,
- distinguish “not provided” from “set to null/delete”,
- avoid generic object upsert mapping,
- be careful with ORMs/mappers.

---

## 19. Collections and Tombstones

Collections can generate tombstones.

Examples:

```sql
UPDATE users SET tags = tags + {'vip'} WHERE user_id = ?;
```

May be okay for small bounded set.

But replacing a whole collection:

```sql
UPDATE users SET tags = ? WHERE user_id = ?;
```

can tombstone old collection entries.

Large collections are dangerous because:

- many cells,
- tombstones per element,
- read amplification,
- compaction pressure.

Use child table for unbounded collections.

---

## 20. UDT/Frozen Value Overwrite

Overwriting frozen composite values can create tombstones for old value.

ScyllaDB KB explicitly lists INSERT/UPDATE operations that overwrite whole composite type values like map, list, or UDT as tombstone generators.

Design:

- keep UDT small and stable,
- avoid high-frequency overwrite of large frozen structures,
- split mutable fields into columns/table,
- avoid storing large mutable document as frozen blob/UDT.

## 21. Delete Patterns

### 21.1 Rare Delete

Example:

```text
delete user account after legal workflow
```

Fine if planned.

### 21.2 TTL Expiry

Example:

```text
idempotency keys expire after 7 days
```

Fine if table designed for TTL.

### 21.3 High-Churn Delete

Example:

```text
queue processed tasks deleted constantly
```

Danger.

### 21.4 Range Delete

Example:

```text
delete old events within partition by time
```

Can create range tombstones; use carefully.

### 21.5 Derived View Cleanup

Example:

```text
delete old open-case row when case closes
```

Common but must monitor tombstone rate and partition bounds.

---

## 22. Derived Tables and Tombstones

Derived views often need delete old key:

```text
case moved from assignee A to B
delete row in A queue
insert row in B queue
```

If many moves:

```text
tombstone-heavy assignee partitions
```

Mitigations:

1. bucket derived view by day/status,
2. tolerate stale rows and validate on read,
3. periodic rebuild,
4. short-lived bucket windows,
5. avoid storing highly volatile dimensions in primary key,
6. design state transition rate limits.

---

## 23. Validate-on-Read to Reduce Deletes

Instead of deleting every stale derived row immediately:

```text
read candidates from derived view
check current state
filter stale
```

Pros:

- fewer deletes,
- less tombstone churn,
- easier async projection.

Cons:

- extra reads,
- stale candidate overfetch,
- higher latency,
- possible user-visible lag.

Good when:

- candidate list is bounded,
- stale ratio low,
- action validates source,
- derived view is not authoritative.

Monitor stale ratio.

---

## 24. Periodic Rebuild Instead of Continuous Cleanup

For very high-churn derived views, consider:

```text
build new table/window periodically
switch pointer
drop old table/window later
```

Example:

```text
open_queue_snapshot_by_assignee_hour
```

This can reduce tombstone churn but increases architecture complexity and staleness.

Useful for:

- dashboards,
- approximate queues,
- search-like candidate lists,
- batch-generated views.

Not good for strict real-time assignment unless semantics allow.

---

## 25. Queue Anti-Pattern Deep Dive

Naive queue table:

```sql
CREATE TABLE tasks_by_queue (
    queue_name text,
    created_at timestamp,
    task_id uuid,
    payload text,
    PRIMARY KEY (queue_name, created_at, task_id)
);
```

Worker:

```text
SELECT first 10
process
DELETE processed
```

Problems:

- queue_name low cardinality,
- hot partition,
- tombstones at head,
- race between workers,
- visibility timeout hard,
- retry/claim logic needs LWT,
- processed deletes accumulate,
- p99 degrades over time.

Better:

- use RabbitMQ/Kafka/SQS for queue,
- use ScyllaDB for task state lookup,
- if using worklist, bucket by time/hash and validate/claim by ID,
- avoid deleting from hot head repeatedly.

---

## 26. Soft Delete vs Hard Delete

Soft delete:

```text
set deleted_at/status = DELETED
```

Hard delete:

```sql
DELETE ...
```

Soft delete avoids immediate tombstone but retains data.

Pros:

- easier audit,
- no zombie/tombstone issue for logical delete,
- can filter in app if query bounded,
- legal hold friendly.

Cons:

- data remains,
- queries must exclude deleted,
- storage grows,
- privacy deletion may require hard delete/anonymization.

Hard delete is needed for:

- privacy/legal deletion,
- storage removal,
- TTL expiry,
- derived cleanup.

Often use both:

```text
source audit: soft delete/legal state
derived views: hard delete/expire
```

---

## 27. Legal Retention and TTL

TTL is dangerous for legally controlled data.

If audit table has TTL:

```text
data may expire automatically
```

But legal hold may require retention.

For regulatory/case systems:

- no TTL on source audit events unless policy mandates,
- explicit retention service,
- legal hold flag,
- deletion approval workflow,
- immutable archive,
- backup retention alignment,
- derived views may TTL if rebuildable.

TTL is schema-level deletion policy. Treat as compliance artifact.

---

## 28. Privacy Deletion Across Duplicates

If PII appears in multiple derived tables:

```text
case_current_by_id
open_cases_by_assignee
notifications_by_user
search index
analytics export
backups
```

Deleting only source table is insufficient.

Need:

- data inventory,
- derived deletion/anonymization workflow,
- external projection deletion,
- backup retention policy,
- audit record of deletion,
- reconciliation to ensure no reappearance.

In ScyllaDB, each delete creates tombstones in each table.

Plan capacity for privacy deletion campaigns.

---

## 29. Backup/Restore and Resurrection

Restoring old backups can reintroduce deleted data if not handled carefully.

ScyllaDB backup docs warn that restoring schema backups created by older cqlsh versions may lead to data resurrection or data loss.

General principle:

```text
backup/restore must preserve schema, tombstone, and repair assumptions.
```

If you restore SSTables/data from before deletes into a cluster where tombstones are gone, old data can reappear.

Disaster recovery plan must include delete semantics.

---

## 30. gc_grace_seconds = 0?

Sometimes engineers set:

```sql
WITH gc_grace_seconds = 0
```

to purge tombstones faster.

This is dangerous if table has deletes/TTL and replicas can miss them.

When might it be acceptable?

- table never deletes and has no TTL,
- data can be fully rebuilt,
- CL ALL used for tombstone-generating operations and operational assumptions hold,
- single-node/non-replicated dev/test,
- ephemeral derived table where resurrection is acceptable and rebuild handles it.

ScyllaDB KB notes you can avoid data resurrection and repair-every-gc_grace requirement if tombstones are generated with operations using CL ALL. But CL ALL has availability cost and must be used consistently for tombstone-generating operations.

Default advice:

```text
do not reduce gc_grace_seconds casually.
```

---

## 31. Changing gc_grace_seconds

Changing gc_grace_seconds is operationally sensitive.

ScyllaDB has KB guidance for changing it and for flushing old tombstones, including running repair first and setting gc_grace_seconds based on time since last repair.

Application engineer takeaway:

```text
gc_grace_seconds is not just a performance knob.
It is a delete correctness knob.
```

Before changing:

- understand delete/TTL workload,
- confirm repair schedule,
- know node downtime risk,
- know backup/restore implications,
- coordinate with DB operations.

---

## 32. Tombstone Warnings

ScyllaDB can warn about tombstone-heavy reads.

Treat warnings seriously.

They indicate queries may be scanning many deleted/expired cells.

Common causes:

- TTL-heavy table queried beyond live window,
- queue-like deletes,
- derived view churn,
- large collection overwrite,
- range deletes,
- low-cardinality partition with many deletes,
- `LIMIT` over tombstone-heavy clustering range.

Fix usually requires data model/lifecycle change, not only tuning.

---

## 33. Large Partition + Tombstones

Large partition plus tombstones is especially bad.

Example:

```text
open_cases_by_status:
partition status=OPEN
millions rows
many cases close -> tombstones
query LIMIT 100
```

Reads scan tombstones and old rows.

Mitigate:

- bucket by day/hash/tenant,
- avoid low-cardinality partition,
- validate-on-read,
- rebuild view,
- use search/OLAP if query broad.

## 34. Time-Series TTL Design

Good TTL time-series:

```sql
CREATE TABLE login_attempts_by_user_day (
    tenant_id uuid,
    user_id uuid,
    bucket_day date,
    attempt_time timestamp,
    attempt_id uuid,
    success boolean,
    PRIMARY KEY ((tenant_id, user_id, bucket_day), attempt_time, attempt_id)
) WITH default_time_to_live = 7776000;
```

Properties:

- bucket by day,
- fixed TTL,
- queries recent days,
- old buckets not queried,
- compaction can align.

Bad:

```text
same table stores attempts TTL 90d and permanent audit events no TTL
```

Split tables.

---

## 35. Idempotency TTL Design

Idempotency table:

```sql
CREATE TABLE command_idempotency_by_id (
    tenant_id uuid,
    command_id uuid,
    status text,
    result_ref text,
    created_at timestamp,
    PRIMARY KEY ((tenant_id, command_id))
) WITH default_time_to_live = 604800;
```

TTL means:

```text
after 7 days, same command_id can be accepted again
```

Is that okay?

For UI retry, maybe.

For financial/regulatory command, maybe not.

Design retention based on duplicate risk window.

Maybe store permanent command audit separately.

---

## 36. Notifications TTL Design

Notification feed:

```text
TTL 30 days
```

Good if notifications are ephemeral.

But if notification includes legal notice:

```text
source record must be elsewhere without TTL
```

Feed table can expire display item, but source/audit remains.

Do not let UX feed table become only source of legally important notification.

---

## 37. TTL and Update

Updating a TTL row can affect TTL semantics.

If you update a column without TTL, depending table/default TTL and statement, you may create cells with different TTL behavior.

Be explicit.

Avoid mixing:

```text
some columns TTL
some columns no TTL
same row
```

unless you understand storage/read semantics.

For simple TTL table, prefer:

```text
table default TTL
full row inserted with same retention
```

---

## 38. Range Deletes vs Drop Bucket/Table

For large expiry by time, deleting ranges row-by-row/range-by-range can create many tombstones.

Alternatives:

- TTL + TWCS,
- separate table per retention class,
- drop old table/keyspace if partitioned physically by time/environment,
- archive and rebuild,
- application stops querying expired buckets.

CQL does not have cheap “drop partition without tombstone” semantics for ordinary rows.

Deleting many rows is still tombstone creation.

---

## 39. Tombstone Impact on Java APIs

API design should prevent tombstone-heavy reads.

Bad:

```http
GET /notifications?from=2020-01-01&to=2026-06-21
```

on TTL 30-day table.

Better:

```text
max range within retention
reject old ranges
use export/archive for historical
```

Repository should enforce:

```java
if (from.isBefore(now.minus(retention))) {
    throw new RangeOutsideRetentionException();
}
```

Do not let clients query expired tombstone ranges.

---

## 40. Java Mapper Pitfalls

Generic mappers can create tombstones by writing nulls.

Bad pattern:

```java
bound.setString("middle_name", user.getMiddleName()); // null means delete
```

Better:

- only bind column when intentionally updating,
- use separate update statements,
- distinguish optional absent vs explicit delete,
- avoid `UPDATE every column` pattern,
- test generated CQL.

For partial update APIs:

```text
PATCH field omitted != set field null
```

This is both API semantics and tombstone control.

---

## 41. API Delete Semantics

Expose delete carefully.

Options:

```text
soft delete
hard delete
expire/TTL
anonymize
archive then delete
remove from derived views only
```

Example:

```http
DELETE /cases/{id}
```

For regulatory case, this is ambiguous.

Better commands:

```http
POST /cases/{id}/close
POST /cases/{id}/archive
POST /cases/{id}/anonymize
POST /cases/{id}/legal-hold
```

Each has defined storage behavior.

## 42. Deleting Derived Rows Safely

To delete derived row, you need full primary key.

Example:

```sql
DELETE FROM open_cases_by_assignee_day_bucket
WHERE tenant_id = ?
  AND assignee_id = ?
  AND bucket_day = ?
  AND bucket_id = ?
  AND due_at = ?
  AND case_id = ?;
```

If you do not know old `assignee_id`, `bucket_day`, `bucket_id`, `due_at`, you cannot delete precisely without scan.

Therefore store old key fields in:

- current state,
- event old/new payload,
- projection metadata,
- command context.

---

## 43. Stale Derived Rows and Tombstone Trade-Off

For derived views:

```text
delete old row immediately
```

vs:

```text
leave stale row and validate on read
```

Immediate delete:

- fresher view,
- tombstones.

Validate-on-read:

- fewer tombstones,
- stale candidate filtering.

Choose based on:

- churn,
- read QPS,
- stale tolerance,
- partition size,
- action correctness.

Often:

```text
candidate list can be stale, action validates source
```

works well.

## 44. Tombstones and Materialized Views/Indexes

Deleting from base table also affects indexes/materialized views.

This can create additional tombstones/work in view/index storage.

If table has high delete/TTL churn plus indexes/MVs, write/compaction amplification grows.

Part 017 covers indexes/MVs.

For now:

```text
indexes/views do not make delete free; they multiply lifecycle work.
```

## 45. Tombstones and Counters

Counters have special storage semantics. Avoid frequent counter delete/recreate patterns.

If aggregate needs retention, consider:

- time-bucketed aggregate rows,
- event-derived recompute,
- OLAP,
- approximate counters.

## 46. Operational Playbook: Tombstone Incident

Symptoms:

```text
read latency spike
tombstone warnings
timeouts on specific table/query
compaction backlog
disk high
```

Investigation:

```text
1. Identify table/query.
2. Identify partition key(s).
3. Check query range and LIMIT.
4. Check TTL/delete rate.
5. Check tombstone warnings/logs.
6. Check compaction strategy/backlog.
7. Check partition size.
8. Check recent deployment/migration.
9. Check mapper writes nulls/collection overwrites.
10. Check repair/gc_grace schedule.
```

Mitigations:

- stop offending query,
- limit range,
- reduce polling,
- throttle deletes/backfill,
- run/adjust compaction with DBA guidance,
- rebuild table with better model,
- fix mapper/update statements,
- adjust TTL/compaction strategy carefully.

## 47. Data Model Fix Patterns

### 47.1 Bucket More Narrowly

```text
status -> status + day + bucket
```

### 47.2 Avoid Delete-Heavy Primary Key

If field changes often, do not put it in primary key of high-churn view unless necessary.

### 47.3 Validate on Read

Reduce deletes.

### 47.4 Separate Retention Classes

Different TTL table.

### 47.5 Immutable Append + Current Snapshot

Avoid updating/deleting event history.

### 47.6 External Queue/Search

Use system with better lifecycle semantics for queue/search.

### 47.7 Rebuild Table

Create v2 schema and backfill without tombstone baggage.

## 48. Table Rebuild to Escape Tombstones

If table is tombstone-polluted:

```text
create new table with better key/TTL
dual-write
backfill only live data
switch reads
drop old table later
```

This can be cleaner than trying to purge enormous tombstone debt.

Derived tables are good candidates for rebuild because they are rebuildable.

Source tables require much more care.

## 51. Common Misconceptions

### Misconception 1: “DELETE removes data immediately.”

No. It writes tombstone.

### Misconception 2: “TTL is free cleanup.”

No. TTL creates tombstone lifecycle and compaction work.

### Misconception 3: “LIMIT avoids tombstone scans.”

Not necessarily. LIMIT controls returned rows, not all scanned tombstones.

### Misconception 4: “gc_grace_seconds is just disk cleanup.”

No. It is delete correctness window.

### Misconception 5: “Repair is optional if we use QUORUM.”

No. Repair remains necessary for replica convergence and tombstone safety.

### Misconception 6: “Setting gc_grace_seconds=0 improves performance safely.”

Only in special cases. It can cause zombie data.

### Misconception 7: “Collections are harmless small JSON.”

Large or overwritten collections can generate tombstones and read amplification.

### Misconception 8: “Soft delete and hard delete are same.”

No. Soft delete is application state. Hard delete creates tombstones.

## 52. Mental Model Compression

Remember:

```text
DELETE = write tombstone.
TTL expiry = future tombstone.
Tombstone = deletion evidence.
gc_grace_seconds = how long deletion evidence must survive for replica safety.
Repair = how deletion evidence reaches replicas that missed it.
Compaction = when old data and deletion evidence can be physically purged.
```

And:

```text
The safest delete is the one your data model avoids needing at high frequency.
```

## 53. Summary

Deletes and TTL are fundamental but dangerous if misunderstood.

Key lessons:

1. Delete is a write, not immediate physical removal.
2. Tombstones prevent deleted data from resurrecting.
3. TTL creates tombstone behavior after expiry.
4. Tombstones affect read latency and disk usage.
5. `LIMIT` does not necessarily prevent tombstone scan.
6. `gc_grace_seconds` protects against zombie data.
7. Repair must run within grace window for delete-heavy tables.
8. Compaction purges tombstones only when safe and when relevant SSTables compact.
9. Multiple TTL values in same table can make expiry inefficient.
10. TTL depends on coordinator time; clock sync matters.
11. Null writes, collection overwrites, and UDT overwrites can create tombstones.
12. Queue-like delete-heavy patterns are dangerous.
13. Derived views need old key cleanup or stale validation strategy.
14. Legal retention and TTL must be designed together.
15. Privacy deletion must cover all duplicate/derived/external copies.
16. Changing gc_grace_seconds is a correctness decision, not just tuning.
17. Java mappers must distinguish absent fields from explicit null/delete.
18. Tombstone-heavy tables may need rebuild with better model.
