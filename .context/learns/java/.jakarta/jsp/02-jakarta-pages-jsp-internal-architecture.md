## 1. Tujuan Pembelajaran

Setelah bagian ini, kamu diharapkan mampu:

1. Menjelaskan apa yang sebenarnya terjadi saat file `.jsp` diakses.

3. Membaca JSP sebagai bentuk lain dari servlet yang dihasilkan oleh container.
4. Memahami implicit objects seperti `request`, `response`, `session`, `application`, `pageContext`, `out`, `config`, `page`, dan `exception`.

8. Menilai kapan JSP aman dipakai, kapan harus direfaktor, dan bagaimana membaca legacy JSP tanpa tersesat.

## 2. Baseline Mental Model

Jakarta Pages, yang secara historis dikenal sebagai JSP, adalah spesifikasi template engine untuk aplikasi web Java. File `.jsp` terlihat seperti HTML yang disisipkan expression, directive, dan tag. Namun secara runtime, JSP tidak dieksekusi sebagai “file HTML”. JSP diproses oleh container menjadi servlet.

Mental model paling penting:

```text
.jsp source file
    |
    | 1. translation
    v
Generated Java servlet source
    |
    | 2. compilation
    v
Generated servlet class
    |
    | 3. class loading + instantiation
    v
Servlet object
    |
    | 4. request handling
    v
HTTP response body
```

Jadi, JSP adalah:

```text
Template syntax -> generated servlet -> servlet lifecycle -> response writer
```

Implikasinya besar:

1. Semua prinsip servlet tetap berlaku.
2. Threading model servlet tetap berlaku.
3. Response commit tetap berlaku.
4. Request/session/application scope tetap berlaku.
5. Classloader dan redeployment issue tetap berlaku.
6. Error handling tetap berada di atas Servlet API.
7. JSP bisa terlihat declarative, tetapi tetap berakhir sebagai Java code yang berjalan di container.

## 3. Kenapa JSP Ada?

Servlet murni kuat, tetapi buruk untuk menulis HTML besar.

Contoh servlet murni:

```java
@WebServlet("/hello")
public class HelloServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest request, HttpServletResponse response)
            throws IOException {
        response.setContentType("text/html;charset=UTF-8");
        PrintWriter out = response.getWriter();
        out.println("<html>");
        out.println("<body>");
        out.println("<h1>Hello</h1>");
        out.println("</body>");
        out.println("</html>");
    }
}
```

Masalahnya:

1. HTML bercampur sebagai string Java.
2. Readability buruk.
3. Designer/front-end sulit bekerja.
4. Escaping mudah salah.
5. Struktur halaman sulit dilihat.
6. Perubahan view memerlukan edit Java code.

JSP membalik arah:

```jsp
<%@ page contentType="text/html;charset=UTF-8" %>
<!doctype html>
<html>
<body>
    <h1>Hello ${user.name}</h1>
</body>
</html>
```

Secara authoring, JSP lebih nyaman karena HTML menjadi bentuk utama. Tetapi secara runtime, JSP tetap menjadi servlet.

Kesimpulan mental model:

```text
Servlet = Java-first response generation
JSP     = HTML-first response generation that becomes servlet
```

## 4. JSP sebagai Abstraction di Atas Servlet

JSP bukan pengganti Servlet API. JSP bergantung pada Servlet API.

Secara konseptual:

```text
HTTP request
    |
    v
Servlet container
    |
    +--> Filter chain
    |
    +--> Servlet dispatching
            |
            +--> Controller servlet
            |       |
            |       +--> request.setAttribute(...)
            |       +--> forward to JSP
            |
            +--> JSP servlet generated by JSP engine
                    |
                    +--> renders response
```

Dalam aplikasi yang sehat, JSP biasanya berperan sebagai **view**, bukan controller utama.

Pola yang disarankan:

```text
Request
  -> Controller Servlet / MVC Controller
  -> Service Layer
  -> View Model prepared in request scope
  -> Forward to JSP
  -> JSP renders only presentation
```

Pola yang buruk:

```text
Request
  -> JSP
      -> reads request parameter
      -> calls database
      -> mutates session
      -> contains business rule
      -> renders HTML
```

JSP mampu melakukan banyak hal, tetapi kemampuan itu bukan berarti semua pantas dilakukan di JSP.

## 5. Dua Fase Besar JSP: Translation Phase dan Request Phase

Spesifikasi JSP/Jakarta Pages menjelaskan bahwa JSP page melewati dua fase besar:

1. **Translation phase** — JSP diterjemahkan menjadi servlet class.
2. **Request phase** — servlet hasil translation menangani request.

### 5.1 Translation Phase

Translation phase terjadi ketika container perlu mengubah file `.jsp` menjadi Java servlet source.

Trigger-nya bisa berbeda tergantung container/configuration:

1. Saat request pertama ke JSP.
2. Saat aplikasi startup jika JSP precompiled.
3. Saat file JSP berubah di development mode.
4. Saat container mendeteksi generated class stale.

Input translation:

```text
/WEB-INF/views/user/list.jsp
```

Output konseptual:

```text
Generated Java source:
org.apache.jsp.WEB_002dINF.views.user.list_jsp.java
```

Nama class dan lokasi file generated berbeda antar container. Di Tomcat/Jasper, generated JSP source dan class biasanya berada di working directory container.

### 5.2 Compilation Phase

Setelah translation menghasilkan Java source, source tersebut dikompilasi menjadi `.class`.

```text
list.jsp
  -> list_jsp.java
  -> list_jsp.class
```

Compilation error bisa muncul dari:

1. Syntax JSP salah.
2. Java code/scriptlet salah.
3. Import class tidak ditemukan.
4. Tag library tidak ditemukan.
5. Method/property tidak valid.
6. Dependency compile classpath berbeda dengan runtime.

### 5.3 Class Loading dan Instantiation

Generated servlet class dimuat oleh web application classloader.

```text
Generated class loaded by webapp classloader
    -> instantiated as servlet
    -> initialized
```

Karena masuk ke classloader aplikasi, JSP juga bisa terkena problem:

1. stale class setelah redeploy,
2. classloader leak,
3. dependency conflict,
4. perbedaan class antara container dan aplikasi,
5. incompatible `javax.*` vs `jakarta.*`.

### 5.4 Request Phase

Setelah menjadi servlet, setiap request diproses oleh method generated servlet.

Secara konseptual:

```java
public final class list_jsp extends HttpJspBase {
    public void _jspService(HttpServletRequest request,
                            HttpServletResponse response)
            throws IOException, ServletException {
        // generated rendering logic
    }
}
```

JSP container menghasilkan method semacam `_jspService(...)`, lalu mengisi body method tersebut dengan:

1. setting content type,
2. inisialisasi implicit objects,
3. menulis static HTML ke writer,
4. mengevaluasi EL,
5. menjalankan tag handlers,
6. menjalankan scriptlet jika ada,
7. mengirim output ke response.

## 6. Contoh Transformasi JSP ke Servlet

Misal JSP sederhana:

```jsp
<%@ page contentType="text/html;charset=UTF-8" %>
<!doctype html>
<html>
<body>
    <h1>Hello</h1>
    <p>User: ${user.name}</p>
</body>
</html>
```

Secara konseptual, generated servlet akan mirip:

```java
public final class hello_jsp extends HttpJspBase {

    @Override
    public void _jspService(HttpServletRequest request,
                            HttpServletResponse response)
            throws IOException, ServletException {

        response.setContentType("text/html;charset=UTF-8");

        PageContext pageContext = ...;
        JspWriter out = pageContext.getOut();

        out.write("<!doctype html>\n");
        out.write("<html>\n");
        out.write("<body>\n");
        out.write("    <h1>Hello</h1>\n");
        out.write("    <p>User: ");
        out.write(evaluateExpression("${user.name}", pageContext));
        out.write("</p>\n");
        out.write("</body>\n");
        out.write("</html>\n");
    }
}
```

Ini bukan kode real dari semua container, tapi cukup dekat untuk mental model.

Hal penting:

1. Static HTML menjadi `out.write(...)`.
2. EL menjadi panggilan evaluator/resolver.
3. Tag menjadi tag handler invocation.
4. Directive mengubah generated servlet metadata/behavior.
5. Scriptlet menjadi Java code yang dimasukkan ke generated servlet.

## 7. JSP Lifecycle dan Servlet Lifecycle

JSP punya lifecycle sendiri sebagai artifact template, tetapi setelah menjadi servlet, ia mengikuti lifecycle servlet.

```text
JSP file created/changed
    |
    v
Translation
    |
    v
Compilation
    |
    v
Class loading
    |
    v
Instantiation
    |
    v
jspInit()
    |
    v
_jspService(request, response) per request
    |
    v
jspDestroy()
```

### 7.1 `jspInit()`

Dipanggil sekali saat JSP servlet diinisialisasi.

Dapat digunakan untuk inisialisasi resource, tetapi jarang direkomendasikan di JSP modern. Resource sebaiknya dikelola di listener, CDI bean, service, atau container-managed component.

### 7.2 `_jspService()`

Dipanggil untuk setiap request.

Ini area utama generated rendering logic.

Poin penting:

```text
Multiple requests may execute _jspService concurrently on the same JSP servlet instance.
```

Artinya:

1. Jangan simpan mutable request-specific state di field JSP declaration.
2. Jangan membuat shared mutable object tanpa synchronization.
3. Jangan mengandalkan instance field untuk data user.

### 7.3 `jspDestroy()`

Dipanggil saat JSP servlet dihancurkan.

Bisa terjadi saat:

1. web application undeploy,
2. container shutdown,
3. JSP recompilation/reload,
4. classloader refresh.

## 8. Generated Servlet dan Thread Safety

Ini salah satu jebakan klasik JSP.

Misal JSP legacy:

```jsp
<%! private int counter = 0; %>

<%
    counter++;
%>

<p>Counter: <%= counter %></p>
```

Declaration `<%! ... %>` menjadi field atau method di generated servlet.

Secara konseptual:

```java
public final class counter_jsp extends HttpJspBase {
    private int counter = 0;

    public void _jspService(HttpServletRequest request,
                            HttpServletResponse response) {
        counter++;
        out.write("Counter: " + counter);
    }
}
```

Karena satu servlet instance bisa melayani banyak request paralel, `counter++` tidak thread-safe.

Masalahnya bukan hanya race condition. Lebih buruk:

1. Data user A bisa memengaruhi user B.
2. State tidak ter-reset antar request.
3. Nilai bisa korup dalam concurrency tinggi.
4. Bug sulit direproduksi.

Rule of thumb:

```text
Di JSP, jangan gunakan declaration untuk mutable state.
```

Jika perlu state:

| Kebutuhan | Tempat yang lebih tepat |
|---|---|
| Data untuk satu render | local variable / request attribute |
| Data satu request lintas controller-view | request scope |
| Data user lintas beberapa request | session scope, dengan hati-hati |
| Shared read-only config | application scope / CDI singleton / config service |
| Shared mutable state | service dengan concurrency control eksplisit |

## 9. Struktur Generated Servlet Secara Konseptual

Generated servlet biasanya memiliki elemen berikut:

```text
Generated JSP servlet
├── imports
├── class declaration
├── static metadata
├── dependency/tag helper fields
├── jspInit()
├── jspDestroy()
└── _jspService(request, response)
    ├── create PageContext
    ├── initialize implicit objects
    ├── set response content type
    ├── write template text
    ├── evaluate EL
    ├── invoke tag handlers
    ├── execute scriptlets
    ├── handle exceptions
    └── release PageContext
```

Kenapa ini penting untuk engineer advanced?

Karena saat produksi bermasalah, stack trace JSP sering mengarah ke generated servlet, bukan file JSP asli.

Contoh stack trace:

```text
org.apache.jasper.JasperException: An exception occurred processing [/WEB-INF/views/user/list.jsp] at line [42]

39: <c:forEach var="user" items="${users}">
40:   <tr>
41:     <td>${user.name}</td>
42:     <td>${user.department.name}</td>
43:   </tr>
44: </c:forEach>
```

Atau stack trace generated class:

```text
at org.apache.jsp.WEB_002dINF.views.user.list_jsp._jspService(list_jsp.java:184)
```

Engineer yang baik tidak berhenti di “JSP error”. Ia tahu bahwa:

1. line JSP dipetakan ke generated source,
2. generated source membantu membaca transformasi real,
3. error bisa berasal dari EL resolver, tag handler, atau Java scriptlet,
4. failure terjadi dalam servlet request handling.

## 10. JSP Implicit Objects

JSP menyediakan beberapa object yang otomatis tersedia di page.

| Object | Type konseptual | Scope / Fungsi |
|---|---|---|
| `request` | `HttpServletRequest` | Data request saat ini |
| `response` | `HttpServletResponse` | Response saat ini |
| `session` | `HttpSession` | Session user, jika enabled |
| `application` | `ServletContext` | Context aplikasi |
| `out` | `JspWriter` | Writer output JSP |
| `pageContext` | `PageContext` | Hub utama JSP context |
| `config` | `ServletConfig` | Konfigurasi servlet JSP |
| `page` | `Object` | Instance servlet page saat ini |
| `exception` | `Throwable` | Hanya tersedia pada error page |

Mari kita bedah satu per satu secara mental model.

## 11. `request`: Boundary Request Saat Ini

`request` mewakili HTTP request saat ini.

Dari JSP kamu bisa membaca:

```jsp
<p>URI: <%= request.getRequestURI() %></p>
```

Namun pada aplikasi modern, JSP sebaiknya tidak banyak membaca raw parameter langsung.

Kurang baik:

```jsp
<%
String status = request.getParameter("status");
%>
```

Lebih baik:

```java
// Controller
String status = request.getParameter("status");
UserListViewModel vm = userService.search(status);
request.setAttribute("model", vm);
request.getRequestDispatcher("/WEB-INF/views/users.jsp").forward(request, response);
```

```jsp
<!-- JSP -->
<p>Status: ${model.statusLabel}</p>
```

Alasannya:

1. Parsing input berada di controller.
2. Validasi berada sebelum view.
3. JSP hanya render state yang sudah diputuskan.
4. Test controller-view contract lebih mudah.
5. View tidak menjadi tempat business decision.

---

## 12. `response`: Boundary Output HTTP

`response` adalah object untuk mengatur response.

Bisa digunakan untuk:

```jsp
<%
response.setHeader("X-Example", "value");
%>
```

Namun hati-hati: JSP sering sudah mulai menulis body. Setelah response committed, banyak perubahan tidak bisa dilakukan.

Response committed berarti header/status sudah dikirim ke client.

Setelah committed:

1. status code tidak bisa diubah dengan aman,
2. redirect bisa gagal,
3. forward bisa gagal,
4. header bisa tidak efektif,
5. error page bisa tidak bisa diterapkan sempurna.

Anti-pattern:

```jsp
<html>
<body>
Some output...
<%
if (!authorized) {
    response.sendRedirect("/login");
}
%>
</body>
</html>
```

Masalah:

1. Output mungkin sudah masuk buffer.
2. Jika buffer flush, response committed.
3. Authorization terlambat dilakukan.
4. View punya tanggung jawab controller/security.

Lebih baik:

```text
Filter / Security layer / Controller
    -> check authorization
    -> redirect / error / forward
    -> JSP only if allowed
```

## 13. `session`: User Conversation State

`session` adalah `HttpSession`. Dalam JSP, object ini tersedia jika session enabled.

Default JSP biasanya dapat membuat session jika belum ada, kecuali dinonaktifkan:

```jsp
<%@ page session="false" %>
```

Ini penting. JSP yang tampak stateless bisa tanpa sadar membuat session.

Dampak accidental session creation:

1. Memory server naik.
2. Sticky session diperlukan.
3. Load balancer behavior berubah.
4. Session replication cost naik.
5. Anonymous user pun punya session.
6. Cacheability response turun.

Jika halaman benar-benar public/stateless:

```jsp
<%@ page session="false" %>
```

Namun kalau `session="false"`, object `session` tidak tersedia sebagai implicit object.

Contoh yang akan gagal:

```jsp
<%@ page session="false" %>
<%= session.getAttribute("user") %>
```

Design rule:

```text
Session harus eksplisit. Jangan biarkan JSP menciptakan session hanya karena default behavior.
```

---

## 14. `application`: Shared Servlet Context

`application` adalah `ServletContext`.

Ia bersifat shared untuk seluruh web application.

Contoh:

```jsp
<p>App version: ${applicationScope.appVersion}</p>
```

Shared berarti:

1. semua user melihat data yang sama,
2. semua request bisa mengakses,
3. mutable state harus thread-safe,
4. data bisa hidup selama webapp hidup,
5. data bisa hilang saat redeploy.

Cocok untuk:

1. read-only application metadata,
2. feature flag snapshot,
3. public config ringan,
4. cache kecil yang dikelola dengan disiplin.

Tidak cocok untuk:

1. user-specific data,
2. request-specific data,
3. business transaction state,
4. mutable collection tanpa concurrency control,
5. cache besar tanpa eviction.

---

## 15. `out`: `JspWriter`, Buffering, dan Output

`out` adalah `JspWriter`, bukan sekadar `PrintWriter` biasa.

Ia terkait dengan JSP buffering.

```jsp
<%
out.write("Hello");
%>
```

JSP menulis output ke buffer terlebih dahulu. Buffer memberi kesempatan container untuk:

1. mengumpulkan output sebelum dikirim,
2. mengubah header/status sebelum commit,
3. forward ke resource lain selama belum committed,
4. menampilkan error page jika terjadi exception sebelum committed.

Page directive bisa mengatur buffer:

```jsp
<%@ page buffer="16kb" autoFlush="true" %>
```

Jika buffer penuh dan `autoFlush="true"`, container flush otomatis.

Jika buffer penuh dan `autoFlush="false"`, bisa terjadi exception.

Mental model:

```text
JSP out.write(...)
    -> JSP buffer
    -> Servlet response buffer
    -> network socket
```

Begitu buffer flush sampai response committed, kamu kehilangan fleksibilitas.

---

## 16. `pageContext`: Hub Semua Context JSP

`pageContext` adalah object sentral JSP.

Ia menyediakan akses ke:

1. request,
2. response,
3. session,
4. application,
5. out,
6. exception,
7. scoped attributes,
8. include/forward helper,
9. EL context integration.

Contoh:

```jsp
<%
pageContext.setAttribute("message", "Hello", PageContext.REQUEST_SCOPE);
%>
```

Atau:

```jsp
${requestScope.message}
```

PageContext penting karena JSP punya beberapa scope:

```text
page scope
request scope
session scope
application scope
```

Saat EL mengevaluasi `${message}`, pencarian biasanya mengikuti scope order tertentu:

```text
page -> request -> session -> application
```

Implikasi:

1. Nama attribute bisa shadowing.
2. `${user}` bisa berasal dari scope yang tidak kamu kira.
3. Debugging view state perlu tahu asal scope.
4. Nama attribute harus disiplin.

---

## 17. `config` dan `page`

`config` adalah `ServletConfig` untuk generated JSP servlet.

Jarang digunakan langsung di JSP modern.

`page` adalah reference ke servlet instance saat ini, mirip `this`.

Penggunaan langsung `page` hampir selalu unnecessary dan biasanya menandakan JSP terlalu dekat dengan servlet internals.

---

## 18. `exception`: Khusus Error Page

`exception` hanya tersedia di JSP yang dideklarasikan sebagai error page:

```jsp
<%@ page isErrorPage="true" %>
```

Contoh:

```jsp
<%@ page isErrorPage="true" contentType="text/html;charset=UTF-8" %>
<h1>Something went wrong</h1>
<p><%= exception.getMessage() %></p>
```

Namun menampilkan `exception.getMessage()` ke user bisa membocorkan informasi internal.

Lebih aman:

```jsp
<%@ page isErrorPage="true" contentType="text/html;charset=UTF-8" %>
<h1>Something went wrong</h1>
<p>Please contact support with reference id: ${requestScope.correlationId}</p>
```

Exception detail harus masuk log, bukan HTML publik.

## 19. Scope Attribute dan Lookup

JSP dan EL sangat bergantung pada attribute.

Controller:

```java
request.setAttribute("users", users);
request.setAttribute("pageTitle", "User List");
```

JSP:

```jsp
<h1>${pageTitle}</h1>
<c:forEach var="user" items="${users}">
    ${user.name}
</c:forEach>
```

Scope utama:

| Scope | Lifetime | Cocok untuk | Risiko |
|---|---|---|---|
| page | selama render JSP | temporary rendering value | jarang terlihat dari luar |
| request | satu HTTP request | view model | hilang setelah forward selesai |
| session | banyak request per user | login state, user prefs | memory bloat, stale data |
| application | selama app hidup | global config ringan | shared mutable state |

Attribute lookup tanpa scope eksplisit bisa membingungkan:

```jsp
${user}
```

Lebih eksplisit:

```jsp
${requestScope.user}
${sessionScope.currentUser}
${applicationScope.appVersion}
```

Untuk sistem besar, gunakan naming convention:

```text
requestScope.model
requestScope.form
requestScope.errors
sessionScope.currentUser
applicationScope.buildInfo
```

## 20. Page Directive: Mengubah Kontrak Generated Servlet

Page directive memengaruhi translation dan runtime behavior JSP.

Syntax:

```jsp
<%@ page attribute="value" %>
```

Contoh:

```jsp
<%@ page contentType="text/html;charset=UTF-8" pageEncoding="UTF-8" session="false" %>
```

Directive bukan HTML. Directive adalah instruksi ke JSP container.

---

## 21. `contentType` dan `pageEncoding`

Dua attribute ini sering disamakan, padahal berbeda.

```jsp
<%@ page contentType="text/html;charset=UTF-8" pageEncoding="UTF-8" %>
```

### 21.1 `pageEncoding`

`pageEncoding` memberitahu container encoding source file JSP.

Jika file `.jsp` disimpan sebagai UTF-8 tetapi container membaca sebagai ISO-8859-1, karakter non-ASCII bisa rusak saat translation.

### 21.2 `contentType`

`contentType` mengatur MIME type dan charset response.

```http
Content-Type: text/html;charset=UTF-8
```

Rule praktis:

```jsp
<%@ page pageEncoding="UTF-8" contentType="text/html;charset=UTF-8" %>
```

Gunakan konsisten di seluruh aplikasi.

Failure mode:

1. karakter Indonesia/Chinese/Japanese rusak,
2. form submit encoding mismatch,
3. database benar tetapi tampilan rusak,
4. output JSON/XML dari JSP salah charset,
5. browser menebak encoding berbeda.

---

## 22. `session`

```jsp
<%@ page session="false" %>
```

Jika `true`, JSP dapat menggunakan implicit object `session` dan bisa membuat session.

Jika `false`, JSP tidak membuat session dan `session` tidak tersedia.

Decision table:

| Jenis Halaman | `session` |
|---|---|
| public landing page | `false` |
| login page | biasanya `false` atau hati-hati |
| authenticated dashboard | `true` boleh |
| static-ish error page | `false` jika bisa |
| fragment include yang tidak perlu user state | `false` jika berdiri sendiri |

Namun kalau JSP di-include dari halaman lain, semantics bisa dipengaruhi oleh parent/request context. Jangan hanya mengandalkan directive sebagai security boundary.

---

## 23. `buffer` dan `autoFlush`

```jsp
<%@ page buffer="16kb" autoFlush="true" %>
```

`buffer` menentukan ukuran buffer JSP.

`autoFlush` menentukan apakah buffer otomatis di-flush saat penuh.

Large JSP dengan banyak output bisa memenuhi buffer. Jika flush terjadi terlalu awal, response committed.

Risiko response committed:

```text
Cannot forward after response has been committed
Cannot call sendRedirect() after response has been committed
Header ignored because response is already committed
```

Rule:

1. Redirect/forward/security decision harus dilakukan sebelum rendering.
2. JSP jangan melakukan decision terlambat setelah menulis output.
3. Buffer bukan solusi arsitektur; buffer hanya memberi ruang teknis.

## 24. `errorPage` dan `isErrorPage`

Halaman normal bisa menunjuk error page:

```jsp
<%@ page errorPage="/WEB-INF/views/error.jsp" %>
```

Error page:

```jsp
<%@ page isErrorPage="true" %>
```

Namun dalam aplikasi modern, error mapping lebih sering dikelola melalui `web.xml`, framework MVC, atau global exception handler.

```xml
<error-page>

</error-page>

<error-page>
    <error-code>404</error-code>
    <location>/WEB-INF/views/errors/404.jsp</location>
</error-page>
```

JSP error page cocok untuk rendering error, bukan memutuskan error taxonomy.

## 25. `isELIgnored`

```jsp
<%@ page isELIgnored="false" %>
```

Ini sering muncul di migrasi aplikasi lama.

Failure mode:

```html
Hello ${user.name}
```

Tampil literal di browser, bukan nama user.

Dalam aplikasi modern, EL biasanya enabled.

## 26. `import`


Ini mengimpor class untuk scripting element.

Namun jika JSP modern menghindari scriptlet, kebutuhan `import` juga turun drastis.

Contoh smell:

```jsp
<%@ page import="java.sql.*" %>
```

Ini hampir pasti view sudah menjadi controller/service.

## 27. `trimDirectiveWhitespaces`

Contoh:

1. HTML output lebih bersih,
2. mengurangi whitespace tak perlu,
3. berguna untuk XML/CSV/text output.

## 28. Include Directive vs Include Action

Ini topik penting karena keduanya tampak mirip, tetapi berbeda secara lifecycle.

### 28.1 Static Include: Include Directive

```jsp
<%@ include file="/WEB-INF/views/common/header.jspf" %>
```

Static include terjadi saat translation phase.

Mental model:

```text
main.jsp + header.jspf
    -> digabung sebagai source JSP
    -> diterjemahkan menjadi satu servlet
```

Karakteristik:

1. Include dilakukan sebelum compilation.
2. Parent dan included fragment menjadi satu translation unit.
3. Variable/scriptlet bisa saling terlihat.
4. Perubahan included file bisa memerlukan retranslation parent.
5. Cocok untuk fragment statis atau directive-level include.

Risiko:

1. coupling kuat,
2. naming conflict,
3. sulit memahami asal variable,
4. compile error menyebar,
5. perubahan fragment memengaruhi banyak generated servlet.

### 28.2 Dynamic Include: `jsp:include`

```jsp
<jsp:include page="/WEB-INF/views/common/header.jsp" />
```

Dynamic include terjadi saat request phase.

Mental model:

```text
main.jsp rendered
    -> at runtime dispatch include to header.jsp
    -> header.jsp renders into same response
```

Karakteristik:

1. Include dieksekusi runtime per request.
2. Included page punya lifecycle sendiri.
3. Lebih loose coupling daripada static include.
4. Bisa mengirim parameter.
5. Cocok untuk fragment dinamis.

Contoh parameter:

```jsp
<jsp:include page="/WEB-INF/views/common/header.jsp">
    <jsp:param name="activeMenu" value="users" />
</jsp:include>
```

### 28.3 Decision Table

| Kebutuhan | Pilihan |
|---|---|
| Include fragment statis kecil | static include |
| Include layout fragment yang butuh runtime data | dynamic include |
| Butuh menghindari coupling compile-time | dynamic include |
| Butuh shared directive/import | static include, hati-hati |
| Butuh parameterisasi | dynamic include atau tag file |
| Butuh reusable component | tag file/custom tag lebih baik |

Advanced rule:

```text
Static include adalah source composition.
Dynamic include adalah runtime dispatch composition.
```

## 29. Forward vs Include

JSP/Servlet dispatch punya dua operasi penting:

1. forward,
2. include.

### 29.1 Forward

Forward menyerahkan control ke resource lain.

```jsp
<jsp:forward page="/WEB-INF/views/login.jsp" />
```

Setelah forward, resource target bertanggung jawab menghasilkan response.

Namun forward dari JSP sering smell. Biasanya controller yang harus memutuskan forward.

### 29.2 Include

Include menyisipkan output resource lain ke response saat ini.

```jsp
<jsp:include page="/WEB-INF/views/common/footer.jsp" />
```

Perbedaan mental:

```text
forward = replace current responsibility
include = compose output
```

## 30. JSP File Placement: Public vs `/WEB-INF`

Jika JSP berada di web root:

```text
/src/main/webapp/users.jsp
```

Maka user bisa mengakses langsung:

```text
/users.jsp
```

Jika JSP berada di `/WEB-INF`:

```text
/src/main/webapp/WEB-INF/views/users.jsp
```

Maka user tidak bisa mengakses langsung via URL. JSP hanya bisa dicapai melalui server-side forward/include.

Pola enterprise yang direkomendasikan:

```text
/src/main/webapp/WEB-INF/views/
    users/list.jsp
    users/detail.jsp
    errors/500.jsp
```

Controller:

```java
request.getRequestDispatcher("/WEB-INF/views/users/list.jsp")
       .forward(request, response);
```

Manfaat:

1. User tidak bypass controller.
2. Authorization/validation flow lebih terkontrol.
3. View tidak menjadi endpoint publik.
4. URL publik bisa bersih dan stabil.
5. Layout internal bisa berubah tanpa mengubah URL.

## 31. JSP Fragments: `.jspf`

Konvensi `.jspf` sering dipakai untuk JSP fragment.

Contoh:

```text
/WEB-INF/views/fragments/header.jspf
/WEB-INF/views/fragments/footer.jspf
```

Biasanya digunakan dengan static include:

```jsp
<%@ include file="/WEB-INF/views/fragments/header.jspf" %>
```

Namun `.jspf` bukan magic universal. Perlakuannya tergantung container dan mapping. Gunakan `/WEB-INF` agar tidak bisa diakses langsung.

## 32. Scriptlet: Kenapa Berbahaya Secara Arsitektur

Scriptlet adalah Java code dalam JSP:

```jsp
<%
    String name = request.getParameter("name");
    out.write(name);
%>
```

Masalah scriptlet bukan hanya “kuno”. Masalahnya struktural.

### 32.1 Scriptlet Mengaburkan Boundary

JSP seharusnya view.

Scriptlet membuat JSP bisa menjadi:

1. controller,
2. service,
3. validator,
4. repository,
5. security gate,
6. formatter,
7. transaction script.

Akibatnya:

```text
UI file becomes mixed execution surface
```

### 32.2 Scriptlet Sulit Dites

Java code di JSP:

1. tidak mudah unit test,
2. bergantung container,
3. error muncul saat JSP compiled,
4. branch logic tersembunyi di view,
5. tidak mudah dianalisis statis.

### 32.3 Scriptlet Mendorong Security Mistake

Contoh:

```jsp
<%= request.getParameter("name") %>
```

Jika `name` berisi:

```html
<script>alert(1)</script>
```

Maka raw output bisa menjadi XSS.

Lebih baik gunakan output escaping context-aware. Minimal untuk HTML text:

```jsp
<c:out value="${param.name}" />
```

Namun bahkan `c:out` tidak cukup untuk semua context seperti JavaScript string, CSS, atau URL attribute. Security JSP akan dibahas khusus di part 11.

### 32.4 Scriptlet Meningkatkan Coupling ke Java Version

Scriptlet dikompilasi sebagai bagian dari generated servlet. Saat migrasi Java/container, error scriptlet bisa muncul di runtime translation.

Legacy JSP dengan Java code lama bisa gagal saat:

1. pindah dari Java 8 ke 17/21/25,
2. pindah `javax.*` ke `jakarta.*`,
3. dependency package berubah,
4. access reflection/module berubah,
5. container JSP compiler lebih strict.

---

## 33. Declaration: Lebih Berbahaya dari Scriptlet Biasa

Declaration:

```jsp
<%!
    private List<String> cache = new ArrayList<>();
%>
```

Menjadi field/method di generated servlet.

Bahaya:

1. shared antar request,
2. shared antar user,
3. tidak thread-safe,
4. lifecycle tidak jelas bagi pembaca,
5. bisa menjadi memory leak,
6. sulit diobservasi.

Declaration yang masih relatif aman:

```jsp
<%!
    private static final String VERSION = "1.0";
%>
```

Tetapi tetap lebih baik pindahkan constant ke Java class/config jika dipakai luas.

Rule:

```text
Declaration di JSP hanya boleh dibaca sebagai legacy construct, bukan pattern baru.
```

## 34. Expression `<%= ... %>`

Expression:

```jsp
<p><%= user.getName() %></p>
```

Secara generated servlet:

```java
out.print(user.getName());
```

Problem utama:

1. raw output,
2. escaping tidak otomatis context-aware,
3. Java object harus tersedia di scripting context,
4. coupling kuat ke Java API,
5. null handling manual.

Lebih baik:

```jsp
<c:out value="${user.name}" />
```

Atau jika framework/tag library punya escaping yang sesuai, gunakan itu.

## 35. JSP Standard Actions

JSP punya standard actions dengan XML-like syntax.

Contoh umum:

```jsp
<jsp:include page="header.jsp" />
<jsp:forward page="login.jsp" />
<jsp:param name="x" value="y" />
<jsp:useBean id="user" class="com.example.User" scope="request" />
<jsp:setProperty name="user" property="name" />
<jsp:getProperty name="user" property="name" />
```

Pada sistem modern, yang paling sering masih relevan:

1. `jsp:include`,
2. `jsp:param`,
3. kadang `jsp:forward`.

Yang umumnya legacy:

1. `jsp:useBean`,
2. `jsp:setProperty`,
3. `jsp:getProperty`.

Kenapa `jsp:useBean` legacy?

Karena ia mendorong JSP membuat/mengisi object model sendiri.

Contoh:

```jsp
<jsp:useBean id="user" class="com.example.User" scope="request" />
<jsp:setProperty name="user" property="*" />
```

Ini bisa mengikat request parameter langsung ke bean. Risiko:

1. mass assignment,
2. validation bypass,
3. tipe/coercion error,
4. view menjadi input binding layer,
5. security boundary lemah.

Dalam arsitektur modern:

```text
Controller binds input -> validates -> creates view model -> forwards to JSP
```

## 36. Taglib Directive

Tag library dideklarasikan dengan directive:

```jsp
<%@ taglib prefix="c" uri="jakarta.tags.core" %>
```

Atau pada legacy Java EE:

```jsp
<%@ taglib prefix="c" uri="http://java.sun.com/jsp/jstl/core" %>
```

Taglib directive memetakan prefix ke tag library descriptor.

Mental model:

```text
prefix c
    -> tag library URI
    -> TLD metadata
    -> tag handler classes / tag files
    -> generated servlet invokes tag logic
```

Taglib detail akan dibahas di part 7–9. Di part ini cukup pahami bahwa tag bukan HTML biasa. Tag bisa menjadi Java handler invocation di generated servlet.

## 37. EL dalam JSP Generated Servlet

EL expression seperti:

```jsp
${user.department.name}
```

Tidak menjadi Java source sederhana seperti:

```java
user.getDepartment().getName()
```

Ia dievaluasi oleh EL engine dengan resolver chain.

Konseptual:

```text
Expression: user.department.name
    -> resolve 'user' from page/request/session/application/CDI/etc.
    -> resolve 'department' property
    -> resolve 'name' property
    -> coerce to String/output type
    -> write output
```

Implikasi:

1. EL lebih fleksibel daripada direct Java call.
2. EL error bisa terjadi saat render.
3. Property resolution bisa melibatkan reflection/introspection.
4. Scope shadowing bisa menyebabkan data salah.
5. Null/coercion behavior harus dipahami.

EL detail akan dibahas di part 5–6.

## 38. JSP Buffering dan Response Commit Deep Dive

Ketika JSP menulis output, output tidak selalu langsung dikirim ke client.

Layering konseptual:

```text
JSP generated servlet
    -> JspWriter buffer
    -> ServletResponse buffer
    -> HTTP connector
    -> TCP socket
    -> client browser
```

### 38.1 Kenapa Buffer Penting?

Buffer memungkinkan:

1. header masih bisa diubah sebelum body dikirim,
2. redirect masih bisa dilakukan sebelum commit,
3. forward masih bisa dilakukan sebelum commit,
4. error page masih bisa menggantikan response,
5. output bisa lebih efisien.

### 38.2 Kapan Response Committed?

Response bisa committed ketika:

1. buffer penuh dan flush,
2. `out.flush()` dipanggil,
3. `response.flushBuffer()` dipanggil,
4. response selesai normal,
5. container memutuskan mengirim header/body.

### 38.3 Contoh Failure

```jsp
<%@ page buffer="1kb" %>
<%
for (int i = 0; i < 10000; i++) {
    out.write("data");
}
response.sendRedirect("/other");
%>
```

Kemungkinan error:

```text
IllegalStateException: Cannot call sendRedirect() after the response has been committed
```

Root cause bukan redirect-nya, tetapi desain flow-nya: redirect diputuskan setelah rendering dimulai.

Rule:

```text
Control-flow decision before rendering.
Rendering after decision.
```

## 39. Error Handling dalam JSP

Error JSP bisa muncul di beberapa fase.

| Fase | Contoh Error | Waktu Muncul |
|---|---|---|
| Translation | tag tidak valid, directive salah | first access/startup |
| Compilation | Java scriptlet error, missing import | first access/startup |
| Class loading | dependency conflict | first access/startup |
| Initialization | error di `jspInit` | servlet init |
| Request render | NullPointer, EL error, tag error | setiap request |
| Flush/commit | redirect/forward after commit | saat output |

Ini penting untuk troubleshooting.

Jika error hanya muncul di request tertentu, kemungkinan:

1. data-specific EL issue,
2. null nested property,
3. tag handler condition,
4. invalid view model,
5. session state corrupt,
6. branch tertentu di scriptlet.

Jika error muncul saat first access semua user, kemungkinan:

1. translation/compilation failure,
2. missing class,
3. invalid taglib,
4. incompatible dependency,
5. namespace mismatch.

Di environment development atau staging, kamu bisa melihat generated JSP servlet source.

```text
$CATALINA_BASE/work/Catalina/localhost/<app>/org/apache/jsp/
```

```text
WEB_002dINF/views/users/list_jsp.class
```

Gunanya:

2. Melihat bagaimana include/tag/EL diterjemahkan.

6. Mengidentifikasi tag pooling/helper logic.

Namun jangan bergantung pada generated source sebagai API stabil. Itu implementation detail container.

## 41. JSP Precompilation

JSP bisa dikompilasi sebelum deployment/runtime request.

Manfaat:

1. Error JSP ditemukan saat build/deploy, bukan saat user pertama akses.
2. Cold-start request lebih stabil.
3. Production tidak melakukan compile JSP saat traffic pertama.
4. Deployment lebih predictable.

Trade-off:

1. Build lebih lama.
2. Pipeline lebih kompleks.
3. Container-specific tooling.
4. Perlu memastikan generated artifacts cocok dengan target runtime.

Untuk sistem enterprise, precompilation layak dipertimbangkan terutama jika:

1. JSP banyak,
2. traffic first-hit penting,
3. environment immutable,
4. production tidak boleh compile source dynamic,
5. compliance mengharuskan artifact deterministic.

## 42. JSP Reload dan Development Mode

Di development, container sering mengecek timestamp JSP untuk retranslation.

Flow:

```text
Request to JSP
    -> check whether JSP source changed
    -> if changed, retranslate/recompile
    -> load new generated servlet
    -> serve request
```

Di production, behavior ini bisa mahal jika terlalu sering check.

Risiko development reload:

1. generated class stale,
2. race saat recompile,
3. inconsistent state antar request,
4. old classloader belum GC,
5. memory leak jika redeploy berkali-kali.

Production recommendation:

1. gunakan deployment immutable,
2. hindari edit JSP langsung di server,
3. pertimbangkan precompile,
4. disable aggressive reload check bila sesuai container,
5. monitor classloader/metaspace.

## 43. JSP dan Classloader

JSP generated class hidup di web application classloader.

Saat redeploy:

```text
old webapp classloader should be discarded
new webapp classloader created
new JSP classes loaded
```

Classloader leak bisa terjadi jika old classloader masih direferensikan oleh:

1. static field,
2. background thread,
3. ThreadLocal,
4. JDBC driver registration,
5. logging framework,
6. custom tag static cache,
7. singleton yang tidak dibersihkan.

JSP bisa berkontribusi pada leak jika:

1. declaration membuat static mutable cache,
2. custom tag menyimpan classloader-bound objects,
3. scriptlet membuat thread/resource,
4. page menyimpan object besar di application scope.

Rule:

```text
JSP must not own long-lived infrastructure resources.
```

Resource harus dikelola di layer aplikasi/container yang jelas lifecycle-nya.

## 44. JSP dan `javax.*` vs `jakarta.*`

Legacy JSP/Servlet API menggunakan `javax.servlet.*` dan `javax.servlet.jsp.*`.

Modern Jakarta menggunakan `jakarta.servlet.*` dan `jakarta.servlet.jsp.*`.

Contoh legacy:

```jsp
<%@ page import="javax.servlet.http.HttpServletRequest" %>
```

Modern:

```jsp
<%@ page import="jakarta.servlet.http.HttpServletRequest" %>
```

Namun pada JSP modern yang menghindari scriptlet/import, perubahan ini sering lebih sedikit terlihat di file `.jsp`.

Area yang sering terdampak:

1. custom tag handler class,
2. servlet/filter/listener imports,
3. TLD class references,
4. custom EL resolver,
5. old JSTL taglib URI/dependency,
6. JSF/Faces namespaces,
7. web.xml schema,
8. third-party library binary compatibility.

Part 28 akan membahas migration playbook secara khusus. Untuk part ini, cukup pahami bahwa JSP generated servlet mengikuti API container. Container `jakarta.*` tidak bisa menjalankan library `javax.*` secara langsung jika binary API tidak cocok.

## 45. JSP sebagai View dalam MVC

JSP paling sehat saat menjadi view.

Pola MVC sederhana:

```text
Browser
  -> /users
  -> UserController
  -> UserService
  -> UserRepository
  -> UserListViewModel
  -> request.setAttribute("model", vm)
  -> forward /WEB-INF/views/users/list.jsp
  -> render HTML
```

Controller:

```java
@WebServlet("/users")
public class UserListServlet extends HttpServlet {
    private UserService userService;

    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        UserListViewModel model = userService.listUsers();
        request.setAttribute("model", model);

        request.getRequestDispatcher("/WEB-INF/views/users/list.jsp")
               .forward(request, response);
    }
}
```

JSP:

```jsp
<%@ page contentType="text/html;charset=UTF-8" pageEncoding="UTF-8" %>
<%@ taglib prefix="c" uri="jakarta.tags.core" %>

<!doctype html>
<html>
<head>
    <title>${model.title}</title>
</head>
<body>
<h1>${model.title}</h1>

<table>
    <c:forEach var="user" items="${model.users}">
        <tr>
            <td><c:out value="${user.name}" /></td>
            <td><c:out value="${user.email}" /></td>
        </tr>
    </c:forEach>
</table>
</body>
</html>
```

Boundary-nya jelas:

| Layer | Tanggung Jawab |
|---|---|
| Controller | HTTP input, auth check, flow decision |
| Service | business use case |
| Repository | data access |
| View model | data siap render |
| JSP | render HTML |

## 46. View Model: Kunci JSP yang Maintainable

JSP sering kacau bukan karena JSP-nya, tetapi karena data yang dikirim ke JSP tidak didesain.

Kurang baik:

```java
request.setAttribute("users", entities);
request.setAttribute("departments", departments);
request.setAttribute("status", status);
request.setAttribute("canApprove", canApprove);
request.setAttribute("error", error);
request.setAttribute("page", page);
request.setAttribute("sort", sort);
```

JSP jadi tahu terlalu banyak.

Lebih baik:

```java
request.setAttribute("model", new UserListPageModel(
    title,
    rows,
    filters,
    pagination,
    actions,
    messages
));
```

Contoh model:

```java
public record UserListPageModel(
    String title,
    List<UserRow> rows,
    FilterState filters,
    PaginationState pagination,
    List<PageAction> actions,
    List<AlertMessage> messages
) {}
```

JSP:

```jsp
<h1>${model.title}</h1>
```

Manfaat:

1. Contract view jelas.
2. JSP lebih sederhana.
3. Testing controller lebih mudah.
4. Rendering logic berkurang.
5. Migration ke templating lain lebih mudah.
6. Security review lebih mudah.

## 47. JSP Internal Architecture dan Security Boundary

Karena JSP menjadi servlet, JSP punya akses kuat ke request/response/session/application. Ini membuatnya powerful sekaligus riskan.

Security boundary yang benar:

```text
Authentication  -> before controller/view
Authorization   -> before action/data exposure
Validation      -> before model mutation
Encoding        -> at output boundary
Audit logging   -> around use case/action
JSP             -> render authorized, prepared model
```

Jangan jadikan JSP sebagai satu-satunya authorization layer.

Contoh buruk:

```jsp
<c:if test="${sessionScope.currentUser.admin}">
    <a href="/admin/delete?id=${user.id}">Delete</a>
</c:if>
```

Ini hanya menyembunyikan link. Endpoint `/admin/delete` tetap harus melakukan authorization.

Rendering condition boleh untuk UX, bukan security enforcement final.

## 48. Failure Mode Umum JSP

### 48.1 JSP Tidak Bisa Compile

Gejala:

```text
JasperException: Unable to compile class for JSP
```

Kemungkinan:

1. scriptlet Java error,
2. import salah,
3. class tidak ada,
4. API `javax/jakarta` mismatch,
5. tag handler dependency missing.

Mitigasi:

1. hindari scriptlet,
2. precompile JSP,
3. pastikan dependency sesuai container,
4. tes semua JSP path.

### 48.2 Taglib Tidak Ditemukan

Gejala:

```text
The absolute uri cannot be resolved in either web.xml or the jar files deployed with this application
```

Kemungkinan:

1. dependency JSTL/Jakarta Tags belum ada,
2. URI salah,
3. versi taglib tidak cocok,
4. JAR tidak masuk runtime classpath,
5. legacy URI dipakai di stack Jakarta baru.

### 48.3 EL Menampilkan Literal

Gejala:

```html
Hello ${user.name}
```

Kemungkinan:

1. EL ignored,
2. file tidak diproses sebagai JSP,
3. extension/mapping salah,
4. container lama/config lama.

### 48.4 `Cannot Forward After Response Has Been Committed`

Kemungkinan:

1. output sudah flush,
2. buffer terlalu kecil,
3. redirect/forward dilakukan dari JSP terlambat,
4. exception terjadi setelah commit.

Mitigasi:

1. pindahkan flow decision ke controller/filter,
2. jangan menulis output sebelum authorization/redirect,
3. gunakan error handling global.

### 48.5 Accidental Session Creation

Gejala:

1. banyak session anonymous,
2. memory naik,
3. sticky session load balancer diperlukan,
4. cache public tidak efektif.

Mitigasi:

```jsp
<%@ page session="false" %>
```

Dan pastikan JSP/tag tidak mengakses session diam-diam.

### 48.6 XSS dari Raw Expression

Gejala:

```jsp
<%= request.getParameter("q") %>
```

Mitigasi:

1. gunakan output escaping,
2. pahami context HTML/JS/URL/CSS,
3. jangan render raw user input,
4. security test rendering.

### 48.7 Session Bloat

Gejala:

1. heap tinggi,
2. GC pressure,
3. slow serialization,
4. cluster replication lambat.

Penyebab:

1. JSP menyimpan object besar ke session,
2. view model masuk session,
3. entity graph masuk session,
4. upload/document temporary masuk session.

Mitigasi:

1. request scope untuk render data,
2. session hanya minimal identity/preference,
3. external cache/storage untuk data besar,
4. TTL dan cleanup.

## 49. Membaca Legacy JSP Secara Sistematis

Saat menemukan JSP legacy besar, jangan langsung refactor membabi buta. Baca sebagai generated servlet mental model.

Checklist:

### 49.1 Identifikasi Role JSP

Tanya:

1. Apakah JSP ini endpoint langsung?
2. Apakah selalu lewat controller?
3. Apakah ada scriptlet membaca request parameter?
4. Apakah ada database/service call?
5. Apakah ada authorization check?
6. Apakah ada mutation state?

Klasifikasi:

```text
Pure view
View + conditional logic
View + controller logic
View + business logic
View + data access
```

Semakin ke bawah, semakin tinggi risiko.

### 49.2 Identifikasi Scope

Cari:

```jsp
request.setAttribute
session.setAttribute
application.setAttribute
pageContext.setAttribute
```

Cari juga EL:

```jsp
${requestScope.x}
${sessionScope.x}
${applicationScope.x}
${x}
```

Mapping:

```text
Attribute name -> writer -> reader -> lifetime -> owner
```

### 49.3 Identifikasi Output Risk

Cari:

```jsp
<%= ... %>
out.write(...)
${...}
escapeXml="false"
```

Tentukan output context:

1. HTML text,
2. HTML attribute,
3. JavaScript string,
4. URL,
5. CSS,
6. raw HTML.

### 49.4 Identifikasi Include Graph

Cari:

```jsp
<%@ include file="..." %>
<jsp:include page="..." />
```

Buat graph:

```text
main.jsp
  -> header.jspf
  -> menu.jsp
  -> messages.jspf
  -> footer.jspf
```

Masalah sering berasal dari fragment, bukan file utama.

### 49.5 Identifikasi Taglib

Cari:

```jsp
<%@ taglib prefix="..." uri="..." %>
```

Mapping:

```text
prefix -> URI -> dependency JAR -> TLD -> handler class
```

Ini penting untuk migrasi `javax` ke `jakarta`.

## 50. Design Heuristics untuk JSP yang Sehat

Gunakan aturan berikut untuk sistem enterprise.

### 50.1 JSP Harus Render, Bukan Decide

Buruk:

```jsp
<%
if (request.getParameter("approve") != null) {
    caseService.approve(id);
}
%>
```

Baik:

```text
POST /case/{id}/approve
    -> controller validates CSRF/auth/input
    -> service approves
    -> redirect detail page
    -> JSP renders result
```

### 50.2 JSP Menerima View Model yang Siap Render

Buruk:

```jsp
${case.assignee.department.manager.displayName}
```

Lebih baik:

```jsp
${model.assigneeManagerName}
```

Alasan:

1. menghindari lazy loading surprise,
2. menghindari N+1 dari view,
3. mengurangi coupling ke domain graph,
4. membuat view contract eksplisit.

### 50.3 JSP Tidak Mengakses Database

Buruk:

```jsp
<%@ page import="java.sql.*" %>
<%
Connection c = DriverManager.getConnection(...);
%>
```

Ini bukan sekadar style buruk. Ini merusak:

1. transaction boundary,
2. connection management,
3. testability,
4. security,
5. performance,
6. observability.

### 50.4 JSP Tidak Menyimpan Object Besar di Session

Buruk:

```jsp
<%
session.setAttribute("searchResults", hugeList);
%>
```

Lebih baik:

1. request scope untuk hasil pencarian,
2. pagination server-side,
3. cache dengan key dan TTL jika perlu,
4. session hanya menyimpan filter ringan.

### 50.5 JSP Tidak Menjadi Security Boundary Tunggal

Buruk:

```jsp
<c:if test="${canDelete}">
  <a href="delete?id=${id}">Delete</a>
</c:if>
```

Baik:

1. JSP menyembunyikan tombol untuk UX.
2. Endpoint delete tetap check authorization.
3. Audit log action di service/controller.
4. CSRF untuk state-changing action.

## 51. JSP dalam Java 8 sampai Java 25

Dari sisi syntax JSP, banyak konsep tetap sama. Yang berubah adalah lingkungan runtime dan dependency.

### 51.1 Java 8 Era

Umumnya:

1. Java EE 7/8,
2. `javax.servlet.*`,
3. JSP 2.x,
4. JSTL 1.2,
5. JSF 2.x,
6. WAR deployment klasik,
7. app server atau servlet container.

Banyak legacy JSP ada di era ini.

### 51.2 Java 11/17 Era

Perubahan penting:

1. Java EE pindah ke Jakarta EE governance,
2. Java 11 menghapus beberapa Java EE modules dari JDK,
3. deployment mulai lebih containerized,
4. Java 17 menjadi baseline banyak framework modern,
5. reflection/classpath lebih perlu disiplin.

### 51.3 Java 21/25 Era

Fokus bukan JSP syntax baru, tetapi runtime architecture:

1. LTS modern,
2. virtual threads di Java 21+ relevan untuk server request handling jika container/framework mendukung,
3. Jakarta EE 11 baseline Java SE 17+,
4. Jakarta namespace matang,
5. legacy JSP perlu migration strategy,
6. operational expectations lebih tinggi: observability, security headers, immutable deployment, zero-downtime rollout.

JSP sendiri tetap server-side rendering model. Jangan mengharapkan JSP berubah menjadi SPA/reactive UI framework hanya karena Java-nya modern.

## 52. JSP dan Virtual Threads: Jangan Salah Mental Model

Java 21 membawa virtual threads sebagai fitur penting. Namun untuk JSP:

```text
Virtual threads do not change JSP semantics.
```

Yang tetap:

1. JSP tetap translated ke servlet.
2. Request tetap masuk container.
3. `_jspService` tetap menghasilkan response.
4. Blocking IO/DB call di view tetap buruk.
5. Session shared mutable state tetap riskan.
6. Scriptlet tetap buruk.

Virtual threads bisa membantu server menangani blocking request lebih efisien jika container/runtime mendukung. Tetapi itu tidak membuat arsitektur JSP yang salah menjadi benar.

Rule:

```text
Concurrency improvement cannot fix layer violation.
```

## 53. JSP vs Template Engine Lain

Untuk memahami posisi JSP, bandingkan dengan template engine lain.

| Teknologi | Model |
|---|---|
| JSP/Jakarta Pages | template compiled to servlet |
| Thymeleaf | template engine integrated with framework, often Spring MVC |
| FreeMarker | text template engine |
| Mustache/Handlebars | logic-light template |
| JSF/Faces Facelets | component tree declaration |
| React/Vue SPA | client-side component rendering |

JSP kuat jika:

1. aplikasi Java EE/Jakarta EE klasik,
2. server-side rendering cukup,
3. butuh integrasi servlet/container,
4. tim maintain legacy WAR,
5. halaman admin/internal/regulatory relatif form/table-heavy.

JSP kurang cocok jika:

1. UI sangat interaktif client-side,
2. state UI kompleks di browser,
3. real-time front-end heavy,
4. tim ingin component model modern client-side,
5. API-first architecture penuh.

---

## 54. Kapan JSP Masih Masuk Akal?

JSP masih masuk akal untuk:

1. aplikasi legacy yang masih aktif,
2. internal admin console,
3. server-rendered CRUD screens,
4. government/regulatory back office,
5. form-heavy workflow,
6. low-JavaScript controlled environment,
7. aplikasi yang ingin WAR sederhana,
8. migrasi bertahap dari Java EE ke Jakarta EE.

Namun syaratnya:

1. tanpa scriptlet baru,
2. pakai JSTL/custom tags dengan disiplin,
3. view model jelas,
4. JSP di `/WEB-INF`,
5. controller/service boundary jelas,
6. escaping/security diperhatikan,
7. session minimal,
8. precompile/test untuk production.

---

## 55. Kapan JSP Tidak Masuk Akal?

JSP bukan pilihan ideal untuk:

1. aplikasi public modern dengan UX sangat dinamis,
2. micro-frontend architecture,
3. offline-first app,
4. heavy client-side state,
5. real-time collaboration UI,
6. design system kompleks berbasis client components,
7. team yang tidak punya Jakarta/Servlet expertise,
8. sistem baru yang sudah API-first dan SPA-native.

Tetapi untuk top engineer, poinnya bukan “JSP kuno”. Poinnya adalah mampu menilai fit berdasarkan constraints.

Pertanyaan arsitektural yang lebih baik:

```text
Apakah server-side rendering dengan servlet-based template memenuhi kebutuhan produk, operasi, security, dan maintainability?
```

Bukan:

```text
Apakah JSP modern atau tidak?
```

## 56. Mini Case Study: Regulatory Case List Page

Bayangkan halaman daftar case enforcement.

Kebutuhan:

1. User login.
2. Hanya role tertentu boleh lihat case tertentu.
3. Filter by status, assignee, due date.
4. Pagination.
5. Ada indikator SLA overdue.
6. Ada action button tergantung permission.
7. Semua action harus audit-able.

### 56.1 Desain Buruk

```text
/case-list.jsp accessed directly
    -> JSP reads request params
    -> JSP queries DB
    -> JSP checks role from session
    -> JSP renders action buttons
    -> JSP performs approve if param action=approve
```

Masalah:

1. Authorization tersebar.
2. Query di view.
3. Action via GET mungkin terjadi.
4. CSRF risk.
5. Audit tidak jelas.
6. Test sulit.
7. Pagination/filter logic tersembunyi.
8. Session coupling kuat.

### 56.2 Desain Lebih Sehat

```text
GET /cases
    -> CaseListController
    -> parse filter
    -> check authorization
    -> CaseQueryService.search(...)
    -> build CaseListPageModel
    -> forward /WEB-INF/views/cases/list.jsp
    -> JSP render only

POST /cases/{id}/approve
    -> CSRF check
    -> authorization check
    -> CaseWorkflowService.approve(...)
    -> audit log
    -> redirect /cases/{id}
```

JSP hanya menerima:

```java
public record CaseListPageModel(
    List<CaseRow> rows,
    FilterView filter,
    PaginationView pagination,
    List<AlertMessage> messages
) {}

public record CaseRow(
    String caseNo,
    String subject,
    String statusLabel,
    String assigneeName,
    boolean overdue,
    List<RowAction> actions
) {}
```

JSP render:

```jsp
<c:forEach var="row" items="${model.rows}">
    <tr class="${row.overdue ? 'overdue' : ''}">
        <td><c:out value="${row.caseNo}" /></td>
        <td><c:out value="${row.subject}" /></td>
        <td><c:out value="${row.statusLabel}" /></td>
        <td><c:out value="${row.assigneeName}" /></td>
        <td>
            <c:forEach var="action" items="${row.actions}">
                <a href="${action.href}"><c:out value="${action.label}" /></a>
            </c:forEach>
        </td>
    </tr>
</c:forEach>
```

Tetap perlu hati-hati: `action.href` harus dibangun aman oleh server, bukan raw user input.

## 57. Operational Checklist untuk JSP Production

Sebelum production, cek:

### 57.1 Build/Deployment

- [ ] JSP bisa diprecompile atau minimal semua path diuji.
- [ ] Tidak ada dependency `javax` di runtime Jakarta modern kecuali memang stack legacy.
- [ ] Taglib dependency lengkap.
- [ ] JSP tidak diedit langsung di server production.
- [ ] Generated JSP tidak disimpan sebagai source utama.

### 57.2 Security

- [ ] JSP berada di `/WEB-INF` jika harus lewat controller.
- [ ] Tidak ada raw user input via `<%= ... %>`.
- [ ] Output escaping sesuai context.
- [ ] Form state-changing memakai CSRF token.
- [ ] Authorization tidak hanya dilakukan dengan hide/show button.
- [ ] Error page tidak leak stack trace.
- [ ] Sensitive page punya cache-control sesuai.

### 57.3 Performance

- [ ] Tidak ada database call di JSP.
- [ ] Tidak ada nested loop besar tanpa pagination.
- [ ] Tidak ada large object graph di session.
- [ ] Response size wajar.
- [ ] Compression diaktifkan pada layer yang tepat.
- [ ] First-hit JSP compilation dipertimbangkan.

### 57.4 Maintainability

- [ ] Scriptlet tidak dipakai untuk code baru.
- [ ] View model jelas.
- [ ] Include graph bisa dipahami.
- [ ] Taglib/custom tag terdokumentasi.
- [ ] Naming attribute konsisten.
- [ ] JSP tidak memegang business workflow.

### 57.5 Observability

- [ ] Error JSP masuk log dengan correlation id.
- [ ] Slow request bisa dipetakan ke view/controller.
- [ ] 4xx/5xx error page termonitor.
- [ ] Session count dan size termonitor.
- [ ] Redeploy/classloader leak dimonitor bila app server long-running.

## 58. Anti-Pattern dan Refactoring Direction

| Anti-pattern | Gejala | Refactoring |
|---|---|---|
| Scriptlet controller | JSP baca parameter dan decide flow | pindah ke servlet/controller |
| JSP database access | import `java.sql.*` | service/repository layer |
| Session as data store | object besar di session | request scope/cache/pagination |
| Raw expression output | `<%= userInput %>` | escaping tag/function |
| Direct JSP access | `/page.jsp` public | move to `/WEB-INF`, forward only |
| Static include spaghetti | banyak include saling bergantung | tag files/layout abstraction |
| Scope shadowing | `${user}` ambiguous | explicit `requestScope.model` |
| Authorization by hiding link | hanya `c:if` di view | enforce in endpoint/service |
| Post action in JSP | mutation saat render | POST controller + PRG |
| Large nested graph access | `${a.b.c.d}` | flattened view model |

## 59. Top 1% Engineer Mental Model

Engineer biasa melihat JSP sebagai:

```text
HTML + Java/EL
```

Engineer yang lebih kuat melihat JSP sebagai:

```text
A servlet-generation mechanism with template syntax, scope lookup, tag execution, buffering, and container lifecycle constraints.
```

Engineer top-tier akan bertanya:

1. Apa boundary antara HTTP flow, business use case, dan rendering?
2. Data mana yang masuk ke request/session/application scope?
3. Apakah JSP ini bisa diakses langsung atau harus lewat controller?
4. Apakah output sudah encoded sesuai context?
5. Apakah rendering bisa menyebabkan DB lazy load/N+1?
6. Apakah JSP membuat session tanpa sengaja?
7. Apakah include/tag graph manageable?
8. Apakah error terjadi di translation, compilation, atau request phase?
9. Apakah generated servlet punya mutable shared state?
10. Apakah migration `javax` ke `jakarta` akan menyentuh JSP, taglib, atau custom handler?
11. Apakah response commit bisa membuat error handling gagal?
12. Apakah desain ini masih tepat, atau harus dipindah ke Faces/SPA/template lain?

## 60. Ringkasan

Jakarta Pages/JSP adalah template engine yang diterjemahkan menjadi servlet. Pemahaman ini mengubah cara kita membaca JSP:

1. JSP bukan file HTML biasa.
2. JSP melewati translation dan request phase.
3. Static text, EL, tag, dan scriptlet semuanya berakhir sebagai generated servlet behavior.
4. Implicit objects adalah shortcut ke Servlet/JSP runtime context.
5. Page directive mengubah kontrak generated servlet.
6. Include directive dan `jsp:include` berbeda fase: source composition vs runtime dispatch.
7. Buffering menentukan kapan response committed.
8. Scriptlet dan declaration berbahaya karena mencampur view dengan Java execution dan shared servlet state.
9. JSP sehat jika digunakan sebagai view yang menerima prepared view model.
10. JSP berbahaya jika menjadi controller, service, repository, dan security layer sekaligus.

Mental model akhir:

```text
Request enters container
  -> controller prepares model and decides flow
  -> JSP generated servlet renders model
  -> tags/EL evaluate against scoped context
  -> JspWriter buffers output
  -> response commits to client
```

Jika kamu memahami pipeline ini, kamu tidak hanya bisa menulis JSP. Kamu bisa membaca legacy JSP, mendiagnosis production issue, merancang migration strategy, menghindari security bug, dan memutuskan apakah JSP masih tepat untuk sebuah sistem.
