## 0. Posisi Materi Ini Dalam Seri

Sampai bagian sebelumnya, kita sudah membahas:

1. bagaimana Jakarta Faces bekerja sebagai component-based MVC framework;
2. bagaimana Facelets membangun component tree;
3. bagaimana backing bean dan CDI scope menentukan state boundary;
4. bagaimana lifecycle memproses request;
5. bagaimana komponen input/output/command/data bekerja;
6. bagaimana conversion, validation, navigation, state, Ajax, composite component, dan custom extension bekerja.

Bagian ini masuk ke sisi yang sering diremehkan: **security di layer UI server-side**.

Di aplikasi enterprise, security bukan hanya login, role, JWT, session cookie, atau filter authorization. Layer view juga bisa menjadi titik bocor melalui:

- XSS dari output yang salah context;
- raw HTML yang dianggap aman padahal berasal dari user;
- `rendered="false"` yang dikira authorization;
- hidden field yang dipercaya sebagai fakta bisnis;
- view state yang terlalu besar, bocor, atau mudah rusak;
- Ajax partial response yang membawa exception/debug detail;
- stale page yang tetap menampilkan action lama;
- role-based menu yang berbeda dari backend enforcement;
- file upload/download yang tampak hanya UI problem tapi sebenarnya security problem.

Mental model utama bagian ini:

> Jakarta Faces membantu membangun UI stateful dengan banyak mekanisme bawaan, tetapi Faces tidak menggantikan security architecture. Faces hanya menyediakan rendering, lifecycle, component state, validation, dan sebagian protection surface. Authorization, data ownership, trust boundary, output context, dan business invariant tetap harus didesain eksplisit.

## 1. Security Mental Model Untuk Jakarta Faces

### 1.1 Faces bukan sekadar HTML generator

Pada JSP biasa, view biasanya seperti ini:

```text
request model -> JSP template -> HTML response
```

Pada Faces, alurnya lebih kaya:

```text
HTTP request
  -> FacesServlet
  -> restore/build component tree
  -> decode submitted values
  -> convert
  -> validate
  -> update backing model
  -> invoke action/listener
  -> render component tree
  -> HTML + javax/jakarta.faces.ViewState
```

Security berarti semua titik berikut perlu dipahami:

```text
1. data masuk dari browser
2. request parameter didecode oleh component
3. value dikonversi
4. value divalidasi
5. model backing bean diupdate
6. action dipanggil
7. service/domain dipanggil
8. data dirender kembali
9. component tree state disimpan untuk request berikutnya
```

Bug security bisa terjadi di setiap titik.

---

### 1.2 Trust boundary Faces

Boundary trust utama:

| Boundary | Contoh | Harus Dipercaya? |
|---|---|---|
| Browser request parameter | input text, hidden input, select value | Tidak |
| Ajax request parameter | execute/render target, source component | Tidak sepenuhnya |
| View state token | `jakarta.faces.ViewState` / `javax.faces.ViewState` | Harus divalidasi oleh framework, jangan dipakai sebagai business fact |
| Rendered menu/action | tombol approve muncul/tidak muncul | Tidak cukup untuk authorization |
| Backing bean field | value hasil postback | Tergantung sumber; tetap validasi domain |
| Session bean | current user, wizard state | Bisa stale/corrupt secara logic |
| Service/domain layer | invariant bisnis | Harus menjadi enforcement final |

Prinsipnya:

> Faces component tree adalah mekanisme UI. Domain rule dan authorization rule tidak boleh bergantung pada tampilan.

---

### 1.3 Secure Faces architecture ringkas

Desain aman biasanya seperti ini:

```text
Facelets XHTML
  - render escaped output by default
  - show/hide action based on permission for UX only
  - include CSRF/view state generated by Faces
  - avoid trusting hidden fields

Backing Bean
  - holds view/form model
  - performs UI-level guard
  - maps user action into command
  - does not become final authorization authority

Application Service
  - re-loads authoritative entity
  - checks ownership/permission/action eligibility
  - validates current workflow state
  - applies transaction
  - writes audit trail

Domain/Policy Layer
  - owns invariant
  - owns transition legality
  - owns escalation and lifecycle rule
```

Dalam sistem regulasi/case management, ini sangat penting karena UI sering menampilkan tombol berbeda sesuai status case. Tombol yang hilang bukan berarti action tidak bisa dipanggil secara crafted request.

## 2. XSS Dalam Jakarta Faces

### 2.1 Apa yang Faces bantu

Komponen seperti `h:outputText` melakukan escaping by default.

Contoh aman:

```xml
<h:outputText value="#{caseDetail.subject}" />
```

Jika `caseDetail.subject` berisi:

```html
<script>alert(1)</script>
```

maka output yang diharapkan adalah text escaped, bukan script yang dieksekusi.

---

### 2.2 Apa yang Faces tidak otomatis selesaikan

Faces tidak otomatis menyelesaikan semua XSS karena XSS bergantung pada **browser parsing context**.

Context berbeda membutuhkan perlakuan berbeda:

| Context | Contoh | Risiko |
|---|---|---|
| HTML body | `<span>DATA</span>` | HTML injection |
| HTML attribute | `<div title="DATA">` | quote/event injection |
| JavaScript string | `var x = 'DATA';` | JS injection |
| URL | `<a href="DATA">` | `javascript:` URL, open redirect |
| CSS | `style="color: DATA"` | CSS injection |
| Raw HTML | rich text/comment body | stored XSS |

OWASP menekankan bahwa output encoding harus disesuaikan dengan context, karena HTML, JavaScript, URL, dan CSS diparse berbeda oleh browser.

---

### 2.3 `escape="false"`: sharp knife

Faces memungkinkan:

```xml
<h:outputText value="#{article.bodyHtml}" escape="false" />
```

Ini berarti Faces tidak melakukan HTML escaping.

Ini hanya boleh dipakai jika semua kondisi berikut benar:

1. data benar-benar HTML yang memang ingin dirender;
2. HTML sudah disanitasi server-side dengan allowlist sanitizer;
3. sanitizer aware terhadap tag, attribute, URL protocol, dan CSS;
4. data tidak berasal langsung dari user tanpa sanitization;
5. hasil sanitization disimpan/ditandai sebagai trusted formatted content;
6. ada regression test dengan payload XSS.

Anti-pattern:

```xml
<h:outputText value="#{comment.text}" escape="false" />
```

Ini membuka stored XSS jika comment berasal dari user.

---

### 2.4 Raw HTML dari database

Banyak aplikasi enterprise memiliki kolom seperti:

- `description_html`;
- `email_template_body`;
- `announcement_body`;
- `case_notes_rich_text`;
- `correspondence_template`.

Masalahnya, nama kolom “HTML” sering membuat engineer berpikir data itu aman.

Yang benar:

```text
HTML from DB is not automatically trusted.
```

DB hanya tempat penyimpanan. Security bergantung pada:

- siapa yang bisa menulis data itu;
- apakah sanitization dilakukan saat write;
- apakah sanitization dilakukan ulang saat render;
- apakah sanitization policy berubah;
- apakah data lama pernah ditulis sebelum sanitization ada;
- apakah HTML mengandung link eksternal, image, inline style, event handler.

---

### 2.5 Safe rich text pipeline

Pipeline yang lebih defensible:

```text
User input rich text
  -> server-side sanitizer allowlist
  -> normalized safe HTML
  -> persisted as sanitized HTML
  -> rendered with escape=false only in dedicated component/tag
  -> CSP reduces impact if sanitizer fails
  -> regression tests with XSS payloads
```

Jangan menyebarkan `escape="false"` langsung di banyak halaman.

Lebih baik buat composite component:

```xml
<app:safeHtml value="#{caseDetail.safeDescriptionHtml}" />
```

Lalu implementasinya menjadi satu tempat untuk audit.

---

### 2.6 Attribute injection

Contoh berbahaya:

```xml
<div class="#{bean.dynamicClass}">...</div>
```

Jika `dynamicClass` berasal dari user, value bisa mengandung karakter yang mengubah struktur attribute.

Untuk class/style dinamis, gunakan allowlist:

```java
public String getSeverityCssClass() {
    return switch (severity) {
        case HIGH -> "badge badge-high";
        case MEDIUM -> "badge badge-medium";
        case LOW -> "badge badge-low";
    };
}
```

Jangan:

```java
public String getSeverityCssClass() {
    return requestParamSeverity;
}
```

---

### 2.7 JavaScript context

Hindari menyisipkan data user langsung ke inline JavaScript:

```xml
<script>
  const name = '#{bean.userInput}';
</script>
```

Masalah:

- HTML escaping bukan JavaScript escaping;
- quote bisa memutus string;
- payload bisa menutup script;
- CSP akan sulit diperketat jika inline script tersebar.

Alternatif lebih aman:

1. render data sebagai JSON dengan encoder JSON yang benar;
2. taruh data di `data-*` attribute yang sudah HTML-attribute-escaped;
3. external JS membaca data dari DOM;
4. hindari inline event handler.

Contoh pattern:

```xml
<h:panelGroup id="casePanel"
              layout="block"
              pt:data-case-id="#{caseDetail.caseId}"
              pt:data-status="#{caseDetail.statusCode}">
    ...
</h:panelGroup>
```

Lalu JavaScript external membaca `dataset`.

---

### 2.8 URL context

Link dinamis harus diperlakukan hati-hati.

Berbahaya:

```xml
<h:outputLink value="#{bean.redirectUrl}">
    Open
</h:outputLink>
```

Jika `redirectUrl` bisa berisi:

```text
javascript:alert(1)
```

atau domain eksternal tidak valid, ini berisiko.

Gunakan allowlist:

```java
public URI getSafeExternalUrl() {
    URI uri = URI.create(rawUrl);
    if (!Set.of("https").contains(uri.getScheme())) {
        throw new IllegalArgumentException("Unsupported URL scheme");
    }
    if (!allowedHosts.contains(uri.getHost())) {
        throw new IllegalArgumentException("Unsupported host");
    }
    return uri;
}
```

Untuk URL internal, lebih baik generate dari route/view id/action, bukan dari input user mentah.

## 3. CSRF Dalam Jakarta Faces

### 3.1 CSRF mental model

CSRF terjadi ketika browser user yang sedang login dipaksa mengirim request ke aplikasi tanpa niat user.

Contoh sederhana:

```html
<form action="https://target.example.gov/case/approve" method="post">
  <input type="hidden" name="caseId" value="123">
</form>
<script>document.forms[0].submit()</script>
```

Jika aplikasi hanya bergantung pada session cookie, browser akan mengirim cookie secara otomatis.

---

### 3.2 Faces form dan view state

Faces form biasanya menyertakan hidden field view state:

```html
<input type="hidden" name="jakarta.faces.ViewState" value="...">
```

Pada era lama bisa bernama:

```html
<input type="hidden" name="javax.faces.ViewState" value="...">
```

View state membantu framework restore component tree dan dapat berperan dalam mengurangi CSRF karena attacker tidak mudah membuat postback valid tanpa token/state yang benar. Namun dari sisi architecture, jangan hanya berpikir:

```text
Faces ViewState exists -> CSRF solved forever.
```

Yang perlu dipastikan:

1. implementation/container mengaktifkan proteksi state sesuai best practice;
2. state token tidak bocor ke third-party origin;
3. sensitive action tetap POST dan memiliki server-side authorization;
4. SameSite cookie diset dengan tepat;
5. custom non-Faces endpoints tetap punya CSRF protection;
6. Ajax endpoints tidak membuka bypass;
7. GET tidak dipakai untuk mutasi.

---

### 3.3 GET harus aman dan idempotent

Buruk:

```text
/case/approve.xhtml?id=123
```

Jika membuka URL itu melakukan approval, maka:

- crawler bisa memicu action;
- prefetch browser bisa memicu action;
- link preview bisa memicu action;
- user bisa tertipu klik link;
- CSRF lebih mudah.

Lebih baik:

```text
GET  /case/detail.xhtml?id=123      -> render detail
POST Faces command approve button   -> mutasi approval
```

Server tetap harus cek permission dan workflow state.

---

### 3.4 SameSite cookie bukan pengganti CSRF token

`SameSite=Lax` atau `Strict` membantu mengurangi pengiriman cookie pada cross-site request tertentu.

Namun SameSite tidak menggantikan:

- CSRF token;
- view state validation;
- origin/referer validation untuk endpoint sensitif;
- server-side authorization;
- idempotent GET rule.

SameSite juga punya trade-off dengan:

- SSO;
- cross-domain app switcher;
- embedded flows;
- identity provider redirects;
- third-party integrations.

## 4. View State Security

### 4.1 Apa itu view state

Faces perlu menyimpan state component tree antar request. State ini dipakai untuk:

- restore component tree;
- mencocokkan component id;
- memproses submitted values;
- menjaga local values;
- menangani validation messages;
- mendukung postback dan Ajax partial lifecycle.

State bisa disimpan:

| Mode | Lokasi | Trade-off |
|---|---|---|
| Server-side | Session/server memory | payload kecil, memory/session pressure |
| Client-side | hidden field | payload besar, perlu integrity/confidentiality protection |

---

### 4.2 Client-side state harus dianggap sensitif

Jika client-side state saving aktif, hidden field bisa menjadi besar dan berisi representasi state component tree.

Risiko:

- state tampering jika tidak ditandatangani;
- information disclosure jika tidak dienkripsi;
- payload bloat;
- replay/stale page behavior;
- WAF/proxy limit;
- log leakage jika body request dilog sembarangan;
- performance cost encode/decode.

Prinsip:

```text
Client-side state must have integrity protection.
Sensitive data should not be placed in component state.
```

Jangan letakkan secret di component attribute/view field hanya karena “tidak terlihat”.

---

### 4.3 Server-side state bukan berarti bebas risiko

Server-side state menyimpan token kecil di browser dan state asli di server/session.

Risiko:

- session memory pressure;
- view map terlalu besar;
- banyak tab membuka banyak view;
- session replication overhead;
- sticky session dependency;
- `ViewExpiredException` jika state hilang;
- stale authorization display.

Untuk aplikasi high-traffic, view state size dan jumlah active views per session harus dipantau.

---

### 4.4 Jangan simpan data sensitif di view state

Buruk:

```java
private String temporaryPassword;
private String apiToken;
private FullCaseFile caseFileWithConfidentialNotes;
```

Jika bean view-scoped dan serializable, field ini bisa masuk ke state/session.

Lebih baik:

- simpan secret di server-side secure storage;
- simpan identifier minimal di view;
- reload authoritative data saat action;
- gunakan short-lived token khusus jika perlu;
- jangan render secret ke HTML;
- jangan taruh sensitive object graph dalam view scope.

---

### 4.5 View state bukan business authority

Jika hidden view state berisi status lama:

```text
case status = SUBMITTED
available action = APPROVE
```

lalu user postback setelah case sudah berubah menjadi `CANCELLED`, service tetap harus menolak approval.

Benar:

```java
public void approve() {
    caseService.approve(currentUser.id(), command.caseId());
}
```

Di service:

```java
@Transactional
public void approve(UserId userId, CaseId caseId) {
    CaseRecord record = repository.lockById(caseId);

    permissionPolicy.requireCanApprove(userId, record);
    workflowPolicy.requireTransition(record.status(), APPROVE);

    record.approve(userId, clock.now());
    auditTrail.record(...);
}
```

Jangan:

```java
public void approve() {
    if (buttonRendered) {
        case.status = APPROVED;
    }
}
```

## 5. Authorization Dalam Faces

### 5.1 `rendered` bukan authorization

Contoh umum:

```xml
<h:commandButton value="Approve"
                 action="#{caseBean.approve}"
                 rendered="#{caseBean.canApprove}" />
```

Ini baik untuk UX. User yang tidak berhak tidak melihat tombol.

Namun ini **bukan** enforcement final.

Kenapa?

1. request bisa dibuat manual;
2. page bisa stale;
3. user bisa membuka page sebelum role dicabut;
4. entity status bisa berubah setelah render;
5. permission bisa berubah setelah tab dibuka;
6. `canApprove` mungkin hanya UI approximation.

Action method dan service harus tetap enforce.

---

### 5.2 Authorization harus dicek di beberapa level

Layered authorization:

| Layer | Tujuan |
|---|---|
| Routing/filter/security | User harus login dan punya akses umum ke page |
| View rendering | UX: tampilkan action/menu yang relevan |
| Backing bean action | Guard awal, pesan user-friendly |
| Service/domain | Enforcement final dan transactional |
| Repository/query | Data scoping/ownership filter |
| Audit | Bukti keputusan security |

Faces layer tidak boleh menjadi satu-satunya tempat rule permission.

---

### 5.3 Menu authorization vs page authorization

Menu bisa menyembunyikan link:

```xml
<ui:fragment rendered="#{security.canView('CASE_APPROVAL')}">
    <h:link outcome="/case/approval-list" value="Approval Queue" />
</ui:fragment>
```

Tapi page tetap harus diproteksi.

Jika user mengetik URL langsung:

```text
/case/approval-list.xhtml
```

server harus memblokir jika tidak berhak.

---

### 5.4 Row-level authorization

Dalam data table:

```xml
<h:dataTable value="#{caseList.rows}" var="row">
    <h:column>
        <h:commandButton value="Assign"
                         action="#{caseList.assign(row.id)}"
                         rendered="#{row.assignable}" />
    </h:column>
</h:dataTable>
```

`row.assignable` membantu UI.

Tetapi service harus tetap cek:

```java
assignmentService.assign(currentUser, caseId, officerId);
```

Dengan rule:

- current user punya permission assign;
- case berada di unit yang sama;
- status case memungkinkan assignment;
- officer valid;
- tidak ada conflicting lock;
- audit tercatat.

## 6. Hidden Fields, Select Items, and Tampering

### 6.1 Hidden input bukan trusted storage

Faces mendukung hidden input:

```xml
<h:inputHidden value="#{bean.caseId}" />
```

Hidden bukan berarti aman. User bisa mengubahnya via DevTools atau crafted request.

Hidden field boleh dipakai untuk membawa identifier, tetapi:

- harus divalidasi;
- harus dicek ownership/permission;
- harus dicek against server-side authoritative data;
- tidak boleh membawa price/status/role/decision yang dipercaya langsung.

---

### 6.2 Select item tampering

Contoh:

```xml
<h:selectOneMenu value="#{bean.selectedOfficerId}">
    <f:selectItems value="#{bean.availableOfficers}" var="o"
                   itemValue="#{o.id}" itemLabel="#{o.name}" />
</h:selectOneMenu>
```

User bisa submit `selectedOfficerId` yang tidak ada di dropdown.

Maka service harus cek:

```java
assignmentPolicy.requireOfficerAssignable(caseId, selectedOfficerId, currentUser);
```

Jangan percaya dropdown sebagai allowlist final.

---

### 6.3 Disabled field tidak dikirim, readonly masih dikirim

HTML behavior:

- disabled input biasanya tidak dikirim saat submit;
- readonly input bisa dikirim;
- hidden input selalu bisa dimodifikasi.

Faces developer sering salah mengira disabled field bisa menjadi security control. Tidak bisa.

## 7. Secure Rendering Pattern

### 7.1 Prefer output component yang escaping default

Gunakan:

```xml
<h:outputText value="#{caseDetail.title}" />
```

Hindari:

```xml
#{caseDetail.title}
```

Walaupun Facelets biasanya escape text dalam banyak situasi, `h:outputText` membuat niat lebih eksplisit dan mendukung attribute seperti `escape` secara jelas.

---

### 7.2 Dedicated safe components

Untuk repeated security-sensitive rendering, buat component/tag reusable:

```xml
<app:caseStatusBadge value="#{row.status}" />
<app:userDisplayName user="#{row.assignee}" />
<app:safeExternalLink url="#{row.referenceUrl}" label="Open" />
<app:safeHtml value="#{announcement.safeHtml}" />
```

Manfaat:

- encoding rule terpusat;
- policy allowlist terpusat;
- audit lebih mudah;
- test lebih fokus;
- developer junior tidak harus mengingat semua context.

---

### 7.3 Avoid raw passthrough from user input

Pass-through attributes berguna:

```xml
<h:inputText value="#{bean.query}" pt:placeholder="Search case" />
```

Namun jangan isi pass-through attribute dari untrusted input tanpa kontrol.

Buruk:

```xml
<h:panelGroup pt:onclick="#{bean.userProvidedHandler}" />
```

Lebih baik tidak ada inline event handler sama sekali.

---

### 7.4 Secure message rendering

Faces messages sering berisi value dari input user.

Buruk:

```java
facesContext.addMessage(null,
    new FacesMessage("Invalid value: " + rawUserInput));
```

Jika message dirender tanpa escape oleh custom component, ini bisa XSS.

Lebih aman:

```java
facesContext.addMessage(null,
    new FacesMessage(FacesMessage.SEVERITY_ERROR,
        "Invalid value",
        "Please correct the highlighted field."));
```

Jika perlu menampilkan value, pastikan output component escaping aktif.

## 8. CSP, Headers, and Browser-Side Controls

### 8.1 CSP sebagai defense-in-depth

Content Security Policy membantu mengurangi dampak XSS.

Target modern:

```http
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-...';
  object-src 'none';
  base-uri 'self';
  frame-ancestors 'none';
```

Namun Faces/component libraries kadang menghasilkan inline script. Ini perlu strategi:

- audit generated markup;
- gunakan nonce jika framework/library mendukung;
- kurangi inline JS custom;
- pindahkan custom script ke external file;
- uji Ajax partial rendering;
- uji component library seperti PrimeFaces jika dipakai.

CSP bukan pengganti output encoding.

---

### 8.2 Clickjacking

Untuk halaman authenticated, set:

```http
X-Frame-Options: DENY
```

atau CSP:

```http
Content-Security-Policy: frame-ancestors 'none'
```

Jika ada kebutuhan embed internal, gunakan allowlist yang eksplisit.

---

### 8.3 Cache-Control

Protected pages sebaiknya tidak disimpan di shared cache.

Header umum:

```http
Cache-Control: no-store
Pragma: no-cache
```

Terutama untuk:

- case detail;
- profile;
- document preview;
- approval page;
- admin page;
- error page dengan data sensitif.

Namun hati-hati: terlalu agresif bisa mengganggu performance static assets. Pisahkan policy halaman dinamis dan static resources.

---

## 9. File Upload and Download Security Dalam Faces

### 9.1 File upload

File upload di Faces atau component library harus divalidasi server-side:

- size limit;
- content type tidak hanya dari header;
- extension allowlist;
- magic number bila relevan;
- antivirus/malware scanning jika policy membutuhkan;
- filename normalization;
- path traversal prevention;
- storage outside web root;
- authorization before attach to case;
- audit upload event.

Jangan pernah menyimpan file dengan path mentah dari client.

---

### 9.2 File download

Secure download flow:

```text
GET /document/download?id=...
  -> authenticate
  -> authorize access to document/case
  -> verify document status/classification
  -> stream file
  -> set safe Content-Disposition
  -> set safe Content-Type
  -> audit download
```

Jangan render direct storage path:

```xml
<h:outputLink value="/uploads/#{doc.path}">Download</h:outputLink>
```

Gunakan endpoint controlled.

## 10. Ajax Security

### 10.1 Ajax request tetap request penuh dari sisi security

Faces Ajax terasa kecil karena hanya render sebagian halaman, tetapi tetap bisa:

- memicu validation;
- memicu listener;
- memicu action;
- mengubah server-side state;
- merender data sensitif.

Maka Ajax action tetap harus melewati authorization dan validation.

---

### 10.2 Partial response leakage

Jika exception terjadi dalam Ajax request, response bisa berisi detail error atau stack trace jika error handling buruk.

Pastikan:

- production tidak menampilkan stack trace;
- partial response exception ditangani;
- user mendapat generic error;
- correlation id dicatat;
- log server menyimpan detail untuk debugging.

---

### 10.3 Execute/render misuse

Contoh:

```xml
<f:ajax execute="@this" render="details" />
```

Jika input penting tidak masuk `execute`, validation tidak jalan dan model tidak update.

Security impact:

- user bisa memicu action dengan subset input;
- stale value dipakai;
- validation yang diharapkan tidak terjadi.

Service/domain layer tetap harus validasi invariant final.

---

## 11. Error Handling Security

### 11.1 Jangan bocorkan internal detail

Jangan tampilkan:

- stack trace;
- SQL;
- entity class name;
- file path;
- server hostname;
- internal module name sensitif;
- token/session id;
- raw request parameter.

Error page aman:

```text
Something went wrong.
Reference ID: ABC-20260618-000123
```

Log server:

```text
correlationId=ABC-20260618-000123
userId=...
requestUri=...
exception=...
stackTrace=...
```

---

### 11.2 Faces ExceptionHandler

Untuk aplikasi Faces serius, pertimbangkan custom `ExceptionHandler` atau mekanisme global error handler untuk:

- mapping exception ke user-friendly page/message;
- logging correlation id;
- handling Ajax exception;
- preventing stack trace leakage;
- distinguishing validation/business/security/system failure.

---

## 12. Session and Authentication Concerns

### 12.1 Session fixation

Setelah login, session id harus diregenerasi oleh security layer/container.

Faces sendiri bukan login framework. Jika integrasi dengan Jakarta Security/Spring Security/Keycloak/custom SSO, pastikan session fixation protection aktif.

---

### 12.2 Session timeout

Faces stateful UI sensitif terhadap timeout.

Failure mode:

1. user membuka form lama;
2. session expired;
3. user submit;
4. view restore gagal;
5. user kehilangan input.

Security/UX pattern:

- tampilkan session timeout warning;
- handle `ViewExpiredException` dengan page yang jelas;
- jangan restore sensitive action otomatis;
- untuk form besar, pertimbangkan draft/autosave dengan authorization ketat.

---

### 12.3 Logout

Logout harus:

- invalidate session;
- clear relevant cookies;
- handle IdP logout jika SSO;
- prevent back-button cached page issue via cache headers;
- handle multiple tabs gracefully.

---

## 13. Data Classification and Secure View Model

### 13.1 View model harus sadar klasifikasi data

Jangan lempar entity penuh ke UI.

Buruk:

```java
private CaseEntity selectedCase;
```

Lalu view bebas mengakses:

```xml
#{caseBean.selectedCase.internalInvestigationNote}
#{caseBean.selectedCase.officerPrivateComment}
#{caseBean.selectedCase.rawAuditMetadata}
```

Lebih baik:

```java
public record CaseDetailView(
    String caseId,
    String title,
    String publicDescription,
    String statusLabel,
    boolean canApprove,
    boolean canAssign,
    List<DocumentRow> documents,
    List<AllowedAction> actions
) {}
```

View model hanya berisi data yang memang boleh dirender untuk user/page tersebut.

---

### 13.2 Avoid accidental EL exposure

EL bisa mengakses getter publik.

Jika entity punya banyak getter, view bisa secara tidak sengaja mengakses data yang tidak seharusnya.

Prinsip:

```text
Do not expose rich domain entities directly to Facelets.
Expose intentional, page-specific view models.
```

---

## 14. Secure Case Management Example

### 14.1 Requirement

Page: case detail regulatory enforcement.

Action:

- approve case;
- assign officer;
- upload supporting document;
- add internal note;
- view audit trail.

Risiko:

- user melihat case dari unit lain;
- tombol approve dimunculkan salah;
- user forged request approve case;
- hidden case id diganti;
- rich text note XSS;
- document path traversal;
- audit trail leakage;
- stale status setelah concurrent update.

---

### 14.2 View

```xml
<h:form id="caseForm">
    <h:messages globalOnly="true" />

    <h:outputText value="#{caseDetail.title}" />
    <h:outputText value="#{caseDetail.statusLabel}" />

    <app:safeHtml value="#{caseDetail.safeDescriptionHtml}" />

    <h:commandButton value="Approve"
                     action="#{caseDetailBean.approve}"
                     rendered="#{caseDetail.canApprove}" />

    <h:commandButton value="Assign"
                     action="#{caseDetailBean.assign}"
                     rendered="#{caseDetail.canAssign}" />
</h:form>
```

`rendered` hanya UX.

---

### 14.3 Backing bean

```java
@Named
@ViewScoped
public class CaseDetailBean implements Serializable {

    private String caseId;
    private CaseDetailView caseDetail;

    @Inject
    private CaseQueryService queryService;

    @Inject
    private CaseCommandService commandService;

    @Inject
    private CurrentUser currentUser;

    public void load() {
        this.caseDetail = queryService.getCaseDetailForUser(currentUser.id(), caseId);
    }

    public String approve() {
        commandService.approve(currentUser.id(), caseId);
        FacesContext.getCurrentInstance().addMessage(null,
            new FacesMessage("Case approved."));
        return "case-detail?faces-redirect=true&id=" + caseId;
    }
}
```

---

### 14.4 Service enforcement

```java
@Transactional
public void approve(UserId userId, String caseId) {
    CaseRecord record = caseRepository.lockById(caseId)
        .orElseThrow(NotFoundException::new);

    permissionPolicy.requireCanAccessCase(userId, record);
    permissionPolicy.requireCanApprove(userId, record);
    workflowPolicy.requireCurrentStatus(record, CaseStatus.SUBMITTED);

    record.approve(userId, clock.now());
    auditTrail.recordApproval(userId, record.id());
}
```

Security properties:

- forged request fails;
- stale page fails;
- hidden id tampering fails;
- concurrent transition fails;
- permission change fails;
- audit remains authoritative.

---

## 15. Security Review Checklist Untuk Faces Page

### 15.1 Rendering

- Apakah semua untrusted text dirender dengan escaping?
- Apakah ada `escape="false"`?
- Jika ada raw HTML, apakah sudah disanitasi dengan allowlist?
- Apakah data user masuk ke JavaScript inline?
- Apakah URL dinamis divalidasi scheme/host?
- Apakah CSS/class dinamis berasal dari allowlist?
- Apakah messages bisa mengandung raw user input?

### 15.2 Form and lifecycle

- Apakah mutasi hanya lewat POST/action?
- Apakah GET bebas side effect?
- Apakah hidden field divalidasi ulang?
- Apakah select item dicek ulang server-side?
- Apakah `immediate=true` tidak melewati validation penting?
- Apakah Ajax `execute` mencakup input yang diperlukan?
- Apakah double submit ditangani?

### 15.3 Authorization

- Apakah page diproteksi di route/security layer?
- Apakah menu visibility bukan satu-satunya proteksi?
- Apakah action method melakukan guard?
- Apakah service/domain enforce permission final?
- Apakah row-level permission dicek ulang?
- Apakah stale page/concurrent update ditangani?

### 15.4 State

- Apakah view scope menyimpan object besar/sensitif?
- Apakah client-side state aman bila digunakan?
- Apakah session memory dipantau?
- Apakah `ViewExpiredException` ditangani aman?
- Apakah multi-tab behavior acceptable?

### 15.5 Browser controls

- Apakah protected pages memakai cache-control yang sesuai?
- Apakah CSP dipertimbangkan?
- Apakah clickjacking dicegah?
- Apakah cookie punya `HttpOnly`, `Secure`, `SameSite` sesuai konteks?
- Apakah error page tidak bocor detail?

### 15.6 Files

- Apakah upload divalidasi server-side?
- Apakah download authorize per document/case?
- Apakah filename aman?
- Apakah storage path tidak dirender langsung?
- Apakah audit upload/download dicatat?

## 16. Common Anti-Patterns

### 16.1 `rendered` as permission

```xml
rendered="#{user.admin}"
```

lalu service tidak cek permission.

Ini salah.

---

### 16.2 Entity directly exposed to Facelets

```xml
#{caseEntity.confidentialInternalNote}
```

Jika getter ada, EL bisa mengaksesnya.

---

### 16.3 `escape=false` everywhere

```xml
<h:outputText value="#{anything}" escape="false" />
```

Ini biasanya tanda tim tidak memahami output context.

---

### 16.4 Trusting hidden status

```xml
<h:inputHidden value="#{bean.status}" />
```

Lalu service memakai status dari form.

Status authoritative harus dari database/current state.

---

### 16.5 GET mutation

```xml
<h:link outcome="approve" value="Approve">
    <f:param name="id" value="#{case.id}" />
</h:link>
```

Approval tidak boleh GET.

---

### 16.6 Debug error in production

Ajax error response berisi stack trace.

Ini information disclosure.

## 17. Java 8 sampai Java 25 Considerations

### 17.1 Java 8 legacy

Kemungkinan stack:

- JSF 2.x;
- Java EE 7/8;
- `javax.faces.*`;
- older component libraries;
- older servlet containers;
- weaker default headers;
- custom login/session handling.

Security focus:

- audit `escape=false`;
- audit raw JS snippets;
- audit old component library vulnerabilities;
- check CSRF/view state settings;
- verify session cookie flags;
- remove scriptlet-era assumptions;
- test with modern browsers.

---

### 17.2 Java 11/17 transition

Transition phase sering mencampur:

- older JSF libraries;
- newer servlet containers;
- partial `javax`/`jakarta` migration;
- custom security filters;
- SSO integration.

Security focus:

- dependency compatibility;
- namespace mismatch;
- transitive old libraries;
- CSP impact;
- SameSite behavior with SSO;
- serialization compatibility for view-scoped beans.

---

### 17.3 Java 21/25 modern Jakarta

Modern stack:

- Jakarta Faces 4.x;
- Jakarta EE 10/11;
- `jakarta.faces.*`;
- CDI-aligned beans;
- Java records useful for immutable view models;
- modern security headers expected;
- stronger dependency scanning expected.

Security focus:

- intentional view models;
- strict CSP where feasible;
- centralized safe components;
- CI security tests for rendered output;
- dependency hygiene;
- state size and session observability;
- avoiding unsafe compatibility hacks from migration.

## 18. Migration Security Checklist: `javax.faces` ke `jakarta.faces`

Saat migrasi, jangan hanya compile-fix imports.

Checklist:

1. cari semua `escape="false"`;
2. cari raw `<script>` dengan EL interpolation;
3. cari dynamic `onclick`, `href`, `style`, `class`;
4. cari `rendered` permission checks;
5. cari action method tanpa service authorization;
6. cari GET mutation;
7. cari hidden field yang membawa status/role/price/decision;
8. cari entity langsung diekspos ke view;
9. cari view-scoped bean yang menyimpan object sensitif/besar;
10. cek client/server state saving config;
11. cek cookie flags;
12. cek error page dan Ajax error;
13. cek component library version;
14. cek sanitizer library;
15. tambah rendered HTML security tests.

Migration adalah momen bagus untuk security hardening.

## 19. Testing Strategy

### 19.1 XSS rendering test

Gunakan payload seperti:

```text
<script>alert(1)</script>
"><img src=x onerror=alert(1)>
javascript:alert(1)
</script><script>alert(1)</script>
```

Test bahwa payload tidak muncul sebagai executable markup di rendered HTML.

---

### 19.2 Authorization test

Test dua hal:

1. tombol/menu tidak muncul jika user tidak punya permission;
2. service/action tetap menolak crafted request.

Jangan hanya test UI visibility.

---

### 19.3 Hidden field tampering test

Scenario:

1. render page untuk case A;
2. ubah hidden/input id menjadi case B;
3. submit;
4. assert service menolak jika user tidak punya akses.

---

### 19.4 Stale page test

Scenario:

1. user buka page status `SUBMITTED`;
2. proses lain ubah status menjadi `CANCELLED`;
3. user klik approve;
4. service menolak transition.

---

### 19.5 Ajax error test

Scenario:

1. trigger exception dalam Ajax action;
2. assert response tidak berisi stack trace;
3. assert user mendapat generic message;
4. assert server log punya correlation id.

## 20. Final Mental Model

Jakarta Faces security harus dilihat sebagai kombinasi beberapa lapisan:

```text
Browser parsing context
  -> output encoding / safe component
  -> component tree and view state
  -> lifecycle conversion/validation
  -> backing bean guard
  -> service/domain authorization
  -> persistence/data ownership
  -> audit and observability
```

Prinsip paling penting:

1. **Escaping default itu bagus, tapi context-aware security tetap perlu.**
2. **`escape=false` harus dianggap exceptional dan diaudit.**
3. **`rendered=false` hanya UX, bukan authorization.**
4. **Hidden field, select value, Ajax parameter, dan view state bukan business authority.**
5. **Service/domain layer harus menjadi enforcement final.**
6. **View model harus intentional, bukan entity leakage.**
7. **Stateful UI berarti state security dan memory security harus dipantau.**
8. **Security header, CSP, SameSite, cache-control, dan error handling adalah defense-in-depth.**
9. **Migration dari `javax` ke `jakarta` harus menjadi security audit opportunity.**
10. **Top-tier engineer tidak hanya membuat UI berjalan; mereka memastikan UI tidak bisa dipakai untuk melanggar invariant sistem.**

---

## 21. Ringkasan Eksekutif

Jakarta Faces memberi banyak mekanisme built-in untuk membangun UI enterprise: component tree, state management, validation, conversion, event handling, navigation, dan Ajax. Namun keamanan tetap harus didesain.

Faces membantu mengurangi beberapa risiko, misalnya output component yang escape by default dan view state untuk postback. Tetapi Faces tidak otomatis menyelesaikan XSS pada semua context, tidak menggantikan CSRF/security architecture untuk semua endpoint, tidak menjadikan hidden field aman, dan tidak menjadikan `rendered` sebagai authorization.

Untuk aplikasi enterprise/regulatory, pola terbaik adalah:

```text
View renders only safe view model.
View may hide unavailable actions for UX.
Backing bean maps user intent into command.
Service reloads authoritative state.
Policy enforces permission and transition.
Domain preserves invariant.
Audit records decision.
```

Jika pola ini konsisten, Faces dapat menjadi UI framework yang sangat produktif sekaligus defensible.

## 22. Latihan Praktis

### Latihan 1 — Audit XSS

Ambil satu halaman Faces yang memiliki:

- `h:outputText`;
- raw EL interpolation;
- `escape=false`;
- dynamic link;
- inline script.

Klasifikasikan setiap output berdasarkan context:

```text
HTML body / HTML attribute / JavaScript / URL / CSS / raw HTML
```

Tentukan encoding/sanitization yang benar.

---

### Latihan 2 — Audit Authorization

Cari semua komponen dengan:

```xml
rendered="#{...can...}"
```

Untuk setiap action, jawab:

1. apakah backing bean mengecek ulang?
2. apakah service mengecek ulang?
3. apakah workflow state dicek ulang?
4. apakah audit dicatat?

---

### Latihan 3 — Tampering Test

Untuk form assignment:

1. render dropdown officer;
2. submit officer id yang tidak ada di dropdown;
3. pastikan service menolak;
4. pastikan error message aman dan tidak bocor internal detail.

---

### Latihan 4 — View State Review

Ukur:

- ukuran hidden view state;
- field di `@ViewScoped` bean;
- jumlah active views per session;
- object graph yang tersimpan.

Tentukan apakah ada data sensitif/besar yang harus dipindah.

---

### Latihan 5 — Error Handling Review

Trigger error pada:

- normal full request;
- Ajax request;
- validation error;
- authorization failure;
- expired view.

Pastikan user-facing response aman dan log server cukup untuk debugging.
