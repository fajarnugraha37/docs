## 0. Ringkasan Besar

Quarkus sering diasosiasikan dengan native image karena startup cepat dan memory footprint kecil.

Namun native image bukan sekadar:

```text
mvn package -Dnative
```

Native image adalah perubahan besar pada model eksekusi Java:

```text
JVM mode:
  bytecode + JVM runtime + JIT + dynamic behavior at runtime

Native image:
  ahead-of-time compiled binary + closed-world analysis + selected runtime support
```

Di JVM mode, banyak hal bisa diputuskan saat runtime:

- classloading,
- reflection,
- dynamic proxy,
- resource lookup,
- service loader,
- serialization,
- annotation scanning,
- framework bootstrap,
- JIT optimization,
- runtime discovery.

Di native image, banyak hal harus diketahui saat build time.

Itulah kenapa Quarkus sangat cocok dengan native image: Quarkus sejak awal memindahkan banyak kerja framework ke build time:

- CDI bean discovery,
- Jandex indexing,
- endpoint/resource discovery,
- generated bytecode,
- reflection metadata,
- native image hints,
- configuration categorization,
- extension build steps,
- static init/runtime init separation.

Native image bukan selalu lebih baik. Native image adalah trade-off:

```text
Startup lebih cepat.
RSS sering lebih rendah.
Cold start lebih bagus.
Container/serverless lebih menarik.
Tetapi build lebih berat.
Dynamic behavior lebih terbatas.
Debugging bisa lebih sulit.
Peak throughput bisa berbeda.
Library compatibility harus diuji.
```

## 1. Mental Model: Native Image Adalah “Java yang Dibekukan Sebelum Runtime”

Bayangkan JVM application biasa:

```text
App starts
  -> framework scans classpath
  -> reads annotations
  -> builds DI graph
  -> creates proxies
  -> discovers REST endpoints
  -> configures ORM
  -> initializes clients
  -> loads resources
  -> JIT optimizes hot paths over time
```

Native image mencoba memindahkan sebanyak mungkin proses tersebut ke build phase:

```text
Build time
  -> analyze application
  -> discover reachable code
  -> initialize safe classes
  -> generate metadata
  -> compile to native executable

Runtime
  -> start binary
  -> run already-prepared application
```

Karena itu native binary bisa start cepat.

Namun konsekuensinya:

```text
Jika sesuatu baru diketahui saat runtime,
native image harus diberi tahu saat build time.
```

Inilah akar masalah native image.

## 2. JVM Mode vs Native Mode

### 2.1 JVM Mode

JVM mode menjalankan bytecode di atas JVM.

Karakteristik:

- JIT compilation,
- dynamic classloading,
- runtime reflection lebih fleksibel,
- runtime proxy generation lebih mudah,
- heap/JIT/metaspace/JVM runtime lebih besar,
- startup lebih lambat dibanding native,
- throughput steady-state sering sangat baik karena JIT,
- debugging/profiling lebih matang.

### 2.2 Native Mode

Native mode menghasilkan executable.

Karakteristik:

- ahead-of-time compilation,
- closed-world analysis,
- startup cepat,
- memory footprint sering lebih kecil,
- runtime dynamic behavior terbatas,
- reflection/resource/proxy harus diketahui,
- build time lebih lama,
- native build membutuhkan GraalVM/Mandrel native-image,
- debugging/profiling berbeda,
- compatibility harus diuji.

### 2.3 Trade-off Table

| Dimension | JVM Mode | Native Mode |
|---|---|---|
| Startup | slower | faster |
| Warm throughput | often strong due JIT | depends workload |
| Memory idle | higher | often lower |
| Build time | faster | slower |
| Dynamic reflection | flexible | constrained |
| Classloading | dynamic | limited |
| Debugging | mature | more specialized |
| Library compatibility | broad | needs validation |
| Serverless/cold start | less ideal | strong |
| Long-running high-throughput service | often excellent | measure |
| Container density | good | often better |
| Build pipeline complexity | lower | higher |

Rule:

```text
Native image is a deployment/runtime strategy, not a universal upgrade.
```

## 3. GraalVM Native Image

GraalVM Native Image is a technology that compiles Java code ahead-of-time into a native executable. The executable contains application classes, needed JDK classes, language runtime pieces, and statically linked native code required at runtime.

Conceptual build:

```text
Java source
  -> javac bytecode
  -> Quarkus augmentation
  -> native-image analysis
  -> AOT compilation
  -> executable
```

Native Image performs:

- static analysis,
- reachability analysis,
- class initialization analysis,
- metadata processing,
- resource inclusion,
- reflection/proxy/JNI config processing,
- code generation,
- native linking.

The important mental model:

```text
Native Image can only include what it can know or what you tell it.
```

---

## 4. Mandrel

Mandrel is a downstream distribution of GraalVM Community Edition tailored to support Quarkus. Its goal is to align GraalVM native-image capabilities with OpenJDK and Red Hat Enterprise Linux libraries to improve maintainability for native Quarkus applications.

In practice:

```text
GraalVM Native Image = upstream native-image technology.
Mandrel = downstream distribution focused on Quarkus ecosystem support.
```

Quarkus supports native builds using GraalVM/Mandrel. Quarkus 3.31 added full Java 25 support, including Java 25 runtime images and native image builds with Mandrel. Later Quarkus 3.36 updated the minimum officially supported GraalVM/Mandrel version for building native executables to 25.0.

Implication:

```text
Always align Quarkus version, Java version, builder image, and Mandrel/GraalVM version.
```

Do not randomly mix:

- Quarkus version,
- Java target,
- Mandrel version,
- builder image,
- container base image.

Native build compatibility is a platform matrix.

## 5. Why Quarkus Fits Native Image

Traditional frameworks often do much runtime discovery:

```text
scan classpath
reflect annotations
create proxies dynamically
infer beans
load configuration dynamically
discover serializers
```

Native image dislikes runtime discovery unless it is explicitly configured.

Quarkus solves this by doing framework work at build time:

```text
Jandex indexes classes.
Extensions produce build items.
Arc/CDI graph is analyzed.
Unused beans removed.
Synthetic beans generated.
REST endpoints discovered.
Hibernate model prepared.
Reflection metadata registered.
Resources registered.
Recorders capture runtime initialization.
```

Thus Quarkus acts like:

```text
framework + application compiler
```

Native image works well when framework behavior is known early.

Quarkus makes more behavior known early.

## 6. Closed-World Assumption

Closed-world assumption means native-image build assumes the universe of reachable code is known at build time.

In JVM mode:

```text
Maybe this class is loaded later.
Maybe reflection accesses this method.
Maybe ServiceLoader finds provider.
Maybe runtime generates proxy.
```

In native image:

```text
If analysis cannot see it, it may not exist in the binary.
```

This is the most important concept.

### 6.1 Reachability

Native image starts from entry points and analyzes reachable code.

If code is reachable:

```text
included
```

If not reachable:

```text
excluded
```

This reduces binary size and memory, but dynamic behavior becomes constrained.

### 6.2 Dynamic Access Problem

Example:

```java
Class<?> clazz = Class.forName(classNameFromConfig);
Object instance = clazz.getDeclaredConstructor().newInstance();
```

At build time, `classNameFromConfig` may not be known.

Native image cannot safely know what to include.

Solution options:

- avoid dynamic classloading,
- use explicit registry,
- register class for reflection,
- use Quarkus extension metadata,
- use build-time known config,
- use static dispatch.

### 6.3 Reflection Problem

Example:

```java
Field field = clazz.getDeclaredField("name");
field.setAccessible(true);
```

In JVM, this can work if module/security allows.

In native image, reflection metadata may not exist unless registered.

Quarkus can automatically register metadata for supported frameworks/extensions.

For application-specific dynamic reflection, you may need explicit configuration.

## 7. Native Image Build Pipeline in Quarkus

Conceptual steps:

```text
1. Compile Java sources.
2. Resolve dependencies and Quarkus platform.
3. Quarkus augmentation runs.
4. Extensions execute build steps.
5. Jandex index used for discovery.
6. Beans/resources/endpoints/model prepared.
7. Generated bytecode/resources produced.
8. Native image metadata produced.
9. GraalVM/Mandrel native-image analyzes reachability.
10. Native executable linked.
```

Important distinction:

```text
Quarkus augmentation is not the same as native-image compilation.
```

Quarkus prepares the application.

GraalVM/Mandrel compiles it into native executable.

## 8. Build Time vs Runtime in Quarkus

Quarkus heavily separates:

1. **Build time**
   - what can be decided while building artifact.

2. **Static init**
   - code executed during native image build and snapshotted into binary image heap where safe.

3. **Runtime init**
   - code executed when binary starts.

4. **Runtime request processing**
   - normal app logic.

This separation matters because doing the wrong thing at build time can freeze wrong state into binary.

## 9. Static Initialization

Static initialization means class initialization happens during native image build.

Example:

```java
public final class SomeConstants {
    static final Pattern POSTAL_PATTERN = Pattern.compile("\\d{6}");
}
```

This is generally safe:

- deterministic,
- no environment dependency,
- no network,
- no secret,
- no current time,
- no random,
- no file system assumption.

If initialized at build time, the initialized state may become part of image heap.

### 9.1 Safe Static Init Examples

Usually safe:

```text
constant lookup tables
regex Pattern
immutable metadata
pure deterministic computation
static enum maps
precomputed serializers if supported
```

### 9.2 Dangerous Static Init Examples

Dangerous:

```java
static final Instant STARTED_AT = Instant.now();
static final String HOST = InetAddress.getLocalHost().getHostName();
static final String SECRET = System.getenv("SECRET");
static final Socket SOCKET = openSocket();
static final Random RANDOM = new SecureRandom();
```

If executed at build time, the runtime binary may contain build machine data or invalid state.

Examples of bad frozen state:

```text
build time timestamp
build machine hostname
wrong environment variable
build-time secret
invalid random seed
opened resource that cannot exist at runtime
```

Invariant:

```text
Build-time initialization must not depend on runtime environment.
```

---

## 10. Runtime Initialization

Runtime initialization means class initialization happens when native executable starts.

Use runtime init when class depends on:

- environment variables,
- current time,
- secrets,
- hostname,
- IP,
- file system,
- sockets,
- random/secure random,
- runtime config,
- container limits,
- system properties,
- certificates,
- runtime-only resources.

Concept:

```text
Initialize pure deterministic things at build time.
Initialize environment-dependent things at runtime.
```

Quarkus extensions often handle this separation for supported libraries.

Application code must avoid static initializers that accidentally read runtime state.

## 11. Class Initialization Trap

The common native error shape:

```text
Classes that should be initialized at run time got initialized during image building
```

This happens when:

- a class is marked runtime-init,
- but some other build-time code touches it,
- causing premature initialization.

Example:

```java
public class RuntimeSecretHolder {
    static final String SECRET = System.getenv("SECRET");
}
```

If any build-time augmentation touches `RuntimeSecretHolder`, secret is read too early.

Fix approaches:

1. remove static initialization,
2. move to runtime bean constructor,
3. use config injection,
4. use `Supplier`,
5. ensure class not referenced during build,
6. configure initialization explicitly,
7. create Quarkus extension recorder if library-level.

Bad:

```java
public final class TokenClient {
    static final String CLIENT_SECRET = System.getenv("CLIENT_SECRET");
}
```

Better:

```java
@ApplicationScoped
public class TokenClient {

    private final TokenClientConfig config;

    public TokenClient(TokenClientConfig config) {
        this.config = config;
    }
}
```

## 12. Image Heap

Native image may snapshot initialized objects into image heap.

Concept:

```text
Objects created during image build can be embedded into executable.
```

Benefit:

- faster startup,
- less runtime work.

Risk:

- object contains runtime-specific state,
- object contains secret,
- object contains file path,
- object contains random state,
- object contains build-machine data,
- object not safe to reuse.

Example safe image heap:

```text
compiled regex
immutable lookup table
static metadata
```

Example unsafe image heap:

```text
open connection
runtime token
temporary directory
current timestamp
system locale assumption if runtime differs
```

## 13. Reflection in Native Image

In JVM:

```java
clazz.getDeclaredMethods()
constructor.newInstance()
field.setAccessible(true)
```

metadata exists by default.

In native image:

```text
reflection metadata must be registered.
```

Quarkus provides mechanisms:

- framework extension auto-registration,
- `@RegisterForReflection`,
- `reflect-config.json`,
- extension build items,
- native image tracing agent output,
- generated metadata.

### 13.1 `@RegisterForReflection`

Conceptual:

```java
import io.quarkus.runtime.annotations.RegisterForReflection;

@RegisterForReflection
public class ExternalDto {
    public String name;
    public String status;
}
```

Use when framework/library needs reflection access and Quarkus cannot infer it.

But do not overuse.

Over-registering:

- increases binary size,
- hides design problems,
- may include unnecessary classes,
- can weaken native optimization.

### 13.2 Prefer Build-Time Known Serialization

For Jackson/JSON-B, Quarkus extensions often register needed metadata for REST DTOs.

But dynamic polymorphic serialization may still need attention.

Rule:

```text
Use explicit DTOs and avoid dynamic object graphs.
```

## 14. Resources in Native Image

Classpath resources are not automatically all included.

If code needs:

```java
getResourceAsStream("/templates/email.html")
```

native image must include that resource.

Quarkus supports native resource inclusion configuration.

Conceptual:

```properties
quarkus.native.resources.includes=templates/**,certs/**
```

Be careful:

- do not include secrets,
- do not include huge unnecessary resource trees,
- verify resource path case sensitivity,
- test native artifact.

Common missing resources:

- templates,
- static files,
- schema files,
- SQL seed scripts,
- certificates,
- font files,
- message bundles,
- localization files.

---

## 15. Dynamic Proxies

Java dynamic proxies can be problematic if proxy interfaces are not known.

Example:

```java
Proxy.newProxyInstance(
    classLoader,
    new Class<?>[] { SomeInterface.class },
    handler
);
```

Native image needs to know proxy interface combinations.

Quarkus extensions register proxies for supported frameworks.

Custom dynamic proxies may need configuration.

Prefer:

- CDI beans,
- generated clients supported by Quarkus,
- static interfaces,
- framework-supported proxies.

Avoid ad-hoc runtime proxy generation unless necessary.

---

## 16. ServiceLoader

`ServiceLoader` dynamically discovers providers from `META-INF/services`.

Native image may need service provider metadata included.

Quarkus extensions often handle service discovery for supported dependencies.

For application custom SPI:

- verify provider discovery in native,
- include resources,
- avoid dynamic provider unknown at build time,
- consider explicit registry.

## 17. JNI and Native Libraries

If code uses JNI/native libraries:

- native library must be available,
- linking/loading must work,
- platform compatibility matters,
- container image must contain required OS libraries,
- native image may need JNI config,
- cross-compilation constraints matter.

Examples:

- image processing,
- PDF/font/native rendering,
- crypto provider,
- compression,
- OS-level integration.

Native executable is not automatically self-contained for every native dependency.

## 18. Serialization Constraints

Serialization libraries may rely on:

- reflection,
- no-arg constructor,
- fields/method metadata,
- polymorphic type resolution,
- proxies,
- dynamic class lookup.

For native:

- use explicit DTOs,
- avoid serializing JPA entities,
- avoid lazy proxies,
- test unknown enum behavior,
- register reflection if needed,
- avoid runtime-generated models,
- prefer records/constructors if supported and tested.

For REST DTOs supported by Quarkus REST/Jackson/JSON-B, much is handled, but edge cases remain.

## 19. Time, Locale, Charset, and Timezone

Native image may need explicit support for:

- timezone data,
- locale data,
- charset,
- resource bundles,
- date/time formatting.

If application depends on:

- Indonesian locale,
- Singapore timezone,
- currency formatting,
- localized messages,
- ICU-like behavior,
- PDF rendering fonts,

test native artifact.

Never assume all JVM locale behavior transfers identically.

## 20. SSL/TLS and Crypto

Native image TLS/crypto can fail due to:

- missing truststore,
- certificate resource not included,
- runtime security provider issue,
- hostname verification config,
- random/secure random initialization,
- native SSL support configuration.

Quarkus has docs for native SSL and TLS registry, but real deployment still needs tests.

Test:

- outbound HTTPS REST client,
- OIDC JWKS fetch,
- mTLS if used,
- Redis TLS,
- Kafka TLS,
- custom truststore,
- certificate rotation.

## 21. Build-Time Config vs Runtime Config

Quarkus has build-time and runtime configuration.

Native image makes this more important.

Build-time config affects generated binary.

Runtime config affects behavior when binary starts.

If you change build-time config at runtime:

```text
it may not take effect
```

Examples of build-time-ish concerns:

- enabling/disabling extensions,
- native resources,
- some datasource/ORM features,
- package type,
- build tool flags,
- reflection/resource config.

Operational implication:

```text
Do not build one native binary expecting every build-time dimension to be mutable at deploy.
```

Document what is fixed at build and what is runtime-configurable.

Typical Maven native build:

```bash
./mvnw package -Dnative
```

Container build:

```bash
./mvnw package -Dnative -Dquarkus.native.container-build=true
```

Gradle equivalent depends on project setup.

Common native build outputs:

```text
target/*-runner
```

Do not assume native build works because JVM build works.

Native build is a separate compilation mode.

## 23. Native Build Resource Needs

Native build is heavier than JVM build.

It needs:

- more memory,
- more CPU,
- longer time,
- correct builder image,
- correct Java/GraalVM/Mandrel version,
- OS libraries/toolchain depending setup.

CI implications:

```text
Native build may need dedicated runner size.
Parallel native builds can exhaust memory.
Build cache matters.
Container build pulls large images.
```

Do not add native build to every PR without capacity planning.

## 24. Debugging Native Build Failure: First Principles

When native build fails, classify:

1. **Missing reflection metadata**
   - NoSuchMethodException, field/method missing, serialization failure.

2. **Missing resource**
   - resource stream null, template missing.

3. **Class init issue**
   - initialized at wrong phase.

4. **Unsupported dynamic behavior**
   - dynamic classloading, proxy, unsafe reflection.

5. **Native library/JNI issue**
   - library not found, symbol missing.

6. **TLS/crypto issue**
   - provider/truststore/SSL failure.

7. **Build resource issue**
   - out of memory, builder version mismatch.

8. **Dependency unsupported**
   - library uses unsupported runtime behavior.

Do not blindly add random native flags.

Find the category first.

## 25. Native Image Tracing Agent

The native image tracing agent is a JVM agent available in GraalVM/Mandrel that tracks dynamic feature usage during regular JVM execution, such as reflection, JNI, dynamic proxies, and classpath resource access. It can output native-image configuration files for later native builds.

Useful for:

- discovering dynamic features,
- generating initial config,
- migrating legacy app,
- understanding reflective libraries.

Limitations:

```text
Agent only records code paths exercised during run.
Unexercised paths remain missing.
Generated config can be too broad.
It is aid, not proof.
```

For Quarkus:

```text
Prefer Quarkus extensions/build items where possible.
Use tracing agent as diagnostic aid.
```

## 26. Native Image and Quarkus Extensions

Quarkus extensions are the preferred way to make frameworks native-friendly.

An extension can:

- inspect Jandex index,
- register classes for reflection,
- register resources,
- register proxies,
- configure runtime init,
- produce generated code,
- configure service providers,
- integrate with native image build.

This is why supported Quarkus extensions are often better than manually wiring random libraries.

Example:

```text
quarkus-rest-jackson handles REST/Jackson integration.
quarkus-hibernate-orm handles ORM integration.
quarkus-oidc handles OIDC integration.
quarkus-redis-client handles Redis integration.
```

If you use a library without Quarkus extension, you own native compatibility.

## 27. Native Image and Dependency Selection

When choosing a library for Quarkus native:

Ask:

1. Does it have a Quarkus extension?
2. Is it tested with GraalVM Native Image?
3. Does it rely on reflection?
4. Does it generate classes at runtime?
5. Does it use dynamic classloading?
6. Does it use JNI/native libs?
7. Does it require resources?
8. Does it use ServiceLoader?
9. Does it initialize environment state statically?
10. Is it needed in request path?

For native-heavy services, library choice is architecture decision.

## 28. Startup vs Throughput

Native image often improves startup.

But throughput story depends on workload.

JVM JIT can optimize hot code over time.

Native AOT compiles ahead of time without the same runtime profiling.

Possible outcomes:

```text
Native startup much faster, memory lower, throughput similar.
Native startup faster, memory lower, peak throughput lower.
Native startup faster, CPU higher.
Native better for short-lived/serverless.
JVM better for long-running high-throughput CPU-heavy service.
```

Measure.

Do not assume.

Metrics:

- startup time,
- readiness time,
- idle RSS,
- RSS under load,
- CPU per request,
- throughput,
- p95/p99 latency,
- GC/native memory behavior,
- image size,
- build time.

## 29. Container and Kubernetes Implications

Native image can improve:

- cold start,
- pod startup,
- scale-out speed,
- memory density,
- serverless latency.

But consider:

- readiness after warmup,
- DB connection burst on fast scale-out,
- cold cache stampede,
- CPU request/limit,
- memory request/limit,
- observability agent overhead,
- TLS/cert resources,
- native debugging,
- image base/distroless compatibility.

Fast startup can create new failure:

```text
50 pods start quickly and all hit DB/Redis/Keycloak at once.
```

Mitigation:

- readiness gating,
- jittered startup,
- connection pool limit,
- cache warmup control,
- pre-stop/drain,
- autoscaling policy.

## 30. Serverless and Native Image

Native image is attractive for serverless because:

- cold start matters,
- memory billing matters,
- process may be short-lived,
- scale-to-zero common.

But serverless also has:

- deployment package constraints,
- runtime platform constraints,
- cold/warm behavior,
- connection reuse constraints,
- external client initialization,
- secret fetch,
- observability overhead.

Native binary helps, but architecture still matters.

## 31. Common Native Image Anti-Patterns

### 31.1 Static Runtime State

```java
static final String SECRET = System.getenv("SECRET");
```

### 31.2 Reflection Everywhere

Dynamic model with no registration.

### 31.3 Serializing Entities

JPA proxies/lazy graph issues.

### 31.4 Random Native Flags

Adding `--initialize-at-run-time=*` without understanding.

### 31.5 No Native Tests

Assuming JVM tests are enough.

### 31.6 Unsupported Library in Hot Path

Choosing library with dynamic behavior and no extension.

### 31.7 Hidden Resource Dependency

Template/config/cert file missing in binary.

### 31.8 Build-Time Secret Leakage

Reading env/secret during build and embedding into binary.

### 31.9 Over-Registering Reflection

Huge binary, hidden design problem.

### 31.10 Treating Native as Always Faster

Not measuring throughput/CPU/tail latency.

## 32. Designing Native-Friendly Application Code

Guidelines:

1. Prefer explicit DTOs.
2. Avoid runtime classpath scanning.
3. Avoid dynamic classloading.
4. Avoid static initialization with runtime state.
5. Avoid reflection-heavy custom frameworks.
6. Use Quarkus extensions.
7. Keep config mapping explicit.
8. Use constructor injection.
9. Keep resource inclusion explicit.
10. Test native early if deployment target is native.
11. Keep external clients simple and supported.
12. Avoid serializing JPA entities.
13. Make polymorphism explicit.
14. Validate TLS/crypto/native resources.
15. Document build-time vs runtime config.

Native-friendly code is also often production-friendly code.

---

## 33. Native Image Suitability Decision

Use native image when:

- cold start matters,
- serverless/scale-to-zero,
- memory density matters,
- many small services,
- short-lived CLI/command mode,
- startup-sensitive autoscaling,
- container cost pressure,
- Quarkus extension ecosystem supports dependencies.

Be cautious when:

- CPU-heavy long-running service,
- heavy dynamic reflection,
- large unsupported legacy library stack,
- runtime plugin architecture,
- many dynamic templates/scripts,
- complex JNI/native libraries,
- build pipeline cannot support native time/resources,
- team lacks native debugging maturity.

Decision matrix:

| Question | If Yes |
|---|---|
| Cold start critical? | native attractive |
| Memory budget tight? | native attractive |
| Service long-running CPU-heavy? | benchmark JVM vs native |
| Many reflection-heavy libs? | native risk |
| Unsupported dynamic plugin system? | native risk |
| Quarkus extensions available? | native easier |
| Need Java 25 native? | align Quarkus/Mandrel version |
| CI capacity limited? | plan native build strategy |

## 34. Implementation Blueprint: Native Readiness Review

Before enabling native:

```text
1. Inventory dependencies.
2. Mark Quarkus-supported extensions.
3. Identify reflection/dynamic classloading.
4. Identify resources/templates/certs.
5. Identify static initializers reading environment.
6. Identify serialization/polymorphism.
7. Identify TLS/crypto/JNI.
8. Build native locally/container.
9. Run native smoke test.
10. Run critical integration tests.
11. Compare JVM vs native performance.
12. Document runtime mode decision.
```

Checklist table:

| Area | Question | Status |
|---|---|---|
| REST DTO | explicit and supported? | |
| Jackson polymorphism | reflection config needed? | |
| Hibernate | Quarkus extension used? | |
| REST client | TLS tested? | |

## 35. Example: Bad vs Good Static Init

### 35.1 Bad

```java
public final class ExternalApiConfig {

    public static final String BASE_URL = System.getenv("EXTERNAL_API_URL");
    public static final String TOKEN = System.getenv("EXTERNAL_API_TOKEN");

    private ExternalApiConfig() {
    }
}
```

Problem:

```text
May be evaluated at build time.
Can embed build environment/token into binary.
Hard to test/change.
```

### 35.2 Good

```java
import io.smallrye.config.ConfigMapping;

@ConfigMapping(prefix = "external.api")
public interface ExternalApiConfig {
    String baseUrl();
    String token();
}
```

Usage:

```java
@ApplicationScoped
public class ExternalApiGateway {

    private final ExternalApiConfig config;

    public ExternalApiGateway(ExternalApiConfig config) {
        this.config = config;
    }

    public void call() {
        String token = config.token();
        // use token at runtime
    }
}
```

Better because:

- config source controlled by Quarkus,
- runtime config can be injected,
- no static secret read,
- testable,
- native-friendly.

## 36. Example: Reflection Risk

Bad:

```java
public Object map(String className, Map<String, Object> values) {
    Class<?> type = Class.forName(className);
    Object instance = type.getDeclaredConstructor().newInstance();

    for (var entry : values.entrySet()) {
        Field field = type.getDeclaredField(entry.getKey());
        field.setAccessible(true);
        field.set(instance, entry.getValue());
    }

    return instance;
}
```

Problems:

- class unknown at build,
- fields unknown,
- constructor metadata needed,
- unsafe mapping,
- security risk.

Better:

```java
public ApplicantDto mapApplicant(Map<String, Object> values) {
    return new ApplicantDto(
            string(values, "name"),
            string(values, "email"),
            string(values, "status")
    );
}
```

Or use compile-time mapper where possible.

## 40. Failure Mode Catalogue

### 40.1 Build Fails

Possible causes:

- unsupported library,
- class init conflict,
- reflection/proxy config,
- missing builder version,
- memory shortage.

### 40.2 Build Succeeds, Startup Fails

Possible causes:

- missing resource,
- runtime config missing,
- TLS/cert issue,
- runtime initialized class fails,
- native library missing.

### 40.3 Startup Succeeds, Endpoint Fails

Possible causes:

- serialization reflection,
- REST client proxy,
- dynamic enum/polymorphism,
- lazy JPA graph,
- validation metadata.

### 40.4 Works Locally, Fails in Container

Possible causes:

- OS library missing,
- truststore path,
- file permissions,
- architecture mismatch,
- locale/timezone,
- environment variables.

### 40.5 Works in JVM, Fails Native Under Load

Possible causes:

- concurrency bug exposed,
- different memory profile,
- TLS/connection pool issue,
- CPU saturation,
- missing fallback path tested only under failure.

## 42. Anti-Pattern Umum

### 42.1 “JVM Works, Native Must Work”

False.

### 42.2 Reading Secrets in Static Fields

Can leak build-time data into binary.

### 42.3 Overusing Reflection Config

Works around symptoms, increases binary and hides design issues.

### 42.4 Ignoring Runtime Init

Static init can freeze wrong state.

### 42.5 No Native Test in CI

Native bug found at release.

### 42.6 Unsupported Library in Critical Path

Native build becomes fragile.

### 42.7 Dynamic Plugin Architecture

Contradicts closed-world assumption unless carefully designed.

### 42.8 No Performance Comparison

Native chosen by assumption, not evidence.

### 42.9 Treating Startup Time as Readiness Time

Fast process start but service not ready.

### 42.10 Missing Resource Files

Templates/certs/messages absent in native executable.

### Latihan 1 — Native Suitability Review

### Latihan 2 — Static Init Audit

Cari masalah dalam code:

public final class AppConstants {

Buat benchmark plan dan decision criteria.

## 44. Ringkasan Invariants

Ingat invariants berikut:

```text
Native image is a different runtime model.
Closed-world assumption means runtime dynamic behavior must be known at build time.
Quarkus fits native because it moves framework work to build time.
GraalVM/Mandrel compiles ahead-of-time to native executable.
Mandrel is tailored to support Quarkus native use cases.
Build-time initialization must not depend on runtime environment.
Runtime secrets must not be read in static initializers.
Reflection/resources/proxies/JNI need explicit support.
Use Quarkus extensions whenever possible.
Native image improves startup and often memory, but throughput must be measured.
Native build success is not native runtime correctness.
JVM tests are necessary but not sufficient for native deployment.
```
