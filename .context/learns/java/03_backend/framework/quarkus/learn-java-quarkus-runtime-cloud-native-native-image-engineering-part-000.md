## 0.1. Posisi Part 0

Part 0 bukan tutorial membuat endpoint `hello`. Part 0 adalah **orientation layer**.

Kalau langsung masuk ke:

```java
@Path("/hello")
public class GreetingResource {
    @GET
    public String hello() {
        return "hello";
    }
}
```

kamu memang bisa menjalankan aplikasi Quarkus, tetapi belum tentu memahami apa yang membuat Quarkus berbeda dari framework Java lain.

Masalah terbesar ketika belajar Quarkus secara dangkal adalah menganggapnya sebagai:

- Spring Boot alternatif,
- Jakarta EE ringan,
- REST framework,
- native image wrapper,
- atau microservice starter kit.

Semua itu sebagian benar, tetapi tidak cukup.

Mental model yang lebih kuat:

> Quarkus adalah runtime engineering platform untuk Java modern yang memindahkan sebanyak mungkin pekerjaan dari runtime ke build time, mengintegrasikan ekosistem Java enterprise ke model cloud-native, reactive-core, container-first, dan native-image-aware.

Artinya, Quarkus harus dipahami dari empat lapisan sekaligus:

1. **Programming model** — CDI, Jakarta REST, Hibernate, security, messaging, config.
2. **Build-time model** — augmentation, indexing, build steps, generated bytecode, extension processing.
3. **Runtime model** — event loop, worker pool, CDI container, lifecycle, datasource, HTTP server, shutdown.
4. **Deployment model** — JVM mode, native mode, container, Kubernetes, probes, secrets, observability, resource envelope.

Tanpa empat lapisan ini, engineer biasanya hanya tahu anotasi dan starter dependency. Dengan empat lapisan ini, engineer bisa mengambil keputusan arsitektural.

## 0.3. Premis Utama Seri Ini

Seri ini berdiri di atas beberapa premis.

### Premis 1 — Quarkus bukan sekadar runtime kecil

Banyak orang menyederhanakan Quarkus menjadi “startup cepat dan memory kecil”. Itu benar, tetapi hanya gejala.

Penyebab teknisnya lebih penting:

- Quarkus melakukan banyak pekerjaan di build time.
- Extension membaca metadata aplikasi saat build.
- CDI container Arc dibootstrap dengan model build-time.
- Banyak reflection/scanning runtime diganti dengan indexing dan generated metadata.
- Native image membutuhkan closed-world assumption.
- Runtime lebih ramping karena banyak keputusan sudah dikunci sebelum aplikasi berjalan.

Konsekuensinya:

- Beberapa konfigurasi tidak bisa diubah bebas saat runtime.
- Dynamic reflection harus dipikirkan lebih awal.
- Library yang normal di JVM belum tentu aman untuk native image.
- Startup lifecycle berbeda dari aplikasi Java klasik.
- Build pipeline menjadi bagian dari runtime behavior.

Dokumentasi Quarkus menyebutkan bahwa beberapa konfigurasi hanya efektif saat build time dan perubahan pada properti tersebut membutuhkan rebuild aplikasi. Ini bukan detail kecil; ini adalah salah satu perbedaan fundamental Quarkus dibanding framework runtime-heavy.

Referensi resmi:

- Quarkus Config Guide — https://quarkus.io/guides/config
- Quarkus Configuration Reference — https://quarkus.io/guides/config-reference

### Premis 2 — Quarkus memakai standar, tetapi tidak identik dengan server Jakarta EE tradisional

Quarkus memanfaatkan model seperti CDI, Jakarta REST, Hibernate, Bean Validation, dan MicroProfile. Namun cara integrasinya berbeda.

Contoh:

- CDI di Quarkus memakai Arc.
- Arc dibootstrap saat build time.
- CDI Portable Extensions tidak dapat digunakan dengan cara yang sama seperti di container CDI tradisional; Quarkus menyediakan extension API sendiri.
- Bean discovery, unused bean removal, synthetic beans, dan build-time augmentation membuat perilaku DI lebih optimal tetapi juga lebih constrained.

Referensi resmi:

- Quarkus CDI Reference — https://quarkus.io/guides/cdi-reference
- Quarkus CDI Integration Guide — https://quarkus.io/guides/cdi-integration

### Premis 3 — Quarkus adalah framework reactive-core, bukan berarti semua aplikasi harus reactive

Quarkus menggunakan reactive engine berbasis Vert.x dan Netty untuk menangani non-blocking I/O. Tetapi ini tidak berarti semua business logic harus ditulis dengan `Uni` dan `Multi`.

Keputusan yang lebih matang:

- Gunakan blocking model ketika domain logic dan persistence blocking lebih sederhana dan cukup efisien.
- Gunakan reactive model ketika pipeline I/O benar-benar non-blocking end-to-end.
- Hindari mencampur blocking call di event loop.
- Gunakan virtual threads sebagai opsi modern untuk mempertahankan gaya imperative pada workload I/O-bound tertentu.
- Ukur dampaknya pada latency, throughput, memory, dan kompleksitas debugging.

Referensi resmi:

- Quarkus Reactive Architecture — https://quarkus.io/guides/quarkus-reactive-architecture
- Quarkus Vert.x Reference — https://quarkus.io/guides/vertx-reference
- Quarkus Virtual Threads — https://quarkus.io/guides/virtual-threads

### Premis 4 — Native image bukan sekadar compile target

Native image bukan hanya mode build:

```bash
./mvnw package -Dnative
```

Native image mengubah asumsi aplikasi:

- classpath tidak diperlakukan seperti JVM biasa,
- reflection perlu metadata,
- resource harus didaftarkan,
- dynamic proxy perlu konfigurasi,
- static initialization menjadi keputusan desain,
- library dengan dynamic classloading bisa bermasalah,
- build menjadi lebih mahal,
- startup lebih cepat,
- memory footprint sering lebih rendah,
- throughput belum tentu selalu lebih tinggi dari JVM warm.

Referensi resmi:

- Building a Native Executable — https://quarkus.io/guides/building-native-image
- Native Reference Guide — https://quarkus.io/guides/native-reference
- Tips for Writing Native Applications — https://quarkus.io/guides/writing-native-applications-tips

### Premis 5 — Quarkus perlu dipelajari sebagai production system, bukan lab framework

Top-tier engineer tidak berhenti di “bisa jalan”. Pertanyaan yang lebih penting:

Referensi resmi:

## 0.4. Realitas Versi: Java 8 sampai Java 25

User meminta seri ini membahas Java dari 8 hingga 25. Untuk Quarkus, ini harus dibaca dengan hati-hati.

### 0.4.1. Java 8

Java 8 penting untuk legacy migration.

Banyak sistem enterprise lama masih punya:

- Java 8,
- Java EE/Jakarta EE lama,
- Spring Boot lama,
- WildFly/WebLogic/WebSphere/Tomcat lama,
- `javax.*`,
- Hibernate 5 atau lebih lama,
- reflection-heavy libraries,
- old JDBC drivers,
- old logging stack,
- XML-heavy integration.

Namun Java 8 bukan baseline realistis untuk Quarkus modern.

Di seri ini Java 8 akan dibahas sebagai:

- sumber migration constraint,
- gaya coding legacy yang perlu diubah,
- library compatibility risk,
- perbedaan language feature,
- perbedaan runtime behavior,
- alasan kenapa migrasi ke Java 17+ sering menjadi prasyarat.

### 0.4.2. Java 11

Java 11 adalah baseline transisi penting untuk banyak aplikasi modern sebelum Java 17.

Tetapi untuk Quarkus 3 modern, Java 11 bukan target utama.

Java 11 akan dibahas untuk:

- memahami migration path dari Quarkus lama,
- dependency modernization,
- module boundary awareness,
- TLS/security provider differences,
- GC/container awareness improvement dibanding Java 8.

### 0.4.3. Java 17

Java 17 adalah baseline modern penting untuk Quarkus 3.

Quarkus 3 menargetkan minimal Java 17 dan merekomendasikan Java 21 pada konteks rilis Quarkus 3 awal.

Referensi resmi:

- Quarkus 3 landing page — https://quarkus.io/quarkus3/

Dampak Java 17:

- records bisa dipakai untuk DTO/config/test data,
- sealed classes bisa membantu domain modeling tertentu,
- pattern matching mulai relevan,
- JVM/container ergonomics lebih baik,
- baseline `jakarta.*` lebih cocok,
- banyak library enterprise modern mulai menganggap Java 17 sebagai minimum.

### 0.4.4. Java 21

Java 21 adalah LTS besar untuk Quarkus modern.

Fitur penting:

- virtual threads,
- sequenced collections,
- record patterns evolution,
- better JVM maturity,
- long-term support adoption enterprise.

Untuk Quarkus, Java 21 penting karena virtual threads membuka pendekatan baru:

> mempertahankan gaya imperative/blocking yang lebih sederhana sambil mengurangi biaya platform thread untuk workload I/O-bound.

Namun virtual threads bukan pengganti semua reactive programming.

Virtual threads cocok ketika:

- workload dominan I/O-bound,
- kode blocking lebih mudah dipahami,
- library blocking harus tetap dipakai,
- simplicity lebih penting daripada pipeline reactive penuh.

Reactive tetap kuat ketika:

- pipeline benar-benar asynchronous end-to-end,
- backpressure eksplisit dibutuhkan,
- streaming/event-driven flow dominan,
- integrasi reactive sudah native.

Referensi resmi:

- Quarkus Virtual Threads — https://quarkus.io/guides/virtual-threads

### 0.4.5. Java 25

Java 25 masuk penting karena Quarkus rilis terbaru sudah menambahkan dukungan penuh Java 25, termasuk runtime image dan native image build dengan Mandrel.

Referensi resmi:

- Quarkus 3.31 release — https://quarkus.io/blog/quarkus-3-31-released/
- Mandrel 25 release — https://quarkus.io/blog/mandrel-25-released/

Dalam seri ini Java 25 akan diposisikan sebagai:

- target modern terbaru,
- area compatibility testing,
- native image evolution,
- runtime image modernization,
- bukan baseline yang otomatis wajib untuk semua enterprise system.

### 0.4.6. Ringkasan strategi versi

| Versi Java | Posisi dalam seri | Sikap praktis |
|---|---|---|
| Java 8 | Legacy source | Dipahami untuk migration, bukan target utama Quarkus modern |
| Java 11 | Transitional runtime | Relevan untuk upgrade path dan dependency modernization |
| Java 17 | Modern baseline | Aman sebagai baseline Quarkus 3 modern |
| Java 21 | Modern LTS kuat | Sangat relevan untuk virtual threads dan production adoption |
| Java 25 | Latest modern target | Dibahas untuk forward-looking engineering dan compatibility |

Ingat invariant ini:

> Quarkus seri ini tidak mengajarkan “satu kode berjalan sama idealnya di Java 8 sampai 25”. Yang diajarkan adalah bagaimana engineer top-tier memahami evolusi constraint dari Java 8 legacy menuju Quarkus modern berbasis Java 17/21/25.

## 0.5. Apa Itu Quarkus dalam Satu Model Besar

Bayangkan aplikasi Java klasik seperti ini:

```text
source code
   |
   v
compile
   |
   v
jar/war
   |
   v
runtime starts
   |
   +--> scan annotations
   +--> discover beans
   +--> inspect classes
   +--> build metadata
   +--> initialize framework
   +--> wire dependencies
   +--> start server
   +--> handle request
```

Banyak pekerjaan terjadi saat runtime.

Quarkus memindahkan banyak pekerjaan itu ke build time:

```text
source code
   |
   v
compile + quarkus augmentation
   |
   +--> index classes
   +--> inspect annotations
   +--> discover beans
   +--> analyze extensions
   +--> generate bytecode/metadata
   +--> prepare runtime init
   +--> prepare native image metadata
   |
   v
optimized application artifact
   |
   v
runtime starts
   |
   +--> run precomputed initialization
   +--> start only needed services
   +--> handle request
```

Dampaknya:

- startup lebih cepat,
- runtime memory bisa lebih kecil,
- classpath scanning runtime berkurang,
- native image lebih feasible,
- konfigurasi build-time menjadi penting,
- extension menjadi komponen inti,
- dynamic behavior perlu dikontrol.

Ini mental model pertama yang harus melekat.

## 0.6. Empat Mesin Utama Quarkus

Quarkus bisa dipahami sebagai gabungan dari empat “mesin”.

```text
+-------------------------------------------------------------+
|                        Quarkus App                          |
+-------------------------------------------------------------+
|  Programming Model                                          |
|  CDI / REST / Hibernate / Config / Security / Messaging     |
+-------------------------------------------------------------+
|  Build-Time Augmentation                                    |
|  Jandex / Build Steps / Build Items / Recorders / Extensions|
+-------------------------------------------------------------+
|  Runtime Engine                                             |
|  Arc / Vert.x / Netty / Agroal / Narayana / SmallRye        |
+-------------------------------------------------------------+
|  Deployment Target                                          |
|  JVM / Native / Container / Kubernetes / Serverless          |
+-------------------------------------------------------------+
```

### Mesin 1 — Programming Model

Ini bagian yang paling terlihat oleh developer:

- `@Path`,
- `@GET`,
- `@Inject`,
- `@ApplicationScoped`,
- `@Transactional`,
- `@ConfigMapping`,
- `@RolesAllowed`,
- `Uni<T>`,
- `Multi<T>`,
- `@Incoming`,
- `@Outgoing`.

Tetapi anotasi bukan inti Quarkus. Anotasi hanya interface menuju mesin build-time dan runtime.

### Mesin 2 — Build-Time Augmentation

Ini salah satu inti Quarkus.

Extension Quarkus memproses aplikasi saat build:

- membaca bytecode,
- membaca annotation,
- membaca config,
- membuat build item,
- mendaftarkan bean,
- menyiapkan reflection metadata,
- menghasilkan bytecode,
- mengatur init-time decision.

Dokumentasi extension Quarkus menyebut extension terdiri dari dua bagian besar: build-time augmentation dan runtime container. Bagian augmentation memproses metadata seperti annotation dan descriptor, lalu menghasilkan bytecode yang akan menjalankan runtime services.

Referensi resmi:

- Writing Your Own Extension — https://quarkus.io/guides/writing-extensions
- Building My First Extension — https://quarkus.io/guides/building-my-first-extension

### Mesin 3 — Runtime Engine

Runtime engine Quarkus mencakup:

- Arc untuk CDI,
- Vert.x/Netty untuk reactive HTTP/I/O,
- Agroal untuk JDBC pooling,
- Narayana untuk transaction,
- SmallRye untuk MicroProfile ecosystem,
- Hibernate ORM/Reactive,
- security runtime,
- scheduler,
- messaging connectors,
- OpenTelemetry/Micrometer.

Runtime ini tidak berdiri sendiri seperti server aplikasi tradisional. Banyak bagiannya sudah disiapkan oleh build-time augmentation.

### Mesin 4 — Deployment Target

Quarkus didesain untuk deployment modern:

- JVM jar,
- fast-jar,
- mutable jar/reaugmentation,
- native executable,
- container image,
- Kubernetes/OpenShift,
- serverless/cloud function style,
- local dev services.

Top-tier engineer harus melihat build artifact sebagai bagian dari desain.

Pertanyaan penting:

- Apakah kita build JVM jar atau native?
- Apakah image punya secret di dalamnya?
- Apakah config build-time sudah benar?
- Apakah native binary compatible dengan target OS/container?
- Apakah startup probe cukup?
- Apakah readiness bergantung pada DB/external service?
- Apakah shutdown graceful aktif?

## 0.11. Quarkus vs Framework Java Lain: Cara Membandingkan yang Benar

Perbandingan framework sering dangkal:

```text
Spring Boot vs Quarkus
Quarkus vs Micronaut
Quarkus vs Jakarta EE
Quarkus vs Helidon
```

Perbandingan yang lebih benar harus memakai dimensi.

### Dimensi 1 — Kapan framework melakukan pekerjaan?

```text
Runtime-heavy:
  banyak scanning, reflection, wiring saat aplikasi start/run

Build-time-heavy:
  banyak scanning, metadata processing, wiring decision saat build
```

Quarkus sangat condong ke build-time-heavy.

### Dimensi 2 — Apa programming model-nya?

Quarkus mendukung banyak standar Java enterprise:

- CDI,
- Jakarta REST,
- Hibernate,
- Bean Validation,
- MicroProfile APIs,
- SmallRye ecosystem.

Jadi ia tidak memutus total dari dunia Java enterprise.

### Dimensi 3 — Bagaimana execution model-nya?

Quarkus punya reactive core berbasis Vert.x/Netty. Tetapi aplikasi bisa tetap blocking, reactive, atau hybrid.

### Dimensi 4 — Bagaimana native image readiness-nya?

Quarkus didesain sejak awal untuk native image. Banyak extension bertugas menyembunyikan kompleksitas native metadata dari developer.

### Dimensi 5 — Bagaimana deployment model-nya?

Quarkus sangat cocok untuk container/Kubernetes/serverless/cold-start-sensitive workload.

### Dimensi 6 — Bagaimana ecosystem dan governance-nya?

Quarkus punya platform BOM dan extension catalog. Dalam enterprise, governance extension dan dependency alignment sangat penting.

Kesimpulan:

> Quarkus tidak otomatis “lebih baik” dari framework lain. Quarkus unggul ketika constraint sistem menghargai startup cepat, resource kecil, container-native lifecycle, build-time optimization, dan native readiness. Quarkus menjadi kurang ideal jika aplikasi sangat bergantung pada dynamic runtime behavior yang sulit dianalisis saat build.

## 0.12. Core Invariants Quarkus

Invariant adalah aturan yang cenderung tetap benar walaupun detail API berubah.

### Invariant 1 — Build time adalah bagian dari runtime behavior

Di Quarkus, build bukan hanya compile.

Build menentukan:

- extension mana aktif,
- bean mana ditemukan,
- bean mana dibuang,
- metadata native image,
- resource inclusion,
- config build-time,
- generated bytecode,
- runtime init strategy.

Implikasi:

> CI/CD pipeline adalah bagian dari desain aplikasi Quarkus.

### Invariant 2 — Tidak semua konfigurasi sama

Ada konfigurasi yang:

- bisa berubah saat runtime,
- hanya efektif saat build time,
- tersedia runtime tetapi read-only,
- membutuhkan rebuild/reaugmentation.

Implikasi:

> Jangan menyamakan environment variable Quarkus dengan environment variable aplikasi runtime-heavy biasa.

### Invariant 3 — Dynamic behavior harus disengaja

Dynamic classloading, reflection, proxy, scanning, dan resource lookup perlu dipikirkan lebih awal, terutama untuk native image.

Implikasi:

> Library yang “jalan di JVM” belum tentu “jalan di native”.

### Invariant 4 — Reactive core tidak berarti semua kode harus reactive

Quarkus bisa menjalankan blocking endpoint, reactive endpoint, virtual-thread endpoint, dan messaging pipeline.

Implikasi:

> Execution model harus dipilih berdasarkan workload, bukan mode framework.

### Invariant 5 — Extension adalah unit integrasi utama

Extension bukan hanya dependency tambahan. Extension adalah cara library berintegrasi dengan build-time augmentation dan native image.

Implikasi:

> Jika library penting tidak punya Quarkus extension, risiko integration/native/runtime behavior harus dinilai lebih serius.

### Invariant 6 — Native image adalah constraint desain, bukan tahap akhir

Native readiness harus dipikirkan sejak:

- memilih library,
- memilih serialization approach,
- memakai reflection,
- mengatur static initialization,
- menulis tests,
- membuat build pipeline.

Implikasi:

> “Nanti saja coba native build di akhir” sering menjadi strategi mahal.

### Invariant 7 — Production lifecycle harus eksplisit

Startup, readiness, liveness, shutdown, health, metrics, tracing, dan logs harus didesain.

Implikasi:

> Aplikasi yang bisa menerima HTTP request belum tentu production-ready.

## 0.15. Mental Model: JVM Mode vs Native Mode

Quarkus bisa dijalankan dalam JVM mode atau native mode.

### JVM mode

Karakteristik:

- startup lebih cepat daripada banyak framework tradisional, tetapi masih JVM,
- warmup JIT bisa meningkatkan throughput,
- dynamic behavior lebih fleksibel,
- debugging lebih familiar,
- build lebih cepat dibanding native,
- memory bisa lebih besar dibanding native,
- cocok untuk service long-running dengan throughput tinggi.

### Native mode

Karakteristik:

- startup sangat cepat,
- memory footprint sering lebih kecil,
- cocok untuk cold start, serverless, scale-to-zero, resource-constrained environment,
- build lebih mahal,
- closed-world assumption,
- reflection/resource/proxy harus jelas,
- debugging bisa lebih sulit,
- peak throughput belum tentu lebih baik dari JVM warm.

### Keputusan praktis

Jangan memakai native hanya karena keren.

Gunakan native jika:

- cold start penting,
- memory per instance sangat dibatasi,
- scale-to-zero atau serverless penting,
- jumlah service sangat banyak dan resource cost signifikan,
- workload tidak membutuhkan dynamic runtime behavior rumit,
- native compatibility sudah terbukti di pipeline.

Gunakan JVM jika:

- service long-running,
- throughput warm penting,
- library dynamic-heavy,
- build time native terlalu mahal,
- debugging/profiling JVM lebih penting,
- cold start bukan bottleneck.

Invariant:

> Native image adalah opsi deployment, bukan bukti kualitas arsitektur. Arsitektur yang baik harus bisa menjelaskan kenapa memilih JVM atau native.

## 0.16. Mental Model: Blocking, Reactive, dan Virtual Threads

Quarkus modern memberi tiga gaya besar.

### Blocking/imperative

Contoh:

```java
@GET
@Path("/{id}")
public CaseDto getCase(@PathParam("id") UUID id) {
    return service.getCase(id);
}
```

Cocok untuk:

- domain logic kompleks,
- JDBC/Hibernate ORM blocking,
- tim yang lebih nyaman imperative,
- transaction boundary eksplisit,
- debugging sederhana.

Risiko:

- thread pool exhaustion,
- blocking call lambat,
- timeout tidak disiplin,
- dependency failure menahan worker.

### Reactive

Contoh:

```java
@GET
@Path("/{id}")
public Uni<CaseDto> getCase(@PathParam("id") UUID id) {
    return service.getCase(id);
}
```

Cocok untuk:

- non-blocking I/O end-to-end,
- streaming,
- high-concurrency I/O,
- messaging pipeline,
- reactive DB client.

Risiko:

- debugging lebih sulit,
- context propagation tricky,
- blocking call di event loop berbahaya,
- transaction semantics berbeda,
- pipeline error handling sering disalahpahami.

### Virtual threads

Contoh konseptual:

```java
@GET
@Path("/{id}")
@RunOnVirtualThread
public CaseDto getCase(@PathParam("id") UUID id) {
    return service.getCase(id);
}
```

Cocok untuk:

- mempertahankan kode blocking sederhana,
- workload I/O-bound,
- mengurangi biaya platform thread,
- migration dari imperative codebase.

Risiko:

- pinning,
- synchronized/blocking native call,
- ThreadLocal assumptions,
- connection pool tetap bottleneck,
- tidak otomatis menyelesaikan backpressure.

Invariant:

> Virtual threads mengurangi biaya thread, bukan menghilangkan biaya database connection, lock, remote call, memory, atau retry storm.

## 0.18. Kesalahan Umum Saat Belajar Quarkus

### Kesalahan 1 — Menganggap Quarkus sama seperti Spring Boot dengan anotasi berbeda

Akibat:

- salah memahami build-time config,
- salah memakai reflection,
- salah ekspektasi dynamic runtime behavior,
- salah debugging bean discovery.

Perbaikan:

- pahami augmentation,
- pahami extension model,
- pahami Arc/CDI build-time behavior.

### Kesalahan 2 — Langsung mengejar native image

Akibat:

- aplikasi JVM belum bersih tapi dipaksa native,
- error reflection/resource muncul terlambat,
- build pipeline lambat,
- root cause sulit dilacak.

Perbaikan:

- mulai dengan JVM correctness,
- buat native compatibility checklist sejak awal,
- jalankan native test di pipeline terpisah,
- nilai library sebelum adopsi.

### Kesalahan 3 — Reactive everywhere

Akibat:

- kode lebih kompleks,
- blocking call tersembunyi,
- transaction sulit,
- error handling tidak konsisten,
- debugging mahal.

Perbaikan:

- pilih reactive hanya jika pipeline I/O non-blocking end-to-end,
- gunakan blocking/virtual threads jika lebih masuk akal,
- ukur dengan workload nyata.

### Kesalahan 4 — Mengabaikan build-time vs runtime config

Akibat:

- config diubah di Kubernetes tetapi tidak berpengaruh,
- environment promotion kacau,
- image harus rebuild tanpa direncanakan,
- incident sulit dilacak.

Perbaikan:

- tandai config build-time,
- dokumentasikan config contract,
- pisahkan build artifact dan runtime deployment config.

### Kesalahan 5 — Mengandalkan Dev Services tanpa memahami production dependency

Dev Services bagus untuk inner loop. Tetapi production dependency tetap harus didesain.

Risiko:

- local test terlalu nyaman,
- auth/db/messaging production behavior tidak sama,
- lifecycle container test tidak mewakili managed service,
- performance/timeout berbeda.

Perbaikan:

- gunakan Dev Services untuk productivity,
- tetap punya integration profile yang mendekati production,
- dokumentasikan perbedaan local vs prod.

### Kesalahan 6 — Menganggap Panache selalu best practice

Panache mempercepat CRUD dan query sederhana. Tetapi untuk domain kompleks, active record bisa mencampur domain, persistence, dan application boundary.

Perbaikan:

- gunakan Panache untuk context yang cocok,
- pakai repository/domain-centric model untuk workflow kompleks,
- jangan biarkan entity menjadi god object.

### Kesalahan 7 — Tidak menguji shutdown dan readiness

Banyak service hanya diuji saat startup. Padahal production failure sering terjadi saat:

- rolling update,
- pod termination,
- DB down,
- Kafka lag,
- Keycloak unreachable,
- config missing,
- slow dependency.

Perbaikan:

- desain readiness/liveness/startup probe,
- aktifkan graceful shutdown jika perlu,
- test SIGTERM behavior,
- test dependency failure.

## 0.22. Learning Contract untuk Seri Ini

Agar hasilnya tajam, setiap part akan mengikuti kontrak ini.

### 1. Problem-first

Setiap part dimulai dari problem engineering, bukan langsung API.

Contoh buruk:

> “Quarkus punya `@ConfigMapping`.”

Contoh baik:

> “Service butuh configuration contract yang type-safe, tervalidasi, aman untuk environment, dan jelas mana yang build-time vs runtime. Quarkus menyediakan SmallRye Config dan config mapping untuk menyelesaikan ini.”

### 2. Mental model before syntax

Sebelum kode, pahami model.

Contoh:

- CDI bukan hanya `@Inject`, tetapi bean lifecycle dan build-time discovery.
- Native bukan hanya `-Dnative`, tetapi closed-world assumption.
- Reactive bukan hanya `Uni`, tetapi execution and backpressure model.

### 3. Mechanism before pattern

Kita tidak hanya menghafal pattern.

Untuk setiap pattern, tanyakan:

- mekanismenya apa,
- constraint-nya apa,
- failure mode-nya apa,
- kapan tidak cocok,
- bagaimana mengujinya.

### 4. Production before demo

Kode demo harus membawa production concern minimal:

- timeout,
- error mapping,
- logging,
- metrics,
- validation,
- transaction,
- test,
- shutdown behavior.

### 5. Decision table

Setiap topik penting akan punya decision table.

Contoh:

| Keputusan | Pilihan A | Pilihan B | Kapan memilih |
|---|---|---|---|
| Runtime | JVM | Native | Berdasarkan cold start, memory, throughput, compatibility |
| Endpoint | Blocking | Reactive | Berdasarkan I/O model dan complexity |
| Persistence | ORM | Reactive SQL | Berdasarkan transaction/domain/query model |

### 6. Failure model

Setiap part harus membahas failure.

Contoh:

- config salah,
- bean tidak ditemukan,
- native image gagal,
- DB pool habis,
- event loop blocking,
- token expired,
- retry storm,
- health check false positive.

### 7. Invariants

Setiap part ditutup dengan invariants.

Tujuannya agar kamu mengingat prinsip, bukan hanya syntax.

## 0.24. Peta Risiko Saat Mengadopsi Quarkus di Enterprise

### Risiko 1 — Migration `javax.*` ke `jakarta.*`

Quarkus 3 bergerak ke Jakarta namespace.

Risiko:

- dependency lama masih memakai `javax.*`,
- generated code lama,
- JAXB/JAX-RS/validation import mismatch,
- library custom belum kompatibel.

Mitigasi:

- inventory dependency,
- static analysis import,
- upgrade library,
- test regression,
- isolate adapter legacy.

### Risiko 2 — Native image compatibility

Risiko:

- reflection runtime gagal,
- resource tidak masuk binary,
- dynamic proxy gagal,
- serialization library bermasalah,
- crypto/TLS provider issue,
- image build lambat.

Mitigasi:

- native compatibility checklist,
- pilih extension resmi jika ada,
- jalankan native test sejak awal,
- hindari library dynamic-heavy tanpa alasan kuat.

### Risiko 3 — Reactive misuse

Risiko:

- event loop blocking,
- stack trace sulit,
- transaction leak,
- backpressure tidak dipahami,
- thread starvation.

Mitigasi:

- pisahkan blocking dan non-blocking path,
- gunakan `@Blocking` jika perlu,
- profiling event loop,
- gunakan virtual threads jika lebih sederhana.

### Risiko 4 — Config drift

Risiko:

- config build-time diubah runtime,
- environment tidak reproducible,
- artifact rebuild diam-diam,
- prod berbeda dari staging.

Mitigasi:

- config catalog,
- build-time config review,
- immutable artifact promotion,
- environment contract.

### Risiko 5 — Extension version drift

Risiko:

- dependency tidak align dengan Quarkus platform,
- transitive conflict,
- native metadata mismatch,
- runtime bug sulit dilacak.

Mitigasi:

- pakai platform BOM,
- audit dependency,
- upgrade terjadwal,
- internal compatibility matrix.

### Risiko 6 — Observability kurang

Risiko:

- startup cepat tetapi incident sulit dianalisis,
- native binary minim visibility,
- logs tidak structured,
- traces tidak propagate,
- metrics tidak actionable.

Mitigasi:

- observability standard sejak template project,
- correlation ID,
- OpenTelemetry,
- health check meaningful,
- runbook.

## 0.28. Latihan Awal: Cara Berpikir Quarkus

Sebelum menulis kode, coba jawab pertanyaan ini.

### Latihan 1 — Build-time vs runtime

Sebuah aplikasi punya config:

```properties
quarkus.datasource.db-kind=postgresql
quarkus.datasource.jdbc.url=jdbc:postgresql://db/prod
```

Pertanyaan:

1. Mana yang kemungkinan build-time?
2. Mana yang runtime?
3. Apa risiko jika image yang sama dipakai untuk environment berbeda?
4. Apa strategi config yang aman?

### Latihan 2 — Native compatibility

Sebuah library membaca semua class di package tertentu saat runtime memakai reflection.

Pertanyaan:

1. Kenapa ini normal di JVM tetapi berisiko di native image?
2. Apa yang perlu disediakan agar native image bisa bekerja?
3. Apakah lebih baik mencari Quarkus extension resmi?
4. Apa test yang harus dibuat?

### Latihan 3 — Reactive or blocking

Endpoint melakukan:

1. validasi request,
2. baca user dari JWT,
3. query DB via Hibernate ORM,
4. panggil external REST API,
5. tulis audit trail,
6. return response.

Pertanyaan:

1. Apakah endpoint ini harus reactive?
2. Apa risiko blocking call?
3. Apakah virtual thread membantu?
4. Timeout mana yang harus lebih kecil: inbound request timeout atau outbound client timeout?
5. Bagaimana transaction boundary-nya?

### Latihan 4 — Production lifecycle

Pod Quarkus menerima SIGTERM saat rolling update, sementara ada request enforcement action sedang diproses.

Pertanyaan:

1. Apakah request boleh diputus?
2. Apakah graceful shutdown aktif?
3. Apakah state transition atomic?
4. Apakah outbox event sudah tertulis?
5. Apakah client akan retry?
6. Apakah retry idempotent?

Latihan-latihan ini akan kembali berulang sepanjang seri.

## 0.30. Ringkasan Invariants Part 0

Pegang prinsip berikut sebelum lanjut:

1. **Quarkus adalah build-time optimized runtime platform, bukan hanya REST framework.**
2. **Build pipeline adalah bagian dari behavior aplikasi.**
3. **Tidak semua config bisa diubah saat runtime.**
4. **Extension adalah mekanisme integrasi utama Quarkus.**
5. **CDI di Quarkus diproses dengan constraint build-time.**
6. **Reactive core tidak berarti semua business logic harus reactive.**
7. **Virtual threads adalah opsi execution model, bukan solusi semua bottleneck.**
8. **Native image adalah constraint desain sejak awal, bukan tombol build di akhir.**
9. **JVM mode dan native mode harus dipilih berdasarkan workload dan operability.**
10. **Production readiness mencakup startup, shutdown, health, observability, tests, security, dan upgrade governance.**
11. **Java 8 relevan sebagai migration source; Java 17/21/25 relevan sebagai Quarkus modern target.**
12. **Top-tier Quarkus engineer memahami mekanisme, trade-off, dan failure mode — bukan hanya anotasi.**
