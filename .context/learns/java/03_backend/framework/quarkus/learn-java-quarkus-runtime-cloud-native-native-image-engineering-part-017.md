## 1. Mental Model: Security Sebagai Rantai, Bukan Filter

Security Quarkus production-grade harus dilihat sebagai rantai:

```text
source code
  -> dependencies
  -> build plugins
  -> generated bytecode
  -> native metadata
  -> container image
  -> SBOM
  -> signing / provenance
  -> deployment manifest
  -> config / secret injection
  -> TLS / mTLS transport
  -> authentication
  -> authorization
  -> crypto usage
  -> logs / traces / audit
  -> runtime patching / rotation
  -> incident response
```

Kekuatan security sistem hanya sekuat link terlemahnya.

Contoh:

- Authorization endpoint kuat, tetapi REST client memakai `trust-all=true` ke payment gateway. Hasilnya MITM risk.
- Token validation kuat, tetapi OIDC client secret masuk ke Git repository. Hasilnya credential compromise.
- RBAC lengkap, tetapi audit log memuat NRIC/token/password. Hasilnya data exposure.
- Container image kecil, tetapi tidak punya SBOM. Hasilnya vulnerability response lambat.
- Native image cepat, tetapi certificate truststore salah. Hasilnya production outage saat calling external API.

Quarkus membantu banyak hal:

- TLS Registry untuk sentralisasi konfigurasi TLS.
- Secret configuration handler.
- Credentials provider.
- OIDC/TLS integration.
- Native image integration.
- CycloneDX SBOM extension.
- Kubernetes/container-image integration.

Tetapi framework tidak bisa menggantikan governance. Quarkus memberi mechanism; engineer menentukan boundary dan operating model.

## 4. TLS Dalam Quarkus: Inbound dan Outbound

TLS di aplikasi Quarkus muncul di dua arah:

1. **Inbound TLS**  
   Client/service lain memanggil Quarkus service melalui HTTPS.

2. **Outbound TLS**  
   Quarkus service memanggil external/internal dependency melalui HTTPS, OIDC, database, messaging, REST client, dan lain-lain.

Masalah umum terjadi ketika engineer hanya mengamankan inbound, tetapi outbound dibiarkan lemah.

```text
Browser/API Gateway -> Quarkus service       : inbound TLS
Quarkus service -> OIDC provider             : outbound TLS
Quarkus service -> REST dependency           : outbound TLS
Quarkus service -> database                  : outbound TLS/mTLS possible
Quarkus service -> Kafka/RabbitMQ            : outbound TLS/mTLS possible
```

### 4.1 Inbound TLS Minimal

Contoh konfigurasi inbound HTTPS dengan certificate file:

```properties
quarkus.http.ssl.certificate.files=/mnt/certs/tls.crt
quarkus.http.ssl.certificate.key-files=/mnt/certs/tls.key
quarkus.http.insecure-requests=disabled
```

Maknanya:

- service hanya menerima HTTPS,
- private key tidak disimpan di repository,
- certificate di-mount dari secret volume,
- HTTP plain ditolak.

Namun pada Kubernetes modern, TLS sering di-terminate di ingress/API gateway/mesh. Dalam kondisi ini, service internal mungkin menerima HTTP dari ingress. Itu bisa diterima hanya jika:

- network boundary jelas,
- ingress melakukan TLS termination secara konsisten,
- internal traffic policy diatur,
- header seperti `X-Forwarded-*` tidak dipercaya sembarangan,
- tidak ada direct bypass ke Pod/Service dari network tidak tepercaya.

### 4.2 Certificate Reload

Quarkus HTTP reference mendukung reload certificate secara periodik ketika certificate/key file berubah. Ini penting untuk certificate rotation tanpa restart aplikasi.

Contoh:

```properties
quarkus.http.ssl.certificate.files=/mount/certs/tls.crt
quarkus.http.ssl.certificate.key-files=/mount/certs/tls.key
quarkus.http.ssl.certificate.reload-period=1h
```

Design implication:

- certificate manager bisa update file,
- Quarkus reload secara periodik,
- restart Pod tidak selalu diperlukan,
- outage window lebih kecil.

Tetapi reload bukan alasan untuk tidak punya deployment rollback. Certificate rotation tetap perlu:

- staging validation,
- expiry monitoring,
- alert sebelum expired,
- dual-trust window,
- audit evidence.

## 5. TLS Registry: Centralized TLS Configuration

Quarkus menyediakan **TLS Registry** untuk menyentralisasi konfigurasi TLS. Tujuannya adalah menghindari konfigurasi certificate/truststore yang tersebar di banyak extension.

Tanpa TLS Registry:

```text
REST client A punya truststore sendiri
REST client B punya truststore sendiri
OIDC client punya TLS config sendiri
Kafka punya TLS config sendiri
Management endpoint punya TLS config sendiri
```

Masalah:

- duplication,
- inconsistent hostname verification,
- trust-all accidentally enabled,
- rotation sulit,
- audit sulit,
- configuration drift.

Dengan TLS Registry:

```text
quarkus.tls.internal-services.*
quarkus.tls.external-government-api.*
quarkus.tls.oidc-provider.*

REST client -> reference tls bucket
OIDC       -> reference tls bucket
management -> reference tls bucket
```

Dokumentasi Quarkus menyebut TLS Registry sebagai extension untuk memusatkan konfigurasi TLS agar komponen aplikasi dapat menggunakan konfigurasi yang konsisten.

### 5.1 Contoh Konsep TLS Bucket

```properties
quarkus.tls.gov-api.trust-store.p12.path=/mnt/trust/gov-api-truststore.p12
quarkus.tls.gov-api.trust-store.p12.password=${TRUSTSTORE_PASSWORD}

quarkus.rest-client.gov-api.tls-configuration-name=gov-api
```

Makna arsitektural:

- nama `gov-api` menjadi identity dari trust policy,
- REST client tidak mengulang detail truststore,
- perubahan trust material bisa dikontrol di satu area,
- audit lebih mudah.

### 5.2 TLS Registry Untuk OIDC

OIDC provider sering dipanggil melalui HTTPS. Dalam beberapa enterprise, OIDC provider juga membutuhkan mTLS client authentication.

Quarkus OIDC configuration dapat mereferensikan TLS Registry configuration dengan property seperti:

```properties
quarkus.oidc.tls-configuration-name=oidc-provider
```

Ini berguna ketika:

- OIDC provider memakai private CA,
- OIDC provider memerlukan client certificate,
- Quarkus harus memvalidasi certificate chain non-public,
- environment berbeda punya CA berbeda.

### 5.3 Rule Penting TLS Registry

Gunakan TLS Registry untuk policy yang punya lifecycle:

- truststore,
- keystore,
- mTLS material,
- certificate reload,
- hostname verification,
- named trust boundary.

Jangan menyebar `trust-all` atau disable hostname verification di masing-masing client.

## 6. mTLS: Ketika TLS Juga Menjadi Service Identity

TLS biasa menjawab:

> Apakah server yang saya hubungi valid?

mTLS menjawab:

> Apakah server valid, dan apakah client juga bisa membuktikan identitasnya dengan certificate?

Dalam service-to-service communication, mTLS berguna untuk:

- internal API high sensitivity,
- OIDC backchannel/token endpoint dengan client certificate,
- database connection yang memerlukan client certificate,
- Kafka/RabbitMQ client authentication,
- east-west traffic protection,
- zero-trust network model.

### 6.1 mTLS vs JWT Service Token

Keduanya tidak saling menggantikan sepenuhnya.

| Mechanism | Membuktikan | Cocok Untuk | Kelemahan |
|---|---|---|---|
| mTLS | Identitas workload/channel | transport-level service identity | certificate lifecycle kompleks |
| JWT client credentials | Identitas aplikasi/caller di application layer | authorization, delegation, scope | token theft/replay risk jika channel lemah |
| mTLS + JWT | workload identity + app permission | high-trust enterprise integration | governance lebih berat |

Top-tier design sering memakai kombinasi:

```text
mTLS verifies: this workload is allowed to connect
JWT verifies : this caller has permission/scope/action
```

### 6.2 mTLS Failure Mode

Common failure:

1. Certificate expired.
2. Truststore belum update CA baru.
3. Hostname mismatch.
4. Client certificate tidak dikirim.
5. Intermediate CA missing.
6. Private key format salah.
7. Keystore password salah.
8. Native image truststore berbeda dari JVM mode.
9. TLS termination di proxy menghilangkan client cert context.
10. Certificate rotation tidak punya overlap window.

Operational checklist:

- monitor certificate expiry,
- support dual CA during rotation,
- use named TLS configuration,
- test mTLS in CI/integration environment,
- log TLS failure reason tanpa membocorkan key/secret,
- document owner dari setiap certificate.

## 7. Jangan Gunakan `trust-all` Di Production

Quarkus REST Client documentation memberi contoh `trust-all` dan disable hostname verification untuk use case tertentu, tetapi explicitly menyatakan konfigurasi seperti itu tidak boleh digunakan di production.

Contoh yang berbahaya:

```properties
quarkus.tls.tls-disabled.trust-all=true
quarkus.tls.tls-disabled.hostname-verification-algorithm=NONE
```

Ini biasanya muncul karena:

- local certificate self-signed,
- private CA belum dipasang,
- certificate mismatch,
- developer ingin cepat jalan,
- testing environment tidak rapi.

Masalahnya, sekali masuk config template, sering terbawa ke production.

Rule:

```text
trust-all boleh hanya untuk local disposable dev,
dengan profile dev eksplisit,
dan harus dicegah oleh CI/config policy agar tidak masuk prod.
```

Contoh guard:

```properties
%dev.quarkus.tls.local-dev.trust-all=true
%prod.quarkus.tls.local-dev.trust-all=false
```

Tetapi guard yang lebih baik adalah tidak memiliki property production dengan nama `local-dev` sama sekali.

## 8. Secret Lifecycle: Dari Source of Truth Sampai Runtime

Secret bukan hanya value. Secret punya lifecycle.

```text
creation
  -> storage
  -> distribution
  -> injection
  -> usage
  -> masking
  -> rotation
  -> revocation
  -> audit
  -> destruction
```

Contoh secret:

- database password,
- OIDC client secret,
- JWT signing key,
- REST API token,
- truststore password,
- keystore password,
- encryption key,
- webhook secret,
- cloud credential.

### 8.1 Anti-Pattern Secret

```text
- secret di application.properties committed ke Git
- secret di Dockerfile ENV
- secret di container image layer
- secret di CI log
- secret di exception message
- secret di trace baggage
- secret di audit metadata
- secret di test fixture public repository
- secret di generated OpenAPI example
- secret di Dev UI exposed environment
```

### 8.2 Better Secret Flow

```text
Secret Manager / Vault / K8s Secret / AWS SSM
  -> mounted file or env var
  -> Quarkus config source
  -> injected into component that needs it
  -> masked in logs
  -> rotated by platform
```

### 8.3 Environment Variable vs Mounted File

| Method | Kelebihan | Risiko |
|---|---|---|
| Env var | mudah, umum | bisa terlihat di process metadata/debug dumps |
| Mounted file | cocok untuk cert/key, bisa reload | permission/path management |
| External secret provider | central governance | runtime dependency dan availability |
| Encrypted config | useful for static encrypted values | key management tetap perlu |

Certificate/private key biasanya lebih cocok sebagai mounted file. Short string secret sering memakai env var atau external provider.

## 9. Quarkus Secrets in Configuration

Quarkus mendukung secret configuration dengan pola `${handler::value}` melalui SmallRye Config SecretKeysHandler. Ini berguna untuk encrypted configuration value.

Contoh konseptual:

```properties
my.secret=${aes-gcm-nopadding::encrypted-value-here}
```

Tujuannya bukan membuat secret magically aman. Tujuannya adalah:

- value yang tersimpan di file tidak plaintext,
- decode/decrypt dilakukan saat runtime,
- handler/key management tetap harus aman.

### 9.1 Kapan Encrypted Config Berguna

Cocok untuk:

- aplikasi yang harus membawa config file terenkripsi,
- environment non-Kubernetes,
- secret kecil yang sulit diintegrasikan ke secret manager,
- transitional migration.

Tidak ideal sebagai pengganti:

- Vault,
- cloud secret manager,
- Kubernetes Secret + sealed secret,
- enterprise KMS.

Karena pertanyaan besarnya tetap:

> Di mana key untuk membuka encrypted config disimpan?

Kalau decryption key juga berada di repository atau image, maka encrypted config hanya memberi ilusi keamanan.

## 10. Credentials Provider

Quarkus menyediakan konsep credentials provider agar credential dapat diperoleh dari provider tertentu dan tidak selalu ditulis langsung sebagai property plain.

Mental model:

```text
consumer component
  -> references credential name/provider
  -> provider resolves actual username/password/token
  -> component receives credential at runtime
```

Keuntungan:

- credential lookup bisa distandarkan,
- rotation lebih mudah,
- source-of-truth bisa diganti,
- config lebih deklaratif,
- credential tidak tersebar di semua module.

Tetapi tetap butuh governance:

- provider availability,
- startup failure behavior,
- refresh behavior,
- masking,
- audit access,
- retry/backoff,
- stale credential handling.

## 11. Secret Rotation Design

Secret rotation sering gagal bukan karena teknologi, tetapi karena dependency graph tidak dipahami.

Contoh database password:

```text
DB user password changed
  -> app secret updated
  -> app reconnects
  -> old connections expire
  -> new connections use new password
```

Failure mode:

- app masih punya old pool connections,
- new connections gagal karena secret belum sinkron,
- all pods restart bersamaan,
- migration job memakai old secret,
- read replica punya credential berbeda,
- rollback memakai image lama dengan config key name lama.

### 11.1 Rotation Dengan Dual Validity

Pattern aman:

```text
1. create new credential while old still valid
2. update secret store
3. roll service gradually
4. verify new credential usage
5. revoke old credential
6. monitor error rate
7. record audit evidence
```

Untuk token/API key, bisa mirip:

```text
old_key valid
new_key created
service supports new_key
traffic switched
old_key revoked
```

Untuk certificate:

```text
old CA trusted
new CA added to truststore
server cert switched
clients verified
old CA removed later
```

### 11.2 Rotation Invariant

> Jangan rotate secret dengan asumsi semua consumer update atomically.

Distributed systems tidak atomic. Rotation perlu overlap window.

## 12. Crypto Hygiene Dalam Quarkus Service

Quarkus tidak mengubah prinsip crypto Java. Tetapi Quarkus native image dan build-time optimization memberi constraint tambahan.

Crypto usage biasanya muncul untuk:

- password hashing,
- token signing,
- payload encryption,
- field-level encryption,
- HMAC webhook verification,
- file integrity,
- data masking,
- certificate handling.

### 12.1 Password Hashing

Jangan pakai:

```text
SHA-256(password)
MD5(password)
Base64(password)
custom encryption for password
```

Gunakan password hashing algorithm yang memang didesain untuk password:

- bcrypt,
- Argon2,
- PBKDF2 dengan parameter kuat.

Quarkus bisa memakai library/security extension yang sesuai. Namun native image compatibility harus dicek jika library memakai reflection/native code.

### 12.2 Signing vs Encryption

Jangan campur mental model:

| Kebutuhan | Mechanism |
|---|---|
| Membuktikan data tidak berubah | hash / MAC |
| Membuktikan siapa yang menandatangani | digital signature |
| Menyembunyikan isi data | encryption |
| Menyimpan password | password hashing |
| Membawa claim trusted | signed token |

Contoh HMAC webhook:

```java
boolean valid = hmacVerifier.verify(
    requestBodyBytes,
    receivedSignature,
    sharedSecret
);

if (!valid) {
    throw new ForbiddenException("Invalid webhook signature");
}
```

Perhatikan:

- compare signature harus constant-time,
- raw body harus sama persis seperti yang ditandatangani,
- secret rotation perlu support multiple active secrets,
- timestamp/nonce perlu mencegah replay.

### 12.3 Algorithm Agility

Jangan hardcode design yang tidak bisa berubah.

Buruk:

```text
column encrypted_value only stores ciphertext
no key id
no algorithm version
```

Lebih baik:

```text
key_id
algorithm
version
ciphertext
created_at
```

Karena suatu hari:

- key rotate,
- algorithm deprecated,
- regulator meminta re-encryption,
- library vulnerability ditemukan.

## 13. JWT Signing Key Rotation

JWT validation sering terlihat mudah:

```text
fetch JWKS
validate signature
check issuer/audience/expiry
map claims
```

Tetapi production challenge ada di rotation.

Failure mode:

- issuer rotate signing key,
- app cache JWKS terlalu lama,
- old token masih valid tetapi old key dihapus terlalu cepat,
- new token diterbitkan sebelum JWKS propagated,
- clock skew terlalu ketat,
- audience berubah antar environment.

Design:

```text
1. Support multiple keys through JWKS.
2. Do not pin single kid unless required.
3. Configure cache with reasonable refresh.
4. Monitor validation failures by reason.
5. Keep issuer/audience explicit per environment.
6. Test rotation scenario.
```

Ingat:

> JWT validation failure harus observable berdasarkan reason, bukan hanya 401 count.

Contoh reason dimension:

- expired,
- invalid issuer,
- invalid audience,
- signature invalid,
- JWKS unavailable,
- unknown key id,
- token malformed,
- not-before violation,
- clock skew.

## 14. Native Image Security: Constraint Yang Sering Terlupakan

Native image mengubah cara aplikasi Java dieksekusi.

Dalam JVM mode:

```text
classes/resources/reflection often resolved dynamically at runtime
```

Dalam native image:

```text
closed-world analysis decides reachable code/resources mostly at build time
```

Security implication:

- reflection-based library harus registered,
- resource seperti truststore/cert/config harus included atau mounted,
- crypto provider behavior bisa berbeda,
- class initialization timing bisa mempengaruhi secret/crypto state,
- SSL truststore handling perlu diperhatikan,
- dynamic classloading tidak bisa diasumsikan.

### 14.1 Reflection and Private Members

Quarkus CDI documentation memperingatkan bahwa GraalVM native executable punya batasan reflection: reflective operation didukung, tetapi member relevan harus didaftarkan eksplisit, dan registrasi reflection bisa memperbesar native executable.

Security impact:

- library security yang memakai reflection mungkin gagal,
- JSON serialization untuk security DTO bisa gagal,
- exception class/error response bisa tidak terserialisasi,
- custom principal/claim object bisa butuh reflection registration,
- private members menambah kebutuhan reflection.

Practice:

- prefer constructor/public members where reasonable,
- avoid magic reflection in security path,
- test native mode untuk auth/security flow,
- jangan hanya test JVM mode.

### 14.2 Resource Inclusion

Native image tidak otomatis membawa semua resource classpath seperti JVM expectation.

Contoh resource yang mungkin perlu diperhatikan:

- PEM file,
- truststore,
- keystore,
- JSON schema,
- template security email,
- public key file,
- localization file,
- policy file.

Quarkus native tips menyediakan property seperti:

```properties
quarkus.native.resources.includes=certs/**,keys/*.pem,policies/*.json
```

Tetapi untuk private key/secret, lebih aman mount runtime secret daripada include ke binary.

Rule:

```text
public/static resource may be included in native image.
private secret/key should not be baked into native executable.
```

### 14.3 SSL With Native Executables

Quarkus memiliki guide khusus SSL dengan native executables dan merekomendasikan TLS Registry untuk pengalaman konsisten antara JVM dan native mode.

Artinya:

- jangan berasumsi truststore JVM mode sama dengan native mode,
- test outbound HTTPS/OIDC/REST client dalam native integration test,
- validate private CA handling,
- validate mTLS client certificate handling,
- validate certificate reload behavior bila digunakan.

## 15. Static Initialization vs Secret Safety

Native image mengenal build-time initialization dan runtime initialization.

Risiko besar:

> Jangan sampai value yang seharusnya runtime secret dibaca saat build dan masuk ke artifact.

Contoh buruk secara konseptual:

```java
@ApplicationScoped
public class BadSecretHolder {
    static final String API_KEY = System.getenv("API_KEY");
}
```

Jika class diinisialisasi saat build native image, ada risiko nilai build environment tertanam atau behavior tidak sesuai runtime.

Lebih aman:

```java
@ApplicationScoped
public class ExternalApiCredentialProvider {

    @ConfigProperty(name = "external.api.key")
    String apiKey;

    public String apiKey() {
        return apiKey;
    }
}
```

Tetapi tetap harus memahami apakah config property build-time atau runtime. Secret harus runtime.

Invariant:

```text
Secret must be resolved at runtime, not baked at build time.
```

## 16. Container Image Security

Quarkus artifact akhirnya sering menjadi container image.

Security image bukan hanya ukuran kecil.

Checklist:

```text
- use trusted base image
- pin image version/digest where appropriate
- run as non-root
- no package manager in runtime image if not needed
- no shell if using distroless/minimal image where possible
- no secret copied into image
- no build cache with secret
- scan image for CVE
- generate SBOM
- sign image if platform supports it
- set resource limits
- expose only needed port
- use read-only filesystem if possible
```

### 16.1 JVM Image vs Native Image

| Aspect | JVM Image | Native Image |
|---|---|---|
| Startup | slower than native | very fast |
| Memory | generally higher | generally lower RSS |
| Debugging | easier | harder |
| Dynamic behavior | richer | constrained |
| Image content | JVM + app | executable + libs |
| Security scan | JDK CVEs included | base/image/native binary concerns |

Native image can reduce attack surface by removing JDK runtime, but that does not automatically make it secure. You still need:

- dependency scan,
- base image scan,
- SBOM,
- TLS/secret validation,
- runtime policy.

### 16.2 Non-Root Runtime

Container should not run as root unless justified.

Kubernetes security context example:

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1001
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  capabilities:
    drop:
      - ALL
```

Potential issue:

- certificate path not readable,
- temp directory not writable,
- log directory not writable,
- native executable permission wrong,
- mounted secret file mode incompatible.

Top-tier approach:

> Harden container, then explicitly provide writable paths needed by app.

## 17. SBOM: Software Bill of Materials

SBOM menjawab:

> Artifact ini berisi dependency apa saja?

Tanpa SBOM, saat vulnerability muncul, pertanyaan sederhana menjadi mahal:

- service mana terdampak?
- versi library apa yang dipakai?
- apakah transitive dependency termasuk?
- apakah artifact yang running sesuai source build?
- apakah image lama masih deployed?

Quarkus menyediakan CycloneDX extension untuk menghasilkan SBOM sesuai CycloneDX specification.

Contoh menambahkan extension:

```bash
quarkus ext add io.quarkus:quarkus-cyclonedx
```

atau Maven:

```bash
./mvnw quarkus:add-extension -Dextensions="io.quarkus:quarkus-cyclonedx"
```

### 17.1 Dependency SBOM vs Build Output SBOM

Secara konsep:

- **Dependency SBOM**: dependency input sebelum build.
- **Artifact/Image SBOM**: apa yang benar-benar menjadi output/deployed artifact.

Keduanya berguna:

```text
pre-build analysis  -> block vulnerable dependency early
post-build analysis -> prove deployed artifact composition
```

### 17.2 SBOM Governance

SBOM hanya berguna jika dipakai.

Pipeline governance:

```text
build
  -> generate SBOM
  -> scan dependency vulnerabilities
  -> scan license policy
  -> attach SBOM to artifact
  -> store SBOM in artifact repository
  -> deploy only artifact with SBOM
  -> incident response can query SBOM inventory
```

Anti-pattern:

```text
SBOM generated locally but not stored
SBOM generated but not attached to artifact
SBOM ignored by vulnerability process
SBOM not connected to deployed version inventory
```

## 18. Dependency and Extension Supply Chain

Quarkus extension ecosystem kuat, tetapi dependency tetap adalah supply chain risk.

Governance questions:

1. Extension resmi Quarkus atau third-party?
2. Maintainer aktif?
3. Native image support jelas?
4. Security policy jelas?
5. Dependency transitive besar?
6. License sesuai?
7. Version aligned dengan Quarkus Platform BOM?
8. Ada CVE history?
9. Ada dynamic classloading/reflection heavy behavior?
10. Dipakai di security path?

### 18.1 Extension Classification

Gunakan classification:

| Class | Contoh | Governance |
|---|---|---|
| Core platform | REST, CDI, config, OIDC | standard approved |
| Data | Hibernate, reactive SQL, Redis | approved with DB policy |
| Messaging | Kafka, RabbitMQ | approved with retry/DLQ policy |
| Observability | OTel, Micrometer | approved with data redaction |
| Security sensitive | crypto, auth, secrets | architecture/security review |
| Experimental | preview/third-party | restricted/non-prod until approved |

### 18.2 Pinning and Upgrade

Quarkus Platform BOM membantu alignment. Tetapi upgrade tetap perlu discipline:

```text
1. read release notes
2. run unit/component/integration tests
3. run security auth tests
4. run native build if applicable
5. scan SBOM/CVE delta
6. verify generated config changes
7. canary deploy
8. monitor error/latency/security failures
```

## 19. Build Pipeline Security

CI/CD adalah bagian dari trusted computing base.

Pipeline harus menjaga:

- source integrity,
- dependency integrity,
- secret isolation,
- artifact immutability,
- reproducibility,
- auditability.

### 19.1 Build Secret Handling

Jangan:

```dockerfile
ARG DB_PASSWORD
ENV DB_PASSWORD=$DB_PASSWORD
RUN echo $DB_PASSWORD
```

Jangan copy config prod ke image:

```dockerfile
COPY application-prod.properties /deployments/config/application.properties
```

Lebih baik:

```text
build artifact without runtime secret
inject secret at deploy/runtime
```

### 19.2 Artifact Promotion

Anti-pattern:

```text
build once for dev
rebuild for uat
rebuild for prod
```

Lebih baik:

```text
build once
promote same artifact across env
inject env-specific config at runtime
```

Ini penting untuk Quarkus karena beberapa config bersifat build-time. Kalau build artifact per environment, kamu harus sangat disiplin membedakan:

- build-time property,
- runtime property,
- environment-specific runtime config.

## 20. Kubernetes Runtime Hardening

Quarkus service di Kubernetes harus dikunci dari beberapa sisi.

### 20.1 ServiceAccount and RBAC

Default ServiceAccount sering terlalu luas jika cluster tidak disiplin.

Rule:

```text
Each service gets a dedicated ServiceAccount.
Grant only required Kubernetes API permissions.
Most application services need zero Kubernetes API permission.
```

### 20.2 NetworkPolicy

Tanpa NetworkPolicy, service bisa saja memanggil dependency yang tidak seharusnya.

Design:

```text
allow inbound only from gateway/allowed services
allow outbound only to DB/message broker/OIDC/external gateway
block everything else
```

### 20.3 Secret Volume Permission

Mounted secret should be least-readable.

```yaml
volumes:
  - name: tls-certs
    secret:
      secretName: my-service-tls
      defaultMode: 0400
```

Pastikan user container bisa membaca file tersebut.

### 20.4 Readiness and Security Dependency

Jika service butuh OIDC/JWKS/truststore/secret, readiness harus mencerminkan dependency minimum.

Tetapi hati-hati:

- readiness yang terlalu ketat bisa menyebabkan cascading outage,
- readiness yang terlalu longgar bisa menerima traffic saat security dependency belum siap.

Rule:

```text
readiness should represent whether service can safely handle expected traffic.
```

## 21. Dev UI and Dev Services Security Boundary

Dev UI dan Dev Services sangat berguna untuk produktivitas, tetapi harus dipahami sebagai dev/test tools.

Risk:

- Dev UI expose config/status internal,
- Dev Services auto-start database/broker/Keycloak lokal,
- local profile accidentally resembles prod,
- test credentials reused,
- debug endpoint terbuka.

Rule:

```text
Dev UI and Dev Services are productivity tools, not production control plane.
```

Production checklist:

- dev-only config isolated,
- no dev secret reused in prod,
- no Dev UI exposed publicly,
- no trust-all propagated from dev profile,
- CI checks banned properties.

## 22. Logging, Tracing, and Secret Leakage

Security sering bocor bukan melalui endpoint utama, tetapi melalui observability.

### 22.1 Jangan Log

```text
Authorization header
Cookie
OIDC token
refresh token
client secret
password
private key
full certificate private material
raw request body containing sensitive data
full DB connection string with password
personal identifier if not needed
```

### 22.2 Structured Redaction

Buat redaction policy:

```text
field name contains: password, secret, token, authorization, cookie, key
  -> redact

known sensitive domain fields:
  -> redact or hash
```

Contoh output baik:

```json
{
  "event": "oidc_token_validation_failed",
  "reason": "invalid_audience",
  "issuer": "https://idp.example.gov/realms/main",
  "audience_expected": "case-service",
  "subject_hash": "sha256:...",
  "correlation_id": "..."
}
```

Bukan:

```json
{
  "token": "eyJhbGciOi..."
}
```

### 22.3 Trace Baggage

Jangan taruh sensitive data di trace baggage/header propagation.

Baggage sering menyebar antar service dan masuk backend telemetry.

Aman:

- correlation id,
- request id,
- tenant id jika allowed,
- sanitized actor id/hash.

Tidak aman:

- token,
- role list detail sensitif,
- secret,
- raw personal data,
- authorization decision reason yang terlalu detail untuk user-facing path.

## 23. Audit Trail vs Security Log

Audit trail dan security log berbeda.

| Jenis | Tujuan | Contoh |
|---|---|---|
| Technical log | debugging/operation | TLS handshake failed |
| Security log | threat detection | repeated invalid token |
| Audit trail | accountability/legal trace | officer approved case |

Audit trail harus menjawab:

```text
who did what, to which object, when, from where, under which authority, with what outcome
```

Security log harus menjawab:

```text
what security-relevant event occurred, severity, reason, source, target, mitigation context
```

Jangan campur semua menjadi satu log bebas.

Untuk regulatory system:

```text
Audit event is part of domain evidence.
Security log is part of operational defense.
```

Gunakan pertanyaan berikut sebelum production:

### 24.4 Artifact

- Apakah image discan?
- Apakah image immutable?
- Apakah image berjalan non-root?

- Apakah native security path dites?

- Apakah resource/reflection didaftarkan minimal?
- Apakah secret tidak terbaca saat build?

## 25. Secure Configuration Checklist

Banned in production:

```properties
quarkus.tls.*.trust-all=true
quarkus.tls.*.hostname-verification-algorithm=NONE
quarkus.http.insecure-requests=enabled
quarkus.log.category."io.quarkus.oidc".level=TRACE
quarkus.log.category."org.apache.http".level=TRACE
```

Allowed only with explicit justification:

```text
- HTTP cleartext behind internal gateway
- self-signed/private CA
- client certificate authentication
- custom crypto provider
- broad outbound egress
- debug logging for security packages
```

Required:

```text
- no secret in repository
- no secret in image
- TLS validation for outbound calls
- named TLS config for important external dependency
- certificate expiry alert
- SBOM generated
- vulnerability scan gate
- non-root container
- production profile locked
- native security integration test if native build is shipped
```

## 26. Example: Secure OIDC Client With TLS Registry

Scenario:

```text
case-service calls private Keycloak/OIDC provider
OIDC provider uses private CA
client credentials are stored in secret manager
```

Conceptual config:

```properties
# TLS bucket
quarkus.tls.oidc-provider.trust-store.p12.path=/mnt/trust/oidc-truststore.p12
quarkus.tls.oidc-provider.trust-store.p12.password=${OIDC_TRUSTSTORE_PASSWORD}

# OIDC
quarkus.oidc.auth-server-url=https://idp.internal.example/realms/aceas
quarkus.oidc.client-id=case-service
quarkus.oidc.credentials.secret=${OIDC_CLIENT_SECRET}
quarkus.oidc.tls-configuration-name=oidc-provider
```

Security properties:

- OIDC endpoint certificate is validated.
- Client secret is runtime-injected.
- Trust boundary is named.
- No `trust-all`.
- Rotation can update mounted truststore/secret.

Add operational requirements:

```text
- alert on OIDC TLS failures
- alert on unknown kid/JWKS failures
- monitor OIDC token validation failures by reason
- test certificate rotation
- test secret rotation
```

## 27. Example: mTLS REST Client

Scenario:

```text
Quarkus service calls government external API.
API requires client certificate.
```

Conceptual config:

```properties
quarkus.tls.gov-api.key-store.p12.path=/mnt/certs/gov-client-keystore.p12
quarkus.tls.gov-api.key-store.p12.password=${GOV_API_KEYSTORE_PASSWORD}
quarkus.tls.gov-api.trust-store.p12.path=/mnt/trust/gov-api-truststore.p12
quarkus.tls.gov-api.trust-store.p12.password=${GOV_API_TRUSTSTORE_PASSWORD}

quarkus.rest-client.gov-api.url=https://api.gov.example
quarkus.rest-client.gov-api.tls-configuration-name=gov-api
```

Operational design:

- client certificate mounted from Kubernetes Secret,
- truststore managed by platform/security team,
- certificate expiry monitored,
- key rotation has overlap,
- REST client has timeout/retry/circuit breaker,
- no sensitive response logged.

## 29. Failure Modes and How To Diagnose Them

### 29.1 TLS Handshake Failure

Symptoms:

```text
javax.net.ssl.SSLHandshakeException
PKIX path building failed
unable to find valid certification path
hostname verification failed
bad_certificate
certificate_unknown
```

Likely causes:

- missing CA,
- expired cert,
- wrong hostname/SAN,
- client cert absent,
- wrong truststore,
- TLS registry bucket not referenced,
- native truststore difference.

Diagnosis:

```text
- inspect server certificate chain
- verify truststore contains issuing CA
- verify SAN matches URL hostname
- verify client certificate sent if mTLS
- compare JVM vs native behavior
- check profile-specific config
```

### 29.2 Secret Resolution Failure

Symptoms:

```text
NoSuchElementException config property
invalid credentials
401 from dependency
DB authentication failed
```

Likely causes:

- secret key name mismatch,
- wrong namespace,
- secret mounted path wrong,
- environment variable naming mismatch,
- build-time secret baked incorrectly,
- secret rotated but app not refreshed/restarted,
- old pool connection.

Diagnosis:

```text
- verify config source order
- check mounted secret exists and permission
- do not print secret value
- log secret version/hash/metadata if safe
- validate startup config report with masking
```

### 29.3 Native Image Security Failure

Symptoms:

```text
works in JVM, fails in native
ClassNotFoundException
NoSuchMethodException
serialization empty object
SSL failure only in native
resource not found
```

Likely causes:

- missing reflection registration,
- missing resource includes,
- class initialized at wrong time,
- truststore not available,
- dynamic proxy unsupported,
- security library incompatible.

Diagnosis:

```text
- run native integration tests
- inspect native build warnings
- add minimal reflection/resource config
- prefer Quarkus extension support over raw library hack
- verify TLS registry behavior
```

## 30. Production Security Checklist

### 30.1 Transport

- [ ] Inbound TLS strategy documented.
- [ ] Outbound TLS validates certificates.
- [ ] No production `trust-all`.
- [ ] No production hostname verification disabled.
- [ ] Private CA truststore managed.
- [ ] Certificate expiry monitored.
- [ ] Rotation plan exists.
- [ ] mTLS used where required.

### 30.2 Secrets

- [ ] No secret in Git.
- [ ] No secret in image layer.
- [ ] No secret in logs/traces.
- [ ] Secret source of truth documented.
- [ ] Rotation plan tested.
- [ ] Runtime injection used.
- [ ] Secret masking enabled.
- [ ] Build-time config reviewed.

### 30.3 Crypto

- [ ] No custom crypto unless reviewed.
- [ ] Password hashing uses appropriate algorithm.
- [ ] Signing/encryption/HMAC purposes separated.
- [ ] Key id/version stored where needed.
- [ ] Algorithm agility planned.
- [ ] Constant-time compare for MAC/signature.

### 30.4 Native

- [ ] Native image tested if shipped.
- [ ] TLS/OIDC/REST client works in native.
- [ ] Resources/reflection minimal and explicit.
- [ ] Secret not baked at build time.
- [ ] Native debug procedure known.

### 30.5 Supply Chain

- [ ] Quarkus BOM used.
- [ ] Dependencies scanned.
- [ ] SBOM generated.
- [ ] SBOM attached/stored.
- [ ] Container image scanned.
- [ ] Image runs non-root.
- [ ] Image provenance/signing policy known.
- [ ] Upgrade cadence defined.

### 30.6 Runtime

- [ ] Dedicated ServiceAccount.
- [ ] Least privilege Kubernetes RBAC.
- [ ] NetworkPolicy or equivalent egress control.
- [ ] Read-only filesystem if possible.
- [ ] Resource limits set.
- [ ] Security-relevant metrics/alerts exist.
- [ ] Incident runbook exists.

## 31. Anti-Patterns

### 31.1 “Internal Network Jadi Tidak Perlu TLS”

Private network mengurangi exposure, tetapi tidak menghapus risk:

- compromised workload,
- misconfigured route,
- insider threat,
- service mesh bypass,
- packet capture,
- wrong namespace access.

Better framing:

```text
Internal network may reduce TLS requirement for low-risk traffic,
but high-sensitive service-to-service traffic still needs strong identity and encryption.
```

### 31.2 “Secret Di Env Var Sudah Aman”

Env var lebih baik daripada Git, tetapi bukan otomatis aman.

Risiko:

- debug dump,
- process inspection,
- accidental log,
- broad pod exec permission.

### 31.3 “Native Image Lebih Aman Karena Tidak Ada JVM”

Native image bisa mengurangi surface tertentu, tetapi:

- dependency vulnerability tetap ada,
- logic bug tetap ada,
- crypto misuse tetap ada,
- TLS misconfig tetap ada,
- secret leak tetap ada,
- observability leak tetap ada.

### 31.4 “SBOM Cuma Compliance”

SBOM adalah operational security inventory. Tanpa SBOM, incident response menjadi lambat dan spekulatif.

### 31.5 “Certificate Rotation Nanti Saja”

Certificate pasti expired. Kalau rotation tidak diuji sebelum expiry, production outage hanya masalah waktu.

## 32. Design Decision Matrix

| Decision | Pilih Ini Jika | Hindari Jika |
|---|---|---|
| TLS at ingress only | service internal low sensitivity, mesh/network controlled | direct pod access possible, sensitive east-west traffic |
| App-level TLS | service exposed directly or needs end-to-end TLS | TLS already terminated and duplicate complexity not needed |
| mTLS | service identity required at transport layer | certificate lifecycle tidak siap |
| JWT service token | permission/scope needed at app layer | channel tidak aman |
| TLS Registry | multiple TLS consumers or private CA/mTLS | trivial local-only app |
| Encrypted config | transitional/non-cloud secret handling | decryption key unmanaged |
| Secret manager | enterprise rotation/audit needed | provider availability not designed |
| Native image | startup/memory/cold start critical | dynamic/reflection-heavy app not ready |
| SBOM gate | production/compliance/security needed | almost never avoid for production |

## 33. Mini Case Study: Regulatory Case Service

Scenario:

```text
Case service handles enforcement lifecycle.
It exposes API to internal users.
It calls:
- OIDC provider
- document service
- notification service
- external address API
- database
- Kafka outbox publisher
```

Security design:

```text
Inbound:
- TLS terminated at internal API gateway
- service reachable only from gateway namespace
- JWT bearer token validated by Quarkus OIDC

OIDC:
- private CA through TLS Registry
- JWKS failures monitored
- issuer/audience explicit

Outbound REST:
- document service uses service token + mTLS
- external address API uses TLS Registry and API key from secret manager

Database:
- password from Kubernetes Secret or external secret provider
- DB SSL if required by platform
- pool credentials rotated with rolling deployment

Kafka:
- TLS/SASL credential from secret
- topic ACL restricts service

Container:
- non-root
- read-only filesystem
- no shell if possible
- SBOM generated
- image scanned

Audit:
- domain audit trail separate from technical log
- actor id, authority, action, case id, outcome stored
- token never stored

Native:
- native build optional
- native integration test must cover OIDC, TLS, JSON error contract, audit event
```

Key invariant:

> Regulatory defensibility comes from secure identity, secure transport, explicit authorization, immutable audit evidence, and traceable artifact lineage.

## 35. Invariants

Simpan invariant berikut sebagai pegangan:

1. **Transport security harus eksplisit.** Jangan mengandalkan “network internal” tanpa threat model.
2. **Secret harus runtime concern, bukan build artifact.**
3. **Certificate pasti expired; rotation harus didesain sebelum production.**
4. **`trust-all` adalah local-dev escape hatch, bukan production configuration.**
5. **mTLS membuktikan workload/channel; JWT membuktikan application-level claims/scope.**
6. **Native image mempercepat startup, tetapi memperketat reflection/resource/SSL assumptions.**
7. **SBOM adalah inventory operasional, bukan dokumen compliance pasif.**
8. **Image kecil tidak otomatis aman.**
9. **Audit trail bukan technical log.**
10. **Security harus diuji pada artifact yang benar-benar akan dideploy.**
