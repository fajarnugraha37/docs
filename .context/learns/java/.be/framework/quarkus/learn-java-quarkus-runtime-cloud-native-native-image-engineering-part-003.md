## 1. Problem Besar yang Diselesaikan Quarkus

Framework Java tradisional biasanya melakukan banyak pekerjaan pada saat aplikasi start:

1. scan classpath,
2. baca annotation,
3. baca XML/config descriptor,
4. bangun metadata object,
5. resolve dependency injection,
6. buat proxy,
7. scan entity,
8. scan REST resource,
9. scan validator,
10. init persistence provider,
11. init security provider,
12. init messaging provider,
13. register endpoint,
14. build routing table,
15. wire runtime service,
16. siapkan reflection/proxy/dynamic invocation.

Model seperti ini fleksibel, tapi mahal untuk container/cloud-native:

- startup lebih lambat,
- memory lebih besar,
- runtime memuat class yang hanya dibutuhkan untuk bootstrap,
- banyak metadata object tetap hidup sepanjang aplikasi,
- native image lebih sulit karena banyak dynamic behavior,
- cold start buruk,
- horizontal scaling lebih mahal,
- dev/prod behavior sering tersembunyi.

Quarkus mengambil pendekatan berbeda:

> Sebanyak mungkin pekerjaan yang bisa diketahui pada build-time harus diproses pada build-time, bukan saat aplikasi start.

Ini berarti Quarkus tidak hanya “menjalankan” aplikasi. Quarkus **membangun bentuk runtime aplikasi** sebelum aplikasi benar-benar berjalan.

## 2. Mental Model Utama: Quarkus sebagai Application Compiler

Cara paling produktif memahami Quarkus adalah:

> Quarkus adalah framework + compiler pipeline untuk aplikasi Java cloud-native.

Bukan compiler Java seperti `javac`, tetapi application compiler:

- membaca struktur aplikasi,
- memahami annotation,
- membangun dependency graph,
- menghapus bagian yang tidak digunakan,
- menghasilkan metadata,
- menghasilkan bytecode helper,
- menyiapkan native-image metadata,
- memisahkan build-time state dan runtime state,
- membuat runtime lebih tipis.

Dalam framework tradisional, runtime sering bertanya:

> “Apa saja class yang ada? Annotation apa yang ada? Bean apa yang harus dibuat? Endpoint apa yang harus didaftarkan?”

Dalam Quarkus, pertanyaan itu sebagian besar dijawab saat build:

> “Saat aplikasi start, runtime tidak perlu mencari terlalu banyak. Banyak keputusan sudah dikompilasi menjadi metadata/bytecode yang siap dipakai.”

### Analogi sederhana

Framework runtime-heavy:

```text
Start aplikasi
  -> scan classpath
  -> baca annotation
  -> buat metadata
  -> resolve dependency
  -> register service
  -> aplikasi siap
```

Quarkus:

```text
Build aplikasi
  -> scan/index class
  -> build model aplikasi
  -> run extension build steps
  -> generate metadata/bytecode
  -> prune unused parts
  -> package optimized artifact

Start aplikasi
  -> load runtime hasil build
  -> initialize yang memang harus runtime
  -> aplikasi siap
```

Konsekuensinya besar: desain aplikasi Quarkus harus ramah terhadap analisis build-time.

## 3. Vocabulary Internal yang Wajib Dikuasai

Sebelum masuk detail, kuasai istilah berikut.

| Istilah | Makna Praktis |
|---|---|
| Augmentation | Fase build-time saat Quarkus menganalisis aplikasi dan menghasilkan bentuk runtime yang dioptimalkan. |
| Extension | Unit integrasi Quarkus untuk teknologi tertentu: REST, CDI, Hibernate, Kafka, OIDC, Redis, Kubernetes, dan lain-lain. |
| Runtime module | Bagian extension yang ikut masuk ke aplikasi runtime. |
| Deployment module | Bagian extension yang hanya dipakai saat build/augmentation. |
| Build step | Method build-time yang menghasilkan atau mengonsumsi build item. |
| Build item | Objek komunikasi antar build step. |
| Recorder | Mekanisme untuk merekam operasi build-time menjadi bytecode yang akan dijalankan di static init atau runtime init. |
| Static init | Fase initialization yang bisa dilakukan lebih awal; pada native image bisa masuk ke image build. |
| Runtime init | Fase initialization saat aplikasi benar-benar berjalan. |
| Jandex | Index annotation/class metadata agar tidak perlu reflection/classpath scanning berat. |
| Arc | Implementasi CDI Quarkus yang dioptimalkan build-time. |
| Synthetic bean | Bean CDI yang didaftarkan oleh extension, bukan langsung dari class aplikasi biasa. |
| Unused bean removal | Optimasi Arc untuk menghapus bean/interceptor/decorator yang tidak terpakai. |
| Native image metadata | Metadata yang membantu GraalVM/Mandrel memahami reflection, proxy, resource, serialization, dan initialization. |

## 4. Pipeline Besar Quarkus dari Source ke Runtime

Secara konseptual, pipeline Quarkus bisa dibaca seperti ini:

```text
Source code + dependencies
        |
        v
Maven/Gradle + Quarkus plugin
        |
        v
Resolve platform + extensions
        |
        v
Index application and dependencies
        |
        v
Run augmentation
        |
        +--> CDI analysis via Arc
        +--> REST endpoint discovery
        +--> Hibernate entity discovery
        +--> Config model processing
        +--> Security integration
        +--> Messaging channels
        +--> Health/metrics/tracing integration
        +--> Kubernetes/native/container metadata
        |
        v
Generate classes/resources/metadata
        |
        v
Package application
        |
        +--> JVM fast-jar / mutable-jar / legacy-jar
        +--> native image preparation/build
        +--> container image metadata
        |
        v
Runtime start
```

Yang penting: banyak extension berjalan pada fase augmentation. Extension bukan hanya library runtime; extension ikut membentuk aplikasi.

## 5. Traditional Runtime Boot vs Quarkus Augmented Boot

### 5.1 Traditional runtime-heavy model

Pada runtime-heavy framework:

```text
Application startup
  -> runtime scanner mencari class
  -> runtime parser membaca annotation/XML
  -> runtime container membangun graph
  -> runtime membuat proxy
  -> runtime memvalidasi wiring
  -> runtime menyimpan metadata
  -> runtime menjalankan aplikasi
```

Keuntungannya:

- sangat dinamis,
- banyak pattern runtime bisa bekerja,
- plugin bisa melakukan discovery saat aplikasi start,
- cocok untuk server lama yang hidup lama dan jarang restart.

Kelemahannya:

- startup cost tinggi,
- memory cost tinggi,
- banyak class bootstrap ikut masuk runtime,
- sulit untuk native image,
- container scaling kurang efisien,
- runtime failure terlambat ketahuan.

### 5.2 Quarkus build-time model

Pada Quarkus:

```text
Build time
  -> discovery
  -> validation
  -> optimization
  -> generation
  -> pruning

Runtime
  -> execute prepared runtime
```

Keuntungannya:

- startup lebih cepat,
- memory lebih kecil,
- banyak error ketahuan saat build,
- native image lebih feasible,
- classpath runtime lebih bersih,
- extension bisa mengoptimalkan integrasi.

Kelemahannya:

- dynamic runtime pattern lebih terbatas,
- reflection harus eksplisit,
- classloading dinamis perlu hati-hati,
- config build-time tidak bisa diubah sembarangan saat runtime,
- extension perlu menulis build step dengan benar,
- beberapa library Java lama tidak cocok tanpa adaptasi.

## 6. Augmentation: Jantung Internal Quarkus

Quarkus documentation menyebut extension terdiri dari dua bagian besar: build-time augmentation dan runtime container. Augmentation bertugas memproses metadata seperti annotation/XML descriptor; hasilnya dapat berupa recorded bytecode yang langsung menginisialisasi runtime service tanpa mengulang pemrosesan metadata saat startup.

Dalam bahasa praktis:

> Augmentation adalah fase ketika Quarkus “memahami aplikasi” dan menulis ulang sebagian pekerjaan framework menjadi artifact siap runtime.

### 6.1 Apa yang dilakukan augmentation?

Augmentation bisa mencakup:

1. menemukan annotation,
2. membaca konfigurasi build-time,
3. membangun model bean CDI,
4. menemukan REST resource,
5. menemukan entity Hibernate,
6. mendaftarkan route,
7. mendaftarkan feature health/metrics,
8. membuat synthetic bean,
9. mendaftarkan reflective class untuk native image,
10. menambahkan resource untuk runtime classloader,
11. menghasilkan bytecode initializer,
12. menghapus bean tidak terpakai,
13. memvalidasi conflict antar extension,
14. menghasilkan deployment artifact.

### 6.2 Apa yang tidak ideal dilakukan saat augmentation?

Tidak semua hal boleh dilakukan pada build-time. Hindari:

1. membuka koneksi production database,
2. membaca secret runtime yang berubah per environment,
3. membuat network call ke service eksternal,
4. menghasilkan state yang environment-specific tapi dibekukan ke artifact,
5. membaca file runtime yang tidak tersedia saat build,
6. membuat random/security object yang tidak aman dibekukan ke native image,
7. bergantung pada jam saat build untuk logic runtime,
8. menaruh tenant-specific runtime state di static init.

### 6.3 Rule of thumb

Gunakan build-time untuk:

- metadata,
- struktur aplikasi,
- class discovery,
- annotation analysis,
- route/entity/bean model,
- native metadata,
- generated code,
- deterministic initialization.

Gunakan runtime untuk:

- koneksi jaringan,
- secret aktual,
- credential refresh,
- tenant runtime,
- clock-dependent state,
- random/security seed,
- environment-specific service discovery,
- runtime feature flag yang berubah.

## 7. Jandex: Kenapa Quarkus Tidak Bergantung pada Reflection Scanning Biasa

Jandex adalah index metadata class dan annotation. Daripada memuat class dan melakukan reflection scanning pada startup, Quarkus dapat memakai index untuk menjawab pertanyaan seperti:

- class mana yang punya annotation tertentu,
- method mana yang punya annotation tertentu,
- field mana yang diinjeksi,
- class mana yang menjadi entity,
- resource mana yang menjadi endpoint,
- bean mana yang memiliki qualifier,
- interceptor binding mana yang ada.

### 7.1 Perbedaan reflection scanning vs indexing

Reflection scanning runtime:

```text
Load class -> inspect annotation -> build metadata -> keep metadata
```

Jandex/build-time indexing:

```text
Read bytecode metadata -> build index -> extension query index -> generate optimized model
```

Keuntungannya:

1. class tidak harus di-load untuk sekadar membaca annotation,
2. lebih cepat untuk discovery,
3. lebih ramah native image,
4. extension dapat bekerja dari metadata yang stabil,
5. banyak validasi bisa dilakukan saat build.

### 7.2 Implikasi untuk dependency

Tidak semua dependency otomatis di-index. Quarkus meng-index aplikasi dan dependency tertentu. Untuk library internal yang berisi CDI bean, entity, annotation, atau extension metadata, kamu harus memastikan library itu ter-index dengan benar.

Masalah yang sering muncul:

- bean di module lain tidak ditemukan,
- entity tidak terdeteksi,
- custom annotation tidak diproses,
- extension tidak melihat class target,
- test berhasil di satu module tapi gagal di packaging final.

Pola mitigasi:

1. gunakan dependency structure yang jelas,
2. buat `META-INF/jandex.idx` bila perlu,
3. gunakan `quarkus.index-dependency` untuk dependency tertentu,
4. hindari library util yang diam-diam berisi bean/entity,
5. buat module boundary eksplisit.

### 7.3 Mental model

Kalau framework runtime-heavy bertanya:

> “Saya akan cari semuanya saat start.”

Quarkus bertanya:

> “Saya harus tahu struktur aplikasi saat build. Kalau sesuatu tidak terlihat di index/build pipeline, jangan berharap runtime magic menyelamatkan.”

## 8. Build Items: Bahasa Komunikasi Antar Extension

Build item adalah objek yang diproduksi dan dikonsumsi oleh build step. Ini adalah mekanisme penting dalam augmentation.

Bayangkan extension Quarkus sebagai pipeline modular:

```text
BuildStep A produces X
BuildStep B consumes X and produces Y
BuildStep C consumes Y and produces Z
```

Contoh konseptual:

```text
REST extension:
  produces discovered REST resources

OpenAPI extension:
  consumes REST resources
  produces OpenAPI document

Security extension:
  consumes endpoint/security annotations
  produces security metadata

Native extension:
  consumes reflection requirements
  produces native-image config
```

### 8.1 Kenapa build items penting?

Karena extension tidak boleh berkomunikasi dengan global mutable state sembarangan. State build-time harus mengalir lewat build item agar:

1. dependency antar build step jelas,
2. ordering bisa dihitung,
3. build bisa deterministik,
4. extension bisa dikombinasikan,
5. incremental/dev mode lebih aman,
6. native image preparation lebih predictable.

### 8.2 Jenis pemikiran build item

Secara mental, build item bisa dipakai untuk menyatakan:

- “Saya menemukan class ini.”
- “Saya butuh resource ini tersedia saat runtime.”
- “Saya butuh class ini register reflection.”
- “Saya menyediakan capability X.”
- “Saya ingin bean ini ditambahkan.”
- “Saya ingin route ini didaftarkan.”
- “Saya ingin service ini diinisialisasi di static init.”
- “Saya ingin feature ini muncul di Dev UI.”

### 8.3 Anti-pattern build-time communication

Hindari pola seperti:

```java
static List<Object> discoveredThings = new ArrayList<>();
```

Masalah:

- tidak deterministik,
- bermasalah di dev mode reload,
- sulit di-test,
- ordering tidak jelas,
- bisa bocor antar build,
- buruk untuk extension composition.

Top-tier Quarkus extension engineering selalu menjaga komunikasi antar build step lewat build item.

## 9. Build Steps: Unit Kerja Augmentation

Build step adalah method yang dijalankan pada fase build/augmentation. Build step bisa:

1. consume build items,
2. produce build items,
3. membaca index,
4. membaca config build-time,
5. mendaftarkan bean,
6. mendaftarkan route,
7. mendaftarkan reflection metadata,
8. memanggil recorder,
9. menghasilkan generated class/resource.

Contoh konseptual, bukan kode produksi lengkap:

```java
class MyExtensionProcessor {

    @BuildStep
    void registerBeans(BuildProducer<AdditionalBeanBuildItem> beans) {
        beans.produce(AdditionalBeanBuildItem.unremovableOf(MyRuntimeService.class));
    }
}
```

Maknanya:

- saat build, extension menambahkan bean `MyRuntimeService`,
- bean itu nanti tersedia di CDI runtime,
- kalau ditandai unremovable, Arc tidak akan menghapusnya meskipun dianggap tidak terpakai.

### 9.1 Build step bukan runtime service

Kesalahan umum:

> Mengira build step adalah tempat menjalankan logic aplikasi.

Build step adalah logic pembentukan aplikasi, bukan logic bisnis runtime.

Jangan letakkan:

- business rules,
- request handling,
- database transaction,
- tenant runtime decision,
- external call production,
- cache runtime state.

Letakkan di build step:

- discovery,
- metadata processing,
- model generation,
- runtime service registration,
- native metadata,
- validation structural.

## 10. Recorder Pattern: Menjembatani Build-Time dan Runtime

Recorder adalah salah satu konsep paling penting di Quarkus internal architecture.

Masalahnya begini:

- saat build, extension tahu banyak metadata,
- tetapi beberapa object runtime harus dibuat saat aplikasi start,
- untuk native image, sebagian initialization bisa direkam ke image,
- extension tidak boleh asal membuat runtime object pada build processor.

Recorder membantu dengan cara:

> Extension memanggil method recorder saat build, lalu Quarkus merekam pemanggilan itu menjadi bytecode yang akan dieksekusi pada fase initialization yang sesuai.

### 10.1 Mental model recorder

```text
Build step:
  “Saya tahu konfigurasi dan metadata. Tolong nanti saat runtime buat service dengan parameter ini.”

Recorder:
  “Saya menyediakan method untuk membuat service runtime.”

Quarkus:
  “Saya rekam instruksi ini menjadi bytecode initializer.”
```

### 10.2 Static init vs runtime init

Recorder bisa dipakai pada fase berbeda:

- static init,
- runtime init.

Static init cocok untuk state yang deterministic dan bisa dibekukan lebih awal. Runtime init cocok untuk state yang bergantung environment runtime.

### 10.3 Contoh konseptual

```java
@Recorder
public class MyRecorder {
    public RuntimeValue<MyRuntimeService> createService(MyConfig config) {
        return new RuntimeValue<>(new MyRuntimeService(config.mode()));
    }
}
```

Build step:

```java
@BuildStep
@Record(ExecutionTime.RUNTIME_INIT)
MyRuntimeServiceBuildItem create(MyRecorder recorder, MyConfig config) {
    return new MyRuntimeServiceBuildItem(recorder.createService(config));
}
```

Makna:

- build step memutuskan service perlu dibuat,
- recorder method direkam,
- pembuatan service terjadi pada runtime init,
- hasilnya bisa dipakai extension lain.

Kode di atas hanya ilustrasi pola. Detail API bisa berbeda sesuai versi Quarkus.

## 11. Static Init vs Runtime Init: Boundary yang Sering Menentukan Bug Native Image

Quarkus membedakan initialization stage. Ini bukan detail kecil; ini bisa menentukan apakah aplikasi:

- aman di JVM mode,
- aman di native mode,
- deterministic,
- portable antar environment,
- tidak membekukan state yang salah.

### 11.1 Static init

Static init adalah fase initialization yang bisa dilakukan lebih awal. Dalam native image, hasil static init tertentu dapat masuk ke image. Artinya object tertentu bisa dibuat saat image build, lalu dibawa ke executable.

Cocok untuk:

- metadata deterministic,
- precomputed routing table,
- mapping statis,
- serializer/deserializer model,
- immutable config build-time,
- generated helper,
- dependency graph.

Tidak cocok untuk:

- database connection,
- HTTP connection,
- credential runtime,
- random generator yang harus fresh,
- local machine path,
- environment-specific state,
- tenant-specific state,
- file descriptor,
- thread pool runtime,
- clock-based state.

### 11.2 Runtime init

Runtime init terjadi saat aplikasi benar-benar start di target environment.

Cocok untuk:

- datasource connection pool,
- HTTP client connection pool,
- secret loading,
- token provider,
- credential refresh,
- scheduler start,
- messaging connection,
- runtime feature flag,
- service discovery.

### 11.3 Failure mode klasik

Bug native image sering muncul karena object yang seharusnya runtime malah dibuat saat build/static init.

Contoh konseptual:

```java
static final SecureRandom RANDOM = new SecureRandom();
```

Pada JVM biasa, ini mungkin tampak aman. Pada native image, class initialization timing dapat menyebabkan object security-sensitive terjebak di image heap atau memicu error build/runtime.

Prinsip:

> Jangan biarkan state yang harus fresh/environment-specific dibekukan pada build/native image.

## 12. Arc: CDI Quarkus yang Berpikir Build-Time

Arc adalah implementasi CDI di Quarkus. Tetapi jangan memahaminya seperti CDI container klasik yang sepenuhnya runtime-dynamic.

Arc melakukan banyak pekerjaan saat build:

1. menemukan bean,
2. memvalidasi injection point,
3. resolve dependency,
4. generate client proxy/subclass bila perlu,
5. menentukan unused bean,
6. menghapus bean/interceptor/decorator tidak terpakai,
7. mengintegrasikan synthetic bean dari extension,
8. menyiapkan metadata runtime yang minimal.

Dokumentasi Quarkus menjelaskan bahwa Arc mencoba menghapus unused beans, interceptors, dan decorators saat build secara default untuk mengurangi generated classes dan memory. Ini sangat penting untuk memahami perbedaan Arc dengan CDI runtime-heavy.

### 12.1 CDI resolution sebagai build-time graph problem

Pada runtime-heavy CDI:

```text
Startup:
  scan beans
  build metadata
  resolve injection
  keep metadata
```

Pada Arc:

```text
Build:
  index beans
  resolve injection
  validate graph
  generate proxies/helpers
  remove unused nodes

Runtime:
  use prepared graph
```

### 12.2 Unused bean removal

Arc dapat menghapus bean yang dianggap tidak dipakai.

Contoh:

```java
@ApplicationScoped
public class InternalHelper {
    public void doSomething() {}
}
```

Kalau tidak ada injection point, observer, named reference, atau mekanisme yang membuat Arc tahu bean ini dipakai, bean bisa dihapus.

Masalah muncul bila kamu memakai dynamic lookup:

```java
CDI.current().select(InternalHelper.class).get();
```

Arc mungkin tidak selalu bisa mendeteksi penggunaan programmatic lookup seperti ini. Akibatnya bean bisa dianggap unused.

Mitigasi:

- gunakan injection eksplisit,
- gunakan `@Unremovable` bila memang perlu,
- gunakan extension build item untuk menandai bean unremovable,
- hindari dynamic service locator pattern,
- pastikan test mencakup path dynamic lookup.

### 12.3 Top-tier invariant

> Di Quarkus, dependency graph yang eksplisit adalah aset. Dependency graph yang tersembunyi adalah risiko.

## 13. Synthetic Beans: Saat Extension Menambahkan Bean ke CDI

Synthetic bean adalah bean yang dibuat/didaftarkan oleh extension, bukan berasal dari class aplikasi biasa yang discan sebagai CDI bean.

Kenapa perlu synthetic bean?

Karena banyak runtime service tidak cocok ditulis sebagai bean biasa di source aplikasi. Contoh konseptual:

- OIDC client,
- datasource,
- REST client,
- Kafka connector,
- metrics registry,
- health check,
- configuration-derived service,
- generated security policy,
- extension-provided runtime component.

Synthetic bean memungkinkan extension berkata:

> “Berdasarkan config dan metadata yang saya temukan saat build, saya akan menyediakan bean runtime ini ke CDI container.”

### 13.1 Synthetic bean vs normal bean

Normal bean:

```java
@ApplicationScoped
public class PaymentService {}
```

Synthetic bean:

```text
Extension build step registers PaymentClient bean based on config/metadata.
```

### 13.2 Kenapa ini penting untuk engineer aplikasi?

Karena banyak hal yang kamu inject di Quarkus sebenarnya tidak muncul dari class aplikasi langsung. Extension yang menyediakannya.

Contoh:

```java
@Inject
SomeQuarkusProvidedClient client;
```

Di balik layar, extension mungkin:

1. membaca config,
2. menemukan interface client,
3. generate implementation,
4. register synthetic bean,
5. menyiapkan native metadata.

Kalau kamu memahami ini, error injection tidak lagi misterius. Kamu akan bertanya:

- extension apa yang harus ada,
- build item apa yang mendaftarkan bean,
- config apa yang mempengaruhi bean,
- apakah bean removable,
- apakah interface ter-index,
- apakah generated implementation berhasil dibuat.

## 14. Generated Bytecode: Mengganti Runtime Reflection dengan Kode Siap Pakai

Quarkus sering menghasilkan bytecode/helper class saat build. Tujuannya:

1. menghindari reflection runtime,
2. mempercepat startup,
3. mengurangi metadata runtime,
4. membuat native image lebih mudah,
5. mengikat keputusan build-time ke kode yang executable.

### 14.1 Apa yang bisa digenerate?

Secara konseptual:

- CDI client proxy,
- subclass proxy,
- bean supplier,
- route handler,
- serialization helper,
- recorder initializer,
- native-image metadata resource,
- REST client implementation,
- config mapping implementation.

### 14.2 Implikasi debugging

Kadang stack trace menampilkan generated class. Jangan panik. Itu bagian dari model Quarkus.

Yang perlu dipahami:

- source code kamu adalah input,
- generated code adalah hasil augmentation,
- runtime menjalankan kombinasi source compiled classes + generated classes + runtime extension classes.

### 14.3 Mental model

> Semakin eksplisit struktur aplikasi, semakin banyak yang bisa digenerate dengan aman. Semakin dynamic aplikasimu, semakin sulit Quarkus mengoptimalkannya.

## 15. Extension Model: Kenapa Quarkus Punya Runtime dan Deployment Module

Quarkus extension umumnya dipisahkan menjadi dua bagian:

```text
my-extension-runtime
my-extension-deployment
```

### 15.1 Runtime module

Runtime module berisi class yang dibutuhkan saat aplikasi berjalan:

- service runtime,
- client runtime,
- handler,
- config runtime,
- public API untuk aplikasi,
- recorder class tertentu,
- runtime support class.

Runtime module masuk ke aplikasi final.

### 15.2 Deployment module

Deployment module berisi logic build-time:

- build steps,
- processor,
- metadata scanner,
- build item producer/consumer,
- native metadata registration,
- generated class logic,
- augmentation logic.

Deployment module dipakai saat build, bukan saat runtime aplikasi final.

### 15.3 Kenapa dipisah?

Karena Quarkus ingin runtime kecil. Kalau class yang hanya dibutuhkan untuk scanning/build ikut masuk runtime, maka memory dan native image akan lebih berat.

Contoh:

```text
Deployment module:
  - annotation scanner
  - XML parser
  - metadata processor
  - code generator

Runtime module:
  - actual HTTP client
  - request handler
  - lightweight runtime service
```

Framework runtime-heavy sering membawa keduanya ke runtime. Quarkus memisahkannya.

### 15.4 Implikasi untuk dependency

Dependency yang hanya perlu build-time jangan masuk runtime.

Contoh:

- annotation processor helper,
- metadata parser,
- bytecode generation tooling,
- descriptor reader,
- testing-only helper.

Kalau salah masuk runtime:

- artifact membesar,
- native image membesar,
- startup/memory memburuk,
- vulnerability surface bertambah,
- dependency conflict lebih banyak.

## 16. Capability System: Extension Tidak Hidup Sendirian

Dalam aplikasi besar, beberapa extension bisa menyediakan feature yang saling terkait atau conflict.

Contoh konseptual:

- extension B membutuhkan JSON support,
- extension C menyediakan alternative JSON support,

### 16.1 Kenapa capability penting?

Karena Quarkus platform adalah ecosystem extension. Tanpa capability system, integrasi antar extension menjadi fragile.

- build steps,
- build items,
- generated metadata,
- native image registrations,
- config namespace,
- possible capabilities,
- possible conflicts,
- runtime services.

## 17. Classpath dan Dependency Resolution di Quarkus

Quarkus Maven/Gradle tooling membedakan runtime extension dependencies dan deployment/build-time extension dependencies. Developer aplikasi biasanya hanya menambahkan runtime artifact extension; Quarkus menyelesaikan deployment side saat build.

Contoh dependency aplikasi:

```xml
<dependency>
    <groupId>io.quarkus</groupId>
```

### 17.1 Kenapa ini penting?

Karena kalau kamu melihat dependency tree, kamu perlu membedakan:

- dependency aplikasi,
- runtime dependency,
- deployment dependency,
- test dependency,
- native build dependency.

Saat troubleshooting, pertanyaan yang benar:

1. Dependency ini muncul di runtime classpath atau build classpath?
2. Extension deployment module versi berapa yang dipakai?
3. Ada conflict antara extension platform BOM dan dependency override?
4. Ada library yang dibutuhkan build-time tapi tidak ter-index?
5. Ada dependency yang membuat native image gagal?

### 17.2 Kesalahan umum

- override dependency extension tanpa memahami platform BOM,
- mencampur versi Quarkus extension yang tidak align,
- menambahkan deployment artifact manual,
- menambahkan library reflection-heavy tanpa native metadata,
- memasukkan annotation processor/helper ke runtime,

### 18.1 Contoh kategori build-time config

- index behavior,

## 19. Native Image Metadata Preparation

Quarkus sangat erat dengan native image. Salah satu tugas extension adalah membantu native image mengetahui hal-hal yang tidak bisa dianalisis otomatis.

Native image memakai closed-world assumption:

> Semua reachable code dan metadata runtime harus diketahui saat build native image.

Masalah Java tradisional:

- reflection,
- dynamic proxy,
- classpath scanning,
- resource loading,
- serialization,
- JNI,
- service loader,
- dynamic classloading.

Quarkus extension bisa mendaftarkan metadata seperti:

- reflective class,
- reflective method/field,
- resource include,
- proxy definition,
- class initialization rule,
- runtime initialized class,
- service provider.

### 19.1 Kenapa extension resmi penting?

Kalau memakai extension resmi, biasanya banyak native metadata sudah disiapkan.

Kalau memakai library biasa tanpa extension:

- JVM mode mungkin berjalan,
- native image mungkin gagal build,
- native executable mungkin build sukses tapi gagal runtime,
- reflection path mungkin hilang,
- resource file mungkin tidak ikut masuk,
- serializer mungkin gagal.

Top-tier rule:

> Untuk aplikasi native Quarkus, dependency bukan hanya soal API. Dependency harus dinilai dari native-image compatibility dan extension support.

## 23. Contoh Mini: Membaca Aplikasi Quarkus sebagai Build-Time Graph

Misal aplikasi punya:

```java
@Path("/cases")
@ApplicationScoped
public class CaseResource {

    @Inject
    CaseService service;

    @GET
    public List<CaseDto> list() {
        return service.list();
    }

```

```java
@ApplicationScoped
public class CaseService {

    @Inject
    CaseRepository repository;

    public List<CaseDto> list() {
        return repository.findOpenCases();
    }
}
```

```java
@ApplicationScoped
public class CaseRepository {
    public List<CaseDto> findOpenCases() {
        // query
    }
}
```

Cara runtime-heavy melihat:

```text
Saat startup, scan classpath, temukan @Path, temukan @ApplicationScoped,

```

Cara Quarkus melihat:

```text
Build:

  gunakan graph dan route yang sudah disiapkan
```

Top-tier insight:

> Endpoint `/cases` bukan hanya method Java. Di Quarkus, ia adalah hasil kerja beberapa extension: REST, Arc/CDI, config, validation, serialization, security, observability, dan mungkin native metadata.

## 24. Contoh Mini: Kenapa Dynamic Classpath Scanning Buruk di Quarkus

Misal kamu membawa library lama:

```java
public class LegacyPluginLoader {
    public List<Object> loadPlugins() {
        Reflections reflections = new Reflections("com.company.plugins");
        return reflections.getTypesAnnotatedWith(Plugin.class)
                .stream()
                .map(type -> type.getDeclaredConstructor().newInstance())
                .toList();
    }
}
```

Di JVM mode, ini mungkin jalan.

Di native mode, masalah muncul:

- class mungkin tidak reachable,
- constructor reflection tidak terdaftar,
- annotation metadata mungkin tidak tersedia,
- resource scanning tidak sama,
- dynamic classloading terbatas,
- startup cost meningkat,
- Quarkus tidak bisa mengoptimalkan graph.

Versi Quarkus-friendly:

```java
@ApplicationScoped
public class PluginRegistry {

    private final List<Plugin> plugins;

    @Inject
    public PluginRegistry(List<Plugin> plugins) {
        this.plugins = List.copyOf(plugins);
    }

    public List<Plugin> plugins() {
        return plugins;
    }
}
```

Atau extension-friendly:

```text
Build step menemukan @Plugin lewat Jandex
  -> register synthetic registry
  -> generate metadata
  -> runtime tinggal memakai registry siap pakai
```

Prinsip:

> Jangan mencari struktur aplikasi saat runtime jika struktur itu bisa diketahui saat build.

## 25. Contoh Mini: Bean Removal dan Service Locator Trap

Anti-pattern:

```java
@ApplicationScoped
public class RiskScoringService {
    public int score(CaseData data) {
        return 100;
    }
}
```

```java
public class DynamicCaller {
    public Object call(String className) {
        Class<?> clazz = Class.forName(className);
        return CDI.current().select(clazz).get();
    }
}
```

Masalah:

- Arc tidak melihat injection eksplisit,
- bean bisa dianggap unused,
- native image tidak menyukai `Class.forName` dinamis,
- authorization/audit sulit,
- dependency graph tidak jelas,
- test coverage mudah bolong.

Lebih baik:

```java
public interface RiskScorer {
    String type();
    int score(CaseData data);
}
```

```java
@ApplicationScoped
public class DefaultRiskScorer implements RiskScorer {
    public String type() { return "default"; }
    public int score(CaseData data) { return 100; }
}
```

```java
@ApplicationScoped
public class RiskScorerRegistry {

    private final Map<String, RiskScorer> scorers;

    @Inject
    public RiskScorerRegistry(List<RiskScorer> scorers) {
        this.scorers = scorers.stream()
                .collect(Collectors.toUnmodifiableMap(RiskScorer::type, Function.identity()));
    }

    public RiskScorer get(String type) {
        RiskScorer scorer = scorers.get(type);
        if (scorer == null) {
            throw new IllegalArgumentException("Unknown scorer type: " + type);
        }
        return scorer;
    }
}
```

Lebih eksplisit, lebih testable, lebih native-friendly, dan lebih mudah diaudit.

## 31. Design Pattern yang Quarkus-Friendly

### 31.1 Explicit registry pattern

Cocok untuk strategi/plugin yang diketahui saat build.

```java
public interface CaseTransitionHandler {
    CaseStatus from();
    CaseStatus to();
    void handle(CaseTransition transition);
}
```

```java
@ApplicationScoped
public class CaseTransitionRegistry {

    private final Map<Key, CaseTransitionHandler> handlers;

    @Inject
    public CaseTransitionRegistry(List<CaseTransitionHandler> handlers) {
        this.handlers = handlers.stream()
                .collect(Collectors.toUnmodifiableMap(
                    h -> new Key(h.from(), h.to()),
                    Function.identity()
                ));
    }

    public CaseTransitionHandler resolve(CaseStatus from, CaseStatus to) {
        CaseTransitionHandler handler = handlers.get(new Key(from, to));
        if (handler == null) {
            throw new IllegalStateException("No transition handler for " + from + " -> " + to);
        }
        return handler;
    }
}
```

Mengapa baik:

- dependency eksplisit,
- Arc melihat semua handler,
- native-friendly,
- mudah test,
- audit transition jelas.

### 31.2 Config mapping pattern

```java
@ConfigMapping(prefix = "case.workflow")
public interface CaseWorkflowConfig {
    Duration escalationGracePeriod();
    int maxReopenCount();
    boolean strictApprovalMode();
}
```

Mengapa baik:

- type-safe,
- fail fast,
- mudah validasi,
- jelas untuk runtime/config owner.

### 31.3 Extension-backed infrastructure pattern

Untuk cross-cutting internal platform, jangan copy-paste filter di semua service. Buat extension/internal library yang:

- register correlation filter,
- register standardized error mapper,
- register audit event helper,
- register OpenTelemetry attributes,
- register security policy adapter,
- register native metadata.

- register additional beans,
- register request filter,
- register error mapper,
- register native reflection metadata bila perlu,
- validate annotation usage,
- produce build item untuk audit metadata,
- expose Dev UI diagnostics.

Runtime module:

- audit writer,
- context extractor,
- masking service,
- event serializer,
- correlation provider,
- runtime config.

Benefit:

- policy konsisten,
- build-time validation,
- native-friendly,
- service code lebih bersih,
- audit defensibility meningkat,
- upgrade terpusat.

Top-tier insight:

> Quarkus internal architecture memungkinkan organisasi membangun internal platform, bukan hanya aplikasi individual.

## 34. Latihan Mental Model

### Latihan 1 — Klasifikasi Build-Time vs Runtime

Untuk setiap item berikut, tentukan apakah build-time, static init, runtime init, atau runtime operation:

4. Generate CDI proxy.
5. Menentukan datasource kind.
6. Membuat token OIDC client credential.
7. Register class untuk reflection native image.
8. Membaca request header `Authorization`.
9. Membuat routing table REST.
10. Mengirim audit event setelah case approved.

### Latihan 2 — Debug Bean Missing

Kasus:

```text
Unsatisfied dependency for type RiskScoringService
```

Langkah analisis:

1. Apakah class punya scope?
2. Apakah package `jakarta.enterprise.context.ApplicationScoped` benar?
3. Apakah module ter-index?
4. Apakah dependency masuk runtime classpath?
5. Apakah bean dihapus?
6. Apakah qualifier injection cocok?
7. Apakah bean hanya dibuat untuk profile tertentu?
8. Apakah ada extension yang seharusnya mendaftarkan synthetic bean?

### Latihan 3 — Native Image Failure

Kasus:

```text
ClassNotFoundException saat native executable memanggil plugin tertentu
```

Analisis:

1. Apakah plugin ditemukan via `Class.forName`?
2. Apakah class reachable saat native build?
3. Apakah reflection metadata ada?
4. Apakah resource plugin masuk image?
5. Apakah ServiceLoader dipakai?
6. Apakah ada Quarkus extension untuk library itu?
7. Bisa diganti menjadi CDI registry eksplisit?

## 35. Production Checklist Part Ini

Gunakan checklist ini saat review aplikasi Quarkus:

### 35.1 Build-time clarity

- [ ] Extension yang digunakan jelas dan align dengan Quarkus platform BOM.
- [ ] Tidak ada dependency override liar.
- [ ] Library internal yang berisi bean/entity ter-index.
- [ ] Build-time config dipahami dan tidak diubah sembarangan di runtime.
- [ ] Build pipeline menjalankan package/augmentation, bukan hanya unit test.

### 35.2 CDI/Arc clarity

- [ ] Dependency graph eksplisit.
- [ ] Tidak overuse `CDI.current()`.
- [ ] Tidak overuse `Class.forName` untuk service runtime.
- [ ] Bean removal dipahami.
- [ ] Bean yang memang dynamic/entry point ditandai dengan benar.

### 35.3 Native readiness

- [ ] Library reflection-heavy diidentifikasi.
- [ ] Resource inclusion dicek.
- [ ] Dynamic proxy dicek.
- [ ] Class initialization risk dicek.
- [ ] Native test pipeline tersedia bila native mode target production.

### 35.4 Extension/platform readiness

- [ ] Cross-cutting concern tidak copy-paste berlebihan.
- [ ] Ada peluang membuat internal Quarkus extension untuk policy bersama.
- [ ] Runtime module dan deployment module dipisah bila membuat extension.
- [ ] Build item digunakan untuk komunikasi build-time.
- [ ] Tidak memakai static mutable state untuk build-step communication.

## 36. Ringkasan Invariants

Ingat invariants berikut:

1. **Quarkus memindahkan kerja runtime ke build-time sebanyak mungkin.**
2. **Augmentation adalah jantung Quarkus.**
3. **Extension adalah participant dalam build pipeline, bukan library pasif.**
4. **Deployment module untuk build-time; runtime module untuk runtime.**
5. **Build item adalah bahasa komunikasi antar build step.**
6. **Recorder menjembatani keputusan build-time dengan initialization runtime/static init.**
7. **Jandex membuat annotation/class discovery lebih murah dan build-time friendly.**
8. **Arc/CDI mengoptimalkan dependency graph saat build dan bisa menghapus unused beans.**
9. **Dynamic runtime behavior harus eksplisit agar aman untuk Quarkus dan native image.**
10. **Native image memperbesar pentingnya build-time determinism.**
11. **Build-time config membentuk artifact; runtime config mengoperasikan artifact.**
12. **Quarkus-friendly design berarti dependency graph jelas, initialization boundary jelas, dan runtime magic minimal.**
