## 0. Posisi Part Ini Dalam Seri

Pada Part 001 sampai Part 004, kita sudah membangun fondasi bahwa Quarkus bukan sekadar framework runtime biasa. Quarkus bekerja dengan cara:

1. banyak keputusan dipindahkan ke **build time**;
2. extension tidak hanya menambahkan dependency, tetapi ikut melakukan augmentation;
3. dev mode, Dev UI, continuous testing, dan Dev Services mempercepat feedback loop;
4. native image compatibility menuntut dependency lebih disiplin daripada aplikasi JVM tradisional.

Part ini membahas lapisan yang sering terlihat “administratif”, padahal untuk sistem enterprise justru sangat menentukan umur panjang aplikasi:

- bagaimana project disusun;
- bagaimana Maven/Gradle dikendalikan;
- bagaimana BOM menjaga kompatibilitas;
- bagaimana extension dipilih;
- bagaimana dependency drift dicegah;
- bagaimana upgrade dilakukan tanpa chaos;
- bagaimana organisasi membuat standar internal untuk banyak service Quarkus.

Kalau Part 003 membahas **bagaimana Quarkus bekerja dari dalam**, maka Part 005 membahas **bagaimana sebuah tim harus mengelola project Quarkus agar cara kerja internal itu tidak dirusak oleh dependency dan struktur yang kacau**.

## 1. Masalah Nyata yang Diselesaikan Part Ini

Di project kecil, struktur build sering dianggap tidak penting. Selama `mvn quarkus:dev` atau `./gradlew quarkusDev` berjalan, dianggap selesai.

Di project enterprise, masalahnya berbeda.

Aplikasi Quarkus biasanya bergantung pada banyak hal:

- REST layer;
- CDI;
- validation;
- security/OIDC;
- Hibernate ORM;
- database driver;
- migration tool;
- messaging connector;
- Redis/cache;
- OpenTelemetry;
- Kubernetes extension;
- native image metadata;
- internal shared libraries;
- organization-wide logging/security/audit library.

Masalah yang muncul bukan lagi “bagaimana menambahkan dependency”, tetapi:

1. **Apakah semua extension kompatibel dengan versi Quarkus yang dipakai?**
2. **Apakah dependency third-party aman untuk native image?**
3. **Apakah versi library dikunci oleh BOM atau liar di setiap module?**
4. **Apakah plugin Quarkus, BOM, dan extension catalog memakai versi yang konsisten?**
5. **Apakah multi-module project mengaburkan boundary domain?**
6. **Apakah shared library internal membawa transitive dependency yang merusak augmentation?**
7. **Apakah upgrade Quarkus bisa dilakukan otomatis atau selalu menjadi proyek besar?**
8. **Apakah setiap microservice punya standar build yang sama?**
9. **Apakah struktur project membantu debugging, testing, native build, dan deployment?**
10. **Apakah organisasi punya platform governance atau semua tim bebas memilih kombinasi extension sendiri?**

Part ini menjawab semua pertanyaan itu.

## 2. Mental Model Utama: Quarkus Project Bukan Sekadar Java Project

Project Java biasa biasanya dipahami seperti ini:

```text
source code + dependencies + plugins -> jar
```

Pada Quarkus, modelnya lebih tepat seperti ini:

```text
source code
+ Quarkus platform BOM
+ Quarkus extensions
+ build-time augmentation
+ generated classes
+ config classification
+ native-image metadata
+ runtime dependency graph
-> optimized runnable artifact
```

Artinya, dependency di Quarkus tidak pasif.

Dependency bisa:

- ikut build step;
- menghasilkan bean;
- mendaftarkan reflection metadata;
- menambahkan generated bytecode;
- menambahkan config namespace;
- mempengaruhi native image;
- mempengaruhi dev mode;
- mempengaruhi test bootstrap;
- mempengaruhi extension capability conflict.

Jadi struktur project dan build governance bukan cosmetic. Ia adalah bagian dari runtime behavior.

### 2.1 Analogi Sederhana

Pada Java tradisional:

```text
Dependency = library yang dipanggil saat runtime.
```

Pada Quarkus:

```text
Extension = library + build-time integration + runtime behavior contract.
```

Karena itu, menambahkan extension Quarkus tidak sama dengan menambahkan random Maven dependency.

## 3. Apa Itu Quarkus Platform BOM?

Quarkus platform BOM adalah mekanisme version alignment. Ia memastikan extension dan dependency utama yang berada dalam satu platform release kompatibel satu sama lain.

Tujuan utamanya:

1. menjaga version compatibility antar dependency;
2. menyederhanakan dependency management;
3. memudahkan upgrade;
4. mengurangi dependency conflict;
5. menyelaraskan extension runtime dan deployment artifacts;
6. memberi basis yang stabil untuk dev tools seperti Quarkus CLI, Maven plugin, dan Gradle plugin.

Secara praktis, di Maven kita biasanya melihat pola seperti ini:

```xml
<properties>
    <quarkus.platform.group-id>io.quarkus.platform</quarkus.platform.group-id>
    <quarkus.platform.artifact-id>quarkus-bom</quarkus.platform.artifact-id>
    <quarkus.platform.version>...</quarkus.platform.version>
</properties>

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>${quarkus.platform.group-id}</groupId>
            <artifactId>${quarkus.platform.artifact-id}</artifactId>
            <version>${quarkus.platform.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

Lalu extension ditambahkan tanpa versi eksplisit:

```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-rest</artifactId>
</dependency>

<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-hibernate-orm</artifactId>
</dependency>

<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-jdbc-postgresql</artifactId>
</dependency>
```

Ketiadaan `<version>` bukan kelalaian. Itu justru discipline. Versi diambil dari platform BOM.

### 3.1 Prinsip Penting

Untuk extension yang dikelola oleh Quarkus platform:

```text
Jangan tulis version per dependency.
Gunakan BOM sebagai source of truth.
```

Kalau setiap developer menulis version sendiri, maka BOM kehilangan fungsi governance-nya.

## 4. BOM vs Plugin Version vs Application Version

Sering terjadi kebingungan antara:

1. Quarkus platform version;
2. Quarkus Maven/Gradle plugin version;
3. application version;
4. Java version;
5. dependency third-party version.

Mereka tidak sama.

### 4.1 Quarkus Platform Version

Ini versi platform yang mengatur kompatibilitas extension:

```text
io.quarkus.platform:quarkus-bom:3.x.y
```

Ia menentukan versi extension Quarkus dan dependency terkait.

### 4.2 Quarkus Plugin Version

Plugin digunakan untuk build, dev mode, test augmentation, native build, code generation, dan packaging.

Maven contoh:

```xml
<plugin>
    <groupId>${quarkus.platform.group-id}</groupId>
    <artifactId>quarkus-maven-plugin</artifactId>
    <version>${quarkus.platform.version}</version>
    <extensions>true</extensions>
    <executions>
        <execution>
            <goals>
                <goal>build</goal>
                <goal>generate-code</goal>
                <goal>generate-code-tests</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

Biasanya plugin version disamakan dengan platform version.

### 4.3 Application Version

Ini versi aplikasi kamu sendiri:

```xml
<groupId>com.company.case</groupId>
<artifactId>case-service</artifactId>
<version>1.12.0</version>
```

Jangan campur application version dengan Quarkus version.

### 4.4 Java Version

Contoh:

```xml
<maven.compiler.release>21</maven.compiler.release>
```

Atau:

```xml
<maven.compiler.release>17</maven.compiler.release>
```

Java version menentukan language level dan bytecode target, sementara Quarkus platform menentukan framework/runtime ecosystem.

### 4.5 Third-Party Dependency Version

Untuk dependency yang tidak dikelola Quarkus BOM, versi harus dikelola secara sadar.

Contoh:

```xml
<dependency>
    <groupId>com.company.platform</groupId>
    <artifactId>company-audit-contract</artifactId>
    <version>${company.audit.version}</version>
</dependency>
```

Atur di parent atau internal BOM, jangan tersebar liar.

## 6. Package Structure: Jangan Ikuti Framework, Ikuti Boundary

Anti-pattern umum:

```text
com.company.app.controller
com.company.app.service
com.company.app.repository
com.company.app.entity
com.company.app.dto
```

Struktur ini terlihat rapi, tapi sebenarnya teknis-sentris. Ia mengelompokkan berdasarkan layer teknis, bukan domain atau business capability.

Untuk aplikasi kecil, ini bisa diterima. Untuk domain kompleks, struktur ini cepat memburuk karena semua use case masuk ke folder yang sama.

### 6.1 Struktur Berdasarkan Capability

Lebih baik:

```text
com.company.caseapp
  casefile/
    api/
    application/
    domain/
    persistence/
    messaging/
  inspection/
    api/
    application/
    domain/
    persistence/
  enforcement/
    api/
    application/
    domain/
    persistence/
  shared/
    error/
    security/
    telemetry/
    config/
```

Keuntungan:

- boundary lebih jelas;
- ownership lebih mudah;
- testing lebih terarah;
- future extraction ke microservice lebih realistis;
- dependency antar module/domain lebih mudah diaudit;
- tidak semua logic menumpuk di `service` package.

### 6.2 Struktur Hexagonal yang Lebih Eksplisit

Untuk domain berat:

```text
com.company.caseapp.casefile
  domain/
    model/
    event/
    policy/
    repository/
    service/
  application/
    command/
    query/
    handler/
    workflow/
  adapter/
    inbound/
      rest/
      messaging/
      scheduler/
    outbound/
      persistence/
      externalapi/
      messaging/
```

Di Quarkus, struktur seperti ini tetap cocok, selama CDI discovery dan bean scope dikelola dengan benar.

### 6.3 Prinsip Dependency Direction

```text
api/rest adapter
    -> application use case
        -> domain
        -> domain repository interface
            <- persistence adapter implements repository
```

Jangan sampai domain tergantung pada Quarkus annotation yang tidak perlu.

Contoh domain entity murni:

```java
package com.company.caseapp.casefile.domain.model;

public final class CaseFile {
    private final CaseId id;
    private CaseStatus status;

    public void submit(SubmissionPolicy policy) {
        if (!policy.canSubmit(this)) {
            throw new CaseTransitionRejected("Case cannot be submitted");
        }
        this.status = CaseStatus.SUBMITTED;
    }
}
```

Contoh application service yang pakai CDI:

```java
package com.company.caseapp.casefile.application;

import jakarta.enterprise.context.ApplicationScoped;
import jakarta.transaction.Transactional;

@ApplicationScoped
public class SubmitCaseUseCase {
    private final CaseFileRepository repository;
    private final SubmissionPolicy policy;

    public SubmitCaseUseCase(CaseFileRepository repository, SubmissionPolicy policy) {
        this.repository = repository;
        this.policy = policy;
    }

    @Transactional
    public void submit(SubmitCaseCommand command) {
        CaseFile caseFile = repository.get(command.caseId());
        caseFile.submit(policy);
        repository.save(caseFile);
    }
}
```

Framework annotation ada di application boundary, bukan menyebar tanpa kontrol ke domain core.

## 7. Single Module vs Multi-Module

Tidak semua Quarkus service butuh multi-module. Multi-module menambah overhead. Tetapi untuk aplikasi besar, multi-module bisa membantu governance.

### 7.1 Single Module Cocok Jika

- service kecil sampai sedang;
- domain tidak terlalu banyak;
- tim kecil;
- native build ingin sederhana;
- CI/CD ingin cepat;
- boundary cukup dikelola dengan package.

Struktur:

```text
case-service/
  pom.xml
  src/main/java/...
```

### 7.2 Multi-Module Cocok Jika

- service besar;
- banyak bounded context;
- ada shared kernel;
- ada generated client/server code;
- ada domain module yang ingin bebas framework;
- ada extension internal;
- ada test fixtures;
- ada contract module;
- perlu enforce dependency direction.

Contoh:

```text
case-platform/
  pom.xml
  case-domain/
  case-application/
  case-adapter-rest/
  case-adapter-persistence/
  case-adapter-messaging/
  case-service-app/
  case-test-support/
```

Namun Quarkus application artifact biasanya tetap ada satu module utama yang melakukan bootstrapping:

```text
case-service-app
```

Module lain adalah dependency library.

### 7.3 Multi-Module Trap

Multi-module sering gagal karena dibuat berdasarkan layer teknis global:

```text
api-module
service-module
repository-module
entity-module
common-module
```

Ini biasanya buruk karena:

- semua domain bercampur;
- module saling tergantung;
- `common` menjadi tempat sampah;
- build lambat;
- domain boundary tetap tidak jelas;
- dependency graph makin kusut.

Lebih baik module berdasarkan capability atau architectural boundary yang nyata.

## 8. Parent POM Strategy

Untuk Maven, parent POM berfungsi sebagai governance point.

Contoh parent sederhana:

```xml
<project>
    <modelVersion>4.0.0</modelVersion>

    <groupId>com.company.platform</groupId>
    <artifactId>company-quarkus-parent</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>

    <properties>
        <java.version>21</java.version>
        <maven.compiler.release>${java.version}</maven.compiler.release>
        <quarkus.platform.group-id>io.quarkus.platform</quarkus.platform.group-id>
        <quarkus.platform.artifact-id>quarkus-bom</quarkus.platform.artifact-id>
        <quarkus.platform.version>3.x.y</quarkus.platform.version>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    </properties>

    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>${quarkus.platform.group-id}</groupId>
                <artifactId>${quarkus.platform.artifact-id}</artifactId>
                <version>${quarkus.platform.version}</version>
                <type>pom</type>
                <scope>import</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>

    <build>
        <pluginManagement>
            <plugins>
                <plugin>
                    <groupId>${quarkus.platform.group-id}</groupId>
                    <artifactId>quarkus-maven-plugin</artifactId>
                    <version>${quarkus.platform.version}</version>
                </plugin>
            </plugins>
        </pluginManagement>
    </build>
</project>
```

### 8.1 Parent POM Harus Mengatur

- Java version;
- Quarkus BOM;
- Quarkus plugin;
- compiler settings;
- surefire/failsafe;
- dependency convergence rules;
- Maven Enforcer;
- formatting/checkstyle jika digunakan;
- test plugin behavior;
- container/native profile;
- organization-wide repository policy.

### 8.2 Parent POM Tidak Boleh Menjadi Tempat Sampah

Jangan masukkan semua dependency ke parent jika tidak semua service memakainya.

Buruk:

```xml
<dependencies>
    <dependency>quarkus-rest</dependency>
    <dependency>quarkus-hibernate-orm</dependency>
    <dependency>quarkus-kafka-client</dependency>
    <dependency>quarkus-redis-client</dependency>
    <dependency>quarkus-oidc</dependency>
</dependencies>
```

Jika parent memaksa semua service membawa semua extension, maka:

- startup bisa membengkak;
- augmentation lebih kompleks;
- native image lebih sulit;
- attack surface membesar;
- dependency conflict makin tinggi;
- service kecil menjadi berat.

Lebih baik parent mengatur version dan plugin, bukan memaksa semua dependency runtime.

## 9. Internal BOM Strategy

Untuk organisasi besar, Quarkus BOM saja tidak cukup.

Biasanya butuh:

```text
Quarkus Platform BOM
+ Company Internal BOM
+ Service dependencies
```

Contoh:

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>io.quarkus.platform</groupId>
            <artifactId>quarkus-bom</artifactId>
            <version>${quarkus.platform.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>

        <dependency>
            <groupId>com.company.platform</groupId>
            <artifactId>company-service-bom</artifactId>
            <version>${company.platform.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

Company BOM bisa mengatur:

- internal audit library;
- internal error contract;
- internal security policy client;
- internal API client;
- internal telemetry helper;
- approved Jackson modules;
- approved database drivers jika tidak dari Quarkus BOM;
- approved testing libraries;
- approved plugin versions.

### 9.1 Urutan BOM Penting

Dependency management resolution bisa dipengaruhi urutan dan override.

Prinsip aman:

1. Quarkus platform BOM sebagai dasar;
2. company BOM hanya override jika benar-benar perlu;
3. override harus punya alasan dan test compatibility;
4. jangan override dependency internal Quarkus sembarangan.

### 9.2 Policy untuk Override

Setiap override harus mencatat:

```text
dependency:
old version:
new version:
reason:
security advisory / bug / compatibility:
native image impact:
test evidence:
rollback plan:
owner:
expiry/review date:
```

Ini terdengar berat, tapi sangat penting untuk sistem production.

## 10. Extension Governance

Extension governance berarti organisasi punya aturan tentang extension apa yang boleh dipakai, kapan, dan dengan constraint apa.

Tidak semua extension punya tingkat kematangan yang sama.

Ada extension core Quarkus, ada extension Quarkiverse, ada extension vendor, ada extension internal.

### 10.1 Kategori Extension

```text
Tier 1: Core Quarkus extension
Tier 2: Quarkiverse mature extension
Tier 3: Vendor-supported extension
Tier 4: Internal extension
Tier 5: Experimental/community extension
```

Setiap tier punya risk berbeda.

### 10.2 Checklist Sebelum Menambahkan Extension

Sebelum menambah extension, tanyakan:

1. Apakah extension ini ada di Quarkus platform?
2. Apakah extension ini kompatibel dengan versi Quarkus yang dipakai?
3. Apakah extension ini mendukung native image?
4. Apakah extension ini membawa transitive dependency besar?
5. Apakah extension ini aktif dipelihara?
6. Apakah ada alternatif core Quarkus?
7. Apakah extension ini menambahkan build-time behavior?
8. Apakah extension ini mempengaruhi security posture?
9. Apakah extension ini mempengaruhi observability?
10. Apakah extension ini benar-benar dibutuhkan?

### 10.3 Extension Addition Decision Record

Gunakan template ADR singkat:

```markdown
# ADR: Add quarkus-redis-client

## Context
Service needs distributed cache for postal-code lookup and token caching.

## Decision
Use `io.quarkus:quarkus-redis-client` managed by Quarkus platform BOM.

## Alternatives
- Caffeine local cache
- Direct Jedis/Lettuce dependency
- Infinispan

## Consequences
- Requires Redis availability in prod
- Requires Dev Services or test container for local/test
- Needs timeout/retry discipline
- Cache invalidation and TTL must be explicit

## Native Image Impact
Supported by Quarkus extension; native test required before production native mode.

## Owner
Platform/backend team.
```

## 11. Quarkus Extension vs Plain Java Dependency

Ini salah satu distinction paling penting.

### 11.1 Plain Dependency

Contoh:

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>some-client</artifactId>
    <version>1.2.3</version>
</dependency>
```

Plain dependency hanya masuk classpath. Quarkus tidak otomatis tahu:

- annotation apa yang perlu diproses;
- bean apa yang harus dibuat;
- reflection metadata apa yang diperlukan;
- resource apa yang harus dimasukkan native image;
- class mana yang harus runtime-init;
- config namespace apa yang valid.

### 11.2 Quarkus Extension

Contoh:

```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-rest-client-jackson</artifactId>
</dependency>
```

Extension bisa memberi Quarkus informasi build-time.

Ia bisa:

- mendaftarkan beans;
- mendaftarkan reflection;
- menghasilkan bytecode;
- menambahkan config;
- mengatur native image;
- mengintegrasikan Dev Services;
- menambahkan Dev UI page;
- menambahkan health/metrics/tracing integration.

### 11.3 Prinsip

Jika tersedia Quarkus extension resmi, biasanya pilih extension daripada library mentah.

Contoh:

| Kebutuhan | Pilihan Lebih Baik | Hindari Jika Tidak Perlu |
|---|---|---|
| REST server | `quarkus-rest` | embedded server manual |
| REST client | `quarkus-rest-client-*` | raw OkHttp untuk semua hal |
| ORM | `quarkus-hibernate-orm` | manual Hibernate bootstrap |
| Redis | `quarkus-redis-client` | raw client tanpa integration |
| OIDC | `quarkus-oidc` | custom filter token manual |
| Kafka | reactive messaging/kafka extension | raw consumer loop tanpa lifecycle |

Bukan berarti plain dependency dilarang. Tetapi semakin dekat dependency ke runtime infrastructure, semakin besar alasan memakai extension.

## 12. Maven Tooling Deep Dive

Maven tetap banyak dipakai di ekosistem Java enterprise. Quarkus punya tooling Maven yang matang.

Perintah umum:

```bash
./mvnw quarkus:dev
./mvnw test
./mvnw package
./mvnw package -Dnative
./mvnw quarkus:add-extension -Dextensions="rest,hibernate-orm,jdbc-postgresql"
./mvnw quarkus:update
```

### 12.1 Maven Wrapper Wajib

Gunakan Maven Wrapper:

```text
mvnw
mvnw.cmd
.mvn/wrapper/
```

Alasannya:

- developer tidak perlu install versi Maven yang sama;
- CI konsisten;
- onboarding lebih mudah;
- build reproducibility lebih baik.

### 12.2 Maven Enforcer

Tambahkan Maven Enforcer untuk mencegah drift.

Contoh rule:

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-enforcer-plugin</artifactId>
    <version>3.5.0</version>
    <executions>
        <execution>
            <id>enforce</id>
            <goals>
                <goal>enforce</goal>
            </goals>
            <configuration>
                <rules>
                    <requireJavaVersion>
                        <version>[21,)</version>
                    </requireJavaVersion>
                    <requireMavenVersion>
                        <version>[3.9,)</version>
                    </requireMavenVersion>
                    <dependencyConvergence />
                </rules>
            </configuration>
        </execution>
    </executions>
</plugin>
```

Catatan: dependency convergence bisa terlalu ketat untuk beberapa project. Tetapi minimal jalankan secara berkala di CI atau dependency audit job.

### 12.3 Surefire dan Failsafe

Pisahkan test cepat dan test integrasi.

```text
Unit/component test       -> surefire
Integration/native test   -> failsafe
```

Quarkus test bisa berat jika semua hal di-boot untuk setiap test. Maka layering test penting.

### 12.4 Maven Profile untuk Native

Contoh:

```xml
<profiles>
    <profile>
        <id>native</id>
        <activation>
            <property>
                <name>native</name>
            </property>
        </activation>
        <properties>
            <quarkus.native.enabled>true</quarkus.native.enabled>
            <skipITs>false</skipITs>
        </properties>
    </profile>
</profiles>
```

Jangan jalankan native build di setiap commit jika terlalu mahal. Tapi native build harus masuk gate sebelum release jika production memakai native mode.

## 13. Gradle Tooling Deep Dive

Gradle cocok untuk organisasi yang butuh build lebih fleksibel atau multi-module besar.

Contoh `build.gradle` sederhana:

```groovy
plugins {
    id 'java'
    id 'io.quarkus'
}

repositories {
    mavenCentral()
    mavenLocal()
}

dependencies {
    implementation enforcedPlatform("${quarkusPlatformGroupId}:${quarkusPlatformArtifactId}:${quarkusPlatformVersion}")
    implementation 'io.quarkus:quarkus-rest'
    implementation 'io.quarkus:quarkus-arc'
    testImplementation 'io.quarkus:quarkus-junit5'
    testImplementation 'io.rest-assured:rest-assured'
}

java {
    sourceCompatibility = JavaVersion.VERSION_21
    targetCompatibility = JavaVersion.VERSION_21
}
```

### 13.1 Gradle Wrapper Wajib

Gunakan:

```text
gradlew
gradlew.bat
gradle/wrapper/
```

### 13.2 Version Catalog

Untuk banyak service Gradle, gunakan version catalog:

```toml
[versions]
quarkus = "3.x.y"

[libraries]
quarkus-bom = { module = "io.quarkus.platform:quarkus-bom", version.ref = "quarkus" }

[plugins]
quarkus = { id = "io.quarkus", version.ref = "quarkus" }
```

### 13.3 Gradle Multi-Module

Contoh:

```text
settings.gradle
build.gradle
case-domain/build.gradle
case-application/build.gradle
case-service-app/build.gradle
```

Pastikan hanya module app yang memakai plugin Quarkus jika module lain hanya library Java biasa.

Buruk:

```text
Semua module memakai plugin Quarkus tanpa alasan.
```

Akibatnya:

- build lebih berat;
- augmentation tidak perlu;
- module domain ikut framework-dependent;
- CI lebih lambat.

## 14. Maven vs Gradle: Pilih Berdasarkan Kebutuhan, Bukan Selera

| Aspek | Maven | Gradle |
|---|---|---|
| Enterprise familiarity | Sangat tinggi | Tinggi, tergantung organisasi |
| Convention | Kuat | Fleksibel |
| Multi-module besar | Bisa, tapi verbose | Sangat kuat |
| Build customization | Lebih terbatas | Sangat fleksibel |
| Reproducibility | Bagus jika disiplin | Bagus jika wrapper/cache dikontrol |
| Learning curve | Lebih rendah | Lebih tinggi |
| Quarkus support | Mature | Mature |

### 14.1 Rekomendasi Praktis

Gunakan Maven jika:

- organisasi Java enterprise tradisional;
- banyak developer familiar Maven;
- build tidak terlalu custom;
- simplicity lebih penting.

Gunakan Gradle jika:

- monorepo/multi-module besar;
- build pipeline kompleks;
- butuh custom task;
- butuh build cache serius;
- tim sudah mature dengan Gradle.

Yang buruk bukan Maven atau Gradle. Yang buruk adalah build system tanpa governance.

## 15. Repository Layout untuk Microservices

### 15.1 Polyrepo

```text
github.com/company/case-service
github.com/company/profile-service
github.com/company/payment-service
github.com/company/audit-service
```

Keuntungan:

- ownership jelas;
- pipeline per service;
- release independent;
- dependency lebih kecil.

Kelemahan:

- standar bisa drift;
- shared library versioning lebih rumit;
- cross-service change lebih berat;
- upgrade Quarkus massal lebih sulit.

### 15.2 Monorepo

```text
company-services/
  services/
    case-service/
    profile-service/
    audit-service/
  libraries/
    platform-error/
    platform-audit/
    platform-security/
  build-logic/
```

Keuntungan:

- standardization kuat;
- refactor lintas service lebih mudah;
- build logic bisa shared;
- upgrade platform bisa lebih seragam.

Kelemahan:

- CI bisa kompleks;
- ownership harus disiplin;
- dependency graph bisa membesar;
- butuh tooling lebih matang.

### 15.3 Rule of Thumb

Kalau organisasi belum punya build engineering kuat, polyrepo lebih sederhana. Kalau organisasi punya platform team dan CI matang, monorepo bisa sangat kuat.

## 16. Shared Library: Berguna Tapi Berbahaya

Shared library sering dimulai dengan niat baik:

```text
common-utils
common-model
common-exception
common-security
common-audit
```

Lalu berubah menjadi coupling monster.

### 16.1 Library yang Relatif Aman

- error response contract;
- audit event schema;
- correlation ID helper;
- test fixtures;
- generated API client;
- domain-agnostic value object;
- security policy client;
- observability conventions.

### 16.2 Library yang Berbahaya

- `common-entity`;
- `common-repository`;
- `common-service`;
- `common-controller`;
- `common-domain` tanpa boundary jelas;
- library yang membawa banyak Quarkus extension transitive;
- library yang melakukan runtime scanning/reflection sendiri.

### 16.3 Prinsip Shared Library di Quarkus

Shared library harus:

1. minim transitive dependency;
2. tidak diam-diam membawa extension;
3. native-image aware;
4. punya semantic versioning;
5. punya compatibility matrix dengan Quarkus version;
6. tidak memaksa semua service mengikuti domain model yang sama;
7. tidak menjadi tempat membuang logic yang tidak jelas pemiliknya.

## 17. Application Module vs Library Module

Dalam Quarkus, bedakan:

```text
Application module = module yang di-augment dan menghasilkan runnable artifact.
Library module     = module Java biasa yang digunakan oleh application module.
```

Contoh:

```text
case-domain         -> Java library
case-application    -> Java library, mungkin CDI annotations minimal
case-adapter-rest   -> Java library atau bagian app
case-service-app    -> Quarkus app
```

Application module memiliki:

- Quarkus plugin;
- application.properties;
- extension dependencies;
- generated build output;
- test bootstrap;
- packaging config.

Library module sebaiknya tidak sembarangan memiliki:

- Quarkus plugin;
- runtime config;
- extension dependency;
- app lifecycle assumption.

## 19. `application.properties` Governance

Project structure bukan hanya folder Java. Config juga bagian dari struktur.

Anti-pattern:

```properties
# application.properties berisi semua hal untuk semua env
quarkus.datasource.jdbc.url=jdbc:postgresql://prod-db:5432/app
quarkus.datasource.username=prod_user
quarkus.datasource.password=super-secret
```

Lebih baik:

```properties
quarkus.application.name=case-service

quarkus.datasource.db-kind=postgresql
quarkus.hibernate-orm.database.generation=none
quarkus.flyway.migrate-at-start=true

%dev.quarkus.datasource.devservices.enabled=true
%test.quarkus.datasource.devservices.enabled=true

%prod.quarkus.datasource.jdbc.url=${DB_URL}
%prod.quarkus.datasource.username=${DB_USERNAME}
%prod.quarkus.datasource.password=${DB_PASSWORD}
```

### 19.1 Config Ownership

Setiap config harus punya ownership:

| Config Type | Owner |
|---|---|
| Build-time framework config | Backend/platform engineer |
| Runtime datasource URL | Deployment/platform team |
| Secret value | Secret management system |
| Feature flag | Product/platform decision |
| Observability endpoint | SRE/platform |
| Security issuer/audience | IAM/security team |

### 19.2 Jangan Campur Build-Time dan Runtime Decision

Sebagian config Quarkus bersifat build-time fixed. Mengubahnya di runtime tidak selalu punya efek.

Maka dokumentasikan:

```text
Config A boleh diubah via env var saat deploy.
Config B membutuhkan rebuild image.
```

## 20. Dependency Scope Discipline

Dependency scope menentukan artifact dan classpath.

### 20.1 Scope Umum

```xml
<scope>compile</scope>
<scope>runtime</scope>
<scope>test</scope>
<scope>provided</scope>
```

Untuk Quarkus extension, biasanya cukup dependency normal tanpa scope eksplisit.

### 20.2 Test Dependency Jangan Bocor

Contoh:

```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-junit5</artifactId>
    <scope>test</scope>
</dependency>

<dependency>
    <groupId>io.rest-assured</groupId>
    <artifactId>rest-assured</artifactId>
    <scope>test</scope>
</dependency>
```

Jangan sampai testing framework masuk runtime artifact.

### 20.3 Annotation Processor

Untuk Lombok/MapStruct, kelola annotation processor dengan jelas.

Contoh Maven compiler plugin:

```xml
<annotationProcessorPaths>
    <path>
        <groupId>org.mapstruct</groupId>
        <artifactId>mapstruct-processor</artifactId>
        <version>${mapstruct.version}</version>
    </path>
    <path>
        <groupId>org.projectlombok</groupId>
        <artifactId>lombok</artifactId>
        <version>${lombok.version}</version>
    </path>
</annotationProcessorPaths>
```

Native image tidak langsung peduli annotation processor, tetapi generated code bisa mempengaruhi reflection, serialization, dan CDI visibility.

## 21. Dependency Drift: Musuh Jangka Panjang

Dependency drift terjadi ketika versi dependency berkembang tanpa kontrol.

Contoh tanda-tanda:

- service A Quarkus 3.8;
- service B Quarkus 3.15;
- service C Quarkus 3.2;
- service D override Netty sendiri;
- service E pakai extension Quarkiverse lama;
- service F memakai custom Jackson version;
- service G gagal native build karena library berbeda.

Akibatnya:

- security patch sulit;
- upgrade mahal;
- bug antar service berbeda;
- platform team kehilangan visibility;
- CI/CD tidak konsisten;
- production incident sulit dianalisis.

### 21.1 Cara Mencegah Drift

1. Gunakan parent/BOM internal.
2. Pin Quarkus platform version secara eksplisit.
3. Larang version eksplisit untuk extension yang sudah dikelola BOM.
4. Gunakan dependency audit di CI.
5. Gunakan Renovate/Dependabot dengan grouping strategy.
6. Lakukan upgrade train berkala.
7. Punya compatibility matrix.
8. Catat exception/override dengan expiry date.

## 22. Upgrade Strategy

Quarkus upgrade tidak boleh dilakukan acak.

### 22.1 Upgrade Level

| Level | Contoh | Risiko |
|---|---|---|
| Patch | 3.15.1 -> 3.15.2 | rendah-sedang |
| Minor | 3.15 -> 3.16 | sedang |
| LTS jump | 3.8 LTS -> newer LTS | sedang-tinggi |
| Major | 2 -> 3 | tinggi |
| Java baseline jump | 17 -> 21/25 | sedang-tinggi |
| Native toolchain jump | Mandrel/GraalVM version change | tinggi |

### 22.2 Upgrade Pipeline

```text
1. Read migration guide
2. Update Quarkus platform version
3. Run quarkus update tool if applicable
4. Compile
5. Run unit/component tests
6. Run integration tests
7. Run dev mode smoke test
8. Run native build if applicable
9. Run native integration tests
10. Run dependency convergence/audit
11. Run performance smoke baseline
12. Deploy to lower env
13. Observe logs/metrics/startup/memory
14. Promote gradually
```

### 22.3 Upgrade Branch Strategy

Untuk banyak service:

```text
platform-upgrade/quarkus-3.x.y
```

Lalu rollout bertahap:

```text
pilot service -> medium-risk service -> critical service
```

Jangan mulai dari service paling kompleks.

## 23. Quarkus Update Tool

Quarkus menyediakan mekanisme update yang dapat membantu memodifikasi project mengikuti versi baru. Tool ini membantu banyak pekerjaan mekanis, tetapi tidak menggantikan engineering review.

Gunakan untuk:

- update dependency;
- migrasi konfigurasi tertentu;
- migrasi source code tertentu;
- mengikuti OpenRewrite recipe.

Tetap review manual untuk:

- behavior change;
- security flow;
- native image behavior;
- Hibernate behavior;
- transaction behavior;
- extension compatibility;
- deprecation yang menyentuh domain logic.

Prinsip:

```text
Automation handles syntax and mechanical migration.
Engineer owns semantics and risk.
```

## 24. Native Image Impact pada Dependency Governance

Native image membuat dependency governance lebih ketat.

Di JVM mode, banyak library reflection-heavy masih bisa jalan. Di native mode, library seperti itu bisa gagal saat build atau runtime.

### 24.1 Evaluasi Dependency untuk Native

Untuk setiap dependency baru, tanyakan:

1. Apakah dependency butuh reflection?
2. Apakah ada dynamic classloading?
3. Apakah menggunakan proxy runtime?
4. Apakah membaca resource file dari classpath?
5. Apakah menggunakan JNI?
6. Apakah menggunakan serialization framework khusus?
7. Apakah menggunakan timezone/locale/font/image processing?
8. Apakah sudah punya Quarkus extension?
9. Apakah dokumentasinya menyebut GraalVM/native support?
10. Apakah native integration test sudah lewat?

### 24.2 Native Readiness Matrix

```markdown
| Dependency | Purpose | Quarkus Extension? | Native Tested? | Risk | Owner |
|---|---|---:|---:|---|---|
| quarkus-rest | REST API | yes | yes | low | platform |
| quarkus-hibernate-orm | ORM | yes | yes | medium | backend |
| custom-report-lib | PDF report | no | no | high | reporting team |
```

Library yang aman di JVM belum tentu aman di native.

## 27. Build Reproducibility

Build reproducibility berarti source yang sama menghasilkan artifact yang predictable.

Faktor yang merusak reproducibility:

- dependency version range;
- SNAPSHOT dependency;
- plugin version tidak dikunci;
- generated code tergantung timestamp;
- dependency dari Maven local;
- environment variable tersembunyi;
- profile aktif otomatis;
- remote repository tidak stabil;
- Docker base image `latest`;
- native builder image tidak dikunci.

### 27.1 Rules

1. Jangan pakai version range.
2. Hindari SNAPSHOT untuk production.
3. Pin plugin versions.
4. Pin Quarkus platform version.
5. Pin Java builder image.
6. Pin native builder image.
7. Pakai Maven/Gradle wrapper.
8. Jangan bergantung pada `mavenLocal()` di CI.
9. Archive build metadata.
10. Generate SBOM.

## 28. Container Build Structure

Untuk Quarkus JVM mode, artifact umum:

```text
target/quarkus-app/
  app/
  lib/
  quarkus/
  quarkus-run.jar
```

Quarkus fast-jar layout membantu layering.

Containerfile JVM contoh konseptual:

```dockerfile
FROM eclipse-temurin:21-jre
WORKDIR /work/
COPY target/quarkus-app/lib/ /work/lib/
COPY target/quarkus-app/*.jar /work/
COPY target/quarkus-app/app/ /work/app/
COPY target/quarkus-app/quarkus/ /work/quarkus/
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "/work/quarkus-run.jar"]
```

Untuk native:

```dockerfile
FROM registry.access.redhat.com/ubi9/ubi-minimal
WORKDIR /work/
COPY target/*-runner /work/application
RUN chmod 775 /work/application
EXPOSE 8080
ENTRYPOINT ["/work/application", "-Dquarkus.http.host=0.0.0.0"]
```

### 28.1 Build Governance

- jangan pakai base image `latest`;
- scan image;
- gunakan non-root user jika possible;
- pisahkan build image dan runtime image;
- dokumentasikan JVM vs native artifact;
- pastikan health endpoint aktif;
- pastikan signal handling benar.

## 34. Quarkus Registry dan Extension Discovery

Quarkus dev tools menggunakan extension registry/catalog untuk menemukan extension dan platform. Ini membantu `quarkus create`, `quarkus ext add`, Maven/Gradle plugin, dan tooling lain memahami extension yang tersedia.

Konsekuensinya:

- extension discovery bukan hanya search Maven manual;
- metadata extension penting untuk tooling;
- enterprise dengan network terbatas perlu memikirkan repository/cache;
- offline build butuh dependency dan metadata sudah tersedia di repository internal;
- proxy/artifact repository harus dikonfigurasi benar.

### 34.1 Enterprise Artifact Repository

Untuk organisasi tertutup, gunakan:


Mirror:

- Maven Central;
- Quarkus artifacts;
- approved Quarkiverse artifacts;
- company artifacts;
- plugin artifacts;
- native builder images jika relevan.

Masalah umum di environment enterprise:

- CLI gagal resolve extension catalog;
- Maven bisa build tapi `quarkus ext add` gagal;
- Gradle cache berbeda dari CI;
- proxy memblokir registry metadata;
- developer memakai Maven local yang tidak sama.

## 35. Extension Capability Conflict

Quarkus extension bisa menyediakan capability. Capability membantu Quarkus memahami kontrak teknis yang disediakan extension.

Contoh konseptual:

```text
extension A provides REST capability
extension B also provides REST capability
```

Jika dua extension menyediakan hal yang konflik, build bisa gagal atau behavior ambigu.

### 35.1 Kenapa Ini Penting?

Karena Quarkus melakukan banyak hal saat build. Ia perlu tahu provider mana yang authoritative.

Contoh problem:

- dua REST stack aktif;
- dua JSON provider saling bentrok;
- dua security mechanism tidak sengaja aktif;
- extension lama dan baru dipakai bersamaan;
- library transitive membawa extension yang tidak diinginkan.

### 35.2 Rule

Saat menambah extension, lihat dependency tree:

```bash
./mvnw dependency:tree
```

Atau Gradle:

```bash
./gradlew dependencies
./gradlew dependencyInsight --dependency quarkus-rest
```

Cari dependency yang tidak sengaja masuk.

## 37. Handling Legacy Java 8/11 Libraries

User meminta seri mencakup Java 8 sampai 25. Dalam konteks Quarkus modern, Java 8/11 lebih sering muncul sebagai legacy library atau source aplikasi lama yang dimigrasikan.

Masalah umum:

- library masih memakai `javax.*`;
- bytecode lama masih jalan, tapi API namespace tidak cocok;
- library memakai reflection/classpath scanning;
- library menggunakan deprecated Java APIs;
- library tidak diuji di Java 17/21/25;
- library memakai illegal reflective access;
- library tidak native-compatible.

### 37.1 Migration Strategy

```text
1. Identifikasi library legacy
2. Cek apakah ada versi Jakarta-compatible
3. Cek Java 17+ compatibility
4. Cek Quarkus extension alternative
5. Cek native image impact
6. Bungkus dengan adapter jika belum bisa diganti
7. Tambahkan test kontrak
8. Rencanakan replacement
```

### 37.2 Adapter Pattern untuk Legacy Library

```java
@ApplicationScoped
public class LegacyDocumentRendererAdapter implements DocumentRenderer {
    private final LegacyRenderer renderer;

    public LegacyDocumentRendererAdapter() {
        this.renderer = new LegacyRenderer();
    }

    @Override
    public RenderedDocument render(RenderRequest request) {
        return RenderedDocument.from(renderer.render(request.toLegacyFormat()));
    }
}
```

Tujuannya:

- legacy dependency tidak menyebar;
- replacement mudah;
- native risk bisa diisolasi;
- testing lebih jelas.

## 38. Jakarta Namespace Discipline

Quarkus 3 berada di dunia `jakarta.*`, bukan `javax.*`.

Periksa dependency yang masih membawa:

```text
javax.ws.rs
javax.persistence
javax.validation
javax.enterprise
javax.transaction
javax.servlet
```

Di Quarkus 3, gunakan:

```text
jakarta.ws.rs
jakarta.persistence
jakarta.validation
jakarta.enterprise
jakarta.transaction
jakarta.servlet
```

### 38.1 Mixed Namespace Problem

Jika aplikasi mencampur `javax.*` dan `jakarta.*`, masalah bisa muncul:

- annotation tidak dikenali;
- CDI bean tidak ditemukan;
- validation tidak jalan;
- entity tidak diproses;
- REST resource tidak terekspos;
- compile error;
- runtime mismatch.

### 38.2 Governance Rule

Tambahkan static check atau review rule:

```text
No new javax.* imports in Quarkus 3 application code.
```

Exception hanya untuk library legacy yang dibungkus adapter dan punya migration plan.

## 42. Anti-Pattern Besar

### 42.1 Semua Service Copy-Paste POM Sendiri

Akibat:

- versi drift;
- security patch sulit;
- upgrade tidak seragam;
- knowledge tersebar.

Solusi:

- parent/BOM internal;
- template;
- reusable CI.

### 42.2 Version Ditulis di Semua Dependency

Buruk:

```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-rest</artifactId>
    <version>3.x.y</version>
</dependency>
```

Untuk extension managed, biarkan BOM mengatur versi.

### 42.3 Common Module Menjadi Dumping Ground

Tanda-tanda:

- `common-utils` punya 200 class;
- semua service tergantung padanya;
- ada domain logic di common;
- ada dependency database/security/messaging campur;
- perubahan kecil common memaksa release semua service.

### 42.4 Menggunakan Extension Tanpa ADR

Akibat:

- tidak tahu kenapa extension dipilih;
- sulit remove;
- sulit upgrade;
- risk tidak tercatat.

### 42.5 Native Mode Diputuskan Terlambat

Jika native mode dipertimbangkan setelah semua dependency masuk, sering terlambat.

Native readiness harus dipikirkan sejak dependency selection.

### 42.6 Gradle/Maven Local Magic

Build hanya jalan di laptop satu orang karena:

- ada dependency di local repository;
- ada environment variable tersembunyi;
- ada generated file tidak di-commit;
- ada profile personal.

Ini harus dianggap build failure serius.

## 43. Design Decision Matrix

### 43.1 Single Module atau Multi-Module?

| Kondisi | Pilihan |
|---|---|
| Service kecil | Single module |
| Domain kompleks | Multi-module atau package-by-capability |
| Banyak generated contracts | Multi-module bisa membantu |
| Tim belum matang build tooling | Single module dulu |
| Perlu enforce dependency direction | Multi-module |

### 43.2 Maven atau Gradle?

| Kondisi | Pilihan |
|---|---|
| Enterprise conventional | Maven |
| Build custom/monorepo besar | Gradle |
| Developer skill Maven kuat | Maven |
| Butuh build cache/custom task | Gradle |
| Ingin simplicity | Maven |

### 43.3 Quarkus Extension atau Plain Library?

| Kondisi | Pilihan |
|---|---|
| Ada extension resmi | Extension |
| Library hanya domain utility murni | Plain library OK |
| Butuh native support | Extension lebih aman |
| Butuh lifecycle/config/CDI integration | Extension |
| Library legacy | Adapter + migration plan |

### 43.4 JVM Mode atau Native Mode dari Perspektif Project Structure?

| Kondisi | Dampak Struktur |
|---|---|
| JVM only | dependency lebih fleksibel |
| Native target | dependency harus native-audited |
| Cold start critical | native readiness sejak awal |
| Throughput long-running | JVM bisa lebih cocok |
| Memory constrained | native perlu dipertimbangkan |

## 47. Invariants yang Harus Dipegang

1. Quarkus project adalah build-time augmented application, bukan Java classpath biasa.
2. BOM adalah governance mechanism, bukan formalitas Maven.
3. Extension adalah integration contract, bukan hanya dependency.
4. Jangan menulis version eksplisit untuk extension yang sudah dikelola platform BOM.
5. Parent/BOM internal harus mengatur versi, bukan memaksa semua runtime dependency.
6. Struktur package harus mengikuti boundary, bukan sekadar layer teknis global.
7. Multi-module hanya berguna jika dependency direction jelas.
8. Shared library harus kecil, eksplisit, dan tidak membawa coupling tersembunyi.
9. Native readiness adalah dependency selection problem sejak awal.
10. Upgrade harus diperlakukan sebagai engineering process, bukan trial-and-error.
11. Dev Services meningkatkan feedback loop, tetapi lifecycle-nya harus jelas.
12. Build harus reproducible di CI, bukan hanya berhasil di laptop.
13. ADR kecil lebih baik daripada keputusan dependency yang hilang dalam sejarah commit.
14. Top engineer tidak hanya membuat service jalan; mereka membuat service bisa dirawat, di-upgrade, diamankan, dan dioperasikan.

## 48. Ringkasan Part 005

Pada part ini kita membahas bahwa project Quarkus harus dikelola sebagai unit runtime/build/platform, bukan hanya kumpulan source code.

Hal paling penting:

- Quarkus platform BOM menjaga compatibility.
- Plugin version, platform version, Java version, dan application version adalah konsep berbeda.
- Extension harus dipilih dengan governance.
- Struktur project harus mengikuti domain/capability boundary.
- Parent POM atau Gradle convention harus mengatur build discipline.
- Internal BOM membantu organisasi menjaga konsistensi banyak service.
- Shared library harus diawasi agar tidak menjadi coupling monster.
- Native image membuat dependency governance lebih ketat.
- Upgrade strategy harus sistematis.
- Service template dan internal platform adalah tanda maturity organisasi.

Jika Part 003 membuat kita memahami Quarkus sebagai framework + compiler, Part 005 membuat kita memahami bahwa **build dan dependency governance adalah bagian dari architecture**.
