# Part 17 — `ClassLoader`, `Package`, `Module`, `Layer`: Runtime Boundaries and Encapsulation

> Series: `learn-java-lang-dom-sax-core-runtime-platform-contracts`  
> File: `17-classloader-package-module-layer-runtime-boundaries.md`  
> Scope: Java 8–25  
> Main packages/classes: `java.lang.ClassLoader`, `java.lang.Package`, `java.lang.Module`, `java.lang.ModuleLayer`, `java.lang.Class`, `java.util.ServiceLoader`, JVM loading/linking model

---

## 1. Tujuan Part Ini

Pada bagian sebelumnya kita sudah melihat `Class<T>` sebagai runtime type token. Tetapi `Class<T>` tidak berdiri sendiri. Di production system, sebuah class tidak hanya ditentukan oleh namanya. Sebuah class ditentukan oleh kombinasi:

```text
runtime class identity = binary name + defining ClassLoader
```

Sejak Java 9, identitas dan aksesibilitas class juga dipengaruhi oleh module system:

```text
runtime access boundary = package + module readability + exports/opens + class loader graph
```

Bagian ini membahas boundary tersebut secara serius.

Kita akan membangun mental model untuk menjawab pertanyaan-pertanyaan seperti:

- Kenapa dua object dengan class name sama bisa menyebabkan `ClassCastException`?
- Kenapa reflection yang berjalan di Java 8 bisa gagal di Java 17/21/25?
- Kenapa framework memakai context class loader?
- Apa perbedaan classpath dan module path?
- Apa arti unnamed module, automatic module, named module?
- Apa bedanya `exports` dan `opens`?
- Kapan `ModuleLayer` berguna?
- Kenapa plugin system, app server, build tool, test runner, servlet container, OSGi-like runtime, dan agent sering berurusan dengan class loader?
- Bagaimana class loader leak terjadi?
- Bagaimana merancang boundary runtime yang stabil untuk library dan enterprise application?

Tujuan akhir part ini bukan supaya kamu hafal seluruh JPMS, melainkan supaya kamu punya **runtime boundary intuition**: tahu di mana object, class, package, module, resource, service provider, dan reflective access hidup.

## 2. Mental Model Utama

### 2.1 Java source code bukan runtime boundary final

Di source code, kita melihat package dan class:

```java
package com.example.payment;

public class PaymentService { }
```

Tetapi saat runtime, JVM melihat lebih banyak dimensi:

```text
Class identity:
  binary name: com.example.payment.PaymentService
  defining loader: AppClassLoader / custom loader / plugin loader / container loader
  module: named module / unnamed module
  package: runtime package owned by loader + module
  protection domain / code source: relevant historically/security/tools
```

Jadi `com.example.payment.PaymentService` tidak cukup untuk menentukan identity.

Dua class bisa punya nama yang sama tetapi berbeda loader:

```text
PluginLoader A defines com.acme.Plugin
PluginLoader B defines com.acme.Plugin

A's com.acme.Plugin != B's com.acme.Plugin
```

Akibatnya:

```java
// Bisa gagal walaupun nama class terlihat sama
Plugin p = (Plugin) objectFromOtherLoader;
```

Karena bagi JVM, type tersebut berbeda.

---

### 2.2 ClassLoader menjawab “dari mana bytecode datang?”

`ClassLoader` adalah objek yang bertanggung jawab memuat class. Dalam model umum, class loader menerima binary name lalu menemukan atau menghasilkan bytecode yang akan didefinisikan sebagai class.

Contoh sumber bytecode:

- file `.class` di filesystem;
- JAR di classpath;
- module path;
- network location;
- generated bytecode;
- in-memory byte array;
- instrumentation/agent transformed bytecode;
- plugin bundle;
- application server deployment unit.

Diagram sederhana:

```text
Java code refers to com.example.Foo
        |
        v
ClassLoader.loadClass("com.example.Foo")
        |
        v
find/define class bytes
        |
        v
Class<?> object created by JVM
        |
        v
linking + possible initialization
```

Important: loading class berbeda dari initializing class. Ini sudah disentuh di Part 3, tetapi di sini penting karena class loader sering memicu efek samping ketika class di-resolve atau diinisialisasi.

---

### 2.3 Package adalah runtime namespace, bukan hanya folder source

Package di source sering dianggap folder. Di runtime, `Package` membawa metadata seperti:

- name;
- specification title/version/vendor;
- implementation title/version/vendor;
- sealing information.

Runtime package juga relevan untuk module membership dan access checks.

Dalam JPMS, package tidak boleh sembarangan “terpecah” antar module yang sama-sama resolved. Ini berkaitan dengan split package problem.

---

### 2.4 Module adalah boundary eksplisit setelah Java 9

Java 8 memakai classpath-centric world:

```text
classpath = kumpulan JAR/directory yang secara efektif menjadi satu namespace besar
```

Java 9 memperkenalkan Java Platform Module System / JPMS:

```text
module = named unit yang mendeklarasikan dependency dan exposure
```

Module descriptor:

```java
module com.example.billing {
    requires java.sql;
    exports com.example.billing.api;
    opens com.example.billing.model to com.fasterxml.jackson.databind;
    uses com.example.spi.PaymentProvider;
    provides com.example.spi.PaymentProvider with com.example.billing.StripeProvider;
}
```

Module menjawab:

- module ini butuh module apa?
- package mana yang diekspor untuk compile/runtime access biasa?
- package mana yang dibuka untuk deep reflection?
- service apa yang digunakan?
- service implementation apa yang disediakan?

---

### 2.5 ModuleLayer adalah graph module runtime

`ModuleLayer` adalah runtime layer yang berisi graph module. Boot layer adalah layer awal yang berisi module platform dan application modules yang dimuat saat startup.

Secara mental:

```text
Boot ModuleLayer
  ├── java.base
  ├── java.logging
  ├── java.xml
  ├── application modules
  └── ...

Custom Plugin ModuleLayer
  ├── plugin.alpha
  ├── plugin.beta
  └── plugin.shared.api
```

Layer berguna ketika aplikasi ingin membuat runtime module graph baru, misalnya untuk plugin system. Namun `ModuleLayer` bukan “unloadable isolation container” otomatis. Class unloading tetap bergantung pada garbage collection terhadap class loader dan class metadata yang tidak lagi strongly reachable.

## 3. Evolusi Java 8–25

### 3.1 Java 8: classpath world

Di Java 8, aplikasi umumnya berjalan dengan:

```bash
java -cp app.jar:lib/* com.example.Main
```

Boundary utama:

- package visibility;
- public/protected/private access;
- class loader hierarchy;
- Security Manager / permissions, jika dipakai;
- JAR sealing, jika dipakai.

Masalah umum:

- dependency hell;
- duplicate class;
- jar shadowing;
- accidental reliance on internal JDK APIs;
- weak encapsulation;
- reflection bebas terhadap banyak hal;
- classpath ordering issue;
- split package tidak dicegah secara kuat.

---

### 3.2 Java 9: JPMS dan strong encapsulation mulai masuk

Java 9 memperkenalkan:

- `module-info.java`;
- module path;
- named modules;
- unnamed module untuk classpath code;
- automatic modules untuk non-modular JAR di module path;
- `java.lang.Module`;
- `java.lang.ModuleLayer`;
- stronger encapsulation;
- platform modularization.

Mulai Java 9, semua class berada di module tertentu:

```java
String.class.getModule();        // module java.base
Document.class.getModule();      // module java.xml
MyClass.class.getModule();       // named/unnamed module depending deployment
```

---

### 3.3 Java 16–17+: stronger encapsulation makin nyata

Banyak aplikasi Java 8 yang bergantung pada reflective access ke JDK internals mulai gagal atau memerlukan flags seperti:

```bash
--add-opens java.base/java.lang=ALL-UNNAMED
--add-exports java.base/sun.nio.ch=ALL-UNNAMED
```

Ini bukan solusi desain jangka panjang. Itu adalah compatibility escape hatch.

Mental modelnya:

```text
exports = mengizinkan ordinary access ke public types pada package
opens   = mengizinkan deep reflection ke package
```

Tanpa `opens`, reflective framework dapat gagal ketika mencoba mengakses field/constructor/method non-public atau melakukan deep reflection.

---

### 3.4 Java 21–25: module model tetap fondasi modern

Sampai Java 25, module/classloader boundary tetap penting, terutama untuk:

- framework modern;
- build tools;
- test runtime;
- plugin architectures;
- app server/container;
- agent/instrumentation;
- native image preparation;
- reflection-restricted environments;
- security hardening;
- migration dari Java 8 ke 17/21/25.

Yang berubah bukan hanya API, tetapi asumsi desain:

```text
Java 8 assumption:
  Kalau public dan ada di classpath, kemungkinan bisa dipakai/reflected.

Java 9+ assumption:
  Harus ada readability, export/open boundary, dan loader/module compatibility.
```

## 4. `ClassLoader` secara Mendalam

### 4.1 Basic API

Beberapa API penting:

```java
ClassLoader loader = Thread.currentThread().getContextClassLoader();
Class<?> clazz = loader.loadClass("com.example.MyClass");

ClassLoader definingLoader = clazz.getClassLoader();
URL resource = loader.getResource("config/app.yaml");
InputStream in = loader.getResourceAsStream("config/app.yaml");
```

Catatan:

- `Class.getClassLoader()` bisa mengembalikan `null` untuk bootstrap-loaded classes.
- `String.class.getClassLoader()` biasanya `null` karena `String` dimuat bootstrap loader.
- `ClassLoader.getSystemClassLoader()` adalah system/application class loader.
- `ClassLoader.getPlatformClassLoader()` tersedia sejak Java 9.

---

### 4.2 Parent delegation model

Model tradisional:

```text
CustomClassLoader
        |
        v parent
App/System ClassLoader
        |
        v parent
Platform ClassLoader   (Java 9+)
        |
        v parent
Bootstrap ClassLoader  (represented as null)
```

Parent delegation:

```text
loadClass(name):
  1. check already loaded
  2. ask parent
  3. if parent fails, find class locally
```

Tujuannya:

- mencegah core class ditimpa sembarangan;
- menjaga konsistensi platform types;
- mengurangi duplicate loading;
- memungkinkan sharing parent API.

Namun tidak semua environment mengikuti parent-first secara murni. App server, servlet container, plugin runtime, OSGi-like system, dan test runner bisa memakai strategi child-first atau hybrid.

---

### 4.3 Defining loader vs initiating loader

Konsep penting JVM:

- **defining loader**: loader yang benar-benar mendefinisikan class;
- **initiating loader**: loader yang menyebabkan loading class dimulai.

Bagi kebanyakan application-level debugging, yang paling penting adalah defining loader karena class identity ditentukan olehnya.

```java
Class<?> c = Class.forName("com.example.Foo");
System.out.println(c.getClassLoader());
```

---

### 4.4 Class identity problem

Contoh konseptual:

```java
URLClassLoader pluginA = new URLClassLoader(urlsA, apiLoader);
URLClassLoader pluginB = new URLClassLoader(urlsB, apiLoader);

Class<?> aFoo = pluginA.loadClass("com.example.Foo");
Class<?> bFoo = pluginB.loadClass("com.example.Foo");

System.out.println(aFoo == bFoo); // false
```

Walaupun binary name sama, class berbeda karena defining loader berbeda.

Masalah umum:

```text
java.lang.ClassCastException:
  class com.example.Dto cannot be cast to class com.example.Dto
```

Ini terlihat absurd, tetapi biasanya berarti:

```text
com.example.Dto loaded by Loader A
com.example.Dto loaded by Loader B
```

---

### 4.5 Resource loading

Class loader tidak hanya load class, tetapi juga resources.

```java
try (InputStream in = Thread.currentThread()
        .getContextClassLoader()
        .getResourceAsStream("templates/email/welcome.html")) {
    if (in == null) {
        throw new IllegalStateException("Resource not found");
    }
}
```

Pitfall:

```java
MyClass.class.getResourceAsStream("config.yaml");
MyClass.class.getResourceAsStream("/config.yaml");
```

Perbedaan:

- tanpa `/`: relatif ke package `MyClass`;
- dengan `/`: absolut dari classpath/module resource root.

Untuk `ClassLoader.getResourceAsStream`, path tidak diawali `/`.

---

### 4.6 Context class loader

Context class loader adalah workaround historis untuk kasus di mana parent code perlu menemukan child/application classes.

Contoh:

```java
ClassLoader cl = Thread.currentThread().getContextClassLoader();
```

Kenapa perlu?

Misalnya library di parent loader ingin menemukan provider yang berada di application loader. Karena parent tidak tahu child, parent delegation biasa tidak cukup.

Use cases:

- `ServiceLoader`;
- JDBC driver discovery;
- JNDI;
- XML parser provider discovery;
- logging providers;
- app server frameworks;
- plugin systems.

Pattern aman:

```java
ClassLoader old = Thread.currentThread().getContextClassLoader();
try {
    Thread.currentThread().setContextClassLoader(pluginLoader);
    runPluginCode();
} finally {
    Thread.currentThread().setContextClassLoader(old);
}
```

Failure mode:

```java
Thread.currentThread().setContextClassLoader(pluginLoader);
runPluginCode();
// forgot restore
```

Akibat:

- class loader leak;
- resource lookup salah;
- provider salah;
- tenant/plugin contamination;
- test flakiness.

## 5. `Package`: Metadata, Sealing, and Runtime Namespace

### 5.1 Apa itu `Package`?

`java.lang.Package` merepresentasikan metadata package yang diketahui class loader.

```java
Package p = String.class.getPackage();
System.out.println(p.getName()); // java.lang
```

API penting:

```java
p.getName();
p.getSpecificationTitle();
p.getSpecificationVersion();
p.getSpecificationVendor();
p.getImplementationTitle();
p.getImplementationVersion();
p.getImplementationVendor();
p.isSealed();
```

Metadata biasanya berasal dari manifest JAR.

---

### 5.2 Package sealing

JAR sealing adalah mekanisme lama untuk memastikan semua class dalam package tertentu berasal dari code source yang sama.

Manifest example:

```text
Sealed: true
```

Atau per-entry:

```text
Name: com/example/internal/
Sealed: true
```

Kenapa relevan?

- mencegah class injection ke package yang sama dari JAR berbeda;
- menjaga package-private access boundary;
- mendukung integrity model sebelum JPMS.

Namun dalam modern Java, JPMS memberikan boundary yang lebih eksplisit dan kuat.

---

### 5.3 Package-private access dan loader

Package-private access bukan hanya berdasarkan nama package source. Runtime package juga dipengaruhi class loader/module.

Secara praktis:

```text
same runtime package requires:
  same package name
  same defining loader
  compatible module context
```

Jadi dua class dengan package name sama tetapi loader berbeda tidak otomatis berada dalam runtime package yang sama.

---

## 6. `Module`: Named, Unnamed, Automatic

### 6.1 Semua class punya module sejak Java 9

```java
System.out.println(String.class.getModule());
System.out.println(org.w3c.dom.Document.class.getModule());
System.out.println(MyApp.class.getModule());
```

Output konseptual:

```text
module java.base
module java.xml
unnamed module @...
```

Jika aplikasi berjalan di classpath, application classes biasanya berada di unnamed module.

---

### 6.2 Named module

Named module punya `module-info.java`:

```java
module com.example.order {
    requires java.base; // implicit, tidak perlu ditulis
    requires java.sql;

    exports com.example.order.api;
    exports com.example.order.spi;

    opens com.example.order.model to com.fasterxml.jackson.databind;
}
```

Rules utama:

- `java.base` selalu implicitly required;
- package tidak diekspos kecuali `exports`;
- deep reflection butuh `opens`;
- module harus read module lain untuk mengakses public types yang diexport.

---

### 6.3 Unnamed module

Classpath code masuk ke unnamed module.

Karakteristik:

- membaca banyak named modules secara default;
- package-nya secara praktis terbuka/terekspos seperti classpath world;
- berguna untuk backward compatibility;
- kurang memberikan encapsulation eksplisit.

Banyak aplikasi Spring Boot non-modular berjalan di unnamed module.

Ini bukan salah. Tetapi saat memakai Java 17/21/25, masalah bisa muncul ketika classpath code mencoba deep reflection ke JDK internals atau named module yang tidak open.

---

### 6.4 Automatic module

Non-modular JAR yang ditempatkan di module path dapat menjadi automatic module.

Karakteristik umum:

- module name diinfer dari JAR name atau manifest `Automatic-Module-Name`;
- membaca module lain secara luas;
- exports semua package;
- bridge untuk migration.

Pitfall:

- module name bisa tidak stabil jika berasal dari file name;
- split package conflict;
- tidak sekuat explicit module;
- transitive dependency behavior bisa mengejutkan.

Library yang serius sering menambahkan manifest:

```text
Automatic-Module-Name: com.example.mylibrary
```

Ini menjaga module name stabil sebelum library punya `module-info.java`.

---

## 7. Readability, Exports, Opens

### 7.1 Readability

Module A dapat mengakses public types dari Module B hanya jika A reads B dan B exports package tersebut.

```java
module com.example.app {
    requires com.example.lib;
}
```

Mental model:

```text
requires = dependency edge in module graph
```

---

### 7.2 Exports

```java
exports com.example.lib.api;
```

Artinya package tersebut dapat diakses oleh module lain secara ordinary Java access.

Qualified export:

```java
exports com.example.lib.internal.spi to com.example.app;
```

Artinya hanya module tertentu yang bisa mengakses.

---

### 7.3 Opens

```java
opens com.example.lib.model;
```

Artinya package dibuka untuk deep reflection.

Qualified opens:

```java
opens com.example.lib.model to com.fasterxml.jackson.databind;
```

Ini umum untuk serialization/deserialization framework.

---

### 7.4 Exports vs opens

Perbedaan utama:

| Directive | Ordinary compile/runtime access | Deep reflection |
|---|---:|---:|
| `exports p` | yes | no |
| `opens p` | no ordinary API exposure | yes |
| `open module` | all packages open | yes broadly |
| neither | no | no |

Design rule:

```text
exports API package
opens data/model package only to framework that needs reflection
avoid open module unless migration pressure is high
```

---

### 7.5 Reflection failure example

```java
Field f = User.class.getDeclaredField("passwordHash");
f.setAccessible(true);
```

Di Java 8, ini sering berhasil.

Di Java 17/21/25 dengan named module, ini bisa gagal jika package tidak opened kepada module pemanggil.

Possible failure:

```text
java.lang.reflect.InaccessibleObjectException
```

Solusi buruk jangka panjang:

```bash
--add-opens com.example.app/com.example.model=ALL-UNNAMED
```

Solusi desain:

```java
module com.example.app {
    opens com.example.model to com.fasterxml.jackson.databind;
}
```

Atau hindari deep reflection jika tidak diperlukan.

## 8. `ModuleLayer`

### 8.1 Apa itu layer?

`ModuleLayer` adalah kumpulan module runtime yang dibuat dari `Configuration` dan mapping module ke class loader.

Boot layer:

```java
ModuleLayer boot = ModuleLayer.boot();
```

Melihat module:

```java
boot.modules().stream()
    .map(Module::getName)
    .sorted()
    .forEach(System.out::println);
```

---

### 8.2 Kenapa butuh custom layer?

Use cases:

- plugin system;
- scripting runtime;
- dynamically loaded feature pack;
- test isolation;
- modular application container;
- version-separated extension modules.

Namun custom layer adalah advanced tool. Banyak aplikasi tidak memerlukannya.

---

### 8.3 Simplified flow membuat layer

Konseptual:

```java
ModuleFinder finder = ModuleFinder.of(Path.of("plugins"));

ModuleLayer parent = ModuleLayer.boot();
Configuration parentConfig = parent.configuration();

Configuration config = parentConfig.resolve(
    finder,
    ModuleFinder.of(),
    Set.of("com.example.plugin.alpha")
);

ClassLoader scl = ClassLoader.getSystemClassLoader();
ModuleLayer layer = parent.defineModulesWithOneLoader(config, scl);
```

Catatan: untuk isolation yang lebih kuat, bisa gunakan loader berbeda per layer/module. Tetapi semakin kuat isolation, semakin kompleks type sharing dan lifecycle management.

---

### 8.4 Layer bukan automatic unload boundary

Membuat layer baru tidak berarti plugin bisa otomatis di-unload.

Class unloading membutuhkan:

- class loader tidak reachable;
- class objects tidak reachable;
- instances tidak reachable;
- threads tidak menahan context class loader;
- static caches tidak menahan class/classloader;
- ThreadLocal tidak menahan plugin class;
- ServiceLoader/cache tidak menahan provider;
- logging/MBeans/listeners tidak menahan referensi.

Jika salah satu masih reachable, plugin tetap hidup.

---

## 9. Service Loading dan Boundary Provider

### 9.1 ServiceLoader mental model

`ServiceLoader` memungkinkan API dan implementation dipisah.

SPI:

```java
package com.example.payment.spi;

public interface PaymentProvider {
    String providerCode();
    PaymentResult pay(PaymentRequest request);
}
```

Provider:

```java
public final class StripePaymentProvider implements PaymentProvider {
    public String providerCode() {
        return "STRIPE";
    }
}
```

Module descriptor provider:

```java
module com.example.payment.stripe {
    requires com.example.payment.spi;
    provides com.example.payment.spi.PaymentProvider
        with com.example.payment.stripe.StripePaymentProvider;
}
```

Consumer:

```java
module com.example.payment.app {
    requires com.example.payment.spi;
    uses com.example.payment.spi.PaymentProvider;
}
```

Load:

```java
ServiceLoader<PaymentProvider> loader = ServiceLoader.load(PaymentProvider.class);
for (PaymentProvider provider : loader) {
    System.out.println(provider.providerCode());
}
```

---

### 9.2 ServiceLoader with ModuleLayer

Untuk plugin layer:

```java
ServiceLoader<PaymentProvider> loader =
    ServiceLoader.load(pluginLayer, PaymentProvider.class);
```

Ini mencari providers di layer tersebut dan ancestors.

---

### 9.3 Failure modes ServiceLoader

Masalah umum:

- service interface diload oleh loader berbeda dari provider;
- provider class tidak public atau constructor tidak sesuai;
- provider module tidak mendeklarasikan `provides`;
- consumer module lupa `uses`;
- context class loader salah;
- provider melakukan heavy initialization saat iteration;
- ServiceLoader cache menahan class loader;
- duplicate providers tanpa deterministic selection policy.

Design rule:

```text
Keep SPI in parent/shared loader/module.
Load implementations in child/plugin layer.
Return only SPI/domain-neutral types across boundary.
Avoid leaking plugin implementation classes to host.
```

## 10. Classpath vs Module Path

### 10.1 Classpath

Classpath sederhana tapi lemah boundary.

```bash
java -cp app.jar:lib/* com.example.Main
```

Karakteristik:

- ordering matters;
- duplicate class bisa shadowing;
- semua package cenderung accessible;
- no explicit dependency graph;
- cocok untuk banyak aplikasi tradisional;
- masih sangat umum.

---

### 10.2 Module path

```bash
java --module-path mods -m com.example.app/com.example.Main
```

Karakteristik:

- dependency graph resolved;
- strong encapsulation;
- package conflict lebih cepat terdeteksi;
- explicit exports/opens;
- better platform integrity;
- migration cost lebih tinggi.

---

### 10.3 Mixed mode

Banyak real-world app berjalan mixed:

```text
some dependencies on module path
some dependencies on classpath
application in unnamed module
```

Ini bisa valid untuk migration, tetapi perlu hati-hati:

- unnamed module behavior berbeda dari named modules;
- automatic modules bisa memberi nama tidak stabil;
- reflection ke named modules butuh opens;
- split packages bisa muncul;
- tool/build behavior perlu dites lintas JDK.

---

## 11. Split Packages

Split package terjadi ketika package yang sama muncul di lebih dari satu module/JAR boundary yang tidak kompatibel.

Contoh:

```text
lib-a.jar contains com.example.util.Strings
lib-b.jar contains com.example.util.Numbers
```

Di classpath, ini sering “jalan” karena classpath menyatukan namespace.

Di module path, split package dapat gagal karena module system ingin package ownership jelas.

Kenapa berbahaya?

- package-private access ambiguity;
- sealing/integrity issue;
- shadowing;
- unpredictable tooling;
- migration pain.

Design rule:

```text
One package should belong to one module/artifact boundary.
Use clear package namespace per artifact/module.
Avoid placing generated classes into someone else's package unless explicitly designed.
```

## 12. Internal APIs and Encapsulation

### 12.1 JDK internals

Java 8 code sering memakai internal APIs seperti:

```java
sun.misc.Unsafe
sun.reflect.Reflection
com.sun.*
jdk.internal.*
```

Setelah Java 9+, ini makin dibatasi.

Masalahnya bukan hanya “API hilang”. Masalahnya:

```text
internal APIs are not compatibility contracts
```

Mereka bisa berubah tanpa backward compatibility seperti public Java SE API.

---

### 12.2 `--add-exports` dan `--add-opens`

Escape hatches:

```bash
--add-exports java.base/jdk.internal.misc=ALL-UNNAMED
--add-opens java.base/java.lang=ALL-UNNAMED
```

Makna:

- `--add-exports`: membuka ordinary access ke package non-exported;
- `--add-opens`: membuka deep reflection.

Gunakan sebagai:

- migration bridge;
- diagnostic workaround;
- legacy support sementara.

Jangan jadikan desain permanen kecuali benar-benar dikontrol dan didokumentasikan.

## 13. Runtime Boundary Design Patterns

### 13.1 Parent API, child implementation

Pattern plugin umum:

```text
Host/App Loader
  ├── com.example.spi.Plugin
  ├── com.example.api.SharedDto
  └── PluginManager

Plugin Loader A
  └── com.vendor.alpha.AlphaPlugin implements Plugin

Plugin Loader B
  └── com.vendor.beta.BetaPlugin implements Plugin
```

Rule:

- shared interfaces/classes harus berada di parent;
- implementation berada di child;
- host berinteraksi lewat parent-owned types;
- jangan pass child implementation-specific type ke host API;
- jangan simpan plugin class di static global host cache tanpa lifecycle cleanup.

---

### 13.2 Narrow boundary objects

Buruk:

```java
Object execute(Object pluginSpecificRequest);
```

Lebih baik:

```java
PluginResult execute(PluginContext context, PluginCommand command);
```

Dengan `PluginCommand` dan `PluginResult` berada di shared API module.

---

### 13.3 Explicit lifecycle

```java
public interface Plugin extends AutoCloseable {
    void start(PluginContext context);
    PluginResult execute(PluginCommand command);
    void stop();
    @Override void close();
}
```

Lifecycle penting karena plugin bisa membuat:

- threads;
- executor;
- timers;
- HTTP clients;
- DB pools;
- file watchers;
- MBeans;
- metrics;
- log appenders;
- ThreadLocals.

Semua harus dibersihkan agar class loader bisa di-GC.

---

### 13.4 Child-first loader: kapan?

Child-first berguna ketika plugin ingin membawa dependency versinya sendiri.

```text
Plugin A uses jackson 2.x
Plugin B uses jackson 3.x
Host uses jackson 2.y
```

Tetapi child-first berbahaya jika child memuat duplikat API shared.

Rule:

```text
Always parent-first for shared API packages.
Optionally child-first for private implementation packages/dependencies.
```

---

## 14. Class Loader Leaks

### 14.1 Apa itu class loader leak?

Class loader leak terjadi ketika class loader seharusnya bisa mati, tetapi masih direferensikan.

Karena class loader mereferensikan semua class yang didefinisikannya, leak class loader berarti leak seluruh dunia plugin/deployment:

```text
one leaked ClassLoader
  -> all Classes loaded by it
  -> static fields
  -> metadata
  -> possibly instances/resources
```

---

### 14.2 Sumber leak umum

1. Static cache di parent loader:

```java
static final Map<Class<?>, Metadata> CACHE = new ConcurrentHashMap<>();
```

Jika key berasal dari plugin loader, plugin loader tertahan.

2. ThreadLocal:

```java
ThreadLocal<Object> ctx = new ThreadLocal<>();
ctx.set(pluginObject);
```

Jika thread milik pool parent masih hidup, plugin object tertahan.

3. Context class loader:

```java
thread.setContextClassLoader(pluginLoader);
```

Thread hidup lama -> plugin loader leak.

4. Running thread created by plugin.

5. JDBC driver registry / logging registry / MBean server.

6. ServiceLoader cache.

7. Lambda/method reference capturing plugin class.

8. Shutdown hook loaded by plugin.

---

### 14.3 Leak prevention checklist

Saat unload plugin/deployment:

- stop all executors;
- interrupt/join plugin threads;
- clear ThreadLocals;
- restore context class loader;
- unregister JDBC drivers loaded by plugin;
- unregister MBeans;
- remove listeners/callbacks;
- close file watchers;
- close URLClassLoader if applicable;
- clear caches keyed by plugin classes;
- avoid static singletons in plugin that reference host lifecycle incorrectly;
- test with heap dump/class loader analysis.

---

## 15. Failure Modes yang Sering Muncul

### 15.1 `ClassNotFoundException` vs `NoClassDefFoundError`

`ClassNotFoundException` biasanya muncul ketika reflective/dynamic loading gagal:

```java
Class.forName("com.example.Missing")
```

`NoClassDefFoundError` biasanya muncul ketika class yang dibutuhkan saat linking/runtime tidak bisa ditemukan, walaupun compile-time pernah ada.

Contoh:

```text
App compiled with lib-a.jar containing Foo
Runtime missing lib-a.jar
```

Atau class gagal initialization sebelumnya, lalu akses berikutnya menghasilkan error terkait.

---

### 15.2 `NoSuchMethodError` / `NoSuchFieldError`

Biasanya binary incompatibility:

```text
compiled against version 2
runtime has version 1
```

Contoh:

```java
// compile-time library has method x()
obj.x();

// runtime library older, x() does not exist
NoSuchMethodError
```

Root cause sering:

- dependency conflict;
- classpath shadowing;
- transitive dependency mismatch;
- container provides older library;
- fat JAR duplicate;
- plugin child/parent conflict.

---

### 15.3 `LinkageError`

`LinkageError` adalah keluarga error untuk masalah loading/linking binary.

Contoh:

- `NoClassDefFoundError`;
- `NoSuchMethodError`;
- `NoSuchFieldError`;
- `IncompatibleClassChangeError`;
- `UnsupportedClassVersionError`;
- `ClassFormatError`.

Ini umumnya bukan domain exception. Ini deployment/runtime compatibility failure.

---

### 15.4 `UnsupportedClassVersionError`

Contoh:

```text
class file version 65 but runtime only supports 61
```

Artinya class dikompilasi dengan Java lebih baru daripada runtime.

Mapping umum:

```text
Java 8  -> class file 52
Java 11 -> class file 55
Java 17 -> class file 61
Java 21 -> class file 65
Java 25 -> class file 69
```

Design rule:

```text
Use --release for library compilation when targeting older Java.
Test runtime with the minimum supported JDK, not only latest JDK.
```

---

### 15.5 Same class name `ClassCastException`

Gejala:

```text
com.example.User cannot be cast to com.example.User
```

Interpretasi:

```text
Same binary name, different defining loaders.
```

Debug:

```java
static void printType(Object o) {
    Class<?> c = o.getClass();
    System.out.println("class=" + c.getName());
    System.out.println("loader=" + c.getClassLoader());
    System.out.println("module=" + c.getModule());
}
```

---

### 15.6 `InaccessibleObjectException`

Gejala modern Java:

```text
Unable to make field private final ... accessible:
module X does not "opens package" to module Y
```

Root cause:

- reflection framework membutuhkan deep reflection;
- package tidak opened;
- Java 8 assumptions masih terbawa.

Solusi desain:

- gunakan public constructor/accessor bila cukup;
- tambahkan qualified `opens`;
- hindari reflection ke JDK internals;
- upgrade framework yang sudah JPMS-aware;
- gunakan `--add-opens` hanya sebagai migration bridge.

---

## 16. Debugging Runtime Boundary

### 16.1 Print class-loader/module info

```java
public final class RuntimeBoundaryDebug {
    public static void print(Class<?> type) {
        System.out.println("Class      : " + type.getName());
        System.out.println("Loader     : " + type.getClassLoader());
        System.out.println("Module     : " + type.getModule());
        System.out.println("Package    : " + type.getPackageName());
        System.out.println("Protection : " + type.getProtectionDomain().getCodeSource());
    }
}
```

Usage:

```java
RuntimeBoundaryDebug.print(String.class);
RuntimeBoundaryDebug.print(com.example.MyClass.class);
```

---

### 16.2 Check duplicate resources

```java
Enumeration<URL> urls = Thread.currentThread()
    .getContextClassLoader()
    .getResources("com/example/MyClass.class");

while (urls.hasMoreElements()) {
    System.out.println(urls.nextElement());
}
```

Jika lebih dari satu URL, ada duplicate class/resource candidate.

---

### 16.3 Module readability check

```java
Module app = MyApp.class.getModule();
Module target = SomeLibrary.class.getModule();

System.out.println(app.canRead(target));
System.out.println(target.isExported("com.example.lib.api", app));
System.out.println(target.isOpen("com.example.lib.model", app));
```

---

### 16.4 Use JVM logging

Untuk debugging class loading:

Java 9+:

```bash
java -Xlog:class+load=info -jar app.jar
```

Java 8:

```bash
java -verbose:class -jar app.jar
```

Gunakan hati-hati di production karena output besar.

## 17. Design Heuristics untuk Software Engineer Senior

### 17.1 Jangan jadikan classpath sebagai tempat sampah global

Classpath yang berisi banyak JAR tanpa kontrol adalah sumber:

- shadowing;
- dependency conflict;
- runtime-only failure;
- accidental coupling;
- hard-to-debug errors.

Gunakan dependency management yang strict:

- Maven dependency convergence;
- Gradle version catalogs/platforms;
- lockfiles;
- dependency insight;
- enforcer rules;
- minimal transitive exposure.

---

### 17.2 Pisahkan API dan implementation

Buruk:

```text
host depends on plugin implementation classes
```

Baik:

```text
host depends on stable SPI/API
plugin implements SPI
communication via stable DTO/events/contracts
```

---

### 17.3 Module/package naming harus mencerminkan ownership

Contoh buruk:

```text
com.company.common
com.company.util
com.company.model
```

Jika dipakai lintas puluhan module, package seperti ini rentan menjadi dumping ground.

Lebih baik:

```text
com.company.payment.api
com.company.payment.spi
com.company.payment.internal
com.company.caseworkflow.api
com.company.caseworkflow.internal.state
```

---

### 17.4 Jangan expose internal package

Di module descriptor:

```java
exports com.example.billing.api;
// do not export com.example.billing.internal
```

Jika framework butuh reflection:

```java
opens com.example.billing.model to com.fasterxml.jackson.databind;
```

Bukan:

```java
exports com.example.billing.internal;
```

---

### 17.5 Treat `--add-opens` as debt

Jika startup command punya banyak:

```bash
--add-opens java.base/java.lang=ALL-UNNAMED
--add-opens java.base/java.util=ALL-UNNAMED
--add-opens java.base/java.lang.reflect=ALL-UNNAMED
```

Itu sinyal:

- framework lama;
- internal reflection berlebih;
- migration belum selesai;
- future JDK upgrade risk.

Dokumentasikan setiap flag:

```text
Flag: --add-opens java.base/java.lang=ALL-UNNAMED
Reason: legacy framework X version Y uses reflection on java.lang.Class internals
Owner: platform team
Removal plan: upgrade X to version Z
```

## 18. Production Checklist

### 18.1 Untuk aplikasi biasa

- [ ] Dependency tree bersih dari duplicate/conflict major version.
- [ ] Tidak memakai internal JDK API kecuali terdokumentasi dan terkontrol.
- [ ] Runtime JDK version sesuai target bytecode.
- [ ] `--release` dipakai untuk library multi-Java target.
- [ ] Startup flags `--add-opens/--add-exports` terdokumentasi.
- [ ] Resource loading tidak bergantung pada working directory.
- [ ] Classpath/module path behavior dites di target deployment, bukan hanya IDE.
- [ ] Framework version kompatibel dengan target Java 17/21/25.

---

### 18.2 Untuk library

- [ ] Public API package jelas.
- [ ] Internal package tidak diekspos.
- [ ] Automatic module name stabil jika belum modular.
- [ ] Tidak split package dengan artifact lain.
- [ ] Tidak memaksa consumer membuka JDK internals.
- [ ] Reflection optional diberi fallback.
- [ ] Test minimum supported Java version.
- [ ] Binary compatibility dicek antar release.

---

### 18.3 Untuk plugin system

- [ ] SPI/API berada di shared parent loader/module.
- [ ] Plugin implementation berada di isolated loader/layer.
- [ ] Boundary object memakai shared API types.
- [ ] Context class loader selalu restore.
- [ ] Plugin lifecycle explicit: start/stop/close.
- [ ] Plugin-created threads/executors ditutup.
- [ ] ThreadLocal dibersihkan.
- [ ] Registries/listeners/MBeans/JDBC drivers di-unregister.
- [ ] ServiceLoader cache dipahami/dikelola.
- [ ] Class loader unload diuji dengan heap dump/metaspace observation.

## 19. Thought Exercises

### Exercise 1 — Same Name, Different Class

Kamu melihat error:

```text
ClassCastException: com.acme.Order cannot be cast to com.acme.Order
```

Jelaskan:

1. kenapa ini mungkin terjadi;
2. apa yang harus di-print untuk debugging;
3. bagaimana desain boundary yang bisa mencegahnya.

Expected reasoning:

```text
Same binary name but different defining ClassLoader.
Print class, classloader, module, code source.
Move shared API/DTO to parent loader and avoid crossing implementation class boundary.
```

---

### Exercise 2 — Reflection Broken After Java 17 Migration

Sebuah serializer custom berjalan di Java 8 tetapi gagal di Java 17 dengan `InaccessibleObjectException`.

Analisis:

- apakah ini class loader problem atau module openness problem?
- apa solusi cepat?
- apa solusi desain?

Expected reasoning:

```text
Likely module openness problem.
Quick workaround: --add-opens or open package.
Design solution: qualified opens to serializer module, public accessors, or framework upgrade.
```

---

### Exercise 3 — Plugin Unload Tidak Berhasil

Setelah plugin di-disable, metaspace tidak turun dan class plugin masih terlihat di heap dump.

Cari kemungkinan:

- thread masih hidup;
- context class loader tertahan;
- static cache parent;
- ThreadLocal;
- MBean/listener;
- ServiceLoader/provider cache;
- logging appender;
- JDBC driver;
- scheduled executor.

## 20. Ringkasan

`ClassLoader`, `Package`, `Module`, dan `ModuleLayer` membentuk boundary runtime Java modern.

Mental model terpenting:

```text
Class identity is not just the class name.
Class identity = binary name + defining ClassLoader.
```

Sejak Java 9:

```text
Access is not just public/private.
Access also depends on module readability, exports, opens, and layer/loader structure.
```

`Package` bukan hanya folder source; ia adalah runtime namespace dan metadata boundary.

`Module` memberi dependency graph dan encapsulation eksplisit.

`ModuleLayer` memungkinkan graph module runtime tambahan, tetapi tidak otomatis memberi unload/lifecycle yang aman.

Untuk production system, boundary design yang baik berarti:

- API dan implementation dipisah;
- shared types berada di shared loader/module;
- internal package tidak diekspos sembarangan;
- reflection diberi `opens` yang sempit;
- plugin lifecycle explicit;
- class loader leak dicegah;
- dependency conflict dan binary compatibility dikelola.

Engineer yang kuat di Java tidak hanya bertanya “class ini ada atau tidak?”, tetapi juga:

```text
Loaded by whom?
In which module?
Readable by whom?
Exported or opened to whom?
Can it be unloaded?
What references keep it alive?
```

Itulah perbedaan antara memahami Java sebagai bahasa dan memahami Java sebagai runtime platform.
