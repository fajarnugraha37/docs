## 1. Why This Part Exists

At basic level, a gRPC call looks simple:

```java
UserResponse response = userServiceBlockingStub.getUser(request);
```

At production level, that line hides a runtime system:

```text
stub
  -> channel
  -> name resolver
  -> service config
  -> load balancing policy
  -> subchannel(s)
  -> HTTP/2 connection(s)
  -> stream
  -> deadline
  -> retry / hedging / transparent retry
  -> metadata
  -> status / trailers
  -> exception mapping
```

A top-tier engineer does not treat `StatusRuntimeException: UNAVAILABLE` as a generic error. They ask:

```text
Was this error generated by the application server?
By the gRPC library?
By the transport?
By name resolution?
By the load balancer?
By deadline expiration?
By client cancellation?
By retry exhaustion?
By health checking?
By channel state transition?
```

The goal of this part is to make gRPC failure, retry, and load balancing behavior explainable.

Official gRPC documentation defines status as the result returned by every RPC, with an integer code and optional description, and notes that either application code or the gRPC library may generate it. gRPC also documents retry and service config as client behavior controls, and load balancing as a collaboration between name resolver and load balancing policy. These are not optional details; they are the real runtime of a gRPC client.

## 2. Core Mental Model

A gRPC call is not only a remote method call.

It is:

```text
A typed attempt to execute an operation on a remote service,
over a resolved target,
through a channel,
under a deadline,
using a load-balancing policy,
with status-based completion semantics,
and optional retry/hedging behavior constrained by idempotency and buffer limits.
```

This means there are two layers of correctness:

| Layer | Question |
|---|---|
| Business correctness | Did the intended domain operation happen exactly as expected? |
| Transport correctness | Did the remote attempt complete, fail, retry, cancel, or become ambiguous in a controlled way? |

The difficult cases are not clean success or clean validation error. The difficult cases are ambiguous:

```text
Client sent request.
Connection broke.
Client got UNAVAILABLE.
Server may or may not have processed the request.
Client retries.
Business side effect may be duplicated.
```

This is why retry, status mapping, idempotency, and service config must be designed together.

## 3. gRPC Error Model: Status Is the Boundary Contract

Every gRPC call ends with a status.

In Java, errors are usually observed as:

```java
try {
    UserResponse response = stub.getUser(request);
} catch (StatusRuntimeException e) {
    Status status = e.getStatus();
    Metadata trailers = e.getTrailers();
}
```

or with checked style:

```java
try {
    UserResponse response = blockingStub.getUser(request);
} catch (StatusException e) {
    Status.Code code = e.getStatus().getCode();
}
```

The status is the gRPC-level result, not necessarily the business result.

```text
OK
  -> RPC completed successfully.

INVALID_ARGUMENT
  -> caller sent invalid input independent of current system state.

FAILED_PRECONDITION
  -> operation cannot be executed because system state is not suitable.

UNAVAILABLE
  -> service or transport temporarily unavailable.

DEADLINE_EXCEEDED
  -> deadline expired before completion.

CANCELLED
  -> caller or runtime cancelled the operation.

ABORTED
  -> concurrency conflict / transaction conflict / retry at higher level may be appropriate.

NOT_FOUND
  -> requested entity not found.

PERMISSION_DENIED
  -> authenticated principal lacks permission.

UNAUTHENTICATED
  -> no valid authentication.

RESOURCE_EXHAUSTED
  -> quota, rate limit, memory, concurrent stream, or capacity limit.

INTERNAL
  -> invariant broken or unexpected internal error.
```

A top-tier engineer uses status codes as a cross-service contract.

Bad error mapping:

```text
All errors -> UNKNOWN
All validation errors -> INTERNAL
All dependency errors -> INTERNAL
All business rejects -> FAILED_PRECONDITION
All timeouts -> UNAVAILABLE
```

Good error mapping is stable, semantic, and actionable:

```text
Invalid input format             -> INVALID_ARGUMENT
Entity does not exist             -> NOT_FOUND
Caller lacks permission           -> PERMISSION_DENIED
Missing/invalid credentials       -> UNAUTHENTICATED
Optimistic locking conflict       -> ABORTED
Quota exceeded                    -> RESOURCE_EXHAUSTED
Dependency temporarily unavailable -> UNAVAILABLE
Client deadline expired           -> DEADLINE_EXCEEDED
Server invariant broken           -> INTERNAL
```

## 4. Library-Generated vs Application-Generated Status

A common debugging mistake is assuming every status came from server business code.

Some statuses may be generated by the gRPC library.

Examples:

| Status | Could be generated by | Example |
|---|---|---|
| `UNAVAILABLE` | Client transport / server / LB | Connection refused, resolver failure, GOAWAY, backend down |
| `DEADLINE_EXCEEDED` | Client runtime / server | Deadline expired before response |
| `CANCELLED` | Client / server runtime | Caller cancelled, deadline propagation cancelled downstream |
| `RESOURCE_EXHAUSTED` | Server app / runtime / proxy | Quota exceeded, max message size, flow-control pressure |
| `INTERNAL` | App / library | Protocol violation, serialization bug, invariant failure |
| `UNKNOWN` | Library / bridge / app | Exception not mapped properly |

Therefore logging only business exception class is insufficient.

A useful error log should include:

```text
service name
method name
authority / target
status code
status description, sanitized
trailers, sanitized
deadline remaining at start
elapsed time
attempt count
previous retry statuses
channel state if available
resolved backend if available
correlation id / trace id
idempotency key if relevant
request size / response size
```

## 5. Status Code Taxonomy for Production Design

### 5.1 Caller Error

Caller errors should generally not be retried automatically.

```text
INVALID_ARGUMENT
NOT_FOUND, depending on operation
ALREADY_EXISTS, depending on operation
PERMISSION_DENIED
UNAUTHENTICATED, except token refresh flow
OUT_OF_RANGE
```

Example:

```java
throw Status.INVALID_ARGUMENT
        .withDescription("postalCode must be exactly 6 digits")
        .asRuntimeException();
```

Use `INVALID_ARGUMENT` when the input is invalid regardless of current system state.

Do not use it for:

```text
Case already approved
Application period closed
Version conflict
Insufficient workflow state
```

Those depend on system state.

---

### 5.2 State Conflict / Workflow Conflict

Use state-related statuses when the request is structurally valid but cannot be executed now.

```text
FAILED_PRECONDITION
ABORTED
ALREADY_EXISTS
NOT_FOUND
```

A practical distinction:

| Situation | Status |
|---|---|
| Request is valid but entity state does not allow operation | `FAILED_PRECONDITION` |
| Transaction/concurrency conflict; retry at higher level may work | `ABORTED` |
| Create failed because resource already exists | `ALREADY_EXISTS` |
| Target entity does not exist | `NOT_FOUND` |

Example in case-management system:

```text
ApproveCase(caseId=123)

Case is already closed.
-> FAILED_PRECONDITION

Case version changed since reviewer opened it.
-> ABORTED

Case does not exist.
-> NOT_FOUND
```

---

### 5.3 Capacity / Overload / Dependency Failure

These are often retryable, but not blindly.

```text
UNAVAILABLE
RESOURCE_EXHAUSTED
DEADLINE_EXCEEDED
```

Practical mapping:

| Situation | Status |
|---|---|
| Service temporarily down | `UNAVAILABLE` |
| LB has no healthy backend | `UNAVAILABLE` |
| Dependency outage | `UNAVAILABLE` or domain-specific mapped error |
| Per-user quota exceeded | `RESOURCE_EXHAUSTED` |
| Server overloaded | `RESOURCE_EXHAUSTED` or `UNAVAILABLE` |
| Client deadline expired | `DEADLINE_EXCEEDED` |

Do not treat all `RESOURCE_EXHAUSTED` as retryable. A quota failure may only recover after minutes or hours.

## 6. Error Details: Beyond Code and Description

A status code alone may not be enough.

For structured error details, gRPC commonly uses richer error model patterns, especially in Google API ecosystems, where `google.rpc.Status` can carry typed details such as validation errors, retry information, quota failure, or localized message.

A generic Java server may still benefit from a structured trailer model:

```text
grpc-status: INVALID_ARGUMENT
grpc-message: validation failed
x-error-code: CASE_INVALID_STATE
x-error-field: postalCode
x-correlation-id: ...
```

However, avoid leaking sensitive internal details.

Bad:

```text
grpc-message: ORA-00001 unique constraint SYS_C007823 violated on table USER_PASSWORD_RESET_TOKEN
```

Better:

```text
grpc-message: duplicate request
x-error-code: DUPLICATE_OPERATION
x-correlation-id: 01HY...
```

For regulated systems, design error details with these invariants:

```text
1. Stable enough for clients to handle.
2. Specific enough for support and audit.
3. Not leaking internals, secrets, SQL, stack trace, infrastructure names, or PII.
4. Mapped consistently across services.
5. Traceable through logs and audit trail.
```

## 7. Java Error Mapping Pattern

A clean Java server should centralize exception-to-status mapping.

Conceptual example:

```java
public final class GrpcExceptionMapper {

    public static StatusRuntimeException toStatus(Throwable error, Metadata trailers) {
        if (error instanceof ValidationException ve) {
            return Status.INVALID_ARGUMENT
                    .withDescription(safe(ve.publicMessage()))
                    .asRuntimeException(trailers);
        }

        if (error instanceof EntityNotFoundException nf) {
            return Status.NOT_FOUND
                    .withDescription(safe(nf.publicMessage()))
                    .asRuntimeException(trailers);
        }

        if (error instanceof OptimisticLockException ole) {
            return Status.ABORTED
                    .withDescription("concurrent modification detected")
                    .asRuntimeException(trailers);
        }

        if (error instanceof BusinessPreconditionException bpe) {
            return Status.FAILED_PRECONDITION
                    .withDescription(safe(bpe.publicMessage()))
                    .asRuntimeException(trailers);
        }

        if (error instanceof RateLimitExceededException rle) {
            return Status.RESOURCE_EXHAUSTED
                    .withDescription("rate limit exceeded")
                    .asRuntimeException(trailers);
        }

        return Status.INTERNAL
                .withDescription("internal server error")
                .asRuntimeException(trailers);
    }

    private static String safe(String message) {
        if (message == null || message.isBlank()) {
            return "request failed";
        }
        return message.length() <= 512 ? message : message.substring(0, 512);
    }
}
```

Interceptor-style mapping:

```java
public final class ErrorMappingInterceptor implements ServerInterceptor {
    @Override
    public <ReqT, RespT> ServerCall.Listener<ReqT> interceptCall(
            ServerCall<ReqT, RespT> call,
            Metadata headers,
            ServerCallHandler<ReqT, RespT> next) {

        ServerCall.Listener<ReqT> delegate = next.startCall(call, headers);

        return new ForwardingServerCallListener.SimpleForwardingServerCallListener<>(delegate) {
            @Override
            public void onHalfClose() {
                try {
                    super.onHalfClose();
                } catch (Throwable t) {
                    Metadata trailers = new Metadata();
                    call.close(GrpcExceptionMapper.toStatus(t, trailers).getStatus(), trailers);
                }
            }
        };
    }
}
```

Real implementation must be careful not to double-close calls and must integrate with async response observers. The point is architectural: keep status mapping explicit and uniform.

## 8. Client-Side Error Handling Pattern

Client code should not scatter `StatusRuntimeException` decisions everywhere.

Bad:

```java
try {
    return stub.approveCase(request);
} catch (Exception e) {
    return null;
}
```

Better:

```java
public ApprovalResult approveCase(ApproveCaseRequest request, Duration deadline) {
    try {
        ApproveCaseResponse response = stub
                .withDeadlineAfter(deadline.toMillis(), TimeUnit.MILLISECONDS)
                .approveCase(request);

        return ApprovalResult.success(response);

    } catch (StatusRuntimeException e) {
        Status.Code code = e.getStatus().getCode();

        return switch (code) {
            case INVALID_ARGUMENT -> ApprovalResult.rejected("INVALID_INPUT", e.getStatus().getDescription());
            case FAILED_PRECONDITION -> ApprovalResult.rejected("INVALID_STATE", e.getStatus().getDescription());
            case ABORTED -> ApprovalResult.retryableConflict(e.getStatus().getDescription());
            case UNAVAILABLE, DEADLINE_EXCEEDED -> ApprovalResult.temporaryFailure(code.name());
            case UNAUTHENTICATED -> ApprovalResult.authFailure();
            case PERMISSION_DENIED -> ApprovalResult.forbidden();
            default -> ApprovalResult.unknownFailure(code.name());
        };
    }
}
```

Even better, centralize this in a typed client SDK.

```text
Application code should not know transport quirks.
Client SDK should expose domain-oriented outcomes.
```

## 9. Retry in gRPC: What Actually Happens

gRPC supports multiple retry concepts:

```text
transparent retry
configured retry policy
hedging policy
application-level retry
external retry by proxy / service mesh
job/message retry outside gRPC
```

These are different.

### 9.1 Transparent Retry

Transparent retry is the safest form. gRPC can retry in low-level situations where it knows the RPC was not processed by application logic.

Example:

```text
Client starts RPC.
Connection failure occurs before request reaches server application.
gRPC may retry transparently.
Application code may never know.
```

This is not a substitute for a real retry policy.

### 9.2 Configured Retry Policy

A retry policy tells gRPC:

```text
For method X,
retry status codes Y,
up to maxAttempts,
with initialBackoff/maxBackoff/backoffMultiplier,
subject to deadline and retry throttling.
```

Conceptual service config:

```json
{
  "methodConfig": [
    {
      "name": [
        { "service": "case.v1.CaseCommandService", "method": "ApproveCase" }
      ],
      "timeout": "3s",
      "retryPolicy": {
        "maxAttempts": 3,
        "initialBackoff": "100ms",
        "maxBackoff": "1s",
        "backoffMultiplier": 2.0,
        "retryableStatusCodes": ["UNAVAILABLE"]
      }
    }
  ]
}
```

But this is only safe if the operation semantics are safe.

`ApproveCase` may not be safe to retry unless it has an idempotency key or operation id.

---

## 10. Retry Decision Matrix

| RPC Type | Safe automatic retry? | Requirement |
|---|---:|---|
| Read-only query | Usually yes | Deadline, bounded attempts |
| Idempotent update by natural key | Sometimes | Stable idempotency semantics |
| Create with client-generated ID | Usually yes | Unique ID / operation ID |
| Create with server-generated ID | Dangerous | Idempotency key required |
| Payment/approval/submission | Dangerous | Strong duplicate suppression |
| Streaming RPC | Usually no generic retry | Resume protocol required |
| Large upload | Usually no blind retry | Chunk protocol / checksum / resume |

Retry is not a library decision only. It is a business operation decision.

## 11. Idempotency in gRPC

For side-effecting RPCs, add an operation identifier.

```proto
message ApproveCaseRequest {
  string case_id = 1;
  string reviewer_id = 2;
  string decision_note = 3;

  // Stable across retries of the same logical operation.
  string operation_id = 10;

  // Optional optimistic concurrency guard.
  int64 expected_version = 11;
}
```

Server invariant:

```text
If operation_id has already been processed:
  return the same logical result, or a stable duplicate result.

If operation_id has same ID but different fingerprint:
  reject as invalid/dangerous duplicate.
```

Storage pattern:

```text
idempotency_record
- operation_id
- caller_identity
- rpc_method
- request_fingerprint
- status
- response_hash or serialized response
- created_at
- expires_at
```

Pseudocode:

```java
@Transactional
public ApproveCaseResponse approveCase(ApproveCaseRequest request) {
    IdempotencyRecord existing = idempotencyStore.find(request.getOperationId());

    if (existing != null) {
        if (!existing.matchesFingerprint(fingerprint(request))) {
            throw new ValidationException("operation_id reused with different request");
        }
        return existing.toApproveCaseResponse();
    }

    CaseEntity entity = caseRepository.lockById(request.getCaseId());

    if (entity.version() != request.getExpectedVersion()) {
        throw new OptimisticConflictException();
    }

    entity.approve(request.getReviewerId(), request.getDecisionNote());
    caseRepository.save(entity);

    ApproveCaseResponse response = buildResponse(entity);
    idempotencyStore.saveSuccess(request.getOperationId(), fingerprint(request), response);
    return response;
}
```

This converts ambiguous network retry into a deterministic business result.

## 12. Retry Budget and Retry Throttling

Retries consume capacity.

A retry policy with `maxAttempts = 3` can multiply load:

```text
1000 original RPS
x 3 attempts
= up to 3000 backend attempts
```

During outage, retry can become an amplifier.

Use:

```text
small maxAttempts
short deadline
retryable codes only
jittered backoff
retry budget
server pushback if available
circuit breaker outside gRPC if needed
idempotency for side effects
metrics by attempt
```

A retry budget policy says:

```text
Only a bounded fraction of total calls may be retries.
When retry budget is exhausted, fail fast or degrade.
```

This is especially important in service meshes or layered systems where retries may exist at multiple levels.

## 13. Hedging in gRPC

Hedging sends multiple copies of the same request to different backends and uses the first successful response.

Mental model:

```text
Attempt 1 starts at t=0
If no response by hedgingDelay, attempt 2 starts
First successful response wins
Other attempts are cancelled
```

Hedging is useful for tail latency, not for normal failure recovery.

It is dangerous for side effects.

Good candidates:

```text
read-only lookup
cache read
idempotent search
metadata fetch
```

Bad candidates:

```text
approve case
submit application
send email
create payment
consume inventory
mutate workflow state
```

A hedging policy is not “faster retry”. It is intentional duplicate execution.

## 14. Service Config

Service config is how gRPC can configure behavior per service/method.

It may include:

```text
method timeout
retry policy
hedging policy
retry throttling
load balancing config
health checking config
waitForReady
```

Conceptual config:

```json
{
  "loadBalancingConfig": [
    { "round_robin": {} }
  ],
  "methodConfig": [
    {
      "name": [
        { "service": "case.v1.CaseQueryService" }
      ],
      "timeout": "2s",
      "retryPolicy": {
        "maxAttempts": 3,
        "initialBackoff": "50ms",
        "maxBackoff": "500ms",
        "backoffMultiplier": 2.0,
        "retryableStatusCodes": ["UNAVAILABLE"]
      }
    },
    {
      "name": [
        { "service": "case.v1.CaseCommandService", "method": "ApproveCase" }
      ],
      "timeout": "5s"
    }
  ]
}
```

Java channel example:

```java
Map<String, Object> serviceConfig = Map.of(
        "loadBalancingConfig", List.of(Map.of("round_robin", Map.of())),
        "methodConfig", List.of(
                Map.of(
                        "name", List.of(Map.of("service", "case.v1.CaseQueryService")),
                        "timeout", "2s",
                        "retryPolicy", Map.of(
                                "maxAttempts", 3.0,
                                "initialBackoff", "50ms",
                                "maxBackoff", "500ms",
                                "backoffMultiplier", 2.0,
                                "retryableStatusCodes", List.of("UNAVAILABLE")
                        )
                )
        )
);

ManagedChannel channel = ManagedChannelBuilder
        .forTarget("dns:///case-service.default.svc.cluster.local:9090")
        .defaultServiceConfig(serviceConfig)
        .enableRetry()
        .useTransportSecurity()
        .build();
```

Notes:

```text
1. Exact support depends on gRPC Java version.
2. Use official gRPC Java documentation/Javadocs for final syntax and supported features.
3. Do not put unsafe retry policy globally.
4. Prefer method-specific policy.
5. Keep service config under version control.
```

## 15. `waitForReady`

By default, if a channel is in transient failure, a call may fail fast.

`waitForReady` changes behavior:

```text
If channel is not ready,
queue call until channel becomes ready or deadline expires.
```

This can be useful during startup or deployment rollout.

But it can also hide outage and create queues.

Example:

```java
CaseServiceGrpc.CaseServiceBlockingStub stub = CaseServiceGrpc
        .newBlockingStub(channel)
        .withWaitForReady()
        .withDeadlineAfter(3, TimeUnit.SECONDS);
```

Never use `waitForReady` without deadline.

Bad:

```text
withWaitForReady + no deadline
-> unbounded waiting
-> request pile-up
-> memory/thread pressure
```

Good:

```text
withWaitForReady + explicit small deadline
-> tolerate short reconnect window
-> still bound resource usage
```

## 16. Name Resolution

Name resolution answers:

```text
Given target name X, what addresses and service config should the channel use?
```

Examples:

```text
dns:///case-service.default.svc.cluster.local:9090
static:///10.0.1.10:9090,10.0.1.11:9090
xds:///case-service
custom:///logical-service-name
```

Name resolution can return:

```text
addresses
attributes
service config
errors
```

Important distinction:

```text
DNS resolution does not equal load balancing.
```

If a DNS name resolves to multiple IPs but the gRPC client creates one HTTP/2 connection and sends many streams over it, traffic may remain pinned to one backend unless the load balancing policy creates and uses multiple subchannels.

## 17. Load Balancing Model

gRPC client-side load balancing has this flow:

```text
1. Channel has a target name.
2. Name resolver resolves target to addresses and possibly service config.
3. Load balancing policy receives resolved addresses.
4. Policy creates/manages subchannels.
5. Each RPC is assigned to a subchannel according to picker logic.
```

Key terms:

| Term | Meaning |
|---|---|
| Channel | Logical client connection to service target |
| Subchannel | Connection-like object to one backend/address |
| Name Resolver | Resolves target to addresses/config |
| Load Balancing Policy | Picks subchannel for each RPC |
| Picker | Runtime object used to pick a ready subchannel |
| Connectivity State | Channel/subchannel state such as READY or TRANSIENT_FAILURE |

Common policies:

| Policy | Behavior |
|---|---|
| `pick_first` | Connect to first available address and use it |
| `round_robin` | Maintain subchannels and distribute calls among ready backends |
| xDS policies | Control-plane driven LB / traffic management |

For Kubernetes, this matters because a service DNS name may resolve to a ClusterIP rather than individual pod IPs. If you use ClusterIP, the gRPC client sees one address and Kubernetes/load balancer decides backend routing at lower layer. Because gRPC uses long-lived HTTP/2 connections, connection-level balancing may be uneven. For true client-side balancing across pods, you usually need headless service DNS or xDS/service mesh/control-plane integration.

## 18. Channel Connectivity State

A channel transitions through states such as:

```text
IDLE
CONNECTING
READY
TRANSIENT_FAILURE
SHUTDOWN
```

Mental model:

```text
IDLE
  Channel has no active connection attempt.

CONNECTING
  Trying to resolve/connect.

READY
  At least one usable transport is ready.

TRANSIENT_FAILURE
  Connection attempts failed; backoff applies.

SHUTDOWN
  Channel is closed.
```

Do not interpret `READY` as “remote business service is healthy.”

It means the transport has a usable connection. The remote app can still fail every method with `UNAVAILABLE`, `INTERNAL`, or application-level errors.

## 19. gRPC Health Checking

gRPC has a standard health checking service.

Client-side health checking can prevent requests from being sent to a backend until the health service reports serving.

Important distinction:

```text
TCP connect success
  != gRPC channel ready
  != health check serving
  != application dependency healthy
  != method-specific readiness
```

Health check should be designed carefully:

```text
Liveness: should this process be restarted?
Readiness: should this process receive traffic?
Dependency readiness: can critical dependencies serve this method?
Method-specific readiness: can this service serve this operation class?
```

Avoid health checks that are too shallow:

```text
return SERVING if process is alive
```

Avoid health checks that are too deep:

```text
query all dependencies synchronously every second
```

Good health checking is stable, cheap, and aligned with routing decisions.

## 20. `UNAVAILABLE`: The Most Misunderstood Status

`UNAVAILABLE` means the service is currently unavailable or the call could not be completed due to transient transport/service condition.

Possible causes:

```text
DNS resolution failed
no addresses resolved
connection refused
TLS handshake failed
ALPN mismatch
HTTP/2 GOAWAY
backend draining
load balancer no healthy target
service mesh ejection
server overloaded
connection reset
channel in TRANSIENT_FAILURE
proxy closed idle connection
server returned UNAVAILABLE intentionally
```

Therefore this log is not enough:

```text
RPC failed: UNAVAILABLE
```

Better:

```text
rpc.method=case.v1.CaseQueryService/GetCase
rpc.status=UNAVAILABLE
rpc.elapsed_ms=148
rpc.deadline_ms=2000
rpc.attempt=2
rpc.target=dns:///case-service.default.svc.cluster.local:9090
rpc.lb_policy=round_robin
rpc.channel_state=TRANSIENT_FAILURE
rpc.description="io exception"
trace_id=...
```

## 21. `DEADLINE_EXCEEDED` vs `CANCELLED`

`DEADLINE_EXCEEDED` usually means the operation did not complete before the deadline.

`CANCELLED` usually means cancellation was requested by caller or propagated by runtime.

But distributed interpretation is subtle.

Client view:

```text
Client deadline expires.
Client sees DEADLINE_EXCEEDED.
Client cancels stream.
Server may observe cancellation.
```

Server view:

```text
Server may see CANCELLED because client deadline expired and client cancelled.
```

This is why server logs often show `CANCELLED` when client logs show `DEADLINE_EXCEEDED`.

Do not treat all server-side `CANCELLED` as server bug.

Ask:

```text
Did client timeout first?
Did upstream caller cancel?
Did browser/user abort workflow?
Did batch job shut down?
Did load balancer close connection?
```

## 22. `ABORTED` vs `FAILED_PRECONDITION` vs `UNAVAILABLE`

These three are often confused.

Use this decision logic:

```text
Can retrying the exact same request immediately against another backend likely succeed?
  -> maybe UNAVAILABLE

Can retrying at a higher transaction/workflow level after refreshing state succeed?
  -> ABORTED

Is the system state currently unsuitable and must be changed before retry?
  -> FAILED_PRECONDITION
```

Example:

```text
Case approval RPC

DB optimistic lock conflict:
  ABORTED

Case already closed:
  FAILED_PRECONDITION

Case service pod restarting:
  UNAVAILABLE
```

## 23. Load Balancing in Kubernetes

A common expectation:

```text
I have 10 gRPC pods behind a Kubernetes Service.
Therefore calls will be evenly balanced.
```

Not necessarily.

Why?

```text
gRPC uses HTTP/2 long-lived connections.
Many RPCs can multiplex on one connection.
If the client has one connection to a ClusterIP,
L4 balancing happens at connection creation time,
not per RPC.
```

Possible strategies:

### Strategy A — Server-side / L4 balancing via ClusterIP

```text
Client -> ClusterIP -> one backend per connection
```

Pros:

```text
simple
standard Kubernetes Service
no custom client resolver
```

Cons:

```text
uneven distribution with long-lived connections
rebalancing slow after scale events
connection pinning
```

### Strategy B — Headless Service + client-side round_robin

```text
Client resolves pod IPs
Client creates subchannels
round_robin distributes RPCs
```

Pros:

```text
better RPC-level distribution
client aware of multiple backends
```

Cons:

```text
more client config
DNS behavior matters
pod churn handling matters
```

### Strategy C — Service mesh / xDS

```text
Client or sidecar uses control-plane-provided endpoints/policies
```

Pros:

```text
advanced traffic management
health/ejection/retry policy centralized
mTLS integration
```

Cons:

```text
more moving parts
harder debugging
policy interactions
possible double retry/double timeout
```

## 24. Method-Level Policy Design

Do not configure gRPC behavior only at channel level.

Different methods have different semantics.

Example service:

```proto
service CaseQueryService {
  rpc GetCase(GetCaseRequest) returns (GetCaseResponse);
  rpc SearchCases(SearchCasesRequest) returns (SearchCasesResponse);
}

service CaseCommandService {
  rpc SubmitCase(SubmitCaseRequest) returns (SubmitCaseResponse);
  rpc ApproveCase(ApproveCaseRequest) returns (ApproveCaseResponse);
}
```

Policy:

| Method | Deadline | Retry | Hedging | Reason |
|---|---:|---:|---:|---|
| `GetCase` | 1s | yes, small | maybe | read-only |
| `SearchCases` | 3s | limited | no | can be expensive |
| `SubmitCase` | 5s | only with idempotency | no | side effect |
| `ApproveCase` | 5s | only with operation ID | no | regulated state transition |

## 25. Application-Level Retry vs gRPC Retry

When should retry be outside gRPC?

Use gRPC retry for:

```text
small unary call
safe/idempotent semantics
transport-level transient failure
method-specific policy
short backoff
bounded attempts
```

Use application-level retry for:

```text
workflow-aware retry
business conflict resolution
refresh state then retry
chunk upload resume
stream reconnect and resume
compensation logic
manual review path
job processing
```

Example:

```text
ApproveCase returns ABORTED due to version conflict.

gRPC retrying same request is useless.
Application must reload latest case, re-evaluate rule, then decide.
```

## 26. Streaming RPC Retry

Generic retry does not work cleanly for streaming.

For server streaming:

```text
Client receives items 1..500.
Stream breaks.
Can client retry from item 501?
Only if protocol supports cursor/resume.
```

For client streaming:

```text
Client uploaded chunks 1..20.
Connection breaks.
Did server persist chunks 1..20?
Need upload session/chunk id/checksum.
```

For bidirectional streaming:

```text
Both sides have state.
Reconnect requires session model, sequence numbers, replay window, and idempotency.
```

Design resumable streaming explicitly.

```proto
message StreamCursor {
  string stream_id = 1;
  int64 last_seen_sequence = 2;
}
```

Do not rely on generic retry to repair streaming semantics.

## 27. Deadlines and Retry Interaction

Deadline is the outer budget.

Retries must fit inside deadline.

Bad:

```text
Deadline: none
Retry: 5 attempts
Backoff: up to 10s
Result: requests hang, pile up, and overload system
```

Good:

```text
Deadline: 2s
Retry: max 2-3 attempts
Backoff: 50ms, 100ms, jittered
Only UNAVAILABLE
Only read/idempotent operation
```

If each attempt has its own timeout but no overall deadline, total latency can explode.

```text
attempt 1: 2s
backoff: 1s
attempt 2: 2s
backoff: 2s
attempt 3: 2s
= 9s user-visible latency
```

Use one logical deadline per user/workflow operation.

## 28. Authentication Retry

Authentication errors need special handling.

```text
UNAUTHENTICATED
  -> token missing/expired/invalid

PERMISSION_DENIED
  -> token valid but principal lacks permission
```

A client may refresh token once on `UNAUTHENTICATED`, but should not retry endlessly.

Pattern:

```text
1. Call with token.
2. If UNAUTHENTICATED and token may be expired:
   refresh token once.
3. Retry once with new token.
4. If still UNAUTHENTICATED, fail.
```

Do not refresh token for `PERMISSION_DENIED`.

For mTLS + token systems, distinguish:

```text
TLS handshake failure
  -> not a gRPC status from application; channel/connect failure

UNAUTHENTICATED
  -> application/auth layer rejected identity/token

PERMISSION_DENIED
  -> identity accepted, action forbidden
```

## 29. Server Pushback and Rate Limiting

When server is overloaded, it should fail predictably.

Possible statuses:

```text
RESOURCE_EXHAUSTED
UNAVAILABLE
```

Server should include structured hint when possible:

```text
retry after 200ms
quota exceeded until reset time
capacity overloaded
```

Even if clients do not automatically understand all hints, observability and typed SDKs can.

Anti-pattern:

```text
Server keeps accepting unlimited calls.
Thread pool saturates.
Deadline expires.
Clients retry.
Load doubles.
Everything collapses.
```

Better:

```text
Server detects queue saturation.
Rejects early with RESOURCE_EXHAUSTED.
Client respects retry budget/backoff.
Circuit breaker opens if failure rate high.
```

## 32. Case Study: Case Approval Service

Suppose you have:

```proto
service CaseCommandService {
  rpc ApproveCase(ApproveCaseRequest) returns (ApproveCaseResponse);
}
```

Naive design:

```proto
message ApproveCaseRequest {
  string case_id = 1;
  string reviewer_id = 2;
}
```

Naive client policy:

```json
{
  "retryPolicy": {
    "maxAttempts": 5,
    "retryableStatusCodes": ["UNAVAILABLE", "DEADLINE_EXCEEDED"]
  }
}
```

Failure scenario:

```text
Client sends ApproveCase.
Server approves case and commits transaction.
Connection breaks before response trailers reach client.
Client sees UNAVAILABLE.
Client retries.
Server approves again or creates duplicate audit/action/email.
```

Better design:

```proto
message ApproveCaseRequest {
  string case_id = 1;
  string reviewer_id = 2;
  string decision_note = 3;
  int64 expected_version = 4;
  string operation_id = 5;
}
```

Server behavior:

```text
operation_id first time:
  validate expected_version
  approve
  write audit
  save idempotency result
  return response

same operation_id same fingerprint:
  return stored result

same operation_id different fingerprint:
  INVALID_ARGUMENT or FAILED_PRECONDITION

version conflict:
  ABORTED

case already closed:
  FAILED_PRECONDITION
```

Client policy:

```text
Retry only UNAVAILABLE.
Max attempts 2 or 3.
Overall deadline 5s.
No hedging.
Metrics by attempt.
```

This turns network uncertainty into a controlled state machine.

## 33. Case Study: Query Service Load Balancing

Service:

```proto
service CaseQueryService {
  rpc GetCase(GetCaseRequest) returns (GetCaseResponse);
  rpc SearchCases(SearchCasesRequest) returns (SearchCasesResponse);
}
```

Deployment:

```text
10 pods in Kubernetes
client uses dns:///case-query.default.svc.cluster.local:9090
policy: pick_first
```

Observed:

```text
One pod receives most traffic.
Other pods mostly idle.
```

Reason:

```text
HTTP/2 multiplexes many RPCs on one connection.
pick_first may use one ready address.
Kubernetes ClusterIP balances per connection, not per RPC.
```

Possible remediation:

```text
Use headless service and round_robin.
Use service mesh/xDS routing.
Increase channels only as a workaround, not first choice.
Validate with per-pod request metrics.
```

## 34. Production-Grade gRPC Client Wrapper

A mature Java codebase should not let every service create arbitrary channels/stubs.

Create a governed client module:

```text
GrpcClientFactory
GrpcChannelRegistry
GrpcDeadlinePolicy
GrpcRetryPolicy
GrpcErrorMapper
GrpcMetadataInjector
GrpcMetricsInterceptor
GrpcTracingInterceptor
GrpcAuthInterceptor
```

Example factory shape:

```java
public final class GrpcClientFactory {

    public ManagedChannel createChannel(GrpcClientConfig config) {
        ManagedChannelBuilder<?> builder = ManagedChannelBuilder
                .forTarget(config.target())
                .defaultLoadBalancingPolicy(config.loadBalancingPolicy())
                .defaultServiceConfig(config.serviceConfig())
                .enableRetry();

        if (config.tlsEnabled()) {
            builder.useTransportSecurity();
        } else {
            builder.usePlaintext();
        }

        return builder.build();
    }
}
```

A better implementation would also handle:

```text
shutdown lifecycle
health check config
interceptors
metrics
tracing
auth
certificates
channel reuse
per-client config validation
safe defaults
```

## 35. Anti-Patterns

### Anti-pattern 1: Retrying all gRPC errors

```text
Retry on UNKNOWN, INTERNAL, INVALID_ARGUMENT, PERMISSION_DENIED.
```

Why bad:

```text
amplifies bugs
hides contract errors
creates duplicate side effects
wastes capacity
```

---

### Anti-pattern 2: No deadline

```text
stub.getCase(request)
```

Why bad:

```text
unbounded wait
resource leak under partial outage
poor cancellation semantics
hard incident recovery
```

---

### Anti-pattern 3: Global retry config for all methods

```text
All methods retry UNAVAILABLE and DEADLINE_EXCEEDED up to 5 attempts.
```

Why bad:

```text
read and write operations have different semantics
streaming has different semantics
side effects can duplicate
```

---

### Anti-pattern 4: Mapping every exception to UNKNOWN

Why bad:

```text
clients cannot make correct decisions
support cannot diagnose
retry behavior becomes unsafe
```

---

### Anti-pattern 5: Assuming Kubernetes Service gives per-RPC balancing

Why bad:

```text
HTTP/2 long-lived connection breaks the mental model of per-request L4 balancing
```

---

### Anti-pattern 6: Hedging state-changing calls

Why bad:

```text
hedging intentionally duplicates execution
unsafe without idempotency
```

## 36. Design Checklist

For every gRPC service, answer:

```text
1. What are all public methods?
2. Which methods are read-only?
3. Which methods mutate state?
4. Which methods are naturally idempotent?
5. Which methods require operation_id/idempotency key?
6. What deadline applies per method?
7. What status codes are allowed per method?
8. Which statuses are retryable per method?
9. Is retry configured in service config, application code, mesh, or all of them?
10. Is there retry budget/throttling?
11. Is hedging allowed anywhere?
12. How is name resolution performed?
13. What load balancing policy is used?
14. Does resolver return one address or many?
15. Is health checking used?
16. How are errors mapped?
17. Are error details safe and stable?
18. Are metrics/logs/traces enough to debug UNAVAILABLE/DEADLINE_EXCEEDED/CANCELLED?
19. Is channel lifecycle managed centrally?
20. Are shutdown and draining handled?
```
