# Strict Coding Standards: Java Security

> **Document status:** Mandatory coding standard for LLM-generated Java implementation.  
> **Scope:** Java 11, Java 17, Java 21, Java 25, and framework code built on top of Java.  
> **Mode:** Security-first, deny-by-default, evidence-driven.  
> **Audience:** LLM code agents, human implementers, code reviewers, security reviewers.

---

## 1. Purpose

This document defines strict security coding standards that every LLM code agent MUST follow when generating, modifying, or reviewing Java code.

This is not a generic security checklist. It is a behavioral contract for implementation. The agent MUST produce Java code that is secure by construction, explicit about trust boundaries, and reviewable by humans.

The goal is to prevent the most common failure modes in Java code generated by LLMs:

- trusting external input too early
- adding authentication without authorization
- building SQL/JPQL/LDAP/OS commands with concatenation
- using weak cryptography or inventing custom crypto
- disabling TLS validation
- logging secrets or personal data
- deserializing untrusted data
- creating SSRF/path traversal/file upload bugs
- hiding security decisions inside helpers
- relying on deprecated platform mechanisms such as the Security Manager
- producing code that works functionally but violates security invariants

## 3. Baseline Security Principle

Every generated code change MUST answer these questions before implementation:

1. What data enters this code?
2. Who controls that data?
3. What trust boundary is crossed?
4. What validation is required before the data is used?
5. What authority does this code exercise?
6. What can fail securely?
7. What must never be logged?
8. What security test proves the rule?

If the agent cannot answer these questions, it MUST NOT add security-sensitive implementation.

## 4. Mandatory Agent Behavior

### 4.1 Required before writing security-sensitive code

The LLM agent MUST identify whether the change touches any of these areas:

- authentication
- authorization
- session handling
- token handling
- password handling
- cryptography
- TLS/mTLS
- secrets
- database query construction
- file upload/download
- network calls
- XML parsing
- JSON polymorphic binding
- serialization/deserialization
- command execution
- reflection/class loading
- template rendering
- logging/auditing
- dependency or build configuration
- security headers or CORS
- multi-tenant data access
- personally identifiable information
- regulatory or audit-sensitive data

If yes, the agent MUST apply this standard and include security tests or reviewer notes.

### 4.2 Required output for security-sensitive change

For every security-sensitive change, the agent MUST include:

```text
Security Impact:
- Trust boundary:
- Sensitive data involved:
- Authorization rule:
- Validation rule:
- Failure behavior:
- Logging behavior:
- Tests added:
- Residual risk:
```

### 4.3 Forbidden agent behavior

The agent MUST NOT:

- invent a custom security protocol
- invent custom crypto
- disable certificate validation
- disable hostname verification
- suppress security warnings without justification
- log secrets for debugging
- create catch-all authorization rules
- assume authentication means authorization
- assume internal network means trusted
- accept unbounded input
- deserialize untrusted data
- use string concatenation for executable contexts
- add temporary insecure code with comments like `TODO secure later`
- rely on the Java Security Manager as a sandbox

## 5. Security Classification

Every data flow MUST classify data before use.

### 5.1 Data classes

| Class     | Examples                                                    | Rule                                           |
| --------- | ----------------------------------------------------------- | ---------------------------------------------- |
| Public    | published metadata, static labels                           | Still validate shape and encoding              |
| Internal  | service IDs, config flags, workflow state                   | Validate source and integrity                  |
| Sensitive | email, phone, IDs, case data, addresses                     | Minimize, mask, and authorize                  |
| Secret    | passwords, tokens, API keys, private keys                   | Never log; never return; never store plaintext |
| Regulated | audit records, enforcement records, financial/legal records | Must be traceable and tamper-aware             |

### 5.2 Mandatory handling

Sensitive and secret values MUST NOT be stored in:

- plain logs
- exception messages
- query strings
- file names
- metric labels
- thread names
- cache keys visible in diagnostics
- `toString()` output
- actuator/debug endpoints
- client-visible error bodies

## 6. Trust Boundaries

The following are always untrusted unless explicitly proven otherwise:

- HTTP request body, query, path, headers, cookies
- message queue payloads
- Kafka/RabbitMQ/JMS messages
- files and uploaded content
- database content originally supplied by users
- environment variables from deployment
- command-line arguments
- system properties
- DNS responses
- remote service responses
- JWT claims before signature/issuer/audience validation
- XML/JSON/YAML configuration loaded from external sources
- serialized objects
- data from “internal” microservices

Internal traffic is not automatically trusted.

A service-to-service call MUST still validate:

- caller identity
- audience
- token expiry
- tenant/scope/role claims
- request shape
- replay or idempotency requirements where relevant

## 7. Input Validation

### 7.1 Core rule

All externally controlled input MUST be validated at the boundary before entering domain logic.

Validation MUST be:

- allow-list based where possible
- type-specific
- length-bounded
- range-bounded
- canonicalized before security decision
- separated from business mutation
- covered by tests

### 7.2 Forbidden validation patterns

The agent MUST NOT:

```java
// BAD: deny-list only
if (input.contains("../") || input.contains("<script>")) { ... }

// BAD: validation after usage
repository.findByDynamicSql(input);
validate(input);

// BAD: trusting client-provided role/status
if (request.role().equals("ADMIN")) { ... }

// BAD: silently trimming security-sensitive identifiers
String userId = request.userId().trim();
```

### 7.3 Required validation categories

| Input type      | Required checks                                            |
| --------------- | ---------------------------------------------------------- |
| ID              | format, length, allowed chars, existence, tenant ownership |
| Email           | length, syntax, normalization policy, verification state   |
| Enum-like value | parse via strict allow-list                                |
| Date/time       | timezone, range, precision, clock source                   |
| Amount          | scale, range, currency, overflow behavior                  |
| File name       | generated server-side or strict allow-list                 |
| URL             | scheme/host/port allow-list; DNS/IP checks                 |
| Header          | length, encoding, trusted source                           |
| JSON            | schema/DTO binding; unknown field policy                   |
| XML             | XXE disabled; schema policy                                |

### 7.4 Canonicalization

Canonicalize before security checks where equivalent representations are possible.

Examples:

- file path normalization before base-directory containment check
- Unicode normalization before identifier comparison if identifiers can contain non-ASCII
- URL parsing before host allow-list check
- lowercasing host names using `Locale.ROOT`
- trimming only when explicitly part of the input contract

## 8. Output Encoding and Injection Prevention

### 8.1 Executable contexts

Any value inserted into an executable or interpreted context MUST use the correct safe API.

| Context       | Required defense                                    |
| ------------- | --------------------------------------------------- |
| SQL/JDBC      | `PreparedStatement` bind parameters                 |
| JPQL/Criteria | named/positional parameters or Criteria API         |
| LDAP          | escaping via LDAP-safe library/API                  |
| OS command    | avoid shell; pass fixed executable + argument array |
| HTML          | context-aware HTML escaping by template engine      |
| JavaScript    | avoid inline JS; use JSON encoder and CSP           |
| URL           | URI builder/encoder per component                   |
| XML           | safe serializer, XXE disabled for parsing           |
| Regex         | quote user literals with `Pattern.quote`            |
| Log           | structured logging with masking                     |

### 8.2 Forbidden

```java
// BAD: SQL injection
String sql = "select * from users where id = '" + userId + "'";

// BAD: command injection risk
Runtime.getRuntime().exec("convert " + filename);

// BAD: regex injection / ReDoS risk
Pattern.compile(userInput);

// BAD: HTML injection
return "<div>" + request.name() + "</div>";
```

### 8.3 Required

```java
// GOOD: SQL parameter binding
try (PreparedStatement ps = connection.prepareStatement(
        "select * from users where id = ?")) {
    ps.setString(1, userId.value());
}
```

Dynamic identifiers such as table name, column name, direction, or sort field cannot be bound as parameters. They MUST use strict allow-list mapping.

```java
String sortColumn = switch (request.sortBy()) {
    case "createdAt" -> "created_at";
    case "status" -> "status";
    default -> throw new BadRequestException("Unsupported sort field");
};
```

## 9. Authentication

### 9.1 Core rule

Authentication verifies identity. It does not grant permission by itself.

The agent MUST NOT implement custom authentication unless explicitly required and reviewed.

Prefer battle-tested framework integration:

- servlet/Jakarta Security where applicable
- Spring Security where applicable
- Quarkus/OIDC where applicable
- MicroProfile JWT where applicable
- external IdP via OIDC/SAML where applicable

### 9.2 Token validation

JWT/OIDC tokens MUST validate:

- signature
- issuer
- audience
- expiry
- not-before
- algorithm allow-list
- key rotation/JWKS behavior
- tenant or realm where applicable
- token type/use where applicable

Forbidden:

```java
// BAD: decoding is not validation
DecodedJWT jwt = JWT.decode(token);
String userId = jwt.getSubject();
```

### 9.3 Session handling

Session identifiers MUST:

- be generated by secure framework/session manager
- never appear in URL query strings
- use `HttpOnly`
- use `Secure` in HTTPS environments
- use appropriate `SameSite`
- rotate after privilege elevation/login
- expire predictably

## 10. Authorization

### 10.1 Core rule

Every state-changing operation and every sensitive read MUST have explicit authorization.

Authorization MUST be checked server-side against trusted identity and trusted resource state.

### 10.2 Required authorization dimensions

The agent MUST consider:

- actor identity
- role/scope/permission
- tenant or organization boundary
- object ownership
- workflow state
- case/status lifecycle
- field-level sensitivity
- action type
- delegation/impersonation context
- regulatory/audit constraint

### 10.3 Forbidden authorization patterns

```java
// BAD: authentication-only check
if (principal != null) approveCase(caseId);

// BAD: trusting client-provided owner
if (request.ownerId().equals(principal.id())) update();

// BAD: checking role but not object scope
if (principal.hasRole("OFFICER")) viewCase(caseId);

// BAD: filtering after retrieving all data
return repository.findAll().stream()
        .filter(c -> c.ownerId().equals(principal.id()))
        .toList();
```

### 10.4 Required pattern

```java
CaseRecord record = caseRepository.findById(caseId)
        .orElseThrow(NotFoundException::new);

authorization.requireAllowed(
        principal,
        Action.APPROVE_CASE,
        record.authorizationContext()
);

workflow.approve(record);
```

Authorization MUST fail closed.

If authorization state cannot be determined, deny.

## 11. Multi-Tenancy and Data Isolation

### 11.1 Core rule

Tenant isolation MUST be enforced in the server-side data access path.

The agent MUST NOT rely only on UI filtering or client-supplied tenant IDs.

### 11.2 Required

Every tenant-scoped query MUST include tenant or equivalent scope from trusted context.

```java
repository.findByTenantIdAndCaseId(principal.tenantId(), caseId);
```

### 11.3 Forbidden

```java
// BAD: tenant comes from request body
repository.findByTenantIdAndCaseId(request.tenantId(), caseId);

// BAD: global query followed by filtering
CaseRecord record = repository.findById(caseId).orElseThrow();
if (!record.tenantId().equals(request.tenantId())) throw new ForbiddenException();
```

Global lookup followed by authorization can be allowed only if the repository method is specifically designed to avoid data leakage and distinguishability concerns have been reviewed.

## 12. Cryptography

### 12.1 Core rule

Do not design cryptography. Use standard, reviewed algorithms through well-maintained libraries or JDK APIs.

The agent MUST NOT invent:

- custom encryption format
- custom MAC/signature construction
- custom password hashing
- custom random generator
- custom token format
- custom key exchange

### 12.2 Allowed by default

| Need                 | Preferred approach                                             |
| -------------------- | -------------------------------------------------------------- |
| Random token         | `SecureRandom` or framework token generator                    |
| Symmetric encryption | AEAD mode such as AES-GCM with correct nonce handling          |
| Integrity            | HMAC with SHA-256 or stronger, or digital signatures           |
| Password storage     | Argon2id, bcrypt, scrypt, or PBKDF2 per policy/library support |
| TLS                  | JSSE/framework defaults with modern protocol/cipher policy     |
| Key storage          | KMS/HSM/secret manager/keystore, not source code               |

### 12.3 Forbidden algorithms and modes

Forbidden for new code unless explicitly required for legacy compatibility and isolated:

- MD5
- SHA-1 for signatures/integrity
- DES
- 3DES for new encryption
- RC4
- ECB mode
- RSA encryption without OAEP
- raw RSA signatures without standard padding
- homegrown XOR/base64 “encryption”
- non-cryptographic RNG for secrets (`Random`, `ThreadLocalRandom`, `Math.random`)

### 12.4 Required JCA behavior

The agent MUST specify algorithm/mode/padding explicitly for `Cipher`.

Forbidden:

```java
Cipher.getInstance("AES"); // ambiguous, often ECB provider default
```

Required:

```java
Cipher.getInstance("AES/GCM/NoPadding");
```

### 12.5 Nonce/IV rules

For AEAD modes such as GCM:

- nonce/IV MUST be unique for the key
- nonce/IV MUST NOT be reused with the same key
- nonce/IV MAY be stored with ciphertext if not secret
- authentication tag MUST be verified before plaintext is trusted
- key rotation policy MUST exist for high-volume encryption

### 12.6 Password storage

Passwords MUST NOT be:

- stored plaintext
- encrypted reversibly
- hashed with fast hashes such as SHA-256/512 alone
- logged
- sent by email
- returned by API

Password verification MUST use a password hashing/KDF library with:

- per-password salt
- configurable cost/work factor
- algorithm marker/version in stored hash
- constant-time verification where supported
- migration path for old hashes

Preferred order depends on platform policy:

1. Argon2id
2. bcrypt
3. scrypt
4. PBKDF2 when required by FIPS/compliance constraints

### 12.7 Secret comparison

Secret comparison MUST use constant-time comparison where practical.

```java
MessageDigest.isEqual(expected, actual);
```

Do not use `String.equals` for MACs, signatures, or tokens when timing leakage matters.

## 13. Secrets Management

### 13.1 Core rule

Secrets MUST NOT be hardcoded.

Secrets include:

- passwords
- tokens
- API keys
- client secrets
- private keys
- database credentials
- encryption keys
- signing keys
- webhook secrets
- session secrets

### 13.2 Forbidden

```java
private static final String API_KEY = "abc123";
private static final String DB_PASSWORD = "password";
```

Also forbidden:

- secrets in source code
- secrets in test resources committed to VCS
- secrets in exception messages
- secrets in logs
- secrets in metrics labels
- secrets in URL query strings
- secrets in generated docs
- secrets in Docker image layers
- secrets in default config files

### 13.3 Required

Use environment-specific secret sources:

- cloud secret manager
- Kubernetes secret mounted as file/env with least exposure
- Vault/KMS/HSM
- protected CI/CD secret store
- Java keystore/truststore where appropriate

Code MUST support rotation where practical.

### 13.4 Secret redaction

Logging and error-handling code MUST redact fields with names such as:

- `password`
- `secret`
- `token`
- `apiKey`
- `authorization`
- `cookie`
- `set-cookie`
- `privateKey`
- `clientSecret`
- `sessionId`

Redaction MUST happen before values enter log events.

## 14. TLS and Certificate Handling

### 14.1 Core rule

TLS validation MUST remain enabled.

The agent MUST NOT generate trust-all code.

Forbidden:

```java
// BAD: trust-all TrustManager
public void checkServerTrusted(X509Certificate[] chain, String authType) { }

// BAD: disable hostname verification
hostnameVerifier = (host, session) -> true;
```

### 14.2 Required

Outbound HTTPS clients MUST:

- use platform/framework TLS validation
- configure truststore explicitly if custom CA is needed
- enforce hostname verification
- set timeouts
- avoid logging certificates or keys unnecessarily
- avoid global JVM-wide TLS hacks

### 14.3 mTLS

mTLS code MUST distinguish:

- truststore: who we trust
- keystore: our client identity/private key
- key password handling
- rotation process
- environment-specific certificate source

---

## 15. Deserialization and Object Binding

### 15.1 Core rule

Native Java deserialization of untrusted data is forbidden.

Forbidden:

```java
ObjectInputStream in = new ObjectInputStream(request.getInputStream());
Object value = in.readObject();
```

### 15.2 Allowed only with strict controls

Deserialization may be allowed only when all are true:

- data source is trusted or cryptographically authenticated
- class allow-list is enforced
- object graph size/depth is bounded
- no sensitive data is serialized unintentionally
- compatibility/versioning is documented
- tests include malicious payload rejection

### 15.3 ObjectInputFilter

If Java serialization must be used, the code MUST configure an allow-list based `ObjectInputFilter`.

Reject by default.

### 15.4 JSON polymorphism

Polymorphic JSON binding is restricted.

Forbidden by default:

- accepting arbitrary class names from JSON
- enabling default typing globally
- binding request body directly to domain/entity objects

Required:

- explicit DTO types
- allow-listed subtype mapping
- unknown field policy defined
- validation after binding

---

## 16. XML, YAML, and Parser Security

### 16.1 XML

XML parsing of untrusted data MUST disable dangerous features:

- external entities
- external DTDs
- XInclude unless required and reviewed
- entity expansion abuse

The parser MUST have size/depth/resource limits where available.

### 16.2 YAML

YAML parsing is restricted because many YAML libraries support object construction.

Required:

- safe loader
- DTO binding only
- no arbitrary type construction
- size limits
- no secret logging

### 16.3 Template engines

Templates MUST use auto-escaping where available.

Untrusted values MUST NOT be inserted into raw HTML/JS/CSS contexts without context-specific encoding.

## 17. File Upload, Download, and Path Traversal

### 17.1 Upload rule

Uploaded files are untrusted.

Before storing or processing uploaded content, code MUST enforce:

- max size
- allowed content type policy
- allowed extension policy if extension matters
- generated server-side storage name
- safe storage directory
- antivirus/malware scanning where required by system policy
- no direct execution
- no archive extraction without zip-slip defense
- no trust in client-provided filename

### 17.2 Path traversal defense

Path access based on user input MUST:

1. resolve against a fixed base directory
2. normalize/canonicalize
3. verify containment under base directory
4. define symlink policy
5. reject absolute paths unless explicitly allowed

Forbidden:

```java
Path path = Paths.get("/data/uploads/" + request.fileName());
return Files.readAllBytes(path);
```

Required:

```java
Path base = uploadRoot.toRealPath();
Path candidate = base.resolve(requestedName).normalize();
if (!candidate.startsWith(base)) {
    throw new BadRequestException("Invalid file path");
}
```

### 17.3 Download rule

Download endpoints MUST check:

- authentication
- authorization to object
- tenant/scope ownership
- file existence through metadata, not path guessing
- content disposition safety
- content type
- no secret leakage in file name

## 18. SSRF and Outbound Requests

### 18.1 Core rule

User-controlled URLs are dangerous.

Outbound requests to URLs influenced by users MUST use SSRF controls.

### 18.2 Required SSRF controls

- scheme allow-list (`https` by default)
- host allow-list where possible
- port allow-list
- block private, loopback, link-local, multicast, and metadata IP ranges
- validate after DNS resolution
- revalidate after redirects
- disable redirects by default
- set connect/read/request timeout
- cap response body size
- no forwarding of internal credentials to arbitrary hosts

### 18.3 Forbidden

```java
HttpRequest request = HttpRequest.newBuilder(URI.create(userUrl)).GET().build();
```

Unless wrapped in a reviewed SSRF-safe client.

---

## 19. Command Execution and Process Handling

### 19.1 Core rule

Avoid OS command execution.

If required, use fixed executable and argument array. Never invoke shell with user input.

Forbidden:

```java
Runtime.getRuntime().exec("sh -c " + userCommand);
```

Restricted:

```java
new ProcessBuilder("/usr/bin/convert", inputPath.toString(), outputPath.toString())
        .redirectErrorStream(false)
        .start();
```

### 19.2 Required controls

- fixed executable path
- arguments validated independently
- no shell interpolation
- timeout
- output size cap
- working directory fixed
- environment minimized
- exit code handled
- stderr captured safely
- no secret in command-line arguments if visible to process list

---

## 20. Logging, Auditing, and Error Handling

### 20.1 Logging rule

Logs must help investigation without leaking secrets or regulated data.

Do log:

- stable event type
- correlation ID
- request ID
- actor ID or pseudonymous ID where policy allows
- resource ID if not sensitive
- decision outcome
- failure category
- duration

Do not log:

- passwords
- tokens
- authorization headers
- cookies
- full request bodies by default
- private keys
- full PII unless policy allows
- database connection strings with credentials
- stack traces returned to clients

### 20.2 Audit rule

Security-sensitive business events MUST produce audit records when required by system policy.

Examples:

- login success/failure
- privilege changes
- case approval/rejection
- data export
- file download
- security configuration changes
- admin impersonation
- manual override

Audit records SHOULD be tamper-aware and SHOULD include:

- actor
- action
- resource
- timestamp
- outcome
- reason/category
- source system
- correlation ID

### 20.3 Error response rule

Client-facing errors MUST be safe.

Do not expose:

- SQL statements
- stack traces
- internal paths
- hostnames
- credentials
- implementation class names
- policy internals that help attackers

Use generic client message + internal correlation ID.

## 21. Dependency and Supply Chain Security

### 21.1 Core rule

Dependencies are executable trust.

The agent MUST NOT add dependencies casually.

Before adding a dependency, the agent MUST justify:

- purpose
- maintenance status
- license compatibility
- known vulnerabilities
- transitive dependency impact
- alternative in JDK/framework
- update path

### 21.2 Forbidden

- abandoned libraries for security-sensitive features
- random GitHub snippets copied into production
- dependencies from untrusted repositories
- snapshot dependencies in production
- disabling dependency verification
- ignoring CVE findings without documented exception

### 21.3 Required build controls

Projects SHOULD enforce:

- dependency lockfile or version catalog
- vulnerability scanning
- SBOM generation where required
- checksum/signature verification where available
- reproducible build practices where feasible
- no secret in build logs

## 22. Java Platform Security Rules

### 22.1 Security Manager

The Java Security Manager MUST NOT be used as the primary sandbox or isolation mechanism.

Java 17 deprecated the Security Manager for removal. New code MUST assume it is not a viable security boundary.

Use:

- OS/container sandboxing
- process isolation
- network policies
- least-privileged credentials
- JVM module encapsulation
- explicit authorization checks

### 22.2 Reflection and method handles

Reflection that bypasses encapsulation is restricted.

Forbidden by default:

- `setAccessible(true)` for application security boundaries
- reflection-based access to private secrets
- loading classes by user-controlled class names
- invoking methods selected by untrusted input

Allowed only for framework integration with clear tests and review.

### 22.3 Native code and JNI

Native code/JNI is restricted.

The agent MUST NOT add JNI/native library usage unless explicitly requested and security-reviewed.

Risks include:

- memory corruption
- privilege escalation
- library path hijacking
- platform-specific behavior
- bypassing JVM safety

### 22.4 Unsafe and internal APIs

Use of `sun.misc.Unsafe`, `jdk.internal.*`, or `--add-opens` is forbidden by default.

Allowed only for low-level infrastructure code with explicit approval.

## 23. Concurrency Security

### 23.1 Thread-local context

Security context stored in `ThreadLocal` MUST be cleared reliably.

This is especially important with:

- servlet containers
- thread pools
- async execution
- scheduled tasks
- virtual threads migration

Required:

```java
try {
    SecurityContext.set(context);
    action.run();
} finally {
    SecurityContext.clear();
}
```

### 23.2 Async propagation

Async tasks MUST explicitly propagate only necessary security context.

Do not implicitly copy full request context containing secrets.

### 23.3 TOCTOU

Security decisions must avoid time-of-check/time-of-use races.

Example risks:

- checking authorization before resource changes ownership
- validating file path then following symlink later
- checking account status before executing transaction in separate boundary

Mitigate with:

- database constraints
- transactional checks
- locks/version checks
- immutable authorization snapshot where required

---

## 24. Caching Security

### 24.1 Core rule

Caches can leak data across users or tenants.

Cache keys MUST include all authorization-relevant dimensions.

Examples:

- tenant ID
- user role/scope if output differs
- locale if output is security-sensitive
- permission version if policy changes

### 24.2 Forbidden

```java
// BAD: cached by case ID only, but response depends on actor permission
@Cacheable("caseDetails")
CaseDetails getCaseDetails(String caseId) { ... }
```

### 24.3 Required

Do not cache:

- raw tokens
- passwords
- authorization headers
- highly sensitive PII unless encrypted and justified
- per-user authorization results without invalidation plan

---

## 25. API Security

### 25.1 REST endpoint rule

Every endpoint MUST define:

- authentication requirement
- authorization rule
- input DTO
- validation rule
- error response shape
- rate/idempotency considerations
- audit/logging behavior

### 25.2 State-changing endpoints

State-changing endpoints MUST use appropriate HTTP methods and must not be implemented as `GET`.

Forbidden:

```java
@GET
@Path("/cases/{id}/approve")
public Response approve(...) { ... }
```

### 25.3 Mass assignment

Request DTOs MUST NOT bind directly to domain entities or persistence entities.

Forbidden:

```java
public Response update(UserEntity user) { ... }
```

Required:

```java
public Response update(UpdateUserRequest request) { ... }
```

Only allow fields that the caller is allowed to change.

### 25.4 Rate limiting and abuse controls

Security-sensitive endpoints SHOULD define abuse controls:

- login
- password reset
- OTP verification
- token exchange
- data export
- expensive search
- file upload
- webhook receiver

---

## 26. Database Security

### 26.1 Query safety

All SQL/JPQL/native queries MUST use bind parameters for runtime values.

Dynamic identifiers MUST use allow-list mapping.

### 26.2 Least privilege

Database users SHOULD be least-privileged by application role.

Application code MUST NOT require schema-owner privileges in normal runtime.

### 26.3 Sensitive data

Sensitive columns SHOULD define:

- minimization requirement
- encryption/tokenization/masking policy
- access policy
- retention policy
- audit requirement

### 26.4 Migration safety

Migrations MUST NOT:

- print sensitive data
- create overly broad grants
- leave default passwords
- disable constraints without re-enabling
- drop audit/security columns without review

---

## 27. Framework-Specific Rules

### 27.1 Spring Security

If Spring Security is used, the agent MUST NOT:

- disable CSRF blindly
- use `permitAll()` broadly
- rely only on URL-level checks for object-level security
- expose actuator endpoints without protection
- store passwords with `{noop}` outside tests
- bypass method security for sensitive service methods

### 27.2 Jakarta/JAX-RS

If JAX-RS/Jakarta REST is used, the agent MUST:

- keep resources thin
- perform authn/authz via filters/interceptors/services
- avoid returning raw exceptions
- use DTOs
- not expose entities directly
- map errors safely

### 27.3 ORM/JPA

If JPA is used, the agent MUST:

- avoid exposing lazy-loaded entities to serialization
- avoid mass assignment into entities
- include tenant/scope in queries
- use optimistic locking for sensitive state transitions where relevant
- avoid native queries unless justified and parameterized

## 28. Testing Requirements

Security-sensitive code MUST include tests that prove denial paths.

### 28.1 Required security tests

| Area            | Required tests                                                |
| --------------- | ------------------------------------------------------------- |
| Authorization   | allowed actor, denied actor, wrong tenant, wrong state        |
| Validation      | invalid shape, boundary length, unexpected enum, null/missing |
| SQL safety      | injection payload does not alter query behavior               |
| File path       | traversal attempts rejected                                   |
| SSRF            | private/loopback/metadata URL rejected                        |
| Crypto          | wrong key/tag fails; nonce handling tested where possible     |
| Token           | expired/wrong issuer/wrong audience rejected                  |
| Logging         | secrets not present in captured logs                          |
| Error handling  | stack trace/internal info not returned                        |
| Deserialization | disallowed class/payload rejected                             |
| Rate limit      | abuse threshold behavior where applicable                     |

### 28.2 Test names

Security tests MUST name the risk.

Good:

```java
@Test
void shouldRejectCaseApprovalWhenActorBelongsToDifferentTenant() { ... }

@Test
void shouldRejectPathTraversalOutsideUploadRoot() { ... }

@Test
void shouldNotLogAuthorizationHeaderWhenRequestFails() { ... }
```

Bad:

```java
@Test
void testError() { ... }
```

## 29. Review Checklist

A change MUST NOT be approved until these questions are answered.

### 29.1 Trust boundary

- [ ] Have all external inputs been identified?
- [ ] Are inputs validated before use?
- [ ] Are canonicalization rules clear?
- [ ] Are size/range limits enforced?

### 29.2 Authorization

- [ ] Is authentication separate from authorization?
- [ ] Is object-level authorization enforced?
- [ ] Is tenant/scope isolation enforced in query/data access?
- [ ] Does the code fail closed?

### 29.3 Injection

- [ ] Are executable contexts parameterized/escaped correctly?
- [ ] Are dynamic identifiers allow-listed?
- [ ] Are OS commands avoided or safely constructed?

### 29.4 Crypto/secrets

- [ ] No custom crypto?
- [ ] No weak algorithms?
- [ ] No hardcoded secrets?
- [ ] No secret logging?
- [ ] Key/secret rotation considered?

### 29.5 Network/file

- [ ] Timeouts configured?
- [ ] TLS validation intact?
- [ ] SSRF controls present for user-influenced URL?
- [ ] Path traversal controls present?
- [ ] Upload/download authorization checked?

### 29.6 Logging/errors

- [ ] Client errors safe?
- [ ] Logs contain useful context but no secrets?
- [ ] Audit event added where required?

### 29.7 Tests

- [ ] Denial path tests exist?
- [ ] Abuse cases tested?
- [ ] Security regression tests named clearly?

## 31. Common LLM Security Anti-Patterns

### 31.1 “Make it work first, secure later”

Forbidden.

Security must be part of the first implementation.

### 31.2 “Debug log everything”

Forbidden for secrets and sensitive payloads.

### 31.3 “Trust internal service”

Forbidden assumption.

Internal services can be compromised, misconfigured, or called out of context.

### 31.4 “Use admin role for service account”

Forbidden by default.

Service accounts must be least-privileged.

### 31.5 “Catch exception and continue”

Forbidden for auth, crypto, validation, and persistence integrity failures.

Security failures must fail closed.

### 31.6 “Base64 means encrypted”

Forbidden misconception.

Base64 is encoding, not encryption.

### 31.7 “Hash means safe”

Forbidden misconception.

Fast hashes are not password storage.

### 31.8 “HTTPS means SSRF-safe”

Forbidden misconception.

HTTPS does not prevent access to internal hosts.

## 32. Minimum Secure Defaults

| Area          | Default                                  |
| ------------- | ---------------------------------------- |
| Input         | reject unknown/invalid                   |
| Authorization | deny unless explicitly allowed           |
| Error         | generic client message + correlation ID  |
| Logs          | structured, masked, no secrets           |
| SQL           | parameterized                            |
| File path     | base-dir constrained                     |
| URL fetch     | allow-listed + timeout + SSRF checks     |
| TLS           | validation enabled                       |
| Crypto        | approved library/JDK API only            |
| Password      | password KDF, not fast hash              |
| Serialization | native Java deserialization forbidden    |
| Dependencies  | justify before adding                    |
| Tests         | include denial/security regression tests |

## 34. Enforcement Summary

The LLM agent MUST treat this file as mandatory.

A generated Java change is non-compliant if it:

- crosses a trust boundary without validation
- performs sensitive action without explicit authorization
- constructs executable contexts with string concatenation
- disables TLS validation
- uses weak/custom crypto
- hardcodes/logs secrets
- deserializes untrusted Java objects
- performs file/network access without traversal/SSRF controls
- returns internal errors to clients
- lacks security denial-path tests for security-sensitive behavior

When uncertain, the agent MUST choose the safer design, reduce authority, reject input, and ask for explicit human approval only for residual risk—not silently weaken security.
