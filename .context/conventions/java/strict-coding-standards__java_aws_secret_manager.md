# Strict Coding Standards: Java + AWS Secrets Manager

> Status: **strict / enforceable**  
> Audience: LLM code agents, reviewers, maintainers  
> Applies with: `strict-coding-standards__java*.md`, `strict-coding-standards__java_security.md`, `strict-coding-standards__java_network.md`, `strict-coding-standards__java_telemetry.md`, `strict-coding-standards__gradle.md`, `strict-coding-standards__maven.md`

## 0. Purpose

This document is not a tutorial. It is a coding contract. An LLM implementing Java code that touches AWS must follow these rules unless the repository contains a more specific, reviewed architecture decision record.

The default posture is:

- safe by default;
- least privilege by default;
- bounded resource usage by default;
- observable failure by default;
- no hidden network, retry, credential, region, or serialization behavior.

## 1. Global LLM Contract

Before changing code, the LLM MUST identify:

1. AWS service being used.
2. AWS SDK generation and version-management mechanism.
3. region source.
4. credential source.
5. client lifecycle owner.
6. timeout and retry policy.
7. idempotency behavior.
8. data classification of payloads.
9. IAM permissions required.
10. telemetry emitted.
11. tests that prove failure behavior.

If any item is unknown, the LLM MUST mark the implementation as incomplete and add a TODO requiring human review. It MUST NOT silently invent account IDs, ARNs, regions, bucket names, topic names, queue names, KMS keys, secret names, or IAM policies.


## 2. Scope

This file defines strict rules for Java code that retrieves, caches, rotates, validates, logs around, or uses secrets stored in AWS Secrets Manager.

It applies to:

- `SecretsManagerClient`;
- Secrets Manager caching library;
- secret retrieval adapters;
- database credential providers;
- API key/token retrieval;
- rotation-aware integrations.

## 3. Secret Classification

### MUST

- Treat every secret value as highly sensitive.
- Define the secret purpose: database credential, API token, private key, webhook secret, signing key, or configuration containing credentials.
- Define whether the secret is binary or string.
- Define whether the secret is static, rotated, externally managed, or generated by AWS rotation.
- Define the consumer scope and IAM role allowed to read it.

### FORBIDDEN

- Logging secret values.
- Including secret values in exceptions, test snapshots, metrics, traces, or health endpoints.
- Copying secret values into application configuration files.
- Storing secrets in DTOs returned to controllers or public APIs.

## 4. Client and Dependency Rules

### MUST

- Use AWS SDK for Java 2.x for new code.
- Reuse `SecretsManagerClient`.
- Configure timeouts and retry budget.
- Use region and credentials from runtime environment/config.
- Wrap AWS access behind a small adapter such as `SecretsProvider` or `DatabaseCredentialsProvider`.

### FORBIDDEN

- Calling Secrets Manager directly from domain logic.
- Creating client per lookup.
- Hardcoding secret ARNs/names in reusable libraries.

## 5. Retrieval Rules

### MUST

- Retrieve only the secret needed by the component.
- Use exact secret ID/ARN from validated configuration.
- Handle `ResourceNotFound`, `AccessDenied`, throttling, timeout, and decryption failures distinctly.
- Avoid exposing secret existence to unauthorized users.
- Validate secret shape after retrieval.
- Avoid repeated remote calls on hot paths.

### FORBIDDEN

- Listing all secrets to find one by name.
- Using wildcard secret access in runtime roles.
- Returning fallback dummy credentials on retrieval failure.

## 6. Caching Rules

### MUST

- Use client-side caching for frequently accessed secrets.
- Define cache TTL, refresh behavior, and failure behavior.
- Ensure cache invalidation works with rotation policy.
- Avoid caching secrets longer than security policy permits.
- Treat in-memory cache as sensitive memory.

### FORBIDDEN

- Fetching secrets on every request.
- Writing cache contents to disk.
- Exposing cache content through debug endpoints.

## 7. Rotation Rules

### MUST

- Design consumers to survive secret rotation.
- Recreate downstream connections when credential rotation requires it.
- For database credentials, ensure pool refresh behavior is documented.
- Use version stage/rotation metadata only with an explicit compatibility design.
- Test rotation failure and stale credential behavior.

### FORBIDDEN

- Assuming a secret never changes.
- Restart-only rotation behavior unless explicitly accepted operationally.
- Reading only at startup for credentials that must rotate at runtime.

## 8. Secret Value Parsing

### MUST

- Parse JSON secrets into explicit DTOs.
- Validate required fields and reject unknown critical ambiguity.
- Avoid generic `Map<String, Object>` escaping into business logic.
- Avoid `String` spreading across layers; wrap sensitive values in small types where possible.

### FORBIDDEN

- Printing secret JSON during validation failure.
- Treating malformed secret as empty/default config.
- Using Java native deserialization for secret payload.

## 9. IAM and KMS Rules

### MUST

- Grant `secretsmanager:GetSecretValue` only for required secrets.
- Grant KMS decrypt only when custom CMK requires it.
- Use resource policies only with explicit cross-account design.
- Separate secret administration/rotation permissions from application read permissions.

### FORBIDDEN

- Runtime role with `secretsmanager:*`.
- Cross-account secret access without explicit trust boundary review.

## 10. Error and Health Rules

### MUST

- Startup may fail fast if secret is mandatory.
- Health checks must not retrieve and print secret values.
- Runtime failures must be classified: configuration error, permission error, throttling, AWS outage, stale cache.
- Metrics may include secret logical name or category, not value.

### FORBIDDEN

- Health endpoint that returns secret contents.
- Logging full ARN if internal policy classifies account/resource data as sensitive.

## 11. Testing Rules

### MUST

- Test retrieval success, not found, denied, malformed, stale cache, rotation, and timeout.
- Test that logs do not include secret values.
- Test JSON shape validation.
- Use fake client/mock/LocalStack/test account only.

### FORBIDDEN

- Real production secrets in tests.
- Test fixtures with real-looking active tokens.

## 12. Reviewer Checklist

- [ ] no secret logged.
- [ ] no hardcoded secret value.
- [ ] secret name/ARN is config.
- [ ] IAM least privilege.
- [ ] caching policy defined.
- [ ] rotation behavior defined.
- [ ] failure behavior safe.
- [ ] tests check redaction and malformed secret.

## 13. LLM Prompt Contract

For Secrets Manager code, the LLM MUST state:

- secret identifier source;
- secret value schema;
- cache TTL and refresh behavior;
- rotation behavior;
- IAM actions/resources;
- failure behavior;
- redaction tests.
