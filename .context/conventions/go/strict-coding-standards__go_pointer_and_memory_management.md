# Strict Coding Standards — Go Pointer and Memory Management

> **File:** `strict-coding-standards__go_pointer_and_memory_management.md`  
> **Scope:** Go source code written, modified, reviewed, or generated by an LLM/code agent.  
> **Status:** Mandatory merge gate.  
> **Audience:** LLM agents, software engineers, reviewers, maintainers.  
> **Primary concern:** pointer correctness, ownership, aliasing, escape behavior, garbage collection pressure, resource lifetime, unsafe usage, and data-race prevention.

---

## 1. Purpose

This document defines mandatory standards for pointer and memory management in Go. Go has garbage collection, but Go programs can still have memory leaks, aliasing bugs, data races, retained buffers, goroutine leaks, accidental heap allocation, unsafe lifetime violations, and broken ownership contracts.

The LLM/code agent MUST treat memory management as an explicit design concern, not something delegated blindly to the runtime.

This document exists to prevent:

1. accidental shared mutable state,
2. nil pointer panics,
3. interface typed-nil bugs,
4. unsafe pointer lifetime violations,
5. slice/map alias leaks,
6. hidden heap pressure,
7. excessive allocation in hot paths,
8. goroutine and timer leaks,
9. finalizer misuse,
10. incorrect atomic/synchronization logic.

---

## 2. Normative language

- **MUST**: mandatory; violation blocks merge.
- **MUST NOT**: prohibited; violation blocks merge.
- **SHOULD**: expected default; deviation requires written justification.
- **MAY**: allowed when justified.
- **LLM MUST**: mandatory behavior for code-generating agents.

---

## 3. Core model

The agent MUST reason using this model:

- A Go variable holds a value.
- A pointer holds the address of a value.
- Slices, maps, channels, functions, and interfaces are values with reference-like behavior.
- Copying a slice copies the slice header, not the backing array.
- Copying a map copies the map header, not the underlying hash table.
- Copying an interface copies the dynamic type/value pair.
- Copying structs may copy locks, atomics, slices, maps, and resource handles incorrectly.
- Escape analysis decides whether values can live on the stack or must move to the heap.
- Garbage collection reclaims unreachable heap objects, but it does not close files, cancel goroutines, stop tickers, or release external resources deterministically.

The agent MUST NOT assume garbage collection removes the need for explicit ownership and lifecycle design.

## 4. Global LLM behavior rules

### 4.1 Every pointer MUST have a reason

The agent MUST NOT use pointers by habit. Each pointer must be justified by at least one of:

- mutation of caller-owned value is required;
- value is large enough that copying is materially expensive;
- nil is a meaningful state;
- method must satisfy interface requiring pointer receiver;
- synchronization primitive must not be copied;
- resource handle identity matters;
- shared object identity is part of the domain;
- optional field must distinguish absent from zero;
- recursive/self-referential structure is required.

Bad:

```go
type User struct {
    ID   *string
    Name *string
}
```

Better:

```go
type User struct {
    ID   UserID
    Name UserName
}
```

Pointers are not a general-purpose replacement for validation.

### 4.2 Every retained reference MUST define ownership

If a function stores a pointer, slice, map, channel, function, or interface value beyond the call, the agent MUST decide whether it is:

- owned by the callee,
- borrowed temporarily,
- shared read-only,
- shared mutable with synchronization,
- transferred from caller to callee.

If ownership is not documented, the agent MUST copy or redesign.

### 4.3 No hidden global mutable memory

The agent MUST NOT introduce package-level mutable maps, slices, buffers, caches, or pointers unless:

- protected by synchronization,
- bounded in size,
- initialized safely,
- test-isolated,
- documented as process-wide state.

Bad:

```go
var users = map[string]User{}
```

Better:

```go
type UserCache struct {
    mu    sync.RWMutex
    users map[UserID]User
}
```

### 4.4 Memory optimization MUST be measured

The agent MUST NOT introduce unsafe code, object pools, manual buffer reuse, or complex pointer tricks without benchmark/profiling evidence.

Correct optimization order:

1. write clear safe code;
2. add tests;
3. benchmark representative workload;
4. profile allocations and CPU;
5. optimize the measured bottleneck;
6. re-run tests, race tests, and benchmarks;
7. document trade-offs.

## 5. Pointer vs value semantics

### 5.1 Prefer value semantics for small immutable data

The agent SHOULD use values for small immutable or immutable-by-convention types.

Good:

```go
type Point struct {
    X, Y int
}

func Distance(a, b Point) float64
```

Good:

```go
type Money struct {
    currency CurrencyCode
    cents    int64
}
```

Value semantics reduce aliasing risk.

### 5.2 Use pointers for mutable aggregates

Pointer receivers are appropriate when methods mutate the receiver.

```go
type Counter struct {
    n int64
}

func (c *Counter) Inc() {
    c.n++
}
```

The agent MUST NOT mutate value receivers expecting the caller to observe the mutation.

Bad:

```go
type Counter struct{ n int }

func (c Counter) Inc() {
    c.n++ // mutates copy
}
```

### 5.3 Receiver choice MUST be consistent

For a type, the agent SHOULD avoid mixing pointer and value receivers unless there is a clear reason.

Use pointer receivers when:

- any method mutates receiver;
- receiver contains `sync.Mutex`, `sync.Once`, atomics, or other non-copyable fields;
- receiver is large;
- method must avoid copying internal slices/maps/resources;
- identity matters.

Use value receivers when:

- type is small;
- type is immutable;
- copy is safe;
- methods are pure accessors or transformations returning new values.

### 5.4 Pointers to interfaces are prohibited

The agent MUST NOT use `*interface{}` or `*SomeInterface` except for rare reflection/serialization framework integration.

Bad:

```go
func Handle(s *Store) error // Store is an interface
```

Better:

```go
func Handle(s Store) error
```

An interface value already carries dynamic type and dynamic value.

### 5.5 Pointer-to-slice and pointer-to-map MUST be justified

The agent MUST NOT use `*[]T` or `*map[K]V` for ordinary mutation. Slices and maps already have reference-like behavior.

Bad:

```go
func Add(items *[]Item, item Item)
```

Better:

```go
func Add(items []Item, item Item) []Item {
    return append(items, item)
}
```

Pointer-to-slice MAY be used when the function must distinguish nil pointer from empty slice pointer or must replace caller's slice in a specific API pattern, but this requires comment justification.

## 6. Nil handling standards

### 6.1 Nil MUST be a designed state

The agent MUST NOT allow nil accidentally. Nil is allowed only when:

- optional value is intentional;
- lazy initialization is safe;
- zero value usability requires nil internal fields;
- interface implementation convention expects nil;
- external data can explicitly be null.

For every exported pointer field, the agent MUST document whether nil is allowed.

### 6.2 Constructors MUST prevent invalid nil dependencies

Bad:

```go
type Service struct {
    repo Repository
}

func NewService(repo Repository) *Service {
    return &Service{repo: repo}
}
```

Better:

```go
func NewService(repo Repository) (*Service, error) {
    if repo == nil {
        return nil, ErrNilRepository
    }
    return &Service{repo: repo}, nil
}
```

For internal constructors where nil is programmer error, panic MAY be acceptable only if project conventions allow it. Public constructors SHOULD return errors.

### 6.3 Nil receiver methods MUST be deliberate

The agent MUST NOT rely on nil receivers unless the behavior is explicitly part of the API.

Allowed:

```go
func (n *Node) IsEmpty() bool {
    return n == nil
}
```

Forbidden accidental pattern:

```go
func (s *Service) Handle() error {
    return s.repo.Save() // panic when s is nil
}
```

### 6.4 Interface typed-nil MUST be avoided

The agent MUST NOT return typed nil pointers as interface values.

Bad:

```go
func BuildError() error {
    var e *ValidationError = nil
    return e
}
```

Better:

```go
func BuildError() error {
    return nil
}
```

When returning interfaces, ensure nil means real nil interface.

## 7. Ownership and aliasing standards

### 7.1 Borrowed data MUST not escape

If a function receives borrowed data, it MUST NOT store it beyond the function call.

Bad:

```go
func (s *Store) SetPayload(payload []byte) {
    s.payload = payload
}
```

Better:

```go
func (s *Store) SetPayload(payload []byte) {
    s.payload = bytes.Clone(payload)
}
```

### 7.2 Ownership transfer MUST be explicit

If the function takes ownership, document it.

```go
// NewBuffer takes ownership of b. The caller must not mutate b after this call.
func NewBuffer(b []byte) *Buffer {
    return &Buffer{b: b}
}
```

For public APIs, prefer copying unless performance demands ownership transfer.

### 7.3 Shared mutable state MUST be synchronized

Any pointer/slice/map shared across goroutines MUST have a synchronization story.

Allowed synchronization:

- `sync.Mutex` / `sync.RWMutex`,
- channel ownership,
- immutable copy-on-write snapshots,
- `sync/atomic` for narrow scalar state,
- `sync.Map` for appropriate specialized cases.

The agent MUST NOT share mutable state with comments like `// should be safe`.

### 7.4 Immutable snapshots are preferred for read-heavy state

For configuration, routing tables, permission maps, feature flags, and policy snapshots, the agent SHOULD prefer immutable snapshots swapped atomically.

```go
type PolicyStore struct {
    current atomic.Pointer[PolicySnapshot]
}

func (s *PolicyStore) Current() *PolicySnapshot {
    return s.current.Load()
}
```

The snapshot itself MUST be immutable-by-convention and must not expose mutable internals.

## 8. Slice memory standards

### 8.1 Slices MUST be assumed to alias

The agent MUST remember that a slice header references a backing array. Two slices may view overlapping memory.

```go
a := []int{1, 2, 3, 4}
b := a[:2]
c := a[1:3]
```

Mutating `b` may affect `a` and `c`.

### 8.2 Stored slices MUST be cloned

Any slice stored in a struct, cache, event, command, or goroutine closure MUST be cloned unless ownership transfer is explicit.

```go
type Event struct {
    payload []byte
}

func NewEvent(payload []byte) Event {
    return Event{payload: bytes.Clone(payload)}
}
```

### 8.3 Returned slices MUST be cloned or documented as borrowed

Bad:

```go
func (e Event) Payload() []byte {
    return e.payload
}
```

Better:

```go
func (e Event) Payload() []byte {
    return bytes.Clone(e.payload)
}
```

### 8.4 Large-buffer retention MUST be prevented

The agent MUST copy small long-lived subslices from large buffers.

Bad:

```go
token := requestBody[start:end]
user.Token = token // retains entire body
```

Better:

```go
token := bytes.Clone(requestBody[start:end])
user.Token = token
```

### 8.5 Capacity must be capped before append when necessary

When passing a subslice to code that may append, the agent MUST cap capacity if append must not mutate the original backing array.

```go
window := data[start:end:end]
```

### 8.6 Clearing references after delete may be required

When deleting elements from long-lived slices containing pointers or large objects, the agent MUST clear obsolete references so the GC can reclaim them.

```go
copy(items[i:], items[i+1:])
var zero *Item
items[len(items)-1] = zero
items = items[:len(items)-1]
```

For Go versions with `slices.Delete`, the agent MUST understand whether obsolete references are cleared by the library version in use and still write tests or review memory behavior when objects are large.

## 9. Map memory standards

### 9.1 Map values containing pointers require ownership review

When storing pointer values in maps, the agent MUST define who owns the pointed object.

```go
map[UserID]*User
```

This is shared mutable identity. It requires synchronization or immutability.

Prefer value storage when feasible:

```go
map[UserID]User
```

### 9.2 Maps MUST not be exposed directly

Maps inside structs MUST not be returned directly if callers could mutate internal state.

```go
func (r Registry) Snapshot() map[string]Handler {
    out := make(map[string]Handler, len(r.handlers))
    for k, v := range r.handlers {
        out[k] = v
    }
    return out
}
```

### 9.3 Long-lived maps MUST have growth and eviction policy

The agent MUST NOT introduce unbounded maps for caches, deduplication, idempotency, sessions, metrics labels, or request tracking.

Required design fields:

- max size or TTL,
- eviction rule,
- cleanup lifecycle,
- metrics,
- concurrency protection,
- test for cleanup behavior.

## 10. Struct copy safety

### 10.1 Non-copyable structs MUST not be copied after first use

The agent MUST NOT copy structs containing:

- `sync.Mutex`, `sync.RWMutex`, `sync.Once`, `sync.WaitGroup`, `sync.Cond`,
- typed atomics,
- file descriptors,
- network connections,
- database transactions,
- buffers with ownership assumptions,
- slices/maps/channels intended as unique internal state.

Bad:

```go
type Cache struct {
    mu sync.Mutex
    m  map[string]string
}

func (c Cache) Get(k string) string { // copies mutex
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.m[k]
}
```

Better:

```go
func (c *Cache) Get(k string) string {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.m[k]
}
```

### 10.2 Value receiver on structs with locks is prohibited

Any method on a struct containing synchronization primitives MUST use pointer receiver.

### 10.3 Copy methods MUST deep-copy mutable internals

If a type exposes `Clone`, `Copy`, or `Snapshot`, it MUST deep-copy mutable slices/maps/pointers as required.

```go
func (c Config) Clone() Config {
    c.Headers = maps.Clone(c.Headers)
    c.Routes = slices.Clone(c.Routes)
    return c
}
```

## 11. Escape analysis and allocation standards

### 11.1 The agent MUST not force heap allocation unnecessarily

Common heap-forcing patterns to avoid in hot paths:

- returning pointer to large temporary when value would work;
- capturing loop variables or large values in closures;
- boxing concrete values into interfaces unnecessarily;
- using `fmt.Sprintf` in tight loops;
- converting `[]byte` to `string` repeatedly;
- appending without preallocation;
- using reflection for ordinary operations.

### 11.2 Returning pointers is allowed but must be intentional

Returning `&value` is safe in Go because the compiler will allocate appropriately. The issue is not safety but heap pressure and API semantics.

Allowed:

```go
func NewService(repo Repository) *Service {
    return &Service{repo: repo}
}
```

Questionable:

```go
func NewPoint(x, y int) *Point {
    return &Point{X: x, Y: y}
}
```

If `Point` is small and immutable, return value instead.

### 11.3 Hot paths MUST be benchmarked with allocation reporting

For hot paths, benchmarks MUST report allocations.

```bash
go test -bench=. -benchmem ./...
```

The agent MUST NOT claim allocation improvement without benchmark evidence.

### 11.4 Preallocation SHOULD be used when size is known

```go
items := make([]Item, 0, len(input))
```

Preallocation MUST NOT be exaggerated to unbounded or attacker-controlled sizes without validation.

### 11.5 Avoid retaining large object graphs through closures

The agent MUST avoid goroutine closures that capture large request objects, contexts with large values, buffers, or service structs unnecessarily.

Bad:

```go
go func() {
    process(req) // captures entire request
}()
```

Better:

```go
userID := req.UserID
go func(userID UserID) {
    process(userID)
}(userID)
```

## 12. Garbage collection standards

### 12.1 GC is not resource lifecycle management

The agent MUST explicitly close or release:

- files,
- network connections,
- response bodies,
- database rows,
- database transactions,
- tickers/timers,
- subprocess handles,
- temporary directories,
- external buffers or handles.

Bad:

```go
resp, err := http.Get(url)
if err != nil { return err }
body, err := io.ReadAll(resp.Body)
```

Better:

```go
resp, err := client.Get(url)
if err != nil {
    return err
}
defer resp.Body.Close()
body, err := io.ReadAll(resp.Body)
```

### 12.2 Long-lived objects MUST not accidentally retain short-lived data

Review long-lived structs, caches, goroutines, closures, and global variables for references to:

- request bodies,
- large byte slices,
- decoded payloads,
- temporary buffers,
- context values,
- transaction-scoped objects.

### 12.3 Memory budget must be explicit for large workloads

For batch processing, streaming, file parsing, large JSON, image/binary processing, or ETL, the agent MUST define:

- maximum in-memory buffer size,
- streaming strategy,
- backpressure behavior,
- spill-to-disk policy if needed,
- per-worker memory budget,
- maximum concurrency,
- benchmark input size.

### 12.4 Do not tune GC blindly

The agent MUST NOT change `GOGC`, `GOMEMLIMIT`, or runtime memory settings without measurement and rollback strategy.

If memory tuning is proposed, include:

- baseline heap profile,
- allocation profile,
- target workload,
- expected trade-off between CPU and memory,
- production observability metrics.

## 13. Resource lifetime standards

### 13.1 Ownership of closers MUST be clear

If a function receives `io.Closer`, `*os.File`, `io.ReadCloser`, `*sql.Rows`, or similar resource, the agent MUST define who closes it.

```go
// ParseAndClose reads all data from r and closes it before returning.
func ParseAndClose(r io.ReadCloser) (Result, error) {
    defer r.Close()
    ...
}
```

If the function does not own it, do not close it.

### 13.2 Timers and tickers MUST be stopped

Bad:

```go
ticker := time.NewTicker(time.Second)
for range ticker.C { ... }
```

Better:

```go
ticker := time.NewTicker(time.Second)
defer ticker.Stop()
```

### 13.3 Context cancellation MUST be called

When creating cancellable contexts, the cancel function MUST be called.

```go
ctx, cancel := context.WithTimeout(parent, 5*time.Second)
defer cancel()
```

### 13.4 Goroutine lifecycle MUST be bounded

The agent MUST NOT start goroutines without a clear stop condition.

Every goroutine MUST have at least one:

- context cancellation,
- input channel closure,
- parent errgroup cancellation,
- process lifetime justification,
- explicit shutdown method.

## 14. sync.Pool standards

### 14.1 `sync.Pool` MUST not be used as a cache

`sync.Pool` is for temporary objects that may be dropped at any time. The agent MUST NOT use it for correctness, identity, or durable caching.

Allowed:

- temporary buffers,
- encoders/decoders with reset,
- scratch objects in high-throughput paths.

### 14.2 Pooled objects MUST be reset before reuse

```go
buf := pool.Get().(*bytes.Buffer)
buf.Reset()
defer pool.Put(buf)
```

The agent MUST NOT put objects containing sensitive data back into a pool without clearing them.

### 14.3 Do not pool small cheap objects by default

Pooling can increase complexity and memory retention. The agent MUST benchmark before using `sync.Pool`.

## 15. Concurrency memory safety

### 15.1 Data races are prohibited

The agent MUST ensure there is no concurrent unsynchronized read/write or write/write access to the same memory location.

The agent MUST run race tests for concurrency changes:

```bash
go test -race ./...
```

### 15.2 Synchronization MUST establish ownership or happens-before relation

Correct options:

- mutex lock/unlock,
- channel send/receive where appropriate,
- atomic operations,
- wait group for waiting, not for memory ownership alone,
- once initialization,
- condition variables where justified.

The agent MUST NOT assume goroutine scheduling order provides memory visibility.

### 15.3 Atomics MUST be used only for narrow cases

Allowed atomic use:

- counters,
- flags,
- pointers to immutable snapshots,
- sequence numbers,
- lock-free fast path with clear invariants.

The agent MUST NOT build complex lock-free algorithms unless explicitly required and heavily tested.

Bad:

```go
var ready atomic.Bool
var config map[string]string
```

Setting `ready` atomically does not make unsynchronized map mutation safe.

Better:

```go
type ConfigSnapshot struct {
    values map[string]string // immutable after construction
}

var current atomic.Pointer[ConfigSnapshot]
```

### 15.4 Channels do not make pointed data safe automatically

Sending a pointer through a channel transfers access only if the program stops using it elsewhere.

Bad:

```go
job := &Job{}
ch <- job
job.Status = "changed" // concurrent alias risk
```

The agent MUST document ownership transfer or clone the data.

## 16. Unsafe standards

### 16.1 Unsafe is prohibited by default

The agent MUST NOT generate `unsafe` code unless explicitly requested or already present in a low-level package where safe alternatives are insufficient.

### 16.2 Unsafe code MUST be isolated

Unsafe code MUST be placed in the smallest possible package/file and surrounded by safe API boundaries.

Required file comment:

```go
// Package fastbytes contains carefully reviewed unsafe conversions.
// All functions rely on documented Go unsafe.Pointer rules and are covered by tests.
```

### 16.3 unsafe.Pointer rules MUST be followed exactly

The agent MUST NOT:

- store `uintptr` converted from pointer for later use;
- assume object will remain alive without a typed pointer or `runtime.KeepAlive` where needed;
- mutate string memory;
- construct slices/strings pointing to invalid or shorter-lived memory;
- violate alignment requirements;
- assume layout of non-specified runtime/internal types;
- use unsafe to bypass visibility/invariants.

### 16.4 String/byte zero-copy conversion is prohibited by default

Bad:

```go
func BytesToString(b []byte) string {
    return *(*string)(unsafe.Pointer(&b))
}
```

This can violate immutability and lifetime assumptions.

Use safe conversion unless a reviewed low-level package proves necessity.

### 16.5 `runtime.KeepAlive` MUST be used when required

When interacting with finalizers, syscalls, cgo, or unsafe references where the compiler might consider an object dead before the external use completes, the agent MUST use `runtime.KeepAlive` correctly.

## 17. Finalizer and cleanup standards

### 17.1 Finalizers MUST NOT be primary cleanup

The agent MUST NOT rely on finalizers for timely release of files, sockets, locks, transactions, or critical resources.

Finalizers MAY be used as safety nets for native resources, but explicit `Close` is still required.

### 17.2 Finalizers MUST not access unsynchronized mutable state

Finalizers run asynchronously. Any mutable shared state accessed by a finalizer MUST be synchronized.

### 17.3 Cleanup callbacks MUST not resurrect objects or hide lifecycle bugs

If using runtime cleanup/finalization mechanisms available in the project Go version, the agent MUST document lifetime assumptions and test explicit cleanup behavior separately.

---

## 18. cgo and external memory standards

### 18.1 Go pointer passing rules MUST be respected

When cgo is used, the agent MUST follow Go pointer passing rules. It MUST NOT let C retain pointers to Go memory unless allowed by the cgo rules and runtime guarantees.

### 18.2 External memory MUST have explicit free path

Any memory allocated outside Go heap MUST have:

- owner type,
- `Close` or `Free` method,
- idempotent cleanup,
- tests for double close,
- finalizer only as fallback if used.

### 18.3 Do not expose raw external pointers

Wrap external pointers in types that enforce lifecycle and prevent use-after-free.

---

## 19. Buffer management standards

### 19.1 Buffer reuse MUST not leak data

Before reusing buffers that may contain secrets, tokens, PII, credentials, or regulated data, the agent MUST zero or discard them.

```go
for i := range b {
    b[i] = 0
}
```

### 19.2 Shared buffers MUST not cross goroutine boundaries unsafely

If a buffer is reused by producer after sending to consumer, the consumer MUST receive a copy or ownership transfer must be explicit.

Bad:

```go
for scanner.Scan() {
    b := scanner.Bytes()
    ch <- b // scanner may reuse backing buffer
}
```

Better:

```go
for scanner.Scan() {
    ch <- bytes.Clone(scanner.Bytes())
}
```

### 19.3 Large reads MUST be streamed

The agent MUST NOT use `io.ReadAll` for untrusted or potentially large input unless a strict size limit is applied.

```go
r := http.MaxBytesReader(w, req.Body, maxBodyBytes)
```

For files and network payloads, prefer streaming decoders and bounded buffers.

---

## 20. Context value memory standards

### 20.1 Context MUST not carry large objects

The agent MUST NOT store large payloads, DB handles, service dependencies, loggers with heavy state, or mutable domain objects inside `context.Context`.

Allowed context values:

- request ID,
- trace/span context,
- auth principal reference or small immutable claims,
- tenant ID,
- deadline/cancellation.

### 20.2 Context values MUST not retain request bodies

Do not store full request payloads, decoded documents, or large metadata maps in context. This creates retention and unclear ownership.

## 21. Memory-safe API design

### 21.1 APIs MUST state whether data is copied

For public functions accepting or returning mutable data, comments MUST state ownership if not obvious.

```go
// NewMessage copies payload.
func NewMessage(payload []byte) Message

// NewMessageOwned takes ownership of payload. The caller must not mutate payload after calling.
func NewMessageOwned(payload []byte) Message
```

### 21.2 Immutable types SHOULD not expose mutable internals

If a type claims to be immutable, all slices/maps/pointers exposed from it MUST be copies or read-only abstractions.

### 21.3 Builder types MUST not share mutable state with built values

Bad:

```go
func (b *Builder) Build() Config {
    return Config{headers: b.headers}
}
```

Better:

```go
func (b *Builder) Build() Config {
    return Config{headers: maps.Clone(b.headers)}
}
```

## 22. Memory profiling and diagnostics standards

### 22.1 Allocation-sensitive code MUST include benchmarks

Required for parsers, serializers, stream processors, batch jobs, queues, caches, middleware, and high-QPS handlers:

```bash
go test -bench=. -benchmem ./...
```

### 22.2 Suspected memory leaks MUST be investigated with profiles

The agent SHOULD use:

- heap profile,
- allocation profile,
- goroutine profile,
- block/mutex profiles for contention,
- execution trace for lifecycle/concurrency issues,
- runtime metrics for production observability.

### 22.3 Benchmark claims MUST include workload description

The agent MUST NOT say “faster” or “less memory” without stating:

- input size,
- operation count,
- Go version,
- benchmark command,
- before/after `ns/op`, `B/op`, `allocs/op`,
- relevant CPU/memory environment if meaningful.

## 23. Security-sensitive memory standards

### 23.1 Secrets MUST not be casually copied or logged

The agent MUST avoid unnecessary copies of:

- passwords,
- tokens,
- private keys,
- session IDs,
- PII,
- regulated documents.

The agent MUST NOT log secrets, even in debug logs.

### 23.2 Secret byte slices SHOULD be zeroed after use when practical

For high-sensitivity data held in `[]byte`, zero after use if the data lifecycle is controlled.

The agent MUST understand that strings cannot be reliably zeroed and should avoid storing secrets in strings when zeroization is required.

### 23.3 Do not put secrets in long-lived pools

The agent MUST NOT put buffers containing secrets into `sync.Pool` unless they are cleared first.

---

## 24. LLM code review checklist

Before producing final code, the agent MUST review:

- Does every pointer have a reason?
- Can any pointer be replaced by value semantics?
- Are nil states explicit and tested?
- Are slices/maps cloned at ownership boundaries?
- Are internal mutable fields exposed?
- Could a subslice retain a large buffer?
- Could a map/cache grow forever?
- Are structs containing locks/atomics copied?
- Are goroutines bounded by context/channel/shutdown?
- Are timers/tickers stopped?
- Are resources closed deterministically?
- Is any mutable data shared across goroutines?
- Is the race detector relevant and passing?
- Is `unsafe` absent or formally justified?
- Are allocation claims benchmarked?
- Are secrets copied, logged, pooled, or retained?

If any item is uncertain, the agent MUST choose the safer design.

## 25. Required anti-pattern rejection list

The agent MUST reject or refactor these patterns:

1. Pointer fields for every DTO/domain field without optionality.
2. Pointer receiver on tiny immutable type without reason.
3. Value receiver on type containing lock/atomic/resource.
4. Returning internal slice/map directly.
5. Storing caller-provided slice/map without copy or ownership contract.
6. Concurrent map access without synchronization.
7. Goroutine started without cancellation or shutdown path.
8. `time.Tick` in long-lived code without stopping.
9. `io.ReadAll` on unbounded input.
10. Global mutable cache without bounds and lock.
11. `sync.Pool` used as correctness cache.
12. Unsafe string/byte zero-copy conversion.
13. `uintptr` storing pointer across statements or calls.
14. Finalizer used as primary resource cleanup.
15. Context containing large mutable values.
16. Closure capturing large request object unnecessarily.
17. Atomic flag used to pretend complex state is safe.
18. Returning typed nil as interface.
19. Map/slice shared through channel while sender continues mutating it.
20. Memory tuning without profile evidence.

## 26. Merge gate checklist

A change MUST NOT be merged unless all applicable items pass:

- [ ] Pointer usage is justified.
- [ ] Ownership of pointers/slices/maps/resources is explicit.
- [ ] Nil is designed, documented, and tested.
- [ ] Mutable internals are not leaked.
- [ ] Slice/map aliasing is controlled.
- [ ] Large-buffer retention is prevented.
- [ ] Structs with locks/atomics/resources are not copied.
- [ ] Goroutines, timers, tickers, and contexts have cleanup paths.
- [ ] Resources are closed deterministically.
- [ ] Concurrent memory access is synchronized.
- [ ] Race detector is used where concurrency changes exist.
- [ ] `unsafe` is absent or isolated, documented, and reviewed.
- [ ] cgo/external memory has explicit ownership and free path.
- [ ] Allocation-sensitive paths have benchmark evidence.
- [ ] Security-sensitive memory is not logged, pooled uncleared, or retained unnecessarily.
