# Strict Coding Standards — Go gRPC

Status: Mandatory for all Go gRPC server, client, protobuf, interceptor, and RPC implementation.
Audience: LLM coding agents, reviewers, maintainers, and platform owners.
Scope: `google.golang.org/grpc`, protocol buffers, service contracts, deadlines, metadata, status codes, streaming, interceptors, security, observability, and testing.

This standard is a merge gate. gRPC code defines service contracts and distributed failure semantics; incorrect generated or handwritten code must be rejected.

---

## 1. Source authority

Use these sources as the primary authority when resolving ambiguity:

- gRPC official documentation.
- `google.golang.org/grpc` package documentation.
- gRPC-Go documentation for metadata, status, credentials, interceptors, and health checking.
- Protocol Buffers language guide and Go generated-code documentation.
- Project API, error handling, context, telemetry, authentication, authorization, and security standards.

When this document conflicts with service-specific `.proto` or API governance, the stricter compatibility rule wins.

---

## 2. Non-negotiable gRPC principles

LLM-generated Go gRPC code MUST obey these principles:

1. `.proto` files are public contracts and require compatibility discipline.
2. Every client call must have a deadline or inherit a bounded context.
3. Every server handler must observe `ctx.Done()` for long-running work.
4. Errors must be mapped to canonical gRPC status codes.
5. Metadata is transport metadata, not a dumping ground for business input.
6. Authentication and authorization must be enforced through interceptors and/or application policy, never assumed from client-provided metadata alone.
7. Streaming RPCs must define backpressure, cancellation, max message size, and ordering semantics.
8. Interceptors must not hide method-specific errors or swallow context cancellation.
9. Generated code must not be manually edited.
10. Tests must cover deadlines, cancellation, status mapping, metadata, interceptor order, and streaming failure cases.

## 3. Protobuf contract standards

### 3.1 File and package naming

`.proto` files MUST define:

- stable `package`;
- `option go_package`;
- explicit service names;
- versioning strategy when API is public or cross-service;
- comments for public messages, fields, services, and methods.

Example:

```proto
syntax = "proto3";

package enforcement.case.v1;

option go_package = "example.com/project/gen/enforcement/case/v1;casev1";
```

### 3.2 Field numbering

Rules:

- Never reuse field numbers.
- Never reuse removed field names without governance approval.
- Reserve removed field numbers and names.
- Do not change field type incompatibly.
- Do not change scalar presence semantics casually.
- Do not encode business meaning in field number gaps.

Preferred:

```proto
message Case {
  reserved 7, 9;
  reserved "legacy_status", "old_owner";

  string id = 1;
  string status = 2;
}
```

### 3.3 Message design

Messages MUST be explicit DTO contracts.

Rules:

- Do not expose persistence rows directly as protobuf messages.
- Use clear request/response messages per RPC.
- Avoid catch-all `map<string, string>` for structured business data.
- Avoid `google.protobuf.Any` unless a versioned extension contract exists.
- Use enums for stable finite states, but include unknown/default handling.
- Use `google.protobuf.Timestamp` for instants, not strings, unless external contract requires strings.
- Use integer minor units or explicit decimal representation for money.

### 3.4 Enum rules

Enums MUST have an explicit unspecified value at zero.

```proto
enum CaseStatus {
  CASE_STATUS_UNSPECIFIED = 0;
  CASE_STATUS_DRAFT = 1;
  CASE_STATUS_SUBMITTED = 2;
}
```

Go code MUST reject or map unspecified values according to validation policy.

---

## 4. Generated code rules

Generated protobuf/gRPC code MUST:

- be generated by pinned tool versions;
- not be manually edited;
- be regenerated through documented commands;
- be checked for drift in CI;
- live in a predictable generated package path;
- avoid import cycles with application/domain packages.

`protoc`, `protoc-gen-go`, and `protoc-gen-go-grpc` versions MUST be pinned through project tooling.

## 5. Server architecture

### 5.1 Service implementation

gRPC service implementations are transport adapters.

They MAY:

- decode request DTOs;
- validate transport-level required fields;
- map identity/context;
- call application services;
- map application errors to gRPC status;
- build response DTOs.

They MUST NOT:

- contain domain transition logic;
- manually perform authorization by trusting raw metadata;
- leak database/provider errors directly;
- spawn unmanaged goroutines;
- mutate generated request messages shared across goroutines.

### 5.2 Embedding generated server types

When generated code requires embedding, follow current gRPC-Go guidance for forward compatibility.

Rules:

- Implement all required RPC methods intentionally.
- Do not rely on unimplemented default behavior for real methods.
- Do not panic for unimplemented methods in production.

### 5.3 Request validation

Each RPC MUST validate:

- required identifiers;
- enum values;
- pagination limits;
- oneof consistency;
- timestamp ranges;
- repeated field sizes;
- map field sizes;
- state transition preconditions when transport DTO contains transition request.

Use application/domain validation for business invariants.

---

## 6. Client architecture

### 6.1 Client construction

gRPC clients MUST be constructed with explicit configuration.

Rules:

- Use TLS credentials in production.
- Do not use insecure credentials except local tests/dev with explicit config.
- Configure per-call deadlines or require caller context deadline.
- Configure interceptors for trace/log/auth/retry where approved.
- Do not create new connections per request.
- Close `ClientConn` on application shutdown.

### 6.2 Deadline policy

Every outbound RPC MUST have a deadline.

Forbidden:

```go
resp, err := client.GetCase(context.Background(), req)
```

Preferred:

```go
ctx, cancel := context.WithTimeout(parent, cfg.GetCaseTimeout)
defer cancel()

resp, err := client.GetCase(ctx, req)
```

Do not impose a client timeout that exceeds upstream SLA or caller deadline.

## 7. Context and cancellation

### 7.1 Server-side cancellation

Server handlers MUST observe cancellation for:

- database calls;
- outbound RPC/HTTP calls;
- long loops;
- streaming send/receive loops;
- expensive CPU work where feasible.

Preferred:

```go
select {
case <-ctx.Done():
    return nil, status.Error(codes.Canceled, "request canceled")
default:
}
```

Map `context.Canceled` and `context.DeadlineExceeded` deliberately.

### 7.2 Context values

Context values MAY carry transport-scoped data such as:

- authenticated principal;
- tenant context;
- request ID;
- trace context;
- peer information.

Rules:

- Use typed context keys or dedicated accessor functions.
- Do not pass optional method parameters through context.
- Do not store context in structs.

---

## 8. Metadata rules

### 8.1 Metadata purpose

Metadata is for transport-level side-channel information:

- authorization token;
- request ID;
- trace propagation;
- tenant routing marker when approved;
- idempotency key;
- feature/version negotiation when approved.

Metadata MUST NOT carry core business request bodies.

### 8.2 Metadata handling

Rules:

- Normalize expected metadata keys.
- Treat incoming metadata as untrusted input.
- Redact sensitive metadata in logs.
- Do not forward all incoming metadata blindly to downstream services.
- Propagate only allowlisted metadata.
- Use `metadata.AppendToOutgoingContext` or equivalent approved APIs for outbound metadata.

### 8.3 Binary metadata

Binary metadata keys ending in `-bin` MUST contain bytes with explicit encoding/decoding policy.

Do not put arbitrary serialized domain objects in binary metadata.

## 9. Status and error mapping

### 9.1 Use canonical status codes

gRPC methods MUST return errors created or mapped through `status` and `codes`.

Rules:

- `codes.InvalidArgument` for malformed request independent of current system state.
- `codes.NotFound` for absent resource or policy-hidden resource when project uses concealment.
- `codes.AlreadyExists` for create conflict.
- `codes.FailedPrecondition` for state-dependent invalid operation.
- `codes.PermissionDenied` for authenticated but unauthorized.
- `codes.Unauthenticated` for missing/invalid authentication.
- `codes.ResourceExhausted` for quota/rate/size limits.
- `codes.Aborted` for concurrency conflict requiring retry at higher level.
- `codes.Unavailable` for transient dependency outage.
- `codes.DeadlineExceeded` for timeout.
- `codes.Internal` only for unexpected server bugs.

Do not return raw `fmt.Errorf` across service boundary without mapping.

### 9.2 Error details

Structured error details MAY be used when:

- clients are expected to handle them;
- details are versioned;
- no sensitive information is exposed;
- compatibility tests exist.

Do not attach stack traces or internal provider details to gRPC status.

### 9.3 Domain-to-gRPC mapping

Error mapping should be centralized.

```go
func mapError(err error) error {
    switch {
    case err == nil:
        return nil
    case errors.Is(err, domain.ErrNotFound):
        return status.Error(codes.NotFound, "case not found")
    case errors.Is(err, domain.ErrInvalidTransition):
        return status.Error(codes.FailedPrecondition, "invalid case transition")
    default:
        return status.Error(codes.Internal, "internal error")
    }
}
```

## 10. Interceptors

### 10.1 Interceptor responsibilities

Interceptors are appropriate for cross-cutting concerns:

- panic recovery;
- request ID;
- authentication token extraction;
- authorization pre-check where policy is transport-level;
- logging;
- metrics;
- tracing;
- rate limiting;
- validation hooks;
- retry on client side when policy allows.

Interceptors MUST NOT:

- hide method-specific authorization requirements;
- perform complex domain transitions;
- convert all errors to `Internal`;
- log full payloads by default;
- retry non-idempotent methods without policy.

### 10.2 Interceptor order

Server interceptor order SHOULD be deliberate.

Recommended unary server order:

1. panic recovery;
2. correlation/request ID;
3. telemetry/tracing;
4. authentication;
5. authorization/rate-limit;
6. validation;
7. handler.

Client interceptor order SHOULD ensure tracing and deadlines wrap retries correctly according to telemetry policy.

## 11. Streaming RPCs

### 11.1 Streaming contract

Every streaming RPC MUST define:

- stream type: server, client, bidirectional;
- ordering guarantees;
- max message size;
- max stream duration;
- heartbeat/keepalive behavior;
- cancellation behavior;
- backpressure policy;
- partial failure behavior;
- replay/resume semantics if needed.

### 11.2 Streaming implementation

Streaming loops MUST:

- check context cancellation;
- handle `io.EOF` correctly for client streams;
- avoid unbounded buffering;
- avoid goroutine leaks;
- close producer channels on exit;
- propagate send/receive errors;
- not ignore flow-control effects;
- protect shared state.

Example server stream pattern:

```go
for {
    select {
    case <-stream.Context().Done():
        return status.Error(codes.Canceled, "stream canceled")
    case item, ok := <-items:
        if !ok {
            return nil
        }
        if err := stream.Send(mapItem(item)); err != nil {
            return mapError(err)
        }
    }
}
```

## 12. Message size, compression, and resource limits

gRPC servers and clients MUST set resource limits deliberately.

Rules:

- Configure max receive message size for expected payloads.
- Configure max send message size when needed.
- Do not accept unbounded repeated/map fields.
- Validate pagination and batch sizes.
- Use streaming for large result sets.
- Treat compression as resource-risky; consider decompression limits and CPU cost.

---

## 13. Security

### 13.1 Transport security

Production gRPC MUST use TLS or mTLS unless protected by an explicitly approved internal security boundary.

Rules:

- Do not use insecure transport in production.
- Verify server names.
- Configure CA pools deliberately.
- Rotate certificates.
- Use mTLS identity only as one input to authorization, not complete business authorization.

### 13.2 Authentication

Authentication MAY use:

- mTLS;
- bearer tokens;
- OAuth2/OIDC access tokens;
- signed internal service tokens;
- provider-approved credentials.

Rules:

- Validate tokens at boundary.
- Do not trust raw metadata role claims without validation.
- Redact auth metadata.
- Reject missing/invalid credentials with `Unauthenticated`.

### 13.3 Authorization

Authorization MUST be checked per RPC and resource.

Rules:

- Method-level auth is not enough for object-level access.
- Tenant/resource ownership must be verified by trusted data.
- State transitions must verify actor permission and current state.
- Denials must be audited when policy requires.

---

## 14. Load balancing, retries, and idempotency

### 14.1 Retry policy

Client retries MUST be explicit and safe.

Rules:

- Retry only idempotent methods or calls with idempotency key.
- Respect context deadline.
- Retry only retryable codes such as `Unavailable` or selected `ResourceExhausted` according to policy.
- Do not retry `InvalidArgument`, `PermissionDenied`, `Unauthenticated`, or `FailedPrecondition` by default.
- Emit attempt metrics.

### 14.2 Idempotency

Mutating RPCs SHOULD accept idempotency keys when clients may retry.

The idempotency key must be bound to:

- actor/service identity;
- method;
- target resource or aggregate;
- request hash when feasible;
- expiry.

---

## 15. Health, readiness, and reflection

### 15.1 Health

Services SHOULD expose gRPC health checking when used by infrastructure.

Rules:

- Liveness must not perform heavy dependency checks.
- Readiness may include critical dependency state.
- Health status must reflect graceful shutdown.

### 15.2 Server reflection

gRPC server reflection MUST be disabled in production unless explicitly approved.

If enabled:

- restrict access where possible;
- audit exposure;
- ensure it does not reveal sensitive internal APIs.

---

## 16. Observability

Every gRPC service MUST emit:

- RPC count by full method and status code;
- RPC duration histogram by full method;
- in-flight RPC gauge when useful;
- stream message counts where useful;
- deadline exceeded/canceled counts;
- dependency call metrics;
- trace spans with method name and status;
- structured logs with request ID, method, status, duration, and safe actor/tenant fields.

Rules:

- Do not include raw message payloads in logs by default.
- Do not use unbounded metadata values as metric labels.
- Status mapping must be consistent between logs, metrics, and traces.

---

## 17. Testing gate

gRPC code MUST include tests for:

- protobuf compatibility where public/cross-service;
- request validation;
- success response mapping;
- each domain error to status code mapping;
- deadline exceeded;
- context cancellation;
- authentication missing/invalid;
- authorization denied;
- metadata propagation allowlist;
- interceptor order and behavior;
- max message size/repeated field limit;
- unary and streaming behavior;
- client retry/idempotency if enabled;
- TLS/mTLS config where practical;
- no secret leakage in logs.

Recommended approaches:

- in-process gRPC server with `bufconn`;
- fake application services;
- generated client integration tests;
- golden tests for error details;
- race tests for streaming/shared state;
- fuzz tests for metadata and request mapping.

---

## 18. Anti-patterns

Reject code that:

- uses `context.Background()` for outbound RPCs in request flow;
- creates a new `ClientConn` per request;
- manually edits generated protobuf code;
- reuses protobuf field numbers;
- returns raw internal errors from RPC methods;
- maps every error to `codes.Internal`;
- ignores `ctx.Done()` in long-running handlers;
- blindly forwards all incoming metadata downstream;
- logs full request/response messages containing PII or secrets;
- uses insecure transport in production;
- implements authorization only in client code;
- retries mutating RPCs without idempotency;
- uses streaming with unbounded goroutines or buffers;
- enables server reflection in production without approval.

---

## 19. Merge checklist

Before merging Go gRPC code, verify:

- [ ] `.proto` contract follows compatibility rules.
- [ ] Generated code is reproducible and not manually edited.
- [ ] Every outbound RPC has a bounded deadline.
- [ ] Server handlers observe context cancellation.
- [ ] Errors are mapped to canonical status codes.
- [ ] Metadata propagation is allowlisted and redacted.
- [ ] Authn/authz are explicit and tested.
- [ ] Streaming has backpressure and cancellation semantics.
- [ ] Message size and batch limits are configured.
- [ ] Metrics/traces/logs use bounded cardinality.
- [ ] Tests cover deadlines, cancellation, security, status mapping, and streaming failures.
