# Strict Coding Standards — Go Data Types

> **File:** `strict-coding-standards__go_data_type.md`  
> **Scope:** Go source code written, modified, reviewed, or generated by an LLM/code agent.  
> **Status:** Mandatory merge gate.  
> **Audience:** LLM agents, software engineers, reviewers, maintainers.  
> **Primary concern:** type correctness, semantic clarity, API stability, domain modelling, and data boundary safety.

---

## 1. Purpose

This document defines mandatory standards for using Go data types. It is not a tutorial and it is not optional style guidance. It exists to prevent LLM-generated Go code from producing weak domain models, ambiguous API contracts, nil/empty bugs, accidental mutability, unsafe numeric conversions, JSON contract drift, reflection abuse, and hidden compatibility breaks.

An LLM/code agent MUST treat type design as part of system design. A type is not merely a container for data. A type defines:

1. valid state,
2. invalid state,
3. ownership expectation,
4. mutability expectation,
5. serialization contract,
6. compatibility boundary,
7. concurrency risk,
8. testability surface.

If a type cannot express or protect its invariants, the agent MUST add validation, constructors, methods, or boundary-specific DTOs. The agent MUST NOT leave important domain constraints only in comments.

---

## 2. Normative language

The terms below are used intentionally:

- **MUST**: mandatory; violation blocks merge.
- **MUST NOT**: prohibited; violation blocks merge.
- **SHOULD**: expected default; deviation requires written justification.
- **MAY**: allowed when justified by context.
- **LLM MUST**: a specific instruction for code-generating agents.

---

## 3. Source authority

When uncertain, the agent MUST prefer these authorities, in this order:

1. Go language specification.
2. Standard library documentation.
3. Go memory model.
4. Effective Go.
5. Go Code Review Comments.
6. Official Go blog articles from `go.dev`.
7. Existing project conventions, only when they do not contradict correctness or safety.

The agent MUST NOT invent language behavior from analogy with Java, C#, JavaScript, Rust, C++, or Python.

---

## 4. Global LLM behavior rules

### 4.1 No type guessing

The agent MUST NOT introduce a type without deciding:

- whether it is a domain type, DTO, persistence model, internal helper, generic utility, or test fixture;
- whether zero value is valid;
- whether nil is meaningful;
- whether it is immutable-by-convention or mutable;
- whether it crosses a package, module, network, storage, or goroutine boundary;
- whether it needs custom validation or construction;
- whether it must remain backward-compatible.

Bad:

```go
type User struct {
    ID string
    Age int
    Status string
}
```

Better:

```go
type UserID string

type UserStatus string

const (
    UserStatusActive   UserStatus = "ACTIVE"
    UserStatusDisabled UserStatus = "DISABLED"
)

type User struct {
    id     UserID
    age    int
    status UserStatus
}

func NewUser(id UserID, age int, status UserStatus) (User, error) {
    if id == "" {
        return User{}, ErrInvalidUserID
    }
    if age < 0 {
        return User{}, ErrInvalidAge
    }
    switch status {
    case UserStatusActive, UserStatusDisabled:
    default:
        return User{}, ErrInvalidUserStatus
    }
    return User{id: id, age: age, status: status}, nil
}
```

### 4.2 Types MUST encode meaning

The agent MUST create named types for values that carry domain meaning, even when the underlying representation is primitive.

Use named types for:

- identifiers: `UserID`, `OrderID`, `CaseID`;
- statuses: `CaseStatus`, `PaymentStatus`;
- categories: `Role`, `Permission`, `RegionCode`;
- units: `AmountCents`, `DistanceMeters`, `RetryCount`;
- external codes: `CountryCode`, `ErrorCode`, `ReasonCode`;
- state-machine triggers: `CommandType`, `EventType`, `TransitionName`.

The agent MUST NOT represent unrelated concepts as plain `string`, `int`, or `bool` when doing so makes invalid states easy.

Bad:

```go
func Assign(caseID string, officerID string, urgent bool) error
```

Better:

```go
type CaseID string
type OfficerID string
type AssignmentPriority string

const (
    AssignmentPriorityNormal AssignmentPriority = "NORMAL"
    AssignmentPriorityUrgent AssignmentPriority = "URGENT"
)

func Assign(caseID CaseID, officerID OfficerID, priority AssignmentPriority) error
```

### 4.3 Type aliases MUST be rare

The agent MUST distinguish named types from aliases.

Allowed:

```go
type UserID string // new named type
```

Rare and compatibility-driven:

```go
type LegacyUserID = string // alias, not a new domain type
```

The agent MUST NOT use type aliases to create fake domain types. A type alias does not prevent accidental mixing with the aliased type.

Type aliases MAY be used only for:

- migration compatibility,
- package re-export during refactoring,
- controlled API transition,
- generated code compatibility.

Every exported alias MUST have a comment explaining why an alias is required instead of a named type.

## 5. Zero value policy

### 5.1 Every type MUST have a zero-value decision

For every exported type, the agent MUST decide one of these policies:

1. **Zero value is valid and usable.**
2. **Zero value is invalid but detectable.**
3. **Zero value must not be constructed directly.**

The policy MUST be visible from the API shape.

Good zero-value-usable type:

```go
type Counter struct {
    n atomic.Int64
}

func (c *Counter) Inc() { c.n.Add(1) }
func (c *Counter) Value() int64 { return c.n.Load() }
```

Good zero-value-invalid type:

```go
type EmailAddress string

func ParseEmailAddress(s string) (EmailAddress, error) {
    if !strings.Contains(s, "@") {
        return "", ErrInvalidEmailAddress
    }
    return EmailAddress(s), nil
}
```

### 5.2 Constructors MUST enforce invariants

If a type has invariants, the agent MUST provide a constructor or parser.

```go
type PageRequest struct {
    page int
    size int
}

func NewPageRequest(page, size int) (PageRequest, error) {
    if page < 1 {
        return PageRequest{}, ErrInvalidPage
    }
    if size < 1 || size > 100 {
        return PageRequest{}, ErrInvalidPageSize
    }
    return PageRequest{page: page, size: size}, nil
}
```

The agent MUST NOT expose mutable fields that allow callers to bypass invariants.

Bad:

```go
type PageRequest struct {
    Page int
    Size int
}
```

Allowed for DTOs only, not domain objects.

### 5.3 Boolean fields MUST be challenged

The agent MUST NOT introduce boolean fields or parameters without checking whether the domain actually has more than two states.

Bad:

```go
type Approval struct {
    Approved bool
}
```

Better:

```go
type ApprovalDecision string

const (
    ApprovalDecisionPending  ApprovalDecision = "PENDING"
    ApprovalDecisionApproved ApprovalDecision = "APPROVED"
    ApprovalDecisionRejected ApprovalDecision = "REJECTED"
)
```

Booleans are allowed for true binary technical facts such as `enabled`, `closed`, `encrypted`, or `dryRun` when future states are unlikely.

## 6. Primitive type standards

### 6.1 Integer types

The agent MUST choose integer types deliberately.

Default:

```go
var retryCount int
```

Use fixed-width integers when:

- interacting with binary protocols,
- persisting exact schema-width values,
- hashing,
- serialization requires exact width,
- atomic typed operations require it,
- cross-platform consistency matters.

Examples:

```go
type Version int64
type AmountCents int64
type ShardID uint32
```

The agent MUST NOT use `int` for persisted numeric contracts where 32-bit vs 64-bit platform variance could matter.

### 6.2 Numeric conversion MUST be explicit and checked

The agent MUST NOT write unchecked narrowing conversions.

Bad:

```go
size := int32(len(payload))
```

Better:

```go
func checkedInt32(n int) (int32, error) {
    if n > math.MaxInt32 || n < math.MinInt32 {
        return 0, ErrIntegerOverflow
    }
    return int32(n), nil
}
```

All conversions involving money, quantities, lengths, offsets, or protocol fields MUST be range-checked.

### 6.3 Unsigned integers MUST NOT be used to avoid negative checks

The agent MUST NOT use `uint`, `uint64`, or similar merely because a value should not be negative.

Bad:

```go
type Age uint
```

Better:

```go
type Age int

func NewAge(v int) (Age, error) {
    if v < 0 || v > 150 {
        return 0, ErrInvalidAge
    }
    return Age(v), nil
}
```

Use unsigned integers primarily for bitsets, hashes, protocol fields, and values that are naturally unsigned at the representation layer.

### 6.4 Floating point MUST NOT be used for money

The agent MUST NOT use `float32` or `float64` for money, balance, fee, tax, penalty, price, or regulatory amount.

Required alternatives:

- integer minor units: `AmountCents int64`,
- decimal library approved by project standards,
- database numeric/decimal with explicit conversion boundary.

Bad:

```go
type Invoice struct {
    Total float64
}
```

Better:

```go
type AmountCents int64

type Invoice struct {
    Total AmountCents
}
```

### 6.5 `byte` and `rune` MUST be used semantically

The agent MUST use:

- `byte` for raw bytes and binary data;
- `rune` for Unicode code points;
- `string` for immutable UTF-8 text by convention;
- `[]byte` for mutable bytes or binary payloads.

The agent MUST NOT assume `len(s)` equals number of characters.

Bad:

```go
if len(name) > 30 { // byte count, not character count
    return ErrNameTooLong
}
```

Better when business rule is Unicode code points:

```go
if utf8.RuneCountInString(name) > 30 {
    return ErrNameTooLong
}
```

If business rule depends on user-perceived characters, the agent MUST use an approved grapheme cluster library or explicitly document that the limit is code-point-based.

## 7. String standards

### 7.1 Strings MUST be treated as immutable data

The agent MUST NOT attempt to mutate strings using `unsafe` or aliasing tricks.

Required conversions:

```go
b := []byte(s) // creates mutable byte slice copy
s2 := string(b) // creates string from bytes
```

The agent MUST consider the allocation impact of string/byte conversions in hot paths.

### 7.2 Stringly typed domains are prohibited

The agent MUST NOT encode domain state with arbitrary strings.

Bad:

```go
if case.Status == "done" || case.Status == "closed" || case.Status == "finish" {
    // ambiguous
}
```

Better:

```go
type CaseStatus string

const (
    CaseStatusDraft      CaseStatus = "DRAFT"
    CaseStatusSubmitted  CaseStatus = "SUBMITTED"
    CaseStatusInReview   CaseStatus = "IN_REVIEW"
    CaseStatusClosed     CaseStatus = "CLOSED"
)
```

### 7.3 External string codes MUST be isolated

When external systems use string codes, the agent MUST isolate them at the boundary.

```go
type ExternalCaseStatusCode string

type CaseStatus string

func MapExternalStatus(code ExternalCaseStatusCode) (CaseStatus, error) {
    switch code {
    case "SUBM":
        return CaseStatusSubmitted, nil
    case "CLOS":
        return CaseStatusClosed, nil
    default:
        return "", ErrUnknownExternalCaseStatus
    }
}
```

The agent MUST NOT allow external codes to spread into core domain logic.

## 8. Struct standards

### 8.1 Structs MUST have one clear role

A struct MUST be one of:

- domain model,
- command,
- query,
- event,
- DTO/request/response,
- persistence record,
- configuration,
- dependency container,
- internal helper.

The agent MUST NOT create one struct that is simultaneously a DB model, JSON DTO, domain object, and event payload.

Bad:

```go
type Case struct {
    ID        string    `json:"id" db:"id"`
    Status    string    `json:"status" db:"status"`
    CreatedAt time.Time `json:"createdAt" db:"created_at"`
}
```

Better:

```go
type Case struct {
    id        CaseID
    status    CaseStatus
    createdAt time.Time
}

type CaseRecord struct {
    ID        string    `db:"id"`
    Status    string    `db:"status"`
    CreatedAt time.Time `db:"created_at"`
}

type CaseResponse struct {
    ID        string `json:"id"`
    Status    string `json:"status"`
    CreatedAt string `json:"createdAt"`
}
```

### 8.2 Exported fields MUST be intentional

The agent MUST keep fields unexported when invariants must be protected.

Exported fields are allowed for:

- DTOs,
- config structs,
- simple immutable-by-convention value structs,
- generated code,
- structs meant for direct literal construction.

Domain objects with state transitions SHOULD use unexported fields and methods.

### 8.3 Struct tags MUST be boundary-specific

The agent MUST NOT mix unrelated tags casually.

Bad:

```go
type User struct {
    ID string `json:"id" db:"id" validate:"required" form:"id"`
}
```

Better:

```go
type CreateUserRequest struct {
    ID string `json:"id"`
}

type UserRecord struct {
    ID string `db:"id"`
}
```

Struct tags are part of boundary contracts and MUST be reviewed when changed.

### 8.4 Embedded fields MUST not hide dependencies

The agent MUST NOT embed types merely to avoid typing field names.

Allowed:

```go
type NotFoundError struct {
    Resource string
    ID       string
}

func (e NotFoundError) Error() string { ... }
```

Questionable:

```go
type Service struct {
    *Logger
    *Repository
    *Metrics
}
```

Embedding dependencies makes API and ownership unclear. Prefer named fields.

### 8.5 Struct copy behavior MUST be safe

The agent MUST NOT copy structs that contain:

- `sync.Mutex`, `sync.RWMutex`, `sync.Once`, `sync.WaitGroup`, `sync.Cond`,
- `atomic.*` values,
- open files, network connections, transactions,
- buffers with ownership assumptions,
- mutable maps/slices that should not alias.

If a struct must not be copied, the agent SHOULD make this clear by design, comments, tests, or project-approved `noCopy` pattern.

## 9. Array standards

Arrays have value semantics. Assigning or passing an array copies all elements.

The agent SHOULD use arrays only when:

- fixed size is part of the type contract,
- cryptographic digest size is fixed,
- protocol fields are fixed-width,
- stack allocation and exact size are intentional.

Example:

```go
type SHA256Digest [32]byte
```

The agent SHOULD NOT use arrays for general dynamic collections. Use slices.

---

## 10. Slice standards

### 10.1 Slices MUST be treated as descriptors with shared backing arrays

A slice value contains a pointer to an underlying array, length, and capacity. Copying a slice copies the descriptor, not the backing array.

The agent MUST assume slices can alias unless proven otherwise.

Bad:

```go
func StoreUsers(users []User) {
    cache.users = users // caller can mutate backing array later
}
```

Better:

```go
func StoreUsers(users []User) {
    cache.users = slices.Clone(users)
}
```

### 10.2 Input slices MUST not be retained without copying

If a function stores a slice beyond the call, the agent MUST copy it unless ownership transfer is explicitly documented.

```go
func NewReport(rows []Row) Report {
    return Report{rows: slices.Clone(rows)}
}
```

### 10.3 Output slices MUST protect internal state

The agent MUST NOT return internal mutable slices directly.

Bad:

```go
func (r Report) Rows() []Row {
    return r.rows
}
```

Better:

```go
func (r Report) Rows() []Row {
    return slices.Clone(r.rows)
}
```

Exception: performance-critical internal packages MAY return borrowed slices if documented and tested.

### 10.4 Nil vs empty slices MUST be contract-driven

Default internal representation MAY use nil slices.

For public API, JSON, database, or external responses, the agent MUST decide whether `null` or `[]` is required.

- `nil` slice can encode as JSON `null`.
- non-nil empty slice can encode as JSON `[]`.

The agent MUST NOT rely on consumers treating `null` and `[]` as identical unless the contract says so.

### 10.5 Long-lived subslices MUST not retain large arrays

The agent MUST copy small subslices taken from large buffers when the subslice outlives the buffer processing.

Bad:

```go
func ExtractToken(payload []byte) []byte {
    return payload[10:30] // may keep entire payload alive
}
```

Better:

```go
func ExtractToken(payload []byte) []byte {
    return bytes.Clone(payload[10:30])
}
```

### 10.6 Append result MUST be assigned

The agent MUST always assign the result of `append`.

Correct:

```go
items = append(items, item)
```

Incorrect:

```go
append(items, item)
```

### 10.7 Slice capacity MUST be controlled when aliasing matters

When appending to a subslice could mutate another view of the same backing array, the agent MUST use full slice expression or clone.

```go
head := input[:n:n] // capacity capped to length
head = append(head, extra...)
```

## 11. Map standards

### 11.1 Maps MUST be initialized before writes

The agent MUST initialize maps before writing.

```go
m := make(map[string]int)
m["a"] = 1
```

Writing to a nil map is prohibited.

### 11.2 Map lookup MUST handle absence explicitly

The agent MUST use comma-ok when absence is semantically different from zero value.

Bad:

```go
limit := limits[userID]
```

Better:

```go
limit, ok := limits[userID]
if !ok {
    return ErrLimitNotConfigured
}
```

### 11.3 Map iteration order MUST not be relied on

The agent MUST NOT write code or tests that depend on map iteration order.

Required when deterministic output matters:

```go
keys := make([]string, 0, len(m))
for k := range m {
    keys = append(keys, k)
}
slices.Sort(keys)
for _, k := range keys {
    process(k, m[k])
}
```

### 11.4 Maps are not concurrency-safe by default

The agent MUST NOT concurrently read and write a plain map without synchronization.

Allowed options:

- guard with `sync.RWMutex`,
- use channel ownership pattern,
- use `sync.Map` only for its intended access patterns,
- build immutable snapshots.

### 11.5 Maps returned from domain objects MUST be copied

The agent MUST NOT expose internal maps directly.

```go
func (p Permissions) Values() map[Permission]bool {
    out := make(map[Permission]bool, len(p.values))
    for k, v := range p.values {
        out[k] = v
    }
    return out
}
```

## 12. Interface standards

### 12.1 Interfaces belong at the consumer side

The agent SHOULD define interfaces in the package that consumes behavior, not in the package that implements it.

Bad:

```go
package postgres

type Repository interface {
    Save(ctx context.Context, user User) error
}
```

Better:

```go
package user

type Store interface {
    Save(ctx context.Context, user User) error
}
```

### 12.2 Do not create one-method interfaces prematurely

The agent MUST NOT create interfaces solely because mocking might be needed later.

Interfaces are allowed when:

- multiple implementations exist,
- the consumer needs substitution,
- boundary inversion is required,
- testing requires controlled behavior and concrete type is unsuitable.

### 12.3 Interface nil trap MUST be avoided

The agent MUST understand that an interface value is nil only when both its dynamic type and dynamic value are nil.

Bad:

```go
func NewError() error {
    var e *MyError = nil
    return e // non-nil error interface
}
```

Better:

```go
func NewError() error {
    return nil
}
```

The agent MUST NOT return typed nil values as interfaces.

### 12.4 `any` MUST be justified

The agent MUST NOT use `any` to avoid modelling a type.

Allowed:

- logging fields,
- JSON-like dynamic payload at boundary,
- generic helpers,
- adapters around reflection-heavy libraries,
- test utilities.

Not allowed:

```go
func Process(v any) error
```

unless the valid dynamic types are documented and checked.

### 12.5 Marker interfaces are prohibited unless required by a framework

The agent MUST NOT introduce empty interfaces or marker interfaces for domain classification. Use explicit fields, methods, or typed constants.

## 13. Generic type standards

### 13.1 Generics MUST reduce duplication without hiding domain meaning

Generics are allowed for:

- containers,
- algorithms,
- typed constraints,
- reusable test fixtures,
- adapter code,
- functional utilities with clear type parameters.

Generics MUST NOT be used to make business logic abstract before concrete duplication exists.

Bad:

```go
type GenericService[T any] struct { ... }
```

Better:

```go
type Repository[T Aggregate, ID comparable] interface {
    FindByID(ctx context.Context, id ID) (T, error)
    Save(ctx context.Context, aggregate T) error
}
```

only if the repository behavior is truly uniform.

### 13.2 Type constraints MUST be narrow

The agent MUST use the narrowest constraint that expresses required operations.

Bad:

```go
func Contains[T any](items []T, target T) bool
```

This does not compile if `T` must be comparable.

Better:

```go
func Contains[T comparable](items []T, target T) bool
```

### 13.3 Avoid generic domain erasure

The agent MUST NOT erase meaningful domain names behind type parameters.

Bad:

```go
type StateMachine[S comparable, E comparable] struct { ... }
```

Allowed only if the package is an actual state-machine library.

For business code, prefer:

```go
type CaseStateMachine struct { ... }
```

### 13.4 Generic aliases MUST follow alias policy

Generic aliases are subject to the same restrictions as non-generic aliases. They MUST exist for migration or re-export reasons, not convenience.

## 14. Enum-like constants

Go does not have native enums. The agent MUST model finite sets with named types and constants.

```go
type EventType string

const (
    EventTypeCaseSubmitted EventType = "CASE_SUBMITTED"
    EventTypeCaseAssigned  EventType = "CASE_ASSIGNED"
)
```

Every externally received enum-like value MUST be validated.

```go
func ParseEventType(s string) (EventType, error) {
    switch EventType(s) {
    case EventTypeCaseSubmitted, EventTypeCaseAssigned:
        return EventType(s), nil
    default:
        return "", ErrInvalidEventType
    }
}
```

The agent MUST NOT use `iota` for externally serialized values unless the numeric values are explicitly part of a stable protocol.

Allowed internal-only `iota`:

```go
type queueState int

const (
    queueStateIdle queueState = iota
    queueStateRunning
    queueStateStopped
)
```

## 15. Time and duration standards

### 15.1 Use `time.Time` for instants

The agent MUST use `time.Time` for instants and `time.Duration` for durations.

The agent MUST NOT represent time as raw `int64`, `string`, or `float64` inside domain logic unless at a protocol boundary.

### 15.2 Timezone and clock source MUST be explicit

The agent MUST NOT call `time.Now()` deep inside business logic when deterministic tests or domain consistency matter.

Better:

```go
type Clock interface {
    Now() time.Time
}
```

Use injected clocks for:

- expiry logic,
- scheduling,
- state transitions,
- SLA computation,
- retry/backoff tests,
- audit timestamps.

### 15.3 Serialized time format MUST be explicit

DTOs MUST define whether time is encoded as:

- RFC3339 string,
- Unix seconds,
- Unix milliseconds,
- database timestamp,
- external legacy format.

The agent MUST NOT assume all systems agree on timezone, precision, or layout.

## 16. Money, decimal, quantity, and measurement standards

### 16.1 Money MUST be exact

Money MUST be represented as:

- minor-unit integer, or
- approved decimal type.

```go
type Money struct {
    currency CurrencyCode
    cents    int64
}
```

The agent MUST include currency when amount can cross currency boundaries.

### 16.2 Measurements MUST encode units

Bad:

```go
type Limit struct {
    Timeout int
    Size int
}
```

Better:

```go
type Limit struct {
    Timeout time.Duration
    SizeBytes int64
}
```

The agent MUST NOT leave unit meaning only in variable names if type can encode it.

---

## 17. Error type standards

### 17.1 Sentinel errors MUST be stable and comparable

Use sentinel errors when callers need `errors.Is`.

```go
var ErrNotFound = errors.New("not found")
```

The agent MUST NOT create dynamic sentinel errors inside functions.

Bad:

```go
return errors.New("not found")
```

when callers need to compare.

### 17.2 Structured error types MUST expose semantic fields

```go
type ValidationError struct {
    Field string
    Rule  string
}

func (e ValidationError) Error() string {
    return e.Field + ": " + e.Rule
}
```

The agent MUST avoid using string parsing for error classification.

### 17.3 Error wrapping MUST preserve cause

The agent MUST use `%w` when wrapping errors that callers may inspect.

```go
return fmt.Errorf("load user %s: %w", id, err)
```

---

## 18. Channel and function type standards

### 18.1 Channels MUST express ownership direction

The agent SHOULD use directional channels in APIs.

```go
func Produce(ctx context.Context) <-chan Event
func Consume(ctx context.Context, in <-chan Event) error
```

### 18.2 Channel element type MUST be intentional

Avoid `chan any`. Use typed messages.

```go
type WorkerMessage struct {
    JobID JobID
    Payload Payload
}
```

### 18.3 Function types MUST be named when semantic

```go
type RetryPredicate func(error) bool
type TransitionGuard func(ctx context.Context, c Case) error
```

Named function types make policies explicit and testable.

---

## 19. JSON and DTO standards

### 19.1 DTOs MUST be separate from domain types

The agent MUST NOT add JSON tags to domain objects unless the domain object is intentionally also the transport contract.

Required pattern:

```go
type CreateCaseRequest struct {
    Title string `json:"title"`
}

type CaseResponse struct {
    ID     string `json:"id"`
    Status string `json:"status"`
}
```

### 19.2 `omitempty`/`omitzero` MUST be contract-driven

The agent MUST NOT add omission tags casually. Missing, null, empty, and zero can carry different meanings.

For each optional field, define:

- absent means no change,
- null means clear value,
- empty means set empty value,
- zero means actual zero value.

### 19.3 Partial update DTOs MUST distinguish absent from zero

Bad:

```go
type UpdateUserRequest struct {
    Age int `json:"age,omitempty"`
}
```

Better:

```go
type UpdateUserRequest struct {
    Age *int `json:"age,omitempty"`
}
```

Even better for complex patches: use explicit patch operation types.

### 19.4 Unknown fields policy MUST be explicit

For external API request decoding, the agent SHOULD reject unknown fields unless backward compatibility requires ignoring them.

```go
dec := json.NewDecoder(r.Body)
dec.DisallowUnknownFields()
```

### 19.5 DTO validation MUST happen at boundary

The agent MUST validate DTOs before converting to domain commands.

```go
func (r CreateCaseRequest) ToCommand() (CreateCaseCommand, error) {
    title, err := NewCaseTitle(r.Title)
    if err != nil {
        return CreateCaseCommand{}, err
    }
    return CreateCaseCommand{Title: title}, nil
}
```

## 20. Persistence type standards

### 20.1 Persistence records MUST not become domain models

Database records reflect storage shape, not business behavior.

```go
type CaseRecord struct {
    ID        string
    Status    string
    Version   int64
    CreatedAt time.Time
}
```

Domain models reflect rules.

```go
type Case struct {
    id      CaseID
    status  CaseStatus
    version Version
}
```

Mapping MUST validate persisted values.

### 20.2 Nullable database fields MUST be explicit

The agent MUST not conflate SQL NULL with Go zero value.

Allowed approaches:

- `sql.NullString`, `sql.NullTime`, etc.
- pointer fields in record structs,
- project-approved nullable generic type.

### 20.3 Version fields MUST be typed

For optimistic locking, event ordering, or state-machine monotonicity, use a named type.

```go
type Version int64
```

The agent MUST NOT pass raw `int64` version values across layers without semantic naming.

## 21. Domain event and command type standards

### 21.1 Commands MUST represent intent

```go
type SubmitCaseCommand struct {
    CaseID CaseID
    ActorID UserID
    Reason SubmissionReason
}
```

The agent MUST NOT reuse DTOs as commands.

### 21.2 Events MUST represent facts

```go
type CaseSubmittedEvent struct {
    EventID EventID
    CaseID  CaseID
    At      time.Time
    By      UserID
}
```

Events MUST be immutable-by-convention. The agent MUST NOT expose mutable collections inside events without copying.

### 21.3 State-machine types MUST restrict invalid transitions

State and transition types MUST be finite named types.

```go
type CaseStatus string
type CaseAction string
```

Transition functions MUST validate from-state, action, actor, and guard conditions.

## 22. Reflection standards

Reflection MUST be avoided in ordinary business logic.

Allowed:

- serialization adapters,
- dependency injection infrastructure,
- validation framework integration,
- test helpers,
- generic libraries where static typing is insufficient.

The agent MUST NOT use reflection to bypass type modelling.

Bad:

```go
func GetField(v any, name string) any
```

Better:

```go
func (u User) Email() EmailAddress
```

Reflection-heavy code MUST have tests for nils, pointers, unexported fields, zero values, and panic paths.

---

## 23. Unsafe type standards

The agent MUST NOT use `unsafe` for type conversion, string/byte zero-copy tricks, or performance optimization unless all of the following are true:

1. safe implementation is measured and proven insufficient;
2. unsafe version is isolated in a tiny package;
3. behavior is covered by tests and benchmarks;
4. documentation explains the memory and lifetime assumptions;
5. static analysis and race tests pass;
6. reviewer explicitly approves it.

The agent MUST NOT generate unsafe code by default.

---

## 24. Package API compatibility standards

### 24.1 Exported types are contracts

Changing an exported type may be a breaking change. The agent MUST treat these as compatibility-sensitive:

- field addition/removal in exported structs,
- field type changes,
- method signature changes,
- interface method changes,
- constant value changes,
- JSON tag changes,
- database tag changes,
- type alias vs named type changes.

### 24.2 Exported interfaces are especially dangerous

Adding a method to an exported interface breaks all external implementers. The agent MUST avoid exporting broad interfaces.

### 24.3 Constructors preserve evolution

For exported domain types, the agent SHOULD prefer constructors over requiring external struct literals. Constructors allow adding fields later without breaking callers.

---

## 25. Validation standards

### 25.1 Validation belongs at trust boundaries and invariant constructors

The agent MUST validate:

- incoming HTTP/gRPC/message DTOs,
- external system payloads,
- database records loaded into domain models,
- config files/env vars,
- CLI input,
- test fixture builders when invalid fixture would hide bugs.

### 25.2 Validation MUST produce typed errors

String-only validation errors are insufficient when callers need structured responses.

```go
type FieldViolation struct {
    Field string
    Code  string
}

type ValidationErrors []FieldViolation
```

## 26. Testing standards for data types

The agent MUST add table-driven tests for:

- constructor valid/invalid cases,
- parser valid/invalid cases,
- enum validation,
- JSON encoding/decoding shape,
- nil vs empty behavior,
- zero value behavior,
- copy/alias behavior for slices/maps,
- time format and timezone behavior,
- numeric boundary values,
- error wrapping/classification.

Example:

```go
func TestParseCaseStatus(t *testing.T) {
    tests := []struct {
        name    string
        input   string
        want    CaseStatus
        wantErr bool
    }{
        {name: "submitted", input: "SUBMITTED", want: CaseStatusSubmitted},
        {name: "unknown", input: "DONE", wantErr: true},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got, err := ParseCaseStatus(tt.input)
            if tt.wantErr {
                if err == nil {
                    t.Fatal("expected error")
                }
                return
            }
            if err != nil {
                t.Fatalf("unexpected error: %v", err)
            }
            if got != tt.want {
                t.Fatalf("got %q, want %q", got, tt.want)
            }
        })
    }
}
```

## 27. Required anti-pattern rejection list

The agent MUST reject or refactor these patterns:

1. `string` status without named type and validation.
2. `float64` money.
3. DTO reused as domain object.
4. DB record reused as API response.
5. Exported mutable fields on invariant-heavy domain object.
6. `map[string]any` as normal business model.
7. `any` used to avoid modelling.
8. Reflection used for ordinary field access.
9. Public interface created before consumer needs it.
10. Interface returning typed nil error.
11. Slice retained without ownership documentation or copy.
12. Map returned directly from internal state.
13. Boolean parameter controlling unrelated behavior.
14. `iota` persisted externally without explicit stable values.
15. Unchecked numeric narrowing conversion.
16. Timestamp represented as string inside domain logic.
17. Optional JSON field represented in a way that cannot distinguish absent from zero.
18. Struct tags from multiple boundaries on a single domain model.
19. Enum parsing via raw string comparison scattered across code.
20. Generic abstraction that erases domain language.

## 28. LLM implementation checklist

Before creating or modifying Go data types, the agent MUST answer:

- What layer owns this type?
- Is zero value valid?
- Are fields mutable or protected?
- Does this type cross a boundary?
- Does it need JSON, DB, message, or CLI representation?
- Is nil meaningful?
- Are slices/maps copied when crossing ownership boundaries?
- Are numeric conversions checked?
- Are enum-like values validated?
- Are time and money represented precisely?
- Can this exported type evolve without breaking callers?
- Are tests covering invalid states and boundary behavior?

If any answer is unknown, the agent MUST choose the safest explicit design and document the assumption.

## 29. Merge gate checklist

A change MUST NOT be merged unless all applicable items pass:

- [ ] Domain concepts use named types instead of raw primitives.
- [ ] Constructors/parsers enforce invariants.
- [ ] Zero-value behavior is clear.
- [ ] DTO/domain/persistence/event types are separated.
- [ ] JSON optionality is deliberate.
- [ ] Slices/maps do not leak mutable internal state.
- [ ] Numeric conversions are explicit and checked.
- [ ] Money does not use floating point.
- [ ] Time uses `time.Time` / `time.Duration` internally.
- [ ] Interfaces are consumer-owned and minimal.
- [ ] Generics have narrow constraints and do not erase domain meaning.
- [ ] Reflection and `unsafe` are absent or explicitly justified.
- [ ] Tests cover valid, invalid, zero, nil, boundary, and serialization cases.
