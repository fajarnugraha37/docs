# Strict Coding Standards — Go Time and Date Handling

Status: mandatory for all Go implementation, refactoring, review, and test generation performed by an LLM/code agent.

Scope: `time.Time`, `time.Duration`, clocks, dates, time zones, deadlines, timers, tickers, scheduling, temporal persistence, parsing/formatting, JSON/database contracts, tests, and time-based business logic.

This document is not a tutorial. It is a merge gate. Any code generated by an agent MUST comply with these rules unless the human explicitly approves a deviation.

---

## 1. Core Invariants

Every Go implementation that handles time or date MUST preserve the following invariants:

1. Wall-clock time and elapsed time are different concepts.
2. Date-only values and instant timestamps are different concepts.
3. Local time, UTC time, and offset time are different concepts.
4. Time zone and UTC offset are different concepts.
5. `time.Duration` is an elapsed interval, not a calendar period.
6. Calendar arithmetic is not duration arithmetic.
7. Deadlines and timeouts MUST be explicit and propagated through context where applicable.
8. Tests MUST NOT depend on real wall-clock time unless explicitly integration-level.
9. Persisted timestamps MUST have a clear timezone/UTC contract.
10. Business rules involving time MUST define inclusivity, timezone, rounding, and boundary behavior.

## 2. Required Mental Model

### 2.1 Three different temporal concepts

The agent MUST distinguish these concepts in names, types, and comments:

| Concept         | Meaning                                  | Go representation                                        |
| --------------- | ---------------------------------------- | -------------------------------------------------------- |
| Instant         | A precise point in time                  | `time.Time`, usually normalized to UTC                   |
| Local date/time | Human calendar value in a location       | `time.Time` with `*time.Location` plus explicit contract |
| Duration        | Elapsed nanoseconds between instants     | `time.Duration`                                          |
| Calendar period | 1 day, 1 month, next business day        | Domain type or explicit calendar function                |
| Deadline        | Latest instant by which work must finish | `context.Context` deadline or `time.Time`                |
| Schedule        | Rule for future occurrences              | Domain type, not just `Duration`                         |

### 2.2 Wall clock vs monotonic clock

Use wall-clock time for telling time:

- display,
- persistence,
- audit timestamp,
- business effective date,
- user-visible local date/time.

Use monotonic elapsed time for measuring time:

- latency,
- timeout measurement,
- benchmark-like elapsed intervals,
- retry delay calculation,
- performance telemetry.

Preferred elapsed-time patterns:

```go
start := time.Now()
// operation
elapsed := time.Since(start)
```

or:

```go
deadline := time.Now().Add(timeout)
if time.Until(deadline) <= 0 {
	return context.DeadlineExceeded
}
```

Do not measure elapsed time by subtracting Unix timestamps.

## 3. Naming Standards

Temporal variables MUST reveal their meaning.

Preferred names:

```go
createdAt       time.Time
updatedAt       time.Time
submittedAt     time.Time
expiresAt       time.Time
effectiveFrom   time.Time
effectiveUntil  time.Time
deadline        time.Time
timeout         time.Duration
retryDelay      time.Duration
clockSkew       time.Duration
businessDate    CivilDate
localStartTime  time.Time
```

Forbidden ambiguous names:

```go
date time.Time
time time.Time
ts int64
value string
period int
```

`At` suffix SHOULD mean an instant. `Date` SHOULD mean a calendar date. `Duration`, `Timeout`, `Delay`, and `Interval` SHOULD mean elapsed duration.

## 4. UTC and Location Rules

### 4.1 Storage rule

Persisted instants MUST be stored in UTC unless the storage contract explicitly requires otherwise.

Required at boundary:

```go
stored := t.UTC()
```

### 4.2 Display rule

User-facing display MUST convert from stored instant to the user's or business process's intended location.

```go
local := stored.In(userLocation)
```

Do not store user-facing formatted strings as canonical time values.

### 4.3 Location is not offset

The agent MUST NOT use a fixed UTC offset to represent a real-world timezone with daylight saving or historical rule changes.

Bad:

```go
loc := time.FixedZone("EST", -5*60*60)
```

Good:

```go
loc, err := time.LoadLocation("America/New_York")
if err != nil {
	return err
}
```

`time.FixedZone` is allowed only for protocols or data sources that explicitly define fixed offsets.

### 4.4 Local timezone rule

Do not use `time.Local` implicitly in business logic.

Allowed only in:

1. CLI defaults,
2. local developer tools,
3. explicit user preference fallback,
4. tests that intentionally validate local behavior.

Production business logic MUST use an explicit `*time.Location`.

## 5. Date-Only Values

### 5.1 Do not fake date-only casually

If the domain concept is a date without time, do not casually use arbitrary midnight `time.Time` without documenting timezone semantics.

Examples:

- birth date,
- business date,
- filing date,
- billing cycle date,
- license effective date,
- SLA working day,
- public holiday date.

Preferred domain type:

```go
type CivilDate struct {
	Year  int
	Month time.Month
	Day   int
}
```

or a validated wrapper with explicit serialization.

### 5.2 Midnight hazard

Midnight local time may be affected by timezone transitions in some jurisdictions. If using midnight as representation, document:

1. location,
2. validity,
3. conversion behavior,
4. database representation,
5. JSON format.

### 5.3 Date comparison

Compare date-only values as date components, not as instants, unless that is the intended semantics.

Preferred:

```go
func SameCivilDate(a, b time.Time, loc *time.Location) bool {
	ay, am, ad := a.In(loc).Date()
	by, bm, bd := b.In(loc).Date()
	return ay == by && am == bm && ad == bd
}
```

## 6. Duration and Calendar Arithmetic

### 6.1 `time.Duration` is not a month

`time.Duration` is a nanosecond-based elapsed interval. It MUST NOT be used to represent months, calendar years, business days, or "same day next month".

Bad:

```go
nextMonth := t.Add(30 * 24 * time.Hour)
```

Good:

```go
nextMonth := t.AddDate(0, 1, 0)
```

Even `AddDate` MUST be reviewed for end-of-month semantics.

### 6.2 Calendar arithmetic must define boundary behavior

Rules involving dates MUST define behavior for:

1. leap years,
2. month end,
3. daylight saving transitions,
4. nonexistent local times,
5. repeated local times,
6. weekends,
7. public holidays,
8. inclusive/exclusive boundaries,
9. timezone of evaluation.

### 6.3 Day is not always 24 hours in local time

Do not add `24 * time.Hour` to mean "tomorrow" in user/business local time.

Use `AddDate(0, 0, 1)` for calendar next day.

Use `Add(24 * time.Hour)` only for exact elapsed 24-hour intervals.

## 7. Parsing and Formatting

### 7.1 Layout constants

Use named layout constants for non-standard formats.

```go
const CivilDateLayout = "2006-01-02"
```

Do not inline magic layout strings repeatedly.

### 7.2 Prefer standard layouts

Use Go-provided constants when appropriate:

- `time.RFC3339`,
- `time.RFC3339Nano`,
- `time.DateOnly`,
- `time.TimeOnly`,
- `time.DateTime`,
- `time.Kitchen` only for user-facing display where appropriate.

### 7.3 Parsing with location

Use `time.Parse` for strings that include timezone/offset.

Use `time.ParseInLocation` for local date/time strings without offset.

Bad:

```go
t, err := time.Parse("2006-01-02 15:04", input)
```

Good:

```go
t, err := time.ParseInLocation("2006-01-02 15:04", input, loc)
```

### 7.4 Do not ignore parse errors

All parse errors MUST be wrapped with field context.

```go
t, err := time.Parse(time.RFC3339, input)
if err != nil {
	return time.Time{}, fmt.Errorf("parse submitted_at: %w", err)
}
```

### 7.5 Canonical output

Machine-readable timestamp output SHOULD use RFC3339 or RFC3339Nano unless protocol requires another format.

Rules:

1. Use UTC unless the API contract explicitly exposes offset/local time.
2. Keep precision stable.
3. Do not emit locale-specific text for machine APIs.
4. Do not use `String()` as a wire format.

## 8. JSON and API Contracts

### 8.1 `time.Time` JSON behavior

When using default `time.Time` JSON encoding, the API contract MUST state RFC3339-compatible timestamp semantics.

### 8.2 Optional time values

Optional timestamp fields MUST use one of these patterns:

```go
DeletedAt *time.Time `json:"deletedAt,omitempty"`
```

or:

```go
type OptionalTime struct {
	Time  time.Time
	Valid bool
}
```

Do not use zero time as optional value unless the domain explicitly defines zero time as absence.

### 8.3 Zero time rule

`time.Time{}` MUST NOT leak to API responses as a meaningful timestamp unless explicitly documented.

Validation rule:

```go
if t.IsZero() {
	return ErrMissingTimestamp
}
```

### 8.4 Precision rule

APIs MUST define timestamp precision:

- seconds,
- milliseconds,
- microseconds,
- nanoseconds.

Do not silently truncate or round without documentation.

When interoperating with JavaScript clients, beware millisecond precision limitations.

## 9. Database Persistence

### 9.1 Timestamp storage

Database timestamp fields MUST define:

1. timezone semantics,
2. precision,
3. nullability,
4. default source,
5. migration behavior,
6. comparison semantics.

Preferred instant contract:

```text
stored as UTC instant, precision microseconds, nullable only when field is optional
```

### 9.2 Application time vs database time

The source of time MUST be explicit.

Allowed:

1. application clock sets `created_at`,
2. database default sets `created_at`,
3. database trigger sets audit timestamp,
4. event producer timestamp sets event time.

Do not mix sources inconsistently within the same entity lifecycle.

### 9.3 Created/updated timestamps

Rules:

1. `created_at` MUST be immutable after insert.
2. `updated_at` MUST change only when persisted state changes.
3. Soft-delete timestamp MUST be nullable or explicit optional type.
4. Audit timestamps MUST not be overwritten by user input.

### 9.4 Query boundary rule

Time range queries MUST define inclusivity.

Preferred half-open interval:

```sql
WHERE created_at >= $1 AND created_at < $2
```

Avoid inclusive end timestamps such as `23:59:59.999` because precision varies across systems.

## 10. Context, Deadlines, and Timeouts

### 10.1 Context deadline rule

Network, database, external API, queue, and blocking operations MUST use context with deadline or timeout unless the parent context already has one.

```go
ctx, cancel := context.WithTimeout(ctx, 5*time.Second)
defer cancel()
```

### 10.2 Do not stack arbitrary timeouts

Nested functions SHOULD respect caller context rather than inventing unrelated shorter or longer timeouts.

Allowed:

1. top-level request timeout,
2. external dependency-specific timeout derived from config,
3. attempt timeout inside retry policy,
4. transaction timeout.

Document timeout hierarchy.

### 10.3 Always call cancel

Every `context.WithTimeout`, `context.WithDeadline`, or `context.WithCancel` MUST call cancel in the same scope unless ownership is transferred.

```go
ctx, cancel := context.WithTimeout(parent, timeout)
defer cancel()
```

### 10.4 Timeout error handling

Distinguish timeout/cancellation from other errors.

```go
if errors.Is(err, context.DeadlineExceeded) {
	return fmt.Errorf("dependency timeout: %w", err)
}
if errors.Is(err, context.Canceled) {
	return fmt.Errorf("request canceled: %w", err)
}
```

## 11. Timers and Tickers

### 11.1 Timer lifecycle

Every timer created with `time.NewTimer` MUST be stopped when no longer needed.

```go
timer := time.NewTimer(timeout)
defer timer.Stop()
```

When resetting timers, follow documented safe reset patterns. Do not reset timers casually in shared goroutines.

### 11.2 Ticker lifecycle

Every ticker MUST be stopped.

```go
ticker := time.NewTicker(interval)
defer ticker.Stop()
```

Do not use `time.Tick` in long-lived code because it hides ticker lifecycle.

### 11.3 Select with context

Long-running loops MUST listen to context cancellation.

```go
for {
	select {
	case <-ctx.Done():
		return ctx.Err()
	case <-ticker.C:
		if err := doWork(ctx); err != nil {
			return err
		}
	}
}
```

### 11.4 Timer allocation in loops

Do not use `time.After` repeatedly in hot loops when it causes unnecessary allocations or unclear lifecycle.

Prefer reusable timers or context deadlines for performance-sensitive loops.

## 12. Retries, Backoff, and Scheduling

### 12.1 Retry delay rules

Retry policies MUST define:

1. initial delay,
2. maximum delay,
3. maximum attempts or total deadline,
4. jitter behavior,
5. retryable errors,
6. context cancellation behavior,
7. telemetry.

Do not implement infinite retry loops without context and upper bound.

### 12.2 Jitter

Distributed systems SHOULD use jitter for retries, polling, and scheduled background work to avoid thundering herd behavior.

### 12.3 Sleep rule

Do not use `time.Sleep` for synchronization in production logic or unit tests.

Allowed:

1. simple CLI delay,
2. backoff wait respecting context via timer/select,
3. integration test waiting for external system with explicit timeout,
4. benchmark pacing when documented.

Preferred wait:

```go
func sleepContext(ctx context.Context, d time.Duration) error {
	timer := time.NewTimer(d)
	defer timer.Stop()

	select {
	case <-ctx.Done():
		return ctx.Err()
	case <-timer.C:
		return nil
	}
}
```

### 12.4 Scheduling is not just duration

A recurring schedule MUST NOT be represented only as `time.Duration` when it means calendar recurrence.

Examples requiring domain schedule logic:

- every business day at 09:00 Singapore time,
- first Monday of month,
- end of quarter,
- yearly renewal date,
- SLA excluding weekends and holidays,
- cron-like schedules.

## 13. Clock Abstraction

### 13.1 Injectable clock for business logic

Business logic depending on current time MUST accept a clock abstraction or `now time.Time` parameter.

Allowed simple pattern:

```go
type Clock interface {
	Now() time.Time
}

type SystemClock struct{}

func (SystemClock) Now() time.Time { return time.Now() }
```

Or pass `now` directly:

```go
func IsExpired(now, expiresAt time.Time) bool {
	return !now.Before(expiresAt)
}
```

### 13.2 Avoid global time in domain functions

Bad:

```go
func (l License) IsExpired() bool {
	return time.Now().After(l.ExpiresAt)
}
```

Good:

```go
func (l License) IsExpired(now time.Time) bool {
	return !now.Before(l.ExpiresAt)
}
```

### 13.3 Clock in infrastructure

Infrastructure adapters may call `time.Now()` directly for:

1. telemetry timestamps,
2. low-level log enrichment,
3. generated audit record when source-of-time policy says application time,
4. retry elapsed measurement.

Business decision code MUST remain testable with deterministic time.

## 14. Comparisons and Boundaries

### 14.1 Prefer method comparisons

Use:

```go
t.Before(u)
t.After(u)
t.Equal(u)
t.Compare(u)
```

Do not use `==` for semantic time equality unless location and monotonic-field behavior are intentionally part of comparison.

### 14.2 Boundary naming

Use boundary names that reveal inclusivity:

```go
validFromInclusive
validUntilExclusive
```

Preferred interval policy: `[start, end)`.

```go
func ContainsInstant(start, end, t time.Time) bool {
	return !t.Before(start) && t.Before(end)
}
```

### 14.3 Expiration rule

Expiration checks SHOULD use `!now.Before(expiresAt)` for "expired at or after expiry instant".

```go
expired := !now.Before(expiresAt)
```

Do not use `After` alone if equality at deadline should expire.

## 15. Monotonic Field and Serialization

### 15.1 Monotonic field is process-local

A `time.Time` returned by `time.Now()` may include monotonic clock reading. That monotonic component is not meaningful after serialization, database persistence, JSON encoding, or cross-process transfer.

Do not rely on monotonic behavior after time values cross process/storage boundaries.

### 15.2 Strip monotonic reading when needed

When comparing or storing canonical wall-clock instants, normalize intentionally.

Common patterns:

```go
t = t.UTC()
```

or:

```go
t = t.Round(0)
```

Use such normalization when tests or equality comparisons require stable behavior.

### 15.3 Do not persist duration from wall timestamps incorrectly

If elapsed time matters, measure it in-process with monotonic-aware values. Persisting start/end wall timestamps is useful for audit, but elapsed calculation across processes must tolerate wall clock changes.

## 16. Validation Rules

### 16.1 Required timestamp validation

For required timestamps:

```go
if submittedAt.IsZero() {
	return ErrMissingSubmittedAt
}
```

### 16.2 Future/past constraints

Every future/past validation MUST define tolerance for clock skew.

```go
if issuedAt.After(now.Add(maxClockSkew)) {
	return ErrIssuedAtInFuture
}
```

### 16.3 Range validation

Validate temporal ranges:

```go
if !end.After(start) {
	return ErrInvalidTimeRange
}
```

For date-only ranges, compare date-only values, not instants.

### 16.4 External timestamps

Timestamps from clients, messages, files, and external systems MUST be treated as untrusted.

Validate:

1. parseability,
2. timezone/offset presence if required,
3. min/max accepted range,
4. clock skew,
5. precision,
6. business range.

## 17. Logging, Telemetry, and Auditing

### 17.1 Log timestamps

Application logs SHOULD rely on logger-managed timestamps rather than manually attaching many timestamp fields.

If logging domain timestamps, field names MUST be explicit:

```go
slog.Time("submitted_at", submittedAt)
slog.Duration("elapsed", elapsed)
```

### 17.2 Metrics

Metrics SHOULD report durations in consistent units.

Preferred:

- seconds for Prometheus-style histograms,
- milliseconds only when metric naming states `_ms`,
- nanoseconds only for low-level runtime metrics.

### 17.3 Audit timestamps

Audit timestamps MUST be:

1. immutable,
2. source-defined,
3. timezone-normalized,
4. not user-editable,
5. precise enough for ordering requirements.

If ordering must be total across distributed nodes, timestamps alone are insufficient; use version, sequence, or logical clock.

## 18. Regulatory and Workflow Systems

### 18.1 Effective date vs submission date

Regulatory/workflow systems MUST distinguish:

- created at,
- submitted at,
- received at,
- effective from,
- effective until,
- approved at,
- rejected at,
- revoked at,
- expired at,
- archived at.

Do not reuse one timestamp field for multiple lifecycle meanings.

### 18.2 SLA time

SLA calculations MUST define:

1. start event,
2. stop event,
3. pause events,
4. resume events,
5. timezone,
6. business calendar,
7. weekends/holidays,
8. inclusivity,
9. escalation boundary.

Do not implement SLA with raw `time.Since(createdAt)` unless the SLA is truly continuous elapsed time.

### 18.3 State transition timestamps

State machines MUST capture transition timestamps explicitly.

```go
type Transition struct {
	From       State
	To         State
	OccurredAt time.Time
	ActorID    string
	Reason     string
}
```

Do not infer important lifecycle timestamps from current state alone.

## 19. Testing Standards

### 19.1 No real time in unit tests

Unit tests MUST NOT depend on `time.Now()` or `time.Sleep()` unless testing clock integration itself.

Preferred:

```go
now := time.Date(2026, 6, 10, 9, 0, 0, 0, time.UTC)
```

### 19.2 Fixed locations

Tests involving local time MUST use explicit locations.

```go
loc, err := time.LoadLocation("Asia/Singapore")
if err != nil {
	t.Fatal(err)
}
```

Do not depend on the machine's local timezone.

### 19.3 Boundary tests

Temporal logic MUST include tests for:

1. exact start boundary,
2. exact end boundary,
3. just before start,
4. just before end,
5. equality at expiration,
6. zero time,
7. nil optional time,
8. timezone conversion,
9. DST transition when relevant,
10. leap day when relevant,
11. end-of-month when relevant,
12. precision truncation/rounding.

### 19.4 Timer tests

Timer/ticker code SHOULD be tested with injectable clock or deterministic synchronization.

Do not make tests pass by arbitrary sleeps.

### 19.5 Time comparison in tests

Do not compare `time.Now()` results exactly.

Allowed:

```go
if got.Before(before) || got.After(after) {
	t.Fatalf("timestamp out of range")
}
```

For deterministic code, use fixed expected values.

## 20. Common Anti-Patterns

The agent MUST NOT generate these patterns:

### 20.1 Storing local formatted time

```go
CreatedAt string `json:"createdAt"`
```

Use `time.Time` or a validated domain date type.

### 20.2 Adding 24 hours for business tomorrow

```go
tomorrow := today.Add(24 * time.Hour)
```

Use calendar arithmetic with explicit location.

### 20.3 Ignoring timezone during parse

```go
t, _ := time.Parse("2006-01-02 15:04:05", input)
```

Use `ParseInLocation` and handle errors.

### 20.4 Using zero time as hidden null

```go
if entity.DeletedAt == (time.Time{}) { ... }
```

Use `IsZero` or optional type, and define API behavior.

### 20.5 Sleeping in tests

```go
time.Sleep(100 * time.Millisecond)
```

Use fake clock, channels, context, or deterministic synchronization.

### 20.6 Unstopped ticker

```go
for range time.Tick(time.Minute) { ... }
```

Use `NewTicker` and `Stop`.

### 20.7 Ambiguous duration config

```go
Timeout int `json:"timeout"`
```

Use explicit units or parse duration strings:

```go
Timeout time.Duration `json:"-"`
TimeoutSeconds int `json:"timeoutSeconds"`
```

## 21. Configuration Standards

### 21.1 Duration config

Configuration MUST state units explicitly.

Allowed:

```yaml
timeout: 5s
retry_delay: 250ms
```

or:

```yaml
timeout_seconds: 5
```

Forbidden:

```yaml
timeout: 5
```

unless the schema explicitly documents the unit.

### 21.2 Validate configured durations

Validate:

1. non-negative or positive as required,
2. maximum allowed value,
3. minimum useful value,
4. zero behavior,
5. overflow during parsing.

```go
if cfg.Timeout <= 0 {
	return ErrInvalidTimeout
}
```

### 21.3 Dynamic config

When timeouts are dynamically reloadable, ensure in-flight operations use a consistent snapshot. Do not mutate global duration variables without synchronization.

## 22. Review Checklist

Before producing final code, the LLM/code agent MUST verify:

- [ ] Instant, date-only, duration, deadline, and schedule concepts are not conflated.
- [ ] Persisted instants are normalized to UTC or explicitly documented otherwise.
- [ ] User-facing display converts to explicit location.
- [ ] No implicit `time.Local` in business logic.
- [ ] No fixed offset is used where real timezone rules are needed.
- [ ] Date-only values have clear representation and serialization.
- [ ] Calendar arithmetic does not use `24*time.Hour` incorrectly.
- [ ] Time ranges define inclusivity, preferably `[start, end)`.
- [ ] Expiration equality behavior is correct.
- [ ] Parsing uses the right layout and location.
- [ ] Parse errors are handled and wrapped.
- [ ] Optional timestamps do not misuse zero time.
- [ ] Context deadlines/timeouts are propagated.
- [ ] `cancel()` is called for derived contexts.
- [ ] Timers/tickers are stopped.
- [ ] Retry loops have bounds, jitter, and context cancellation.
- [ ] Tests use fixed clocks or injected time.
- [ ] Boundary tests cover DST/leap/end-of-month when relevant.
- [ ] Database queries use half-open intervals where appropriate.
- [ ] Timestamp precision is explicit across API/database boundaries.

## 23. Agent Refusal Rules

The LLM/code agent MUST refuse or ask for human approval before implementing:

1. SLA or regulatory deadline logic without timezone and calendar rules.
2. Business date logic using raw `time.Since` or `24*time.Hour` without justification.
3. Long-lived ticker loops without stop/cancel path.
4. Unit tests that rely on arbitrary sleeps.
5. Timestamp persistence without timezone contract.
6. APIs using ambiguous integer timeout units.
7. Expiration logic without equality behavior.
8. Date-only field represented as `time.Time` without timezone/storage rules.
9. Retry loop without context cancellation and upper bound.
10. Distributed ordering based solely on timestamps when correctness requires total order.
