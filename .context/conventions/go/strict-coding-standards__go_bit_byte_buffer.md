# Strict Coding Standards — Go Bit, Byte, and Buffer Handling

Status: mandatory for all Go implementation, refactoring, review, and test generation performed by an LLM/code agent.

Scope: bits, bytes, binary protocols, buffers, readers/writers, streaming I/O, byte ownership, byte-slice aliasing, endianness, encoders/decoders, large payloads, memory-sensitive processing, and security-sensitive byte handling.

This document is not a tutorial. It is a merge gate. Any code generated by an agent MUST comply with these rules unless the human explicitly approves a deviation.

---

## 1. Core Invariants

Every Go implementation that handles bits, bytes, buffers, or binary I/O MUST preserve the following invariants:

1. A byte slice is mutable shared storage until proven otherwise.
2. A string is immutable bytes, not a mutable buffer and not a character array.
3. A slice is a descriptor over an underlying array; copying a slice value does not copy the backing storage.
4. A buffer returned by a library call may be invalidated after the next read, write, append, reset, scan, or reuse operation unless the API explicitly says otherwise.
5. Read and write operations are allowed to be partial unless the API explicitly guarantees full completion.
6. Binary protocols MUST define byte order, field size, framing, maximum length, and error behavior explicitly.
7. Large or untrusted payloads MUST be streamed or bounded; they MUST NOT be loaded wholesale unless the maximum size is proven small.
8. Security-sensitive byte material MUST have explicit lifecycle rules and MUST NOT be logged, formatted, reused, or retained accidentally.
9. LLM-generated code MUST prefer simple, correct byte handling over premature unsafe optimization.
10. Any use of `unsafe`, zero-copy conversion, memory pooling, or in-place mutation MUST include a correctness justification.

## 2. Required Mental Model

### 2.1 Byte slice model

A `[]byte` value consists of pointer, length, and capacity. It may share backing storage with other slices.

Therefore, code MUST NOT assume any of the following unless proven locally:

```go
b2 := b1
// b2 is NOT an independent copy.
```

Required copy pattern:

```go
owned := append([]byte(nil), input...)
```

or:

```go
owned := make([]byte, len(input))
copy(owned, input)
```

Use the `make/copy` form when capacity must be controlled exactly.

### 2.2 String and byte conversion model

Conversion between `string` and `[]byte` normally copies data.

```go
b := []byte(s)
s2 := string(b)
```

Agent-generated code MUST NOT use unsafe string/byte conversion unless all of these are true:

1. A benchmark proves allocation is a real bottleneck.
2. The source bytes are immutable for the lifetime of the string.
3. The string is not retained beyond the source buffer lifetime.
4. The code is isolated behind a small, tested function.
5. The deviation is documented in code comments.

### 2.3 Bits are not booleans

Bit-level code MUST document whether a bit represents:

- a flag,
- a permission,
- a protocol field,
- a bitmap index,
- a compact state,
- a hash partition,
- a checksum field,
- a binary mask,
- or numeric arithmetic.

Do not use bit tricks where named constants or clear struct fields would be safer.

## 3. Package Selection Rules

### 3.1 Standard packages

Use the following packages by default:

| Concern                                | Preferred package                               |
| -------------------------------------- | ----------------------------------------------- |
| Byte-slice search/compare/manipulation | `bytes`                                         |
| Buffered reader/writer                 | `bufio`                                         |
| Basic reader/writer abstraction        | `io`                                            |
| Binary fixed-width encoding            | `encoding/binary`                               |
| Bit counting/rotation/math             | `math/bits`                                     |
| Hex encoding                           | `encoding/hex`                                  |
| Base64 encoding                        | `encoding/base64`                               |
| Compression stream                     | `compress/*`                                    |
| Cryptographic random bytes             | `crypto/rand`                                   |
| Non-cryptographic random bytes         | `math/rand/v2` only if acceptable               |
| Hashing                                | `hash`, `hash/crc32`, `crypto/*` as appropriate |

### 3.2 Prohibited package misuse

The agent MUST NOT:

1. Use `fmt.Sprintf` for binary encoding.
2. Use `strings.Builder` for binary payloads.
3. Use `bytes.Buffer` as a concurrent queue.
4. Use `bufio.Scanner` for unbounded records or large lines without an explicit buffer limit.
5. Use `io.ReadAll` on untrusted or unbounded inputs.
6. Use `encoding/binary.Write` blindly for public wire protocols without verifying exported fields, padding, field order, and byte order.
7. Use `unsafe` to reinterpret structs as bytes for wire serialization.
8. Use `math/rand` or `math/rand/v2` for secrets, tokens, IDs with security meaning, IVs, nonces, salts, keys, or session material.

## 4. Byte Ownership and Aliasing

### 4.1 API boundary rule

Every function accepting `[]byte` MUST have a clear ownership contract.

Allowed contracts:

```go
// Borrowed: process during call only; do not retain or mutate.
func Parse(input []byte) (Message, error)
```

```go
// Takes ownership: caller must not reuse or mutate after call.
func NewFrameOwned(payload []byte) Frame
```

```go
// Copies input: safe for caller to reuse immediately.
func NewFrame(payload []byte) Frame
```

If the function stores the slice after returning, it MUST either:

1. copy the slice, or
2. document that it takes ownership.

Default rule: copy at API boundaries unless performance requires ownership transfer.

### 4.2 Return value rule

A function returning `[]byte` MUST document whether the result is:

1. owned by caller,
2. borrowed and valid until next method call,
3. immutable by convention,
4. a view into another buffer,
5. pooled and must be released,
6. or secret material requiring clearing.

Do not return internal mutable slices directly from domain objects.

Bad:

```go
func (m Message) Payload() []byte { return m.payload }
```

Good:

```go
func (m Message) Payload() []byte {
	out := make([]byte, len(m.payload))
	copy(out, m.payload)
	return out
}
```

Allowed high-performance variant:

```go
// PayloadView returns a read-only view valid until m is mutated.
// Callers MUST NOT mutate the returned slice.
func (m *Message) PayloadView() []byte { return m.payload }
```

The name MUST include `View`, `Borrowed`, or equivalent when ownership is not transferred.

### 4.3 Append alias rule

The agent MUST NOT append to a slice received from another owner unless capacity sharing is intended.

Bad:

```go
func AddHeader(payload []byte) []byte {
	return append(payload, 0x01)
}
```

Good:

```go
func AddHeader(payload []byte) []byte {
	out := make([]byte, 0, len(payload)+1)
	out = append(out, 0x01)
	out = append(out, payload...)
	return out
}
```

Use full slice expressions to restrict capacity when creating views:

```go
view := input[start:end:end]
```

This prevents append on `view` from mutating data beyond `end`.

## 5. Buffer Usage Standards

### 5.1 `bytes.Buffer`

Use `bytes.Buffer` for dynamic binary or text accumulation when a writer interface is helpful.

Rules:

1. The zero value is valid.
2. Do not copy a non-zero `bytes.Buffer`.
3. Do not share a `bytes.Buffer` across goroutines without external synchronization.
4. Call `Grow` if the expected size is known.
5. Do not keep `buf.Bytes()` after subsequent writes, reads, truncates, or resets.
6. Do not mutate `buf.Bytes()` unless the mutation is local, deliberate, and safe.
7. After using a buffer for secret data, do not rely on `Reset` as secure erasure.

Preferred:

```go
var buf bytes.Buffer
buf.Grow(estimatedSize)
_, _ = buf.Write(header)
_, _ = buf.Write(payload)
out := append([]byte(nil), buf.Bytes()...)
```

If returning bytes from a temporary buffer, copy before returning unless ownership and invalidation are clear.

### 5.2 `bytes.Reader`

Use `bytes.Reader` when reading from an immutable `[]byte` snapshot.

Rules:

1. Do not mutate the underlying slice while the reader is in use.
2. Prefer `bytes.NewReader(input)` over `bytes.NewBuffer(input)` when the input is read-only.
3. Use `io.NewSectionReader` or `bytes.Reader` seeking only when offsets are validated.

### 5.3 `bytes.Builder` does not exist

There is no `bytes.Builder` in the standard library. For binary data use:

- `bytes.Buffer`,
- `make([]byte, 0, n)` + `append`,
- `encoding/binary.Append`,
- or a domain-specific encoder.

For text use `strings.Builder`, not binary buffers.

### 5.4 `strings.Builder` boundary

`strings.Builder` MUST be used only for text. It MUST NOT be used for binary payloads, encrypted bytes, compressed bytes, or protocol frames.

Do not copy a non-zero `strings.Builder`.

## 6. Reader/Writer Correctness

### 6.1 `io.Reader` rule

`Read(p)` may return fewer bytes than `len(p)` even when no error occurred. Code that requires exactly N bytes MUST use `io.ReadFull`.

Bad:

```go
buf := make([]byte, 16)
n, err := r.Read(buf)
if err != nil {
	return err
}
_ = buf[:n] // may be partial
```

Good:

```go
buf := make([]byte, 16)
if _, err := io.ReadFull(r, buf); err != nil {
	return err
}
```

### 6.2 `io.Writer` rule

`Write(p)` may return `n < len(p)` with a non-nil error. Code MUST check both `n` and `err` when correctness depends on full writes.

Preferred for stream copy:

```go
written, err := io.Copy(dst, src)
if err != nil {
	return fmt.Errorf("copy payload: %w", err)
}
_ = written
```

For bounded writes:

```go
if _, err := dst.Write(payload); err != nil {
	return fmt.Errorf("write payload: %w", err)
}
```

If using a custom writer where partial success matters, validate `n` explicitly.

### 6.3 `io.ReadAll` restrictions

`io.ReadAll` is allowed only when one of these is true:

1. The source is trusted and small.
2. The reader is already bounded by `io.LimitReader` or equivalent.
3. The maximum size is enforced by protocol, configuration, or validated metadata.

Required pattern for untrusted input:

```go
const maxBody = 1 << 20 // 1 MiB
body, err := io.ReadAll(io.LimitReader(r, maxBody+1))
if err != nil {
	return nil, fmt.Errorf("read body: %w", err)
}
if len(body) > maxBody {
	return nil, ErrBodyTooLarge
}
```

### 6.4 `io.Copy` rules

Use `io.Copy` or `io.CopyBuffer` for streaming transfers.

Rules:

1. Do not replace `io.Copy` with manual read-all/write-all loops unless needed.
2. Use `io.CopyBuffer` when buffer reuse is required.
3. Never pass a zero-length buffer to `io.CopyBuffer`.
4. Bound copies from untrusted sources with `io.LimitReader`, `http.MaxBytesReader`, or protocol-level limits.
5. Preserve returned byte count when auditing, telemetry, or exact transfer size matters.

### 6.5 Concurrent I/O rule

Unless the concrete type explicitly documents concurrent safety, readers and writers MUST be treated as unsafe for concurrent calls.

The agent MUST NOT concurrently call `Read`, `Write`, `ReadAt`, or `WriteAt` on the same object unless the concrete type is documented safe or the code synchronizes access.

## 7. `bufio` Standards

### 7.1 `bufio.Reader`

Use `bufio.Reader` when:

1. repeated small reads would cause excessive syscalls,
2. peeking or unread behavior is needed,
3. line-oriented reading requires control over long lines,
4. parser logic needs staged reads.

Rules:

1. Do not keep slices returned by methods such as `ReadSlice` after the next read unless copied.
2. Handle `ErrBufferFull` explicitly when using delimiter-based reads.
3. Do not assume `ReadLine` returns a complete line unless `isPrefix == false`.
4. Do not stack many buffered readers around the same underlying reader.

### 7.2 `bufio.Writer`

Use `bufio.Writer` when batching small writes.

Rules:

1. Always call `Flush` and handle its error.
2. Do not assume data is written until `Flush` succeeds.
3. Do not share a writer across goroutines without synchronization.
4. Do not ignore errors from intermediate writes; buffered writers may retain errors.

Required pattern:

```go
bw := bufio.NewWriter(w)
if _, err := bw.Write(payload); err != nil {
	return fmt.Errorf("buffer write: %w", err)
}
if err := bw.Flush(); err != nil {
	return fmt.Errorf("flush output: %w", err)
}
```

### 7.3 `bufio.Scanner`

`bufio.Scanner` is allowed only for bounded token scanning.

The agent MUST NOT use `bufio.Scanner` for:

1. arbitrary-size logs,
2. arbitrary-size CSV rows,
3. NDJSON with unknown row size,
4. binary frames,
5. large generated lines,
6. protocol parsing requiring precise recovery,
7. sequential scans after token-size failure.

If `Scanner` is used, the maximum token size MUST be set explicitly when default size is insufficient:

```go
scanner := bufio.NewScanner(r)
scanner.Buffer(make([]byte, 64*1024), maxTokenBytes)
```

If token-size failure must be recoverable, use `bufio.Reader` instead.

## 8. Binary Encoding and Endianness

### 8.1 Explicit byte order

Every binary encoder/decoder MUST define byte order explicitly.

Allowed:

```go
var order = binary.BigEndian
```

or:

```go
var order = binary.LittleEndian
```

Forbidden:

```go
// implicit native endian
```

Native endian MUST NOT be used in persisted files, network protocols, caches, cross-process IPC, or database binary fields unless the protocol explicitly requires it and is documented.

### 8.2 Protocol layout

Every binary protocol implementation MUST define:

1. magic/header if applicable,
2. version,
3. byte order,
4. field widths,
5. signedness,
6. alignment/padding rules,
7. frame length maximum,
8. checksum/MAC if applicable,
9. unknown-field handling,
10. backward/forward compatibility rules.

### 8.3 Avoid struct dumping

The agent MUST NOT serialize Go structs directly as memory bytes.

Forbidden:

```go
// unsafe cast struct memory to []byte
```

Reason: Go struct layout, padding, alignment, pointer fields, unexported fields, and architecture differences are not a stable wire contract.

Preferred:

```go
buf := make([]byte, 0, 16)
buf = binary.BigEndian.AppendUint32(buf, msg.ID)
buf = binary.BigEndian.AppendUint64(buf, uint64(msg.Timestamp))
```

### 8.4 `encoding/binary` selection

Use fixed-width helper methods where possible:

```go
binary.BigEndian.PutUint32(buf[0:4], value)
value := binary.BigEndian.Uint32(buf[0:4])
```

Use `binary.Encode` / `binary.Decode` for structured values only when:

1. data is fixed-size,
2. all fields are intended for serialization,
3. exported field behavior is understood,
4. padding behavior is intentionally handled,
5. error handling is explicit.

Prefer hand-written encoding for public protocols.

### 8.5 Varint rules

Use varint encoding only when the protocol benefits from variable-length integers.

Rules:

1. Set a maximum encoded length.
2. Reject overlong encodings if canonical form matters.
3. Reject values above domain maximum after decoding.
4. Distinguish signed `Varint` from unsigned `Uvarint`.
5. Never decode varint in an unbounded loop without read limits.

## 9. Bit Manipulation Standards

### 9.1 Named masks

Every bit mask MUST be named.

Bad:

```go
if flags&4 != 0 {
	...
}
```

Good:

```go
const flagCompressed uint32 = 1 << 2

if flags&flagCompressed != 0 {
	...
}
```

### 9.2 Use unsigned types for bit operations

Bit masks SHOULD use unsigned integer types unless a signed protocol field requires otherwise.

Allowed types:

- `uint8` for byte-level flags,
- `uint16` for compact protocol flags,
- `uint32` for common protocol masks,
- `uint64` for large masks,
- custom named unsigned types for domain masks.

Avoid `int` for persisted bit layout.

### 9.3 Shift safety

The agent MUST validate shift counts that come from runtime input.

Bad:

```go
mask := uint64(1) << n
```

Good:

```go
if n >= 64 {
	return 0, ErrInvalidBitIndex
}
mask := uint64(1) << n
```

### 9.4 `math/bits`

Use `math/bits` for:

1. population count,
2. leading/trailing zero count,
3. bit rotation,
4. full-width multiplication/division helpers,
5. optimized bit-level algorithms.

Do not reimplement bit counting loops unless a benchmark and clarity justify it.

### 9.5 Flag type pattern

Preferred:

```go
type FrameFlags uint16

const (
	FrameFlagCompressed FrameFlags = 1 << iota
	FrameFlagEncrypted
	FrameFlagHasChecksum
)

func (f FrameFlags) Has(mask FrameFlags) bool {
	return f&mask == mask
}
```

Rules:

1. Use named type for domain flags.
2. Keep helper methods small and deterministic.
3. Reject unknown reserved bits when protocol requires strictness.
4. Preserve unknown bits only when forward compatibility explicitly requires it.

## 10. Framing and Length Prefixes

### 10.1 Mandatory length limit

Any length-prefixed frame MUST enforce a maximum before allocation.

Bad:

```go
payload := make([]byte, length)
```

Good:

```go
if length > maxFrameSize {
	return nil, ErrFrameTooLarge
}
payload := make([]byte, length)
```

### 10.2 Length type conversion

Conversions from unsigned protocol lengths to `int` MUST validate overflow.

```go
if length > uint32(maxFrameSize) {
	return nil, ErrFrameTooLarge
}
if uint64(length) > uint64(math.MaxInt) {
	return nil, ErrFrameTooLarge
}
buf := make([]byte, int(length))
```

### 10.3 Exact read rule

After validating length, read exactly that many bytes:

```go
payload := make([]byte, length)
if _, err := io.ReadFull(r, payload); err != nil {
	return nil, fmt.Errorf("read frame payload: %w", err)
}
```

### 10.4 Partial frame behavior

Every decoder MUST define partial-frame behavior:

1. EOF before header,
2. EOF during header,
3. EOF during payload,
4. invalid length,
5. unsupported version,
6. checksum mismatch,
7. trailing bytes.

Do not collapse all frame errors into `io.EOF`.

## 11. Memory Allocation and Retention

### 11.1 Preallocation rule

Preallocate when size is known or bounded.

```go
out := make([]byte, 0, headerSize+len(payload))
```

Do not preallocate from untrusted length before validating maximum.

### 11.2 Large backing array retention

Small slices referencing large arrays can keep the whole array alive.

Bad:

```go
small := huge[start:end]
return small
```

Good:

```go
small := append([]byte(nil), huge[start:end]...)
return small
```

### 11.3 Buffer reuse

Buffer reuse is allowed when:

1. ownership is local,
2. previous contents are not retained,
3. no goroutine can access the old contents,
4. sensitive data is handled correctly,
5. capacity growth does not create memory retention.

### 11.4 Pooling rule

Use `sync.Pool` only for high-churn temporary buffers after allocation profiling shows benefit.

Rules:

1. Pooled buffers MUST NOT be used after `Put`.
2. Reset before `Put`.
3. Do not put excessively large buffers back into the pool.
4. Do not rely on `sync.Pool` for resource lifecycle.
5. Do not store secrets in pooled buffers unless cleared first.

Example:

```go
const maxPooledBuffer = 64 << 10

func putBuffer(b []byte) {
	if cap(b) > maxPooledBuffer {
		return
	}
	clear(b)
	pool.Put(b[:0])
}
```

## 12. Security-Sensitive Bytes

### 12.1 Secret material

Secret material includes:

- passwords,
- tokens,
- API keys,
- session IDs,
- private keys,
- symmetric keys,
- nonces when reuse is dangerous,
- MAC keys,
- seed material,
- decrypted plaintext.

Rules:

1. Do not log secrets.
2. Do not include secrets in error messages.
3. Do not convert secrets to `string` unless unavoidable.
4. Do not store secrets in long-lived buffers.
5. Clear temporary byte slices when useful and feasible.
6. Do not use `bytes.Equal` for secret equality when timing may matter; use constant-time comparison.

### 12.2 Constant-time comparison

For MACs, signatures, tokens, and secret-derived values, use `crypto/subtle` or a verified package API.

```go
if subtle.ConstantTimeCompare(got, want) != 1 {
	return ErrInvalidMAC
}
```

Length checks may leak length; ensure protocol does not treat length as secret or normalize length before comparison.

### 12.3 Random bytes

Use `crypto/rand.Reader` or package helpers for security-sensitive random bytes.

```go
buf := make([]byte, 32)
if _, err := io.ReadFull(rand.Reader, buf); err != nil {
	return nil, fmt.Errorf("read random bytes: %w", err)
}
```

Never use pseudo-random packages for cryptographic bytes.

## 13. Encoding Boundaries

### 13.1 Hex

Use hex for debug-safe binary representation, not as an internal binary storage replacement.

Rules:

1. Validate expected length after decoding.
2. Reject malformed input.
3. Do not log full secret hex values.
4. Prefer lowercase canonical encoding unless protocol specifies otherwise.

### 13.2 Base64

Rules:

1. Choose encoding variant explicitly: standard, URL, raw standard, raw URL.
2. Decide padding policy explicitly.
3. Validate decoded size.
4. Do not accept multiple encodings for canonical tokens unless required for compatibility.

### 13.3 Text/binary boundary

Do not mix text and binary concerns in one buffer without a protocol definition.

Examples requiring explicit boundary logic:

- HTTP body with binary content,
- multipart payload,
- CSV containing base64 binary,
- JSON containing byte arrays,
- protobuf/gRPC metadata,
- database `BYTEA`/BLOB columns,
- log redaction of binary data.

## 14. Compression and Decompression

### 14.1 Compression bombs

Any decompression of untrusted input MUST enforce output limits.

```go
zr, err := gzip.NewReader(r)
if err != nil {
	return err
}
defer zr.Close()

limited := io.LimitReader(zr, maxDecompressedBytes+1)
out, err := io.ReadAll(limited)
if err != nil {
	return err
}
if len(out) > maxDecompressedBytes {
	return ErrDecompressedTooLarge
}
```

### 14.2 Streaming decompression

Prefer streaming decompression into bounded processors instead of `ReadAll` when payloads may be large.

### 14.3 Compression level

Compression level MUST be selected based on a documented trade-off:

- speed,
- size,
- CPU cost,
- latency,
- memory,
- compatibility.

Do not maximize compression by default.

## 15. Checksums, Hashes, and MACs

### 15.1 Terminology rule

The agent MUST distinguish:

- checksum: accidental corruption detection,
- hash: digest/fingerprint,
- MAC: authenticated integrity,
- signature: asymmetric authenticity.

Do not use CRC or non-cryptographic hash for security decisions.

### 15.2 Streaming hash

For large payloads, compute hash/MAC while streaming.

```go
h := sha256.New()
if _, err := io.Copy(h, r); err != nil {
	return nil, err
}
sum := h.Sum(nil)
```

### 15.3 Hash storage

Stored digests MUST define:

1. algorithm,
2. encoding,
3. length,
4. version/migration policy,
5. collision/security assumptions.

## 16. Error Handling

### 16.1 Error wrapping

Wrap byte/buffer errors with operation context.

Good:

```go
if _, err := io.ReadFull(r, header); err != nil {
	return fmt.Errorf("read frame header: %w", err)
}
```

Bad:

```go
return err
```

### 16.2 Sentinel errors

Use sentinel errors for domain/protocol conditions:

```go
var ErrFrameTooLarge = errors.New("frame too large")
```

Wrap with `%w` when preserving machine-checkable behavior.

### 16.3 EOF semantics

The agent MUST distinguish:

- clean EOF before a new record,
- unexpected EOF mid-record,
- truncated header,
- truncated payload,
- EOF after partial write/read,
- protocol-level end marker.

Do not treat all EOF as success.

## 17. Concurrency

### 17.1 Shared byte slice rule

A byte slice MUST NOT be mutated by one goroutine while read by another unless protected by synchronization.

Allowed patterns:

1. immutable copied snapshot,
2. ownership transfer over channel,
3. mutex-protected buffer,
4. worker-local buffers,
5. append-only data with no concurrent readers until sealed.

### 17.2 Channel ownership transfer

When sending `[]byte` over a channel, define whether ownership transfers.

Preferred:

```go
// send transfers ownership; sender must not reuse b after send.
out <- b
```

If reuse is needed, send a copy.

### 17.3 Buffer pools and goroutines

A buffer from `sync.Pool` MUST NOT be returned to the pool until all goroutines are done using it.

Use tests with `-race` for any implementation involving shared byte buffers.

## 18. Performance Standards

### 18.1 Optimize only with evidence

Before introducing complex buffer management, the agent SHOULD first implement the simple correct version and add benchmarks.

Optimization requires evidence from:

- `go test -bench`,
- `benchstat`,
- allocation counts,
- pprof heap/cpu profile,
- realistic input sizes.

### 18.2 Allocation budget

For performance-sensitive paths, tests or benchmarks SHOULD assert allocation behavior.

```go
func BenchmarkEncodeFrame(b *testing.B) {
	payload := make([]byte, 1024)
	b.ReportAllocs()
	for b.Loop() {
		_, _ = EncodeFrame(payload)
	}
}
```

### 18.3 Avoid micro-optimizations

The agent MUST NOT introduce:

- unsafe conversions,
- manual loop unrolling,
- custom binary encoders,
- global buffer pools,
- pointer arithmetic,
- architecture-specific paths,

without benchmark evidence and tests.

## 19. Testing Standards

### 19.1 Mandatory tests for binary code

Binary parsers/encoders MUST include tests for:

1. valid minimal frame,
2. valid maximal frame,
3. zero-length payload,
4. unsupported version,
5. invalid flags,
6. truncated header,
7. truncated payload,
8. invalid length,
9. length above maximum,
10. wrong byte order if detectable,
11. checksum/MAC failure if applicable,
12. trailing bytes behavior,
13. round-trip encode/decode,
14. fuzz inputs.

### 19.2 Fuzzing

Any parser handling untrusted bytes SHOULD have a fuzz test.

```go
func FuzzDecodeFrame(f *testing.F) {
	f.Add([]byte{0x01, 0x00})
	f.Fuzz(func(t *testing.T, data []byte) {
		_, _ = DecodeFrame(data)
	})
}
```

Fuzz tests MUST assert no panic, no unbounded allocation, and no infinite loop.

### 19.3 Golden bytes

For stable wire formats, use golden byte fixtures.

Rules:

1. Golden bytes must include hex comments or decoded field explanation.
2. Tests must fail on accidental format changes.
3. Fixtures must include version information.

### 19.4 Race tests

Any byte buffer shared across goroutines MUST be tested with:

```bash
go test -race ./...
```

The agent MUST not claim concurrency safety without either local synchronization or documented immutable ownership.

## 20. Review Checklist

Before producing final code, the LLM/code agent MUST verify:

- [ ] Every `[]byte` API has an ownership contract.
- [ ] No internal mutable slice leaks unintentionally.
- [ ] `append` cannot mutate caller-owned backing arrays unexpectedly.
- [ ] All untrusted lengths are bounded before allocation.
- [ ] All exact-size reads use `io.ReadFull` or equivalent.
- [ ] All buffered writers are flushed and flush errors handled.
- [ ] `Scanner` is not used for unbounded tokens.
- [ ] Endianness is explicit.
- [ ] Bit masks are named and unsigned where appropriate.
- [ ] Shift counts from input are validated.
- [ ] `ReadAll` is not used on unbounded input.
- [ ] Compression/decompression has size limits.
- [ ] Secret bytes are not logged or converted to string unnecessarily.
- [ ] Constant-time comparison is used for secret equality.
- [ ] Buffer pools do not cause use-after-put or secret retention.
- [ ] Tests include malformed/truncated/fuzz cases.
- [ ] Benchmarks exist before unsafe or pooling optimization.

## 21. Agent Refusal Rules

The LLM/code agent MUST refuse or ask for human approval before implementing:

1. unsafe struct-to-byte serialization,
2. unbounded read of external input,
3. binary parser without max frame size,
4. secret comparison with normal equality,
5. use of pseudo-random bytes for cryptographic material,
6. returning mutable internal byte slices without a view/ownership contract,
7. concurrent mutation of shared byte slices without synchronization,
8. ignoring `Flush` errors,
9. ignoring partial read/write behavior in protocol code,
10. unchecked allocation from input-provided length.
